# 0029 — Anonymous WebSocket subscribe for public sessions

- **Date**: 2026-05-19
- **Status**: Accepted

## Context

[ADR 0002](0002-auth-self-hosted-oidc-authelia.md) establishes cookie-only authentication (HttpOnly `aconversa-session` JWT). Every authenticated HTTP route and the existing WS upgrade gate in `apps/server/src/ws/connection.ts` enforce the cookie via the shared `authenticateRequest` primitive: no cookie or an unverifiable cookie produces an HTTP 401 BEFORE the WS handshake completes (lines 86–104 of that file document the contract verbatim, including the explicit reminder that "any future cross-origin audience surface … MUST either be same-origin to the app or carry a different auth primitive (a query-string ticket issued by an authenticated HTTP exchange)").

[ADR 0026](0026-micro-frontend-root-app.md) introduces the audience surface as a public-by-default micro-frontend mountable under `/a/sessions/:id`. [`aud_no_auth_for_public`](../../tasks/refinements/audience/aud_no_auth_for_public.md) landed 2026-05-18 and widened `apps/root/src/surfaces/SurfaceHost.tsx` to honor `SurfaceMeta.requiredAuthLevel: 'public'` — an anonymous browser hitting `/a/sessions/<uuid>` now mounts the audience bundle, sees the placeholder, but `<WsClientProvider>` short-circuits because its `auth.status !== 'authenticated'` guard keeps the socket closed. No live events arrive over WS for anonymous viewers.

Closing this gap means the WS substrate must accept anonymous upgrades for the audience surface AND enforce per-session privacy: an anonymous client may subscribe to a session iff `sessions.privacy = 'public'` AND `sessions.ended_at IS NULL`. The privacy boundary that today is enforced at the cookie-auth gate moves into the visibility predicate; the SQL fragment in `apps/server/src/sessions/visibility.ts:115-132` already encodes "PUBLIC OR host OR participant" — for an anonymous caller the OR-host / OR-participant branches collapse to false and the predicate degenerates cleanly to "PUBLIC only."

The shape of the choice is three-way:

1. **Anonymous WS upgrade** — the WS auth gate accepts a missing/invalid cookie as "anonymous" instead of 401; the `WsConnectionContext.user` field becomes genuinely nullable; every server-side write handler (propose / vote / commit / withdraw / mark-meta-disagreement / catch-up) refuses anonymous callers with a `forbidden` error envelope; the subscribe handler routes anonymous to a strict "public-only" visibility check.
2. **Query-string ticket primitive** — a new HTTP endpoint (`POST /api/sessions/:id/audience-ticket`) issues a short-lived signed token; the audience client fetches the ticket out-of-band, then opens the WS with `?ticket=<jwt>`; the WS auth gate exchanges the ticket for an `AuthUser`-shaped record (possibly with `id: null` or a synthetic anonymous-id).
3. **Server-Sent Events (SSE)** — a separate read-only event-stream endpoint at `GET /api/sessions/:id/events?since=<seq>` that the audience consumes over EventSource; the WS stays authenticated-only.

## Decision

The platform adopts **Option 1: anonymous WS upgrade** as the transport for audience live-event delivery on public sessions.

The wire shape:

- **Connection auth gate** (`apps/server/src/ws/connection.ts` `preValidation`): when the `aconversa-session` cookie is absent OR `authenticateRequest` returns null, the gate no longer throws `ApiError(401, 'auth-required', ...)`. It instead sets `request.authUser = undefined` and proceeds to the upgrade. The origin-allowlist gate (production) still runs first and still rejects off-origin upgrades with HTTP 403 — anonymous upgrade does not relax the origin contract. The cookie-auth gate's existing "valid cookie → real `AuthUser`" path is preserved unchanged; the new branch only covers "no cookie / invalid cookie."
- **Connection context** (`WsConnectionContext.user?: AuthUser`): the type signature was already optional (anticipating anonymous since `ws_auth_on_connect`); this ADR is the first time the optionality is genuinely populated on the wire. Connection handlers that today defensively narrow `user === undefined` and throw "ws-auth-gate bypassed" change to "ws-anonymous-client: writes refused" and emit a `forbidden` error envelope instead of a 500-equivalent.
- **Subscribe handler** (`apps/server/src/ws/handlers/subscribe.ts`): when `connection.user === undefined`, the visibility check uses a new predicate `canSeeSessionAnonymously(executor, sessionId)` that runs `SELECT 1 FROM sessions WHERE id = $1 AND privacy = 'public' AND ended_at IS NULL`. The result is collapsed with the existence-non-leak rule the authenticated path already uses — a private or ended session looks identical to a nonexistent one (`not-found`). The registry call drops the `userId` argument (passing `undefined`); the existing privacy-flip pruner skips anonymous subscribers with a warn log per its `userId === undefined` branch, then a sibling `pruneAnonymousSubscribersForPrivateSession` helper evicts every anonymous subscriber from a session whose privacy flips to `'private'` regardless of the per-subscriber identity (no user-keyed `canSeeSession` to call — anonymous can never see private).
- **Write handlers** (`propose`, `vote`, `commit`, `withdraw-proposal`, `mark-meta-disagreement`, `catch-up`, `snapshot-state`): the existing `connection.user === undefined` branch's "bypassed" error becomes a wire `error` envelope with `code: 'forbidden'` + `inResponseTo: envelope.id`. The connection stays open (per `ws_error_message`'s connection-stays-open invariant); only the single envelope is refused. Anonymous catch-up specifically: forbidden — replay of a public session for an anonymous viewer is a useful feature, but introducing it now widens the security surface (rate-limit accounting, per-anonymous-connection state) past this ADR's scope; deferred to a future leaf if the broadcast-only path proves insufficient.
- **Client-side provider** (`packages/shell/src/ws/WsClientProvider.tsx`): the `WsClientAuthState` discriminator gains an additional opening condition. Today the effect opens iff `auth.status === 'authenticated'`; after this ADR the provider also opens when the surface's `requiredAuthLevel` is `'public'` AND `auth.status === 'unauthenticated'`. The provider grows a new prop `allowAnonymous?: boolean` (default `false`) so audience surfaces opt in explicitly; the moderator and participant surfaces, which never want anonymous WS, keep the default and their behavior is unchanged.

The privacy boundary is enforced at the **server's data layer**, not at the UI layer. An anonymous client that tampers with its TypeScript-narrowed surface to send a raw `propose` envelope receives a wire-level `forbidden`; an anonymous client that subscribes to a session whose privacy is `private` receives `not-found`; an anonymous client subscribed to a session whose privacy flips to `private` is evicted via an `unsubscribed { reason: 'privacy-flipped' }` envelope and the registry entry is dropped.

## Consequences

- **The cookie-on-upgrade contract relaxes for the WS endpoint only.** At the time of this decision the WS upgrade gate was the sole exception to "every HTTP route requires a valid cookie." (Amended 2026-06-13: the anonymous read-only `GET /api/sessions/public` listing endpoint is a second, narrower exception under its own `started_at`-based gate — see Amendments.) The WS upgrade gate's relaxation is targeted: missing/invalid cookie maps to `request.authUser = undefined`, not to a 401. The origin-allowlist gate is unchanged and still rejects off-origin upgrades.
- **`WsConnectionContext.user` becomes genuinely optional.** Every consumer that today narrows `user === undefined` and throws a "wiring bug" error now treats the narrow as a real branch: subscribe → public-only-visibility-check; every write handler → wire `forbidden` error envelope. This is a non-trivial widening of the connection-handler surface; the implementation refinement at [`aud_anonymous_ws_subscribe.md`](../../tasks/refinements/audience/aud_anonymous_ws_subscribe.md) names every call site.
- **A new visibility predicate.** `canSeeSessionAnonymously(executor, sessionId)` joins `canSeeSession(executor, sessionId, userId)` in `apps/server/src/sessions/visibility.ts`. The two predicates compose: `canSeeSessionAnonymously` is strictly stricter (PUBLIC + not-ended only) and is the authoritative oracle for an anonymous caller. The authenticated predicate is unchanged — no signature drift in the existing five call sites.
- **A new privacy-flip pruner branch.** `pruneAnonymousSubscribersForPrivateSession` evicts every anonymous subscriber when a public session flips to private. The existing user-keyed pruner already handles the authenticated branch; the new helper handles the rest. Both are called from `PATCH /api/sessions/:id/privacy`.
- **The `WsClientProvider` opens the socket for public surfaces with anonymous auth.** The opt-in is per-surface via the new `allowAnonymous` prop. Moderator and participant surfaces keep their authenticated-only WS posture; only the audience opts in. The provider's reset-on-unmount semantics are unchanged.
- **Wire-format unchanged.** No new envelope type, no new `WsMessageType` enum entry, no role flag on the subscribe payload. The audience uses the existing `subscribe` envelope verbatim; the server tells "anonymous-or-authenticated" apart by inspecting the connection's `user` field at handler-dispatch time, not by reading a payload field.
- **No query-string ticket primitive needed.** The same-origin cookie-or-no-cookie posture handles both authenticated and anonymous; the OBS browser source loads the audience URL on the same origin as the API, and the WS upgrade goes to that same origin. Cross-origin embedding (audience surface hosted on a third-party page) is deferred — when it surfaces, a ticket primitive can be added without disturbing the same-origin anonymous path.
- **No SSE side-channel.** Adopting Option 3 would mean maintaining two transport paths (WS for moderator + participant, SSE for audience) — twice the broadcast-fanout code, twice the rate-limit machinery, twice the reconnect logic. The "one transport, two auth postures" choice keeps the per-instance broadcast emitter at one call site (`app.wsBroadcast.emit`) and the audience inherits the server's existing fanout for free.
- **Anonymous catch-up is deferred.** An anonymous client that disconnects briefly cannot replay missed events for free — it must re-subscribe and accept that any envelopes emitted during the disconnect window are lost. This is acceptable for the v0 broadcast surface (the audience is a passive viewer; missed events are visually inert). Re-enabling anonymous catch-up requires a per-anonymous-connection rate-limit accounting story (today's per-`connection.user.id` accounting is meaningless for anonymous) — deferred to a future hardening leaf.
- **No tracking concern.** The server does not synthesize an anonymous-user identity; `WsConnectionContext.user` stays `undefined`. The `connectionId` (already minted per upgrade) is logged for diagnostic purposes only and is not stored beyond the connection's lifetime. No anonymous-id cookie, no localStorage handle, no opt-in pseudonym.
- **Tested through committed Cucumber scenarios** per [ADR 0022](0022-no-throwaway-verifications.md). The implementation refinement scopes three new scenarios at the WS protocol seam: (a) anonymous subscribe to a public session → `subscribed` ack + live `event-applied` delivery; (b) anonymous subscribe to a private session → `not-found` rejection envelope; (c) anonymous client that subscribed and then raw-sends a `propose` → `forbidden` rejection envelope (connection stays open). The privacy-flip eviction scenario is pinned at the Vitest layer (in `apps/server/src/ws/subscriptions.test.ts`) because the flip surface is the existing `PATCH /api/sessions/:id/privacy` handler, not the WS protocol.

## Alternatives considered

- **Query-string ticket primitive** (Option 2). Rejected for v0 because the same-origin cookie-or-no-cookie posture is strictly simpler and the ticket adds a second authenticated HTTP round trip (the audience must fetch the ticket before it can open the WS) for no observable benefit on the same-origin deployment shape. The ticket primitive remains the future answer if a cross-origin audience embed becomes a requirement; this ADR does not preclude adding it later. The cookie-on-upgrade comment at `apps/server/src/ws/connection.ts:86-104` already documents the ticket path as the future answer for the cross-origin case.
- **Server-Sent Events** (Option 3). Rejected because maintaining two transports (WS + SSE) doubles the broadcast-emitter call sites, the reconnect logic, the per-connection rate-limit accounting, and the test surface. The audience would also lose access to the per-session subscribe-acknowledge cycle (SSE has no client-to-server ack channel without a parallel HTTP endpoint). The one-transport design keeps the per-instance fanout at a single seam.
- **Synthesize a per-visit anonymous user** (`AuthUser { id: 'anonymous-<random>', screenName: 'Anonymous Viewer' }`). Rejected (mirroring [`aud_no_auth_for_public.md`](../../tasks/refinements/audience/aud_no_auth_for_public.md) Decision §5): a fake identity leaks into logs / events / projections in confusing ways; the per-visit handle is itself a tracking vector that should be opt-in, not default-on; the per-connection-rate-limit accounting that depends on `connection.user.id` becomes brittle (every anonymous connection gets a fresh id; the cap is per-connection anyway). Keeping `user` genuinely `undefined` is the cleanest signal "this is a read-only anonymous viewer."
- **Wire-format role flag** (e.g. `subscribe { role: 'audience' }`). Rejected: the server already discriminates by `connection.user` (undefined = anonymous = audience). A role flag would either be ignored (the server would still read `user`) or would create a second source of truth that could disagree with the first. The single-source-of-truth posture matches [ADR 0021](0021-event-envelope-discriminated-union-with-zod.md)'s narrow-envelope discipline.
- **Per-route auth gate inside the audience surface** ("subscribe only on routes that actually need WS"). Rejected: the audience surface mounts one WS client at the surface boundary per [`aud_ws_client.md`](../../tasks/refinements/audience/aud_ws_client.md) Decision §1 ("one client per surface, mounted at the surface boundary"); per-route gates would multiply the open-socket count and contradict that decision.

## Stack-validation tests

Per [ADR 0022](0022-no-throwaway-verifications.md), the architecture lands behind the real wiring in [`apps/server/src/ws/`](../../apps/server/src/ws/), [`apps/server/src/sessions/visibility.ts`](../../apps/server/src/sessions/visibility.ts), and [`packages/shell/src/ws/WsClientProvider.tsx`](../../packages/shell/src/ws/WsClientProvider.tsx), with the test contracts spelled out in [`tasks/refinements/audience/aud_anonymous_ws_subscribe.md`](../../tasks/refinements/audience/aud_anonymous_ws_subscribe.md). Coverage shape:

- **Cucumber** (`tests/behavior/backend/ws-audience-subscribe.feature`) — the existing feature grows three scenarios at the anonymous wire seam: anonymous-subscribe-public-session-succeeds, anonymous-subscribe-private-session-rejected-with-not-found, anonymous-write-attempt-rejected-with-forbidden. The existing three authenticated scenarios stay unchanged.
- **Vitest** — `apps/server/src/ws/subscriptions.test.ts` grows two cases pinning the anonymous-prune branch; `apps/server/src/ws/handlers/subscribe.test.ts` (if it exists, else added) pins the `canSeeSessionAnonymously` branch; `packages/shell/src/ws/WsClientProvider.test.tsx` grows one case pinning the `allowAnonymous` opt-in opens the socket under `'unauthenticated'`.

## Amendments

- **2026-06-13** — Session discovery (milestone M11) shipped the first anonymous **HTTP** route, `GET /api/sessions/public` ([`apps/server/src/sessions/routes.ts:2336`](../../apps/server/src/sessions/routes.ts)), which corrects the original Consequence that "HTTP routes continue to require a valid cookie." The same anonymous-access *principle* this ADR established — public-only, existence-non-leak (a hidden session is indistinguishable from a nonexistent one), no synthesized anonymous identity — is applied to a second transport, so the extension is recorded here rather than in a new ADR. The HTTP listing gate differs from this ADR's WS subscribe gate in two ways: it is **stricter on lifecycle** — `privacy = 'public' AND started_at IS NOT NULL` ([`routes.ts:2406`](../../apps/server/src/sessions/routes.ts)), the lobby-secrecy rule that hides unstarted public sessions — whereas the WS gate keys on `ended_at IS NULL`; and it is **wider on ended sessions** — it carries no `ended_at` clause, so an ended public session still lists, which is its discovery path to replay. The endpoint returns listing fields only (no host identity, no participant data, no event-log access); anonymous replay-*content* visibility remains the concern of [ADR 0045](0045-audience-replay-surface-visibility-gating.md). The WS Decision and Context above are unchanged; this amendment only narrows the now-overbroad HTTP-cookie Consequence.
- **2026-06-05** — The deferred "anonymous catch-up / replay of a public session for an anonymous viewer" (noted under Decision → Write handlers and Consequences → "Anonymous catch-up is deferred") is taken up by [ADR 0045](0045-audience-replay-surface-visibility-gating.md). ADR 0045 routes anonymous public-session replay through the **HTTP** event-log endpoint (not anonymous WS catch-up), gated by a new `canReplaySessionAnonymously` predicate (`privacy = 'public'`, ended-agnostic — strictly this ADR's `canSeeSessionAnonymously` minus its `ended_at IS NULL` clause), and scopes the build into the `backend.replay_endpoints.anonymous_public_session_log` leaf. The WS anonymous-catch-up path described here remains deferred and unbuilt; replay is served over HTTP instead.
