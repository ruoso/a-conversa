# 0015 — Application Dockerfile: multi-stage with pnpm via Corepack

- **Date**: 2026-05-10
- **Status**: Accepted

## Context

`a-conversa` is delivered as a single Docker image that serves the backend (HTTP + WebSocket) and the frontend bundles (moderator, participant, audience, replay) as static assets. The architectural baseline is recorded in [docs/architecture.md](../architecture.md); the foundation refinement at [tasks/refinements/foundation/dockerfile_app.md](../../tasks/refinements/foundation/dockerfile_app.md) settled the multi-stage shape, the base-image pair (`node:lts` Debian for the build stage, `node:lts-alpine` for runtime), the pnpm-via-Corepack pattern, the single-image-serves-all-surfaces decision, and Vite as the eventual frontend bundler.

The repo today is a pnpm workspace (ADR 0010) with five workspaces: `apps/server`, `apps/{moderator,participant,audience}`, and `packages/shared-types`. None of them has real product code yet — they ship `index.{ts,tsx}` stubs (`export {};`) and empty `scripts` blocks. The downstream tasks `backend.api_skeleton`, `data_and_methodology.schema` (migrations), and the frontend-bundler tasks fill in the runtime behaviour. The application Dockerfile is needed *now* by `foundation.dev_env.compose_file`, even though those inner pieces have not landed: the dev-stack compose file expects an image to depend on.

## Decision

A single `Dockerfile` at the repo root with three stages — `deps` → `build` → `runtime` — and a `.dockerignore` excluding everything the build doesn't need (`node_modules/`, `dist/`, `.git/`, `playwright-report/`, `test-results/`, IDE state, env files, the Husky dispatcher dir).

**Stages**:

- **`deps`** — `node:20-bookworm-slim` (Debian, the LTS we target per ADR 0001). Enables Corepack and activates `pnpm@9.15.4` (matching the `packageManager` pin in root `package.json`, ADR 0010). Copies *only* `package.json`, `pnpm-workspace.yaml`, `pnpm-lock.yaml`, and each workspace's `package.json`, then runs `pnpm install --frozen-lockfile`. This layer is cache-friendly: it invalidates only when the lockfile or a manifest changes, so source-only edits skip re-installation entirely. `HUSKY=0` is set so the `prepare` script no-ops inside the image (no git tree, no hooks needed).
- **`build`** — same Debian base; copies the resolved tree from `deps` (whole `/app` rather than enumerating per-workspace `node_modules/` because pnpm's hoisting layout varies) and layers source on top, then runs `pnpm -r build`. Today that walks five workspaces: `apps/server` and `packages/shared-types` actually emit JS via `tsc -b`; the React frontends run an explicit no-op echo script (`"build": "echo 'no build yet (placeholder; bundler wiring lands with frontend tasks)'"`) added to each placeholder workspace's `package.json` as part of this task. The no-op script is intentional — `pnpm -r build` would otherwise skip those workspaces silently, and a future contributor adding the real bundler will edit one obvious string instead of remembering to wire a new script everywhere.
- **`runtime`** — `node:20-alpine`. Reinstalls dependencies with `pnpm install --frozen-lockfile --prod --ignore-scripts` so devDependencies (TypeScript, ESLint, Vitest, Playwright, Cucumber, etc.) don't ship. Copies only the compiled `dist/` outputs from the build stage (`apps/server/dist`, `packages/shared-types/dist`); source `.ts/.tsx` does not ship. Drops to the `node` user (uid 1000) before `CMD`. Exposes port 3000.

**Entry point — stub today.** The runtime `CMD` is a one-liner that prints a banner identifying the image and the deferred work, then exits 0. The full entry point — apply migrations on startup (per architecture C6), then start the HTTP/WebSocket server, serving the frontend bundles as static assets — is owned by `backend.api_skeleton` together with the migrations task. Documenting the deferral here keeps the image honest: it builds, it boots, it doesn't pretend.

## Consequences

- **`docker build .` succeeds end-to-end on the current placeholder repo.** Verified: image builds, runs, exits 0, prints the banner. Image size **155 MB** (target was < 200 MB), measured with `docker image ls a-conversa-app:dev`.
- **Layer caching is right.** The expensive `pnpm install --frozen-lockfile` layer is keyed on the lockfile + manifests only; editing `apps/server/src/...` reuses the cached `deps` layer. The `build` stage's source-COPY layer invalidates on any source change, but that's unavoidable and the install is already cached.
- **Build on Debian, runtime on Alpine** — settled in the refinement. The Debian build stage gives native modules (`bcrypt`, `pg-native`, anything Postgres-side that may want a C toolchain) the ordinary build environment; the Alpine runtime keeps the shipped image small. If musl-vs-glibc friction shows up later (it sometimes does with native modules built on glibc and run on musl), the fallback is `node:20-bookworm-slim` for runtime as well — same Dockerfile shape, two-line change.
- **pnpm via Corepack, not a separate install step.** Matches ADR 0010's pinning model: the `packageManager` field in root `package.json` is the single source of truth for the pnpm version; Corepack reads it. The `PNPM_VERSION` Docker `ARG` defaults to `9.15.4` and should be bumped in lockstep with the root `package.json` pin.
- **Single image serves all surfaces.** The runtime stage is structured for this: when frontend bundlers land, their `dist/` directories will be added as `COPY --from=build` lines in the runtime stage and the server will serve them as static assets. No separate per-surface image, no nginx sidecar.
- **Placeholder no-op build scripts in the React-frontend workspaces.** A small, deliberate piece of throwaway: `"build": "echo 'no build yet (placeholder; bundler wiring lands with frontend tasks)'"` in `apps/{moderator,participant,audience}/package.json`. They exist purely so `pnpm -r build` succeeds today, and they get replaced — not deleted — by the real bundler scripts when the frontend tasks land.
- **Stub `CMD` is honest about what's missing.** The real entry point lands with `backend.api_skeleton` (server boot) and the migrations task (`data_and_methodology.schema`, which adds `apps/server/migrations/` and the runner). Until then the image starts cleanly under any caller (compose health checks, manual `docker run`, future CI smoke checks) and exits 0; it does not lie about running a server.
- **Image is dev-shaped, not production-final.** `deployment.prod_container` will iterate on this Dockerfile — the same multi-stage shape, but likely with adjustments for image-signing, vulnerability scanning, image-tagging conventions, possibly distroless if pnpm/native deps cooperate by then. This Dockerfile's structure is intended to be the spine prod inherits from, not a throwaway.
- **CI not touched here.** `foundation.ci.ci_build` is the task that wires `docker build` into CI; this task only authors the Dockerfile.
- **Compose / Postgres / mock-OAuth not touched here.** `dockerfile_postgres`, `dockerfile_mock_oauth`, and `compose_file` are separate refinements that depend on this one.

## Amendments

- **2026-05-10** — The `HUSKY=0` env var set in the `deps`, `build`, and `runtime` stages exists specifically to neutralize the `prepare` script that [ADR 0014](0014-pre-commit-hooks-husky-lint-staged.md) (Husky + lint-staged) wires into root `package.json`. Without `HUSKY=0`, `pnpm install` inside the image would invoke `husky` and try to install client-side git hooks against a `.git` tree that the build context excludes (per `.dockerignore`). Adding the explicit backlink so the interaction is discoverable from either ADR. The decision (multi-stage with pnpm via Corepack) is unchanged.
- **2026-05-10** — The `dockerfile_postgres`, `dockerfile_mock_oauth`, and `compose_file` refinements above have all landed: [ADR 0016](0016-postgres-upstream-alpine.md) (Postgres `postgres:16-alpine`), [ADR 0017](0017-mock-oauth-authelia-users-file.md) (Authelia 4.39 in users-file mode), and [ADR 0018](0018-compose-file-three-service-dev-stack.md) (three-service compose stack at `compose.yaml`). The `app` service in compose builds from this Dockerfile and tags it `aconversa/app:dev`; the standalone `docker build -t a-conversa-app:dev .` invocation noted in the Dockerfile header is the manual-build form. The decision (multi-stage Dockerfile) is unchanged.
- **2026-05-10** — The stub `CMD` described in the Decision section has been **replaced** by the real Fastify server entry point: `CMD ["node", "/app/apps/server/dist/index.js"]`. Landed with `backend.api_skeleton.http_server` and [ADR 0023](0023-web-framework-fastify.md) (Fastify 5). The compose `app` service now starts the real server; the healthcheck still targets `/healthz` (which is owned by `backend.api_skeleton.health_endpoint`) and remains unhealthy until that sibling lands, but the container no longer exits 0 in a restart loop — `curl http://localhost:3000/` returns `{ status: 'ok' }`. The runtime stage's COPY layout (which already copies `apps/server/dist` from the build stage) needed no further changes. The decision (multi-stage Dockerfile) is unchanged.
