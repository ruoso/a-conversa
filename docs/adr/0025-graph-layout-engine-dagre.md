# 0025 — Moderator graph layout engine: `dagre` for hierarchical (Sugiyama-style) auto-layout

- **Date**: 2026-05-15
- **Status**: Accepted

## Context

The moderator's `<GraphCanvasPane>` mounts ReactFlow per [ADR 0004](0004-graph-libraries-reactflow-and-cytoscape.md) and renders nodes + edges projected from the per-session event log. Today every emitted node gets a placeholder grid position (`x = (i % 4) * 240`, `y = floor(i / 4) * 140`) keyed by `node-created` arrival order — a deliberate stop-gap pinned in `tasks/refinements/moderator-ui/mod_node_rendering.md` until a real layout engine lands. The grid keeps nodes from stacking, but it carries none of the methodology structure: a `supports` edge from `data` to `claim` is visualized identically regardless of where the two nodes sit, and the moderator cannot read "claim at the top, evidence underneath" from the canvas.

An argument graph's shape is overwhelmingly **hierarchical with occasional cycles**: a claim is supported by data via warrants; rebuttals attach to claims and warrants; meta-disagreement splits and decompositions add structure under existing nodes. Per `docs/methodology.md`, the methodology surfaces a top-down causal / inferential flow at the modal case (claim → warrant → data) plus the structural escape valves (cycles when reasoning loops, meta-disagreement when the dispute itself becomes the record). The first task this layout engine serves — surface "claim on top, evidence below" at a glance — is the methodology's primary reading order.

ReactFlow does not ship an auto-layout pass; the library is interactive-edit-first by design and expects positions to come from the caller. The four candidates surveyed:

- **`dagre` (`@dagrejs/dagre`)** — Sugiyama-style hierarchical layouter; the canonical ReactFlow companion (the official ReactFlow examples use it for exactly this case). Layered top-to-bottom or left-to-right, deterministic given an input graph + ordering, handles DAGs natively, tolerates cycles by treating the input as a "best-effort" DAG (edges that introduce cycles are reversed for layering then drawn in their original direction). Tiny dependency (~50 KB unminified including its `graphlib` peer), no native bindings, runs in under 10 ms on 100 nodes. The MIT-licensed `@dagrejs/dagre` fork (active maintenance) supersedes the original abandoned `dagre@0.8.5` package. The `useReactFlow().fitView` pairs naturally with it: run dagre, hand positions back to ReactFlow, fit-view to frame the result.
- **`elkjs`** — JS port of the Eclipse Layout Kernel; richer algorithm catalogue (layered, force, radial, rectpacking, box) and finer per-edge / per-node options (port constraints, edge bundling). Three trade-offs: (a) ~1 MB minified — 20× dagre's footprint, a non-trivial bundle hit on the moderator surface; (b) async API — every layout call returns a Promise (the worker-thread story is the reason; even synchronous calls go through the same async layer), which complicates the projection's pure-function story and forces every consumer to thread an awaited result; (c) overkill for the argument-graph use case — the layered algorithm dagre also has is the one we'd pick, so the extra weight buys nothing v1 needs. Defensible for v2+ if we surface graph-class options (radial overview, force-clustered exploration) but not today.
- **`d3-force`** — Force-directed simulation; produces round, balanced layouts that look pleasant on densely connected social graphs. Wrong fit for argument graphs: (a) it explicitly does NOT respect hierarchy — a claim-at-top reading is the methodology's primary signal and force-directed layouts produce them only by coincidence; (b) layouts are non-deterministic without a seeded RNG and even with a seed the simulation iterates over many ticks (visible re-arrangement on every insertion); (c) the dense-graph aesthetic is wrong for a graph the moderator is actively editing one node at a time. The `@types/d3-force` reference in the existing lockfile is a transitive dependency, not a moderator-side intent to render force layouts.
- **Manual positioning (moderator places nodes)** — ReactFlow's native idiom; long-term the right escape valve for "the moderator knows best" cases. Not the right default: a session's first node creation should land somewhere readable without requiring the moderator to drag it; and the cost of building the manual-placement UX before the auto-layout exists is the wrong order.

Stability under incremental change is the other axis. A Sugiyama-style layout that re-runs from scratch on every event ticks every existing node to a new position when one node is added — disorienting. Two strategies handle this:

1. **Pin previous positions; only relayout new nodes.** Cache the prior `nodes` array's `(id → {x, y})`; when projecting, reuse the cached position for every existing node and only feed *new* nodes (and their edges) through dagre. Existing nodes never move on incremental events.
2. **Relayout from scratch on every event, accept the jitter.** Simpler; deterministic given input order; visually disorienting.
3. **Relayout from scratch only on user-triggered "tidy up"; auto-layout only new nodes on incremental events.** Hybrid — the moderator gets a "freshly arranged" canvas on demand without losing in-flight context to involuntary motion.

The hybrid (strategy 3) is the methodology-faithful pick: moderators work *across* events on a stable canvas; a major rearrangement is a user-elective action ("the canvas feels cluttered, tidy it up"). The user-triggered "tidy up" affordance itself is a future task (`mod_layout_tidy_action` if/when filed); this ADR commits to making the seam exist (a `relayoutAll()` function exported from the layout module) without shipping the button.

Variable node heights complicate this slightly. The moderator's `<StatementNode>` is a Tailwind card with wording (2-line max) + a kind label + optional decoration rows (facet pills, axiom marks, annotations, vote indicators). Per-facet-state visualization (`mod_per_facet_state_visualization`) explicitly produced cards of varying heights depending on which facets carry status, and per `mod_axiom_mark_decoration` / `mod_vote_indicators_on_graph` further decoration grows the card. Dagre uses fixed per-node dimensions to compute the layered placement; if we hand it the rendered card heights it produces tighter, less-overlapping layouts but requires *measuring* the DOM after a render pass. The pragmatic v1 choice: feed dagre constant placeholder dimensions (`width: 220, height: 90` — the card's `min-w-[12rem] max-w-[18rem]` + the typical two-line wording + kind label + one decoration row) and accept that very tall cards (many decoration rows) will visually overlap the row below. The DOM-measured improvement is a follow-up task `mod_layout_measured_dimensions`; v1 ships the simpler path.

The Refinement-Writer brief in ORCHESTRATOR.md says: "Adding a new graph-rendering dependency is ADR-worthy. Reusing whatever the moderator canvas already uses is not." None of the moderator workspace's existing dependencies (`reactflow`, `react`, `react-dom`, `react-router-dom`, `zustand`, `i18next`, `react-i18next`, `i18next-icu`, `@a-conversa/shared-types`, `@a-conversa/i18n-catalogs`) provide a graph-layout algorithm. The repo-root `package.json`'s `devDependencies` carry `cytoscape@^3.30.0` for the read-only-surface smoke script — but ADR 0004 explicitly forbids using Cytoscape on the moderator surface (different interaction profile, different library), so its bundled layouts are not reusable here. The `@types/d3-force` lockfile entry is a transitive type stub, not a runtime library. Adding a graph-layout library is therefore the v1 unblock.

## Decision

The moderator workspace adds **`@dagrejs/dagre@1.1.4`** (pinned, no caret) as a runtime dependency and routes all auto-layout through it.

**Default orientation**: top-to-bottom (`rankdir: 'TB'`). The methodology surfaces a claim-at-top reading order; placing source nodes (in the supports/rebuts/qualifies graph sense) above their targets matches "claim → warrant → data" as a vertical reading. Per-debate orientation switch (LR / RL / BT) is a future task if real moderation sessions show TB is wrong for some debate shapes; v1 commits to TB.

**Stability strategy**: incremental positions are cached; only *new* nodes pass through dagre on event-driven incremental layout. Existing nodes keep their positions across `node-created` / `edge-created` / `commit` ticks. A `relayoutAll()` function — exported alongside the incremental projection — exists for a future user-triggered "tidy up" action; v1 does not ship the action itself, only the function it will eventually call.

**Re-layout trigger**: auto on every projection pass driven by event-driven `useMemo` deps in `<GraphCanvasPane>`. The pure-function layout pass takes precomputed `Node[]` (today positioned on the grid) + `Edge[]` and returns the same nodes with `position` overwritten by the cached-or-dagre-computed values. The stability cache keys on node ids — a node that was positioned in tick N gets the same `{x, y}` in tick N+1 unless `relayoutAll()` was invoked.

**Dimensions fed to dagre**: constant `width: 220, height: 90` for v1; DOM-measured per-card dimensions are a follow-up task (`mod_layout_measured_dimensions`).

**Library**: `@dagrejs/dagre` (the actively-maintained MIT fork of the original `dagre`). Pinned at `1.1.4`, the latest stable at the time of this ADR.

**Module home**: the new file `apps/moderator/src/graph/layoutEngine.ts` exports a pure `applyLayout(nodes, edges, options)` function plus a `relayoutAll(nodes, edges, options)` companion. `<GraphCanvasPane>` calls `applyLayout` once per `useMemo([events, diagnosticHighlights, sessionId])` tick after `projectNodes` returns. The layout engine does NOT subscribe to the store; the caller decides when to invoke it. Mirrors the `facetStatus.ts` / `diagnosticHighlights.ts` pattern (pure module + caller-driven re-runs).

## Consequences

- **Reads like an argument.** The first node a moderator creates lands at the top; nodes connected via `supports` / `rebuts` / `qualifies` / `bridges-from` / `bridges-to` flow downward in layered ranks. The methodology's claim-at-top reading order surfaces at first paint, on every session, without per-debate configuration.
- **One new top-level moderator dependency.** `@dagrejs/dagre@1.1.4` lands in `apps/moderator/package.json` under `dependencies` (pinned, no caret, matching the repo convention from `mod_graph_canvas_pane`'s `reactflow: 11.11.4` pin). Footprint: ~50 KB minified + its bundled `graphlib`. The moderator bundle grows by a small amount; the bundle-size growth is the expected price of auto-layout. No new ADR-tier consequence on the bundle-sensitive audience surface (audience uses Cytoscape per ADR 0004).
- **Incremental stability over the cycle case.** Dagre's stability under "add one node, rerun" is structurally weak in the worst case; the cache-and-only-layout-new-nodes strategy sidesteps the issue entirely for the common case. The user-triggered "tidy up" path is reserved for when the moderator wants a fresh arrangement — that path *intentionally* moves existing nodes.
- **Cycles render but layered placement is best-effort.** Dagre breaks cycles by reversing offending edges for the layering pass and drawing them in their original direction; the visual result is a layered graph with one or more "back-edges" that visibly travel upward. The moderator's diagnostic-highlighting layer (`mod_diagnostic_highlighting`) already surfaces cycle diagnostics with an amber halo, so the back-edge geometry is supplemented by the methodology-engine's explicit signal — the visual cost of the layered approach is bounded.
- **Variable card heights overlap in v1.** A card with many decoration rows (multiple facet pills + axiom marks + vote indicators) will overflow the constant 90px slot dagre allocates and visually overlap the row below. Accepted v1 trade-off; the `mod_layout_measured_dimensions` follow-up task lifts the limitation by reading `getBoundingClientRect` after the first render and re-laying out.
- **No state-management coupling.** The layout engine is a pure module; the cache lives in a `useRef` inside `<GraphCanvasPane>`. The Zustand WS store is unchanged. The cache is intentionally per-component-instance — re-mounting `<GraphCanvasPane>` (e.g. by route navigation) clears the cache, which is the right semantics for "the moderator left and came back, fresh arrangement is fine."
- **Tested through committed Vitest cases** per [ADR 0022](0022-no-throwaway-verifications.md). The pure-function `applyLayout` is tested directly (no React tree); the integration with `<GraphCanvasPane>` is tested via the existing canvas test suite. A Playwright e2e spec under the UI-stream policy proves the in-browser layout produces a non-overlapping, edge-connected arrangement.
- **Two ADRs receive Amendment lines** (per the amendment-pass rule in `docs/adr/README.md`): ADR 0004 (graph libraries) and ADR 0022 (no throwaway verifications — only if its operational text mentions the moderator graph rendering surface, which it does not directly; the amendment-pass scan is conservative). The 0004 amendment notes that the moderator surface's hierarchical auto-layout uses dagre — Cytoscape's bundled layouts remain reserved for the read-only surfaces.

## Stack-validation tests

Per [ADR 0022](0022-no-throwaway-verifications.md), no throwaway smoke script lands at the ADR layer. The library lands behind the real wiring inside `apps/moderator/src/graph/layoutEngine.ts`, with the test contract spelled out in `tasks/refinements/moderator-ui/mod_layout_engine_choice.md`'s Acceptance criteria — a pure-function Vitest suite over `applyLayout`, a `<GraphCanvasPane>` extension verifying the canvas consumes the dagre-derived positions, and a Playwright spec verifying the in-browser layout produces a stable, non-overlapping arrangement.

## Amendments

- **2026-05-15** — `nodeWidth` adjusted from `220` to `288` during the `mod_layout_engine_choice` implementation pass. The original 220 figure was pinned in this ADR as the "empirical mid-point" between `<StatementNode>`'s `min-w-[12rem] max-w-[18rem]` (192–288 px) range, but the Playwright non-overlap assertion observed real-world wordings hitting the `max-w-[18rem]` upper bound: with a 220 px dagre footprint and 40 px nodesep, cards laid out at `x={0, 260}` while rendered cards were 288 px wide, producing a horizontal overlap. The 288 px figure matches the Tailwind upper bound exactly so the dagre footprint always bounds the actual rendered card. The trade-off is a slightly looser horizontal arrangement for short wordings (a per-card measurement pass is the deferred `mod_layout_measured_dimensions` follow-up). All other dimension constants (`nodeHeight: 90`, `rankSep: 60`, `nodeSep: 40`) and the layout strategy are unchanged.
