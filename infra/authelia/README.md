# Authelia container — `infra/authelia/`

Configuration record for the Authelia OIDC provider used by the local-dev
Compose stack. The actual `docker-compose.yml` wiring lands with the
`foundation.dev_env.compose_file` task; this directory documents the
decisions that task will consume, plus the YAML it will mount.

See [ADR 0002](../../docs/adr/0002-auth-self-hosted-oidc-authelia.md)
for the choice of Authelia as the project's OIDC service, and
[ADR 0017](../../docs/adr/0017-mock-oauth-authelia-users-file.md) for
the dev-mode users-file configuration recorded here.

## Image

- **`authelia/authelia:4.39`** (upstream image, no custom Dockerfile).
- Major-minor pinned at **4.39** (current stable line at the time of
  writing). Bump deliberately when a new minor lands; do not float to
  `latest`.
- The same image is used in production; only the mounted configuration
  and the secret material differ.

## Mount points

The Compose service mounts two paths:

| Container path                       | Source                                    | Purpose                                   |
| ------------------------------------ | ----------------------------------------- | ----------------------------------------- |
| `/config/configuration.yml`          | `infra/authelia/configuration.yml` (ro)   | Main Authelia config                      |
| `/config/users.yml`                  | `infra/authelia/users.yml` (ro)           | Flat-file user database                   |
| `/var/lib/authelia/`                 | named volume `aconversa-authelia-data`    | sqlite DB + filesystem notifier output    |

Both config files are mounted **read-only** so the container's startup
chown attempts (which the upstream entrypoint does unconditionally) fail
loud-but-harmlessly without rewriting the files in the working tree. The
`/var/lib/authelia` volume holds the sqlite storage backend and the
filesystem notifier output — both ephemeral dev state, dropped by
`docker compose down -v`.

The volume's host-side scratch directory (when run outside Compose for
ad-hoc verification) is gitignored; see `.gitignore`'s
`infra/authelia/data/` entry.

## Issuer URL convention

OIDC issuer URLs are environment-dependent and live in `.env.example`
(authored by `foundation.dev_env.env_var_template`). The conventions
this directory commits to:

| Caller                              | Issuer URL                          |
| ----------------------------------- | ----------------------------------- |
| Backend service inside Compose      | `http://authelia:9091`              |
| Browser / host shell                | `http://localhost:9091`             |

Service-to-service traffic uses the Compose-internal DNS name
(`authelia`); browser-side redirects target the host-published port. The
backend is configured with both — the issuer for OIDC discovery, the
authorization endpoint for the user-facing redirect — and these two
values can therefore differ. The cookie domain in `configuration.yml`
(`aconversa.local`) accommodates Authelia's "must contain a period or
be an IP" check; production uses real DNS names and is not constrained
this way.

## OIDC client registration

The application registers as a single generic OIDC client. The shape
committed in `configuration.yml`:

| Field                          | Dev value                                 |
| ------------------------------ | ----------------------------------------- |
| `client_id`                    | `aconversa-app-dev`                       |
| `client_secret` (plaintext)    | `aconversa-app-dev-secret`                |
| `redirect_uris`                | `http://localhost:5173/api/auth/callback`, `http://localhost:3000/api/auth/callback` |
| `scopes`                       | `openid`, `profile`, `email`, `offline_access` |
| `response_types`               | `code`                                    |
| `grant_types`                  | `authorization_code`, `refresh_token`     |
| `token_endpoint_auth_method`   | `client_secret_basic`                     |

The client secret in `configuration.yml` is stored as a `pbkdf2-sha512`
hash of the plaintext above, generated by Authelia's `crypto hash` CLI.
The plaintext is what the application uses to authenticate against the
token endpoint; that value (along with the client ID and the issuer
URLs) lives in `.env.example`, owned by
`foundation.dev_env.env_var_template`.

In production these placeholders are overridden by real client
credentials supplied as deploy secrets. The application's OIDC client
code is identical in dev and prod — only the issuer URL and the secret
material change.

## Dev users and dev password

Twelve dev users are seeded in `users.yml`: **`alice`**, **`ben`**,
**`maria`**, **`dave`**, **`erin`**, **`frank`**, **`grace`**,
**`henry`**, **`ivan`**, **`julia`**, **`kate`**, **`leo`**. All
twelve share a single dev password — **`aconversa-dev`** — so
`make up` produces a working login immediately and Playwright tests
can log any of them in without per-user secret management.

The pool was originally seeded with six users by
`tasks/refinements/foundation/dockerfile_mock_oauth.md`; it was
expanded to twelve by
`tasks/refinements/participant-ui/part_e2e_user_pool_expansion.md`
so the participant-UI Playwright suite could drop
`test.describe.serial` on `tests/e2e/participant-graph-render.spec.ts`
and run its four blocks in parallel again, each with a distinct
`{ creator, debater }` pair.

The shared password is intentional and dev-only: it is documented here,
in the users-file header, and in the ADR. Production never reads this
file; production Authelia uses a database-backed user store and
federated upstream providers (per ADR 0002).

To rotate the dev password (or generate a different one):

```sh
docker run --rm authelia/authelia:latest \
  authelia crypto hash generate argon2 --password 'NEW-DEV-PASSWORD'
```

Paste the resulting `$argon2id$...` hash into each user's `password:`
field in `users.yml`, then update this README with the new plaintext.

## RSA issuer key (dev only)

The OIDC provider needs an RSA private key to sign id_tokens. The
committed `configuration.yml` embeds a dev-only RSA-2048 key inline
(under `identity_providers.oidc.jwks[0].key`). It was generated with:

```sh
docker run --rm authelia/authelia:latest \
  authelia crypto pair rsa generate --directory /tmp \
  --file.private-key oidc.issuer.dev.key
```

This key is not a secret — it lives in the public repo. Production
overrides the JWKS entry via env (or a separate config file overlay)
with material held as a deploy secret. The key ID `dev-issuer-2026` is
used so any token observed in dev is unambiguous about its origin.

## Verifying the container starts

To smoke-test the config without bringing the whole Compose stack up:

```sh
mkdir -p /tmp/authelia-data
docker run --rm \
  -v "$PWD/infra/authelia:/config:ro" \
  -v /tmp/authelia-data:/var/lib/authelia \
  -p 9091:9091 \
  authelia/authelia:4.39 \
  --config /config/configuration.yml
```

Look for `Startup complete` and `Listening for non-TLS connections on
'[::]:9091'` in the log output. The `/.well-known/openid-configuration`
endpoint will return non-200 because the issuer URL is configured for
the Compose hostname rather than `localhost`; that is wired up
end-to-end by `compose_file` + `env_var_template`. Container startup is
the property this directory verifies.
