# Mount-contract types (MountProps, UnmountFn, MountFn, SurfaceModule)

**TaskJuggler entry**: [tasks/27-shell-package.tji](../../27-shell-package.tji) — task `shell_package.shell_mount_contract`
**Effort estimate**: 0.5d
**Inherited dependencies**:

- `!shell_package.shell_pkg_skeleton` (settled — workspace skeleton + Vite library build + `SHELL_PACKAGE_VERSION` shipped 2026-05-16 in commit `9035d9d`; see [`tasks/refinements/shell-package/shell_pkg_skeleton.md`](shell_pkg_skeleton.md)).

## What this task is

Land the TypeScript **mount-contract types** every surface bundle (`apps/moderator/`, `apps/participant/`, `apps/audience/`, `apps/replay-test/`) exports and every host (`apps/root/`'s dispatcher) consumes. Pure types — no runtime code, no React, no DOM. The four exported names:

- **`MountProps`** — the dependency bag the host passes into a surface's `mount()`. Carries the container element, the auth context value, the shared i18n instance, the React Router base path the surface mounts under, and optional shared services (WS client, locale string).
- **`UnmountFn`** — `() => void`. The synchronous cleanup function `mount()` returns; the host invokes it on prefix change to tear the surface down.
- **`MountFn`** — `(props: MountProps) => UnmountFn`. The canonical signature every surface's `mount` export conforms to.
- **`SurfaceModule`** — `{ mount: MountFn; meta?: SurfaceMeta }`. The shape of the object a dynamically-imported surface bundle exposes; `meta` is an optional slot for surface-level metadata (display name, required auth level) the host can read without invoking `mount()`.

Source layout: a new subdirectory `packages/shell/src/mount-contract/` with the type declarations split across one or more `.ts` files (one if the whole thing fits cleanly, otherwise one per type). The root [`packages/shell/src/index.ts`](../../../packages/shell/src/index.ts) re-exports the four public names so consumers `import { MountProps, ... } from '@a-conversa/shell'` and never reach into the subdir.

**Forward-declared placeholder interfaces** ship in the same module for the named services later leaves will fully implement: `AuthContextValue` (real impl in `shell_auth_context`), an `I18n` interface (real impl in `shell_i18n_bootstrap`), and a `WebSocketClient` interface (real impl in `shell_ws_client`). Each placeholder is the **bare-minimum shape consumers need** so the contract is callable today without circular WBS dependencies; the implementing leaves widen or refine — they MUST stay assignable to the placeholder.

A Vitest case at `packages/shell/src/mount-contract/mount-contract.test.ts` pins the canonical "no-op surface" shape compiles and behaves: `const noopSurface: SurfaceModule = { mount: () => () => {} }` typechecks at compile time and the runtime assertion confirms invoking `mount()` returns a callable whose return is `undefined`.

## Why it needs to be done

Every downstream leaf in this group — `shell_auth_context`, `shell_screen_name_form`, `shell_login_logout_components`, `shell_i18n_bootstrap`, `shell_ws_client`, `shell_error_mapper`, `shell_tests` — needs these types to be type-safe. The root-app group (`apps/root/`) imports `MountFn` to type its dynamic-import call sites and `SurfaceModule` to validate the imported module's shape. The moderator-refactor task (`moderator_ui.mod_extract_to_mountable_library`) wraps its existing `main.tsx` bootstrap in a `mount: MountFn` export and deletes the local `createRoot()` / `BrowserRouter` / `initI18n()` calls. Each consumer pins against the contract.

Landing the types first means the rest of the chain can pin signatures against a stable surface area. The placeholder-interface pattern unblocks the order: `shell_auth_context` is the next sibling to land (`depends !shell_pkg_skeleton, backend.auth.session_token_management`) but needs to be type-compatible with `MountProps.auth` — which means `MountProps` has to ship first with a minimal `AuthContextValue` it can later widen to. Same shape for `shell_i18n_bootstrap` and `shell_ws_client`.

The contract also bakes in two architectural decisions [ADR 0026](../../../docs/adr/0026-micro-frontend-root-app.md) is explicit on but that need to be expressed as TypeScript: (a) **surfaces own their own `BrowserRouter`**, mounted under `routerBasePath` the host passes in — encoded as `routerBasePath: string` in `MountProps` rather than passing a Router instance; (b) **unmount is synchronous** — encoded as `UnmountFn = () => void` rather than `() => Promise<void>`. Both decisions show up as type signatures, so picking them wrong here forces a contract-wide migration later.

## Constraints / requirements

- **Types-only.** No runtime code, no React imports, no DOM imports beyond `HTMLElement` (which is a built-in `lib.dom.d.ts` interface). The `dist/` output may emit an empty `.js` file alongside the `.d.ts` (TypeScript erases types at compile time); that's fine — the build pipeline already runs `vite build && tsc --emitDeclarationOnly` per the skeleton's setup.
- **Source layout: `packages/shell/src/mount-contract/`.** New subdirectory. Files inside (split is the author's call):
  - `packages/shell/src/mount-contract/types.ts` — the four exported names + the three placeholder interfaces (or split `auth.ts` / `i18n.ts` / `ws.ts` / `surface.ts` if the author prefers; one file is fine for ~80 lines of types).
  - `packages/shell/src/mount-contract/index.ts` — re-exports the public names.
  - `packages/shell/src/mount-contract/mount-contract.test.ts` — the no-op surface case.
- **Root barrel update.** [`packages/shell/src/index.ts`](../../../packages/shell/src/index.ts) gains `export * from './mount-contract/index.js';` (or per-name re-exports — author's call; per-name is more explicit but `export *` keeps the barrel diff small as more types land). The existing `SHELL_PACKAGE_VERSION` export and comment block stay untouched.
- **The four exported names: `MountProps`, `UnmountFn`, `MountFn`, `SurfaceModule`.** All `export type` (interfaces or type aliases — interfaces for the object-shaped ones so consumers can extend; type aliases for the function signatures and the surface module). No `export interface` for `MountFn` / `UnmountFn` — function aliases are cleaner as `type`.
- **`MountProps` field list (minimum):**
  - `container: HTMLElement` — required, where the surface renders.
  - `auth: AuthContextValue` — required, the host-supplied auth state.
  - `i18n: I18n` — required, the host-supplied i18n instance (placeholder interface).
  - `routerBasePath: string` — required, the URL prefix the surface mounts under (e.g., `'/m'` for moderator; the surface uses `<BrowserRouter basename={props.routerBasePath}>`).
  - `ws?: WebSocketClient` — optional, the host-supplied WS client. Optional because the audience surface may not need WS for v1 (unauthenticated read-only views) and the replay-test surface ships without one.
  - `locale?: string` — optional, the locale string the host resolved (BCP-47, e.g., `'pt-BR'`). The i18n instance already carries this via `i18n.language`; `locale` is a denormalized convenience for surfaces that want it at mount time without reading from `i18n`. Optional so a surface that doesn't care doesn't need to ignore a required field.
- **`UnmountFn = () => void`** — synchronous. No `Promise` return. The host's contract is "I tear down your container now; you have a synchronous chance to clean up React state and event listeners." Surfaces that need async cleanup (e.g., drain an outgoing WS send queue) own the drain themselves and the host does not await.
- **`MountFn = (props: MountProps) => UnmountFn`** — the canonical signature.
- **`SurfaceModule = { mount: MountFn; meta?: SurfaceMeta }`** — `meta` is optional with a thin shape:
  - `SurfaceMeta = { displayName?: string; requiredAuthLevel?: 'public' | 'authenticated' }` — both fields optional. `displayName` is what the host can render in nav (e.g., "Moderator"); `requiredAuthLevel` is a hint the host can use to short-circuit a mount call for an unauthenticated user trying to enter `/m/*`. Both are advisory; the surface and the backend still enforce per-route checks.
- **Forward-declared placeholder interfaces** in the same module:
  - `AuthContextValue` — `{ status: 'loading' | 'unauthenticated' | 'needs-screen-name' | 'authenticated'; user?: { userId: string; screenName: string }; refresh: () => Promise<void> | void; logout: () => Promise<void> | void }`. The discriminant string set matches the moderator's existing `AuthStatus` (`useAuth.ts` line 52). `refresh` and `logout` accept `Promise<void> | void` so the real impl can be async (moderator's `useAuth.ts` returns `Promise<void>`) without forcing every consumer to `await`. Field `user` is optional — `undefined` when not authenticated. No `error` field; the placeholder is the bare minimum, and consumers that want richer error handling cast to a richer type when `shell_auth_context` widens.
  - `I18n` — `{ t: (key: string, vars?: Record<string, unknown>) => string; language: string; changeLanguage: (lang: string) => Promise<void> }`. Matches the bare-minimum subset of `i18next`'s `i18n` interface surfaces invoke. Real impl (likely `i18next.i18n` itself) is assignable to this — checked at consumer compile time.
  - `WebSocketClient` — `{ subscribe: (kind: string, handler: (event: unknown) => void) => () => void; send: (kind: string, payload: unknown) => void }`. `subscribe` returns an unsubscribe function. `event` and `payload` are typed `unknown` (not the moderator's typed `WsEnvelopeUnion`) because the shell does not depend on `@a-conversa/shared-types` in this task — adding that dep is `shell_ws_client`'s job. Consumers narrow at the call site.
- **Test seam (per [ADR 0022](../../../docs/adr/0022-no-throwaway-verifications.md)).** `packages/shell/src/mount-contract/mount-contract.test.ts` contains one Vitest case:
  - Declares `const noopSurface: SurfaceModule = { mount: () => () => {} };` — typechecks at compile time. If a required field of `MountProps` was removed, or `UnmountFn` was changed to `() => Promise<void>`, the type annotation here fails.
  - At runtime calls `const unmount = noopSurface.mount(props)` with a minimal props bag (a real `document.createElement('div')` for `container`, a `{ status: 'loading' }` stub auth, a minimal i18n stub, `routerBasePath: '/'`), then asserts `expect(unmount()).toBeUndefined()`.
  - Pins observable behavior: "a `SurfaceModule` whose `mount` returns a no-op unmount function fully satisfies the contract." Catches the regression where a future leaf accidentally narrows a `MountProps` field to required-and-non-default-able and breaks the no-op shape.
- **No new dependencies.** Stays React-as-peer, no `dependencies`, no `i18next`, no `react-router-dom`. The placeholder pattern is the whole point.
- **ESLint clean.** The flat config at the root (`eslint.config.js`) covers `packages/**/*.{ts,tsx}` automatically; the test file uses Vitest globals (`describe`, `it`, `expect`) per the existing pattern in [`packages/shell/src/index.test.ts`](../../../packages/shell/src/index.test.ts).

## Decisions

- **Forward-declared placeholder interfaces (`AuthContextValue`, `I18n`, `WebSocketClient`) instead of importing the real implementations.** Alternatives:
  - *Import the real types from their implementation modules now.* Rejected — circular WBS dependency: `shell_auth_context`, `shell_i18n_bootstrap`, `shell_ws_client` all depend on `!shell_pkg_skeleton` and land **after** this task. Reaching for them from `shell_mount_contract` either delays this leaf until all three land (defeating the unblock-the-chain rationale) or creates a forward import the implementing leaves have to break.
  - *Use `unknown` slots for the placeholder fields.* Rejected — `unknown` forces every consumer to type-narrow before use, which is exactly the boilerplate the contract is supposed to eliminate. The placeholders document what shape consumers can rely on without committing to the rich implementation.
  - *Chosen:* minimum-shape placeholder interfaces. Implementing leaves widen (`shell_auth_context` may add fields to its real `AuthContextValue` that aren't on the placeholder); the placeholder defines the **floor** of what a surface can count on, not the ceiling. Consumers that want richer types cast at the call site (`props.i18n as unknown as i18next.i18n`).
- **Surfaces own their own React Router; the host passes a `routerBasePath: string`.** Alternatives:
  - *Host owns the only `<BrowserRouter>`, surfaces consume route definitions as props or via nested `<Routes>`.* Rejected — surfaces have complex internal navigation (the moderator's six routes today; participant's invite-flow → lobby → debate chain coming up). Keeping route definitions colocated with the surface that owns them is cleaner than dragging them out into a host config. Cross-surface navigation is rare (it goes through `window.location` and the host catches the prefix change on the next render).
  - *Host passes a Router instance.* Rejected — couples the contract to a specific Router library version. `react-router-dom@7` vs `react-router@6` vs a future library all have different instance shapes; a string base path is library-agnostic and tested in react-router via `BrowserRouter`'s `basename` prop.
  - *Chosen:* `routerBasePath: string` in `MountProps`. The surface does `<BrowserRouter basename={props.routerBasePath}>` inside its `mount()` body. ADR 0026 Decision 2 is explicit on this; the type encodes the decision.
- **`UnmountFn = () => void`, synchronous (not `() => Promise<void>`).** Alternatives:
  - *`() => Promise<void>` so surfaces can await async cleanup.* Rejected — the host's contract is "I tear down your container now; you have a synchronous chance to clean up React state and event listeners." Async cleanup forces the host into an async navigation flow where the **next** surface's mount is blocked on the previous surface's drain — a UX regression (the new prefix's surface should appear instantly).
  - *Return type `void | Promise<void>` so surfaces can opt in to async.* Rejected — half the host code paths still have to handle both; the union type doesn't reduce host complexity, it just shifts the discriminant onto every call site.
  - *Chosen:* `UnmountFn = () => void`. Surfaces that need to drain async work do so in their own queue (background `Promise`s the surface itself owns); the host does not await them. Worst case, an async send completes after the surface unmounts and the response is dropped — which is fine because the user is on a different surface now.
- **`SurfaceModule` includes an optional `meta` slot.** Alternatives:
  - *No `meta` slot.* Rejected — future leaves (analytics breadcrumbs per ADR 0026 surface, required-auth-level hints, surface display names for the host's nav) want a place to land without breaking the contract. Adding `meta` later would force every existing surface module to either accept `meta?:` (the same shape we're proposing now) or break.
  - *`meta` is required.* Rejected — surfaces should be able to ship without metadata (the replay-test surface, internal tooling surfaces, future stub surfaces). Optional with thin shape is the lowest-friction expansion seam.
  - *Chosen:* `meta?: SurfaceMeta` where `SurfaceMeta = { displayName?: string; requiredAuthLevel?: 'public' | 'authenticated' }`. Both `meta` and both fields inside it are optional. Surfaces opt in field-by-field.
- **`routerBasePath` is a string, not a Router instance.** (Restated above — appears in the routing decision; called out separately because it's the load-bearing type-level encoding of ADR 0026 Decision 2.)
- **No `host` callback in `MountProps` for cross-surface navigation primitives** (e.g., "ask the host to navigate to `/p/sessions/abc/lobby`"). Alternative: include one for clean cross-surface jumps. **Deferred** — it would couple the contract to a navigation primitive (the host's Router internals or a `navigate(url)` helper). Surfaces can navigate within their own base path via `react-router-dom`'s `useNavigate`; cross-surface navigation goes through `window.location.assign(url)` which forces the host to re-dispatch on the new prefix. If a sharper primitive turns out to be needed (e.g., a surface needs to navigate cross-surface without a full reload), it lands as a separate optional field on `MountProps` in a later leaf — not in this task.
- **No `unmount` field on `SurfaceModule` (just `mount` returning `UnmountFn`).** Alternative: `SurfaceModule = { mount: MountFn; unmount: UnmountFn }` with a module-level unmount that the host calls. Rejected — a single surface module can be mounted multiple times (in theory; per the URL-prefix dispatcher in ADR 0026 it'll be mounted at most once at a time, but the type shouldn't preclude future cases like a multi-pane host). Returning the unmount from `mount()` keeps the cleanup scoped to the specific mount call, which is the standard React-imperative-mount pattern (the same shape `ReactDOM.createRoot(el).unmount()` follows).
- **Test pins the no-op shape (not a full-MountProps shape, not no test at all).** Alternatives:
  - *No test in this leaf.* Rejected — violates [ADR 0022](../../../docs/adr/0022-no-throwaway-verifications.md); the contract's invariants want a committed regression-pin, and "the no-op surface fully satisfies the contract" is exactly the kind of structural property whose drift would break every consumer at once.
  - *Test that constructs a full `MountProps` with realistic stubs for auth + i18n + WS.* Rejected — overspecifies. The contract's value is letting consumers compile against minimal shapes; a test that demands full stubs misses the regression we actually care about (someone narrowing the contract to require more than the no-op needs).
  - *Chosen:* one Vitest case that declares the no-op `SurfaceModule`, invokes `mount()` against the minimum-viable props bag, asserts the returned unmount call returns `undefined`. Compile-time check catches MountProps narrowing; runtime check catches an UnmountFn shape change (`void` vs anything else).
