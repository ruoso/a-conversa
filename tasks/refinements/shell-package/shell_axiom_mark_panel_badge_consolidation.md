## Why it needs to be done

The participant panel and the canonical shell `<AxiomMarkBadge>` carry the **same visual contract** (rounded-square, centered "A", per-participant chromatic via `axiomMarkColorFor`, memoized component) but ship as two separate React components because three load-bearing surface choices diverge: the prop shape (`{participantId, screenName}` vs `{mark: AxiomMark}`), the i18n key namespace (`participant.detailPanel.axiomMarkBadge.*` vs `methodology.axiomMark.*`), and the testid prefix (`participant-detail-panel-axiom-mark-badge-` vs `axiom-mark-badge-`). The predecessor refinement (`shell_axiom_marks_extraction`) noted explicitly that unifying these is a behavior change rather than a pure refactor and registered this leaf as the discharge path.

Leaving the divergence in place has the same cost as any near-duplicate component in two workspaces: a future methodology change (a new chromatic-palette entry, a different aria-label structure, an accessibility-audit-driven role change) has to land in two component files in two workspaces, and any update that fails to land in both silently desynchronizes the moderator/audience graph surfaces from the participant detail panel. The methodology cares about consistent surfacing of axiom-marks across every UI in the session — the predecessor refinement made that explicit ("a methodology change to axiom-mark semantics … has to be applied in three files in three different apps with three different test suites, and any update that fails to land in all three sites silently desynchronizes the surfaces' rendering of bedrock"). This leaf closes the last copy.

The screen-name vs UUID semantic divergence is real but resolvable: the panel surface has the participants projection locally and wants screen names; the graph surfaces don't (today) and stick with UUIDs. An **optional** `screenName?: string` prop on the canonical badge lets each surface pass what it has — the graph surfaces pass nothing and keep the UUID branch; the panel passes `screenName` and gets the screen-name branch. When the graph surfaces eventually get a participants projection (a separate future capability), they pass `screenName={resolveScreenName(mark.participantId)}` and inherit the screen-name surface without any further refactor here. The two i18n key clusters stay semantically distinct — they're named differently because they ARE different surfaces (participant identity disclosure vs methodology bedrock attribution); a future change to the screen-name copy doesn't need to ripple to the UUID branch, and vice versa.

The testid retarget (from `participant-detail-panel-axiom-mark-badge-${participantId}` to `axiom-mark-badge-${nodeId}-${participantId}`) is a small behavior change observed only by the panel's Vitest suite + one Playwright assertion — both are repo-internal selector seams, not external API. The canonical testid form is already what the moderator + audience graph surfaces emit; aligning the panel surface on the same form means every future Playwright assertion that wants to target "this axiom-mark badge for participant X" uses the same selector shape regardless of which surface mounts the badge.

## Decisions

### §1 — Optional `screenName?: string` prop on the canonical badge; do NOT split into two shell components

Three options for unifying the panel + graph badge surfaces:

- **(A — chosen)** Extend the canonical `<AxiomMarkBadge>` to accept an optional `screenName?: string`. When provided, the badge switches to screen-name presentation (screen-name-keyed tooltip + sr-label, screen-name in `title`); when omitted, the badge uses the existing UUID-keyed methodology presentation. Single component file in shell; one prop interface; one DOM shape; two i18n key clusters selected by a single discriminant. Backward-compatible at every existing graph call site (they pass nothing for `screenName` and inherit byte-for-byte historical behavior). Forward-compatible: when the moderator + audience graph surfaces eventually get a participants projection (separate future work), they pass `screenName={resolveScreenName(mark.participantId)}` and inherit the screen-name surface for free.
- **(B)** Keep two separate shell components — the existing `<AxiomMarkBadge>` for graph surfaces and a new `<AxiomMarkAttributionBadge>` (or similar) for the panel surface. Cost: two shell symbols with near-identical DOM, chromatic palette, glyph, memoization, and visual contract. Future readers see two components doing almost the same thing and ask why. A methodology change to the visual chrome (e.g. a different ring style, a new role attribute, an accessibility-driven prose tweak) has to land in two files; the consolidation gain is partial. The two components would share `axiomMarkColorFor` + the `AxiomMark` type from shell but duplicate the JSX body. Same drift-risk failure mode the predecessor refinement was structured to close.
- **(C)** Drop the panel's screen-name surface entirely: rewire the panel call site to `<AxiomMarkBadge mark={mark} />` (no screen-name) and let the badge fall through to the UUID-keyed methodology tooltip / sr-label. Cost: regresses the panel UX significantly — the predecessor refinement (`part_entity_detail_panel_chromatic_axiom_mark_badge`) made the screen-name surface deliberate (Decision §2 there explicitly chose `{screenName}` over `{participantId}` substitution for the panel); reverting it would surface UUIDs in the panel's tooltip / aria-label even though the participant has the roster locally. Not a refactor, an unwarranted UX regression.

Chosen: (A). The optional prop is the simplest discriminant; it costs ~10 lines of conditional in the badge body and preserves the historical graph-surface contract verbatim. The screen-name and UUID surfaces are different by design (different i18n key clusters, different substitution variable names), but the rendered DOM + visual contract are identical — exactly the case where an optional prop is the right shape. The third-caller policy that drove the predecessor extraction applies here in spirit: the participant panel is structurally a fourth caller of the same badge primitive; consolidating it follows the same mechanical move.

The conditional branch is a single `if (screenName !== undefined)` selector in the body; it does NOT introduce a new component file, a new export from the barrel, or a new caller-visible API surface beyond the optional prop. The badge's testid, `data-participant-id`, chromatic palette, glyph, memoization, and outer DOM shape are all branch-independent — only the two strings (tooltip + sr-label) differ.

### §2 — Relocate the screen-name i18n keys from `participant.detailPanel.axiomMarkBadge.*` to `methodology.axiomMarkBadge.*`

The screen-name keys today live under the participant-app namespace because they were originally consumed only by the participant detail panel. After consolidation the keys are consumed by the shell-resident canonical badge — a methodology-layer primitive that can be mounted from any surface. Three options for where they live:

- **(A — chosen)** Move both keys to `methodology.axiomMarkBadge.{tooltip,srLabel}` as a sibling to the existing `methodology.axiomMark.{tooltip,srLabel}` cluster. Delete the now-unused `participant.detailPanel.axiomMarkBadge.*` parent block from all three catalogs. The two methodology clusters live side-by-side: `methodology.axiomMark.*` for the UUID-keyed surface (graph callers without a roster), `methodology.axiomMarkBadge.*` for the screen-name surface (callers with a roster).
- **(B)** Keep the keys under `participant.detailPanel.axiomMarkBadge.*`; the shell badge reads them at that path. Cost: a shell-resident component now reads from a participant-app-namespaced key cluster — a semantic mismatch that any future reader scanning the catalog by namespace would flag. The catalog's namespace structure is supposed to reflect WHERE keys are consumed; moving the component to shell without moving the keys leaves the namespace lying about the consumer surface.
- **(C)** Merge the two clusters: rename `methodology.axiomMark.*` to `methodology.axiomMark.uuid.*` and `participant.detailPanel.axiomMarkBadge.*` to `methodology.axiomMark.screenName.*`. Cost: ripple — every existing reader of `methodology.axiomMark.{tooltip,srLabel}` (the moderator + audience graph badges; the consolidated shell suite; the pending-mark surface) has to retarget; the catalog parity test rebases; the test fixtures rebase. High blast radius for marginal semantic clarity (the two clusters are already named differently).

Chosen: (A). The methodology-namespace placement matches the canonical component's home; the parent-block deletion is a clean removal (no remaining nested keys); the cross-locale move is symmetric and the parity test stays green. The two methodology clusters are named distinctly enough (`axiomMark` vs `axiomMarkBadge`) that a future reader scanning the catalog sees them as siblings, not synonyms — and the `Badge` suffix correctly hints at the consumer (the badge primitive, not the methodology surface broadly).

The translated values stay byte-for-byte from the participant-namespaced cluster: en-US `"Bedrock by {screenName}"`, pt-BR `"Pedra fundamental de {screenName}"`, es-419 `"Fundamento de {screenName}"`. The participant refinement (`part_entity_detail_panel_chromatic_axiom_mark_badge` Decision §2) settled those translations; this leaf relocates, doesn't retranslate.

### §3 — Unify on the canonical `axiom-mark-badge-${nodeId}-${participantId}` testid; retire the panel-specific prefix

The canonical shell badge emits `data-testid="axiom-mark-badge-${mark.nodeId}-${mark.participantId}"`; the panel-local badge emitted `data-testid="participant-detail-panel-axiom-mark-badge-${participantId}"`. Three options:

- **(A — chosen)** Retire the panel-specific prefix and unify on the canonical shell form. The panel's component-level Vitest cases retarget (a single assertion-body edit per case); the Playwright spec retargets at line 2949. The retargeted selectors are still unique within the panel surface (the panel renders one node at a time; `nodeId` is constant within a section render; `participantId` is uniqued by the dedup walk).
- **(B)** Pass an optional `testIdPrefix?: string` prop on the badge so the panel call site can preserve `participant-detail-panel-axiom-mark-badge-${participantId}`. Cost: introduces a caller-configurable testid for the badge — a leaky abstraction (the testid is the badge's contract with its tests, not its API). Future callers might pass arbitrary prefixes and the badge's selector surface fragments. The predecessor's panel-specific prefix existed because the panel was a different component; now that the panel uses the canonical component, the canonical prefix is the right answer.
- **(C)** Add a panel-specific wrapper component (e.g. `<PanelAxiomMarkBadge>`) inside the participant workspace that re-emits `<AxiomMarkBadge>` underneath with a redundant outer `<span data-testid="participant-detail-panel-axiom-mark-badge-${participantId}">` wrapper. Cost: a new component file in the participant workspace, contradicting the "the panel uses the shell badge directly" goal. Two testids on every panel badge (one on the wrapper, one on the shell badge underneath) — confusing.

Chosen: (A). The testid retarget is a one-time mechanical change in three files: `EntityDetailPanel.test.tsx` (5 selectors), the Playwright spec (1 selector), and the deleted panel-local Vitest file (subsumed). The canonical form is more informative (the `nodeId` qualifier is present, which surfaces a debug context the panel-specific prefix omitted) and aligns the panel surface's per-badge selector with the moderator + audience graph surfaces — a future Playwright assertion targeting "this axiom-mark badge for participant X regardless of surface" uses the same shape.

The container testid `participant-detail-panel-axiom-mark-attribution` (and the section testid `participant-detail-panel-axiom-marks`) are preserved — they sit at the panel-section layer, not the per-badge layer, and continue to give the panel surface its selector identity.

### §4 — The dedup walk retains the first-seen `AxiomMark` reference (not a synthesized mark)

The panel's `AxiomMarkAttributionSection` dedups by participantId — multiple `AxiomMark` rows for the same participant (which can happen as the methodology lets a participant axiom-mark a node through multiple proposal envelopes; only the latest is the active mark, but the projection may surface earlier rows) collapse into one badge per participant. The dedup walk needs an `AxiomMark` per attribution to pass to the new badge's `mark` prop. Three options:

- **(A — chosen)** Retain the **first-seen** `AxiomMark` reference for each participant: `if (!seen.has(mark.participantId)) { seen.add(mark.participantId); attributions.push({ mark, screenName: ... }); }`. Cost: matches the existing walk's first-seen semantics (the predecessor refinement's dedup walk uses `if (seen.has(mark.participantId)) continue;` — first wins). The mark passed to the badge carries the participant's first axiom-mark on this node; the badge only reads `mark.nodeId` + `mark.participantId` (for testid + chromatic color), so the choice of WHICH AxiomMark within the per-participant set is functionally irrelevant — first-seen is the simplest and matches the existing semantics.
- **(B)** Retain the **latest-by-committedAt** `AxiomMark` reference per participant. Cost: a second pass over the data to find the latest; the badge doesn't display the committedAt today (it's a future surface), so the choice has no observable effect. YAGNI.
- **(C)** Synthesize a minimal AxiomMark-shaped object per attribution (e.g. `{ nodeId: selectedNodeId, participantId, committedAt: '' }`). Cost: invents a sentinel value (`committedAt: ''`) for a field the badge doesn't consume today; if a future change to the badge starts reading `committedAt`, the sentinel surfaces as a bug. Cleaner to pass the real first-seen reference and let any future field-read get a real value.

Chosen: (A). One-line change to the dedup walk's body; preserves the existing first-seen semantics; future-proof against badge changes that start reading more `AxiomMark` fields. The `nodeId` on the first-seen mark equals the selected node's id (the panel renders one entity at a time; the bucket the panel walks is `marks` for that selection); the chromatic color is per-participant deterministic regardless of which specific mark instance is passed.

### §5 — No shell-side `<AxiomMarkAttributionRow>` (or similar group wrapper)

The panel section renders a flex row of badges with a heading and a container div. The flex container is one JSX line at the call site (`<div data-testid="participant-detail-panel-axiom-mark-attribution" className="flex flex-wrap items-center gap-1">`); the heading is panel-specific. Three options:

- **(A — chosen)** Keep the flex container + heading inline at the call site. The shell exports only the badge primitive; the panel composes the row around it. One caller; no extracted symbol.
- **(B)** Extract `<AxiomMarkAttributionRow>` (or similar) into shell — takes `{ marks: AxiomMark[], roster: ReadonlyMap<...>, heading: string }` and renders the section internally. Cost: a one-caller shell symbol (only the participant panel uses it today); the audience + moderator graph surfaces don't render this section pattern. Introduces a shell-resident component with the same heading-translation key set this leaf's call-site already resolves via `useTranslation()` — the shell would need its own i18n integration for the heading, duplicating the participant's. Premature abstraction.
- **(C)** Extract just the flex container as a generic `<BadgeFlexRow>` shell primitive. Cost: a one-line wrapper over a Tailwind class set; the abstraction gain is negative (a future reader has to chase the shell symbol to see what classes the container has).

Chosen: (A). The third-caller policy that drove `<AxiomMarkBadge>` into shell hasn't fired for the row pattern (no other surface renders this row). If a future audience or moderator surface adopts the same section pattern, a separate refinement can scope the extraction then.

### §6 — No new Playwright / Cucumber coverage (Vitest + retargeted Playwright are sufficient)

- **(A — chosen)** Vitest at the canonical home (`packages/shell/src/axiom-marks/AxiomMarkBadge.test.tsx`) + the retargeted panel `EntityDetailPanel.test.tsx` cases + the retargeted block-10 Playwright selector. No new e2e fixtures, no new feature files.
- **(B)** Scope a new Playwright spec that targets the participant panel's `screenName`-branch behavior (e.g. observe that the panel badge's tooltip resolves to a screen name post-tap). Cost: the existing block-10 spec already covers the panel surface + the per-badge testid + `data-participant-id` + `title` assertion; the retargeted selector continues to exercise the same flow. A new spec is redundant.
- **(C)** Scope a Cucumber scenario for the i18n-key relocation. Cost: i18n keys are catalog-internal vocabulary, not a wire / broadcast seam; Cucumber is the wrong layer. The cross-locale parity test in `catalogs.test.ts` is the architecturally-correct pin and stays green by construction (the move is symmetric across locales).

Chosen: (A). The consolidation is a refactor + small behavior change (testid prefix, i18n key path); the regression surface is exactly what the existing Vitest + Playwright assertions exercise. The new shell Vitest cases for the `screenName` branch + the retargeted panel cases are the full incremental pin per ADR 0022.
