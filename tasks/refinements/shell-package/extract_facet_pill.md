## What this task is

The single-commit **shell-extraction transition** that lifts `<FacetPill>` + `PILL_BASE_CLASSNAME` + `PILL_STATUS_CLASSNAME` (plus the small inferred type-coupling — see Decisions §1) from `apps/moderator/src/graph/FacetPill.tsx` into `packages/shell/src/facet-pill/`, rewires every consumer (the moderator's two in-workspace call sites + the participant's one cross-workspace call site) to import from `@a-conversa/shell`, and **drops the architecturally-awkward `apps/participant → apps/moderator` workspace dependency** that the predecessor `part_entity_detail_panel` opened as tactical debt.

## Why it needs to be done

Two intertwined reasons:

**Structural debt paydown — break the participant → moderator workspace edge.** The `part_entity_detail_panel` refinement's Decision §6 ("import vs reimplement") settled the trade-off at the moment: importing the moderator's `<FacetPill>` from the participant was the right tactical move because (a) the participant had to honour the predecessor `part_per_facet_state_styling` Decision §6 deferral — "the per-facet pill row's role moves to `part_entity_detail_panel` (the React-driven panel which IS a React surface and CAN host the moderator's `<FacetPill>` vocabulary)" — and (b) reimplementing the per-facet-state palette + the `PILL_*_CLASSNAME` Tailwind vocabulary in the participant workspace would have duplicated ~50 lines of class-string code with NO behavioural difference, creating drift risk every time the moderator retunes the visual vocabulary. The tactical fix was deliberate, time-bounded by the third-caller-policy, and registered as tech debt in the same commit (`728f8d1`, the `extract_facet_pill` `.tji` `note` lines 67-74). It IS time to pay it down: `apps/participant` should not carry a workspace runtime dependency on a peer UI surface ([`apps/participant/package.json:14`](../../../apps/participant/package.json#L14) + [`apps/participant/tsconfig.json:19`](../../../apps/participant/tsconfig.json#L19)) — that edge inverts the natural shell-substrate-vs-surface layering per [ADR 0026](../../../docs/adr/0026-micro-frontend-root-app.md) and means any future participant-side build needs to walk the moderator's source tree (TypeScript project references chase the moderator's full type graph; if the moderator's `ws/` or `routes/` or `layout/` ever break their typecheck, the participant typecheck breaks too — a structurally arbitrary coupling). After this task: participant depends on `@a-conversa/shell` only (already its existing dep at [`apps/participant/package.json:16`](../../../apps/participant/package.json#L16)); the moderator → participant edge does not need to exist.

**The third caller trigger has fired.** The `extract_facet_pill` `.tji` note articulates the deferral policy verbatim: "Deferred-until-third-caller policy: extracting a primitive with only two callers risks ossifying the wrong shape; extracting at three callers gives confidence the API is right. The audience surface is the natural third caller — when it lands per-facet rendering (likely as part of aud_graph_rendering's per-facet state styling work), it will need the same FacetPill vocabulary." The audience surface skeleton landed in commit `cf747fe` and the audience's eventual per-facet-styling leaf (`aud_graph_rendering.*`) is a near-term P1 leaf in the WBS — once its facet rendering work starts, it will be the third caller of `<FacetPill>`. Doing the extraction *now*, on the schedule of this dedicated 0.5d leaf, means the audience's facet-rendering leaf finds `<FacetPill>` already in `@a-conversa/shell` and imports it cleanly; doing it *later* (after the audience's facet leaf opens its own moderator-import or re-implementation path) compounds the workspace-dep cleanup cost. The `.tji` deps (`participant_ui.part_graph_view.part_entity_detail_panel`, `audience.aud_shell.aud_app_skeleton`) gate this leaf at exactly the moment both predecessors have shipped — the trigger condition is met.

This task unblocks: any future `audience.aud_graph_rendering.*` leaf that needs per-facet rendering (it inherits `<FacetPill>` from `@a-conversa/shell` cleanly); future audience `aud_entity_detail_panel`-style work (same primitive available); the orchestrator's tracking of structural-debt registrations (this commit closes the `extract_facet_pill` row in the orchestrator's debt registry).

## Decisions

- **§1. Where do `FacetName` / `FacetStatus` types live after the move?** Alternatives:
  - *Import them from the moderator's `apps/moderator/src/graph/facetStatus.ts`.* Rejected — this is the exact dependency-inversion the task is solving. Shell-package code cannot reach into a surface app.
  - *Import them from `@a-conversa/shared-types`.* Considered — the canonical home for cross-package wire types. Rejected for this leaf because `FacetName` / `FacetStatus` are *client-side projection* types, not wire-protocol types ([`apps/moderator/src/graph/facetStatus.ts:43-49` + `54-56`](../../../apps/moderator/src/graph/facetStatus.ts#L43-L56) — the file's header comment explicitly notes "`@a-conversa/shared-types` does not re-export `FacetStatus`"); widening `shared-types` to include them risks pulling in projection-vocabulary semantics that aren't actually shared with the server (which uses its own internal `FacetStatus` at [`apps/server/src/projection/types.ts`](../../../apps/server/src/projection/types.ts)). The cleanest move would be a `@a-conversa/methodology-types` package; that's its own decision and not in scope here.
  - *Co-locate a small `types.ts` inside `packages/shell/src/facet-pill/`.* Chosen. The shell's FacetPill module owns a tiny pair of string-literal-union type aliases (`FacetName`, `FacetStatus`) — verbatim copies of what the moderator's `facetStatus.ts:43-56` already declares (and what the participant's `facetStatus.ts:64-77` ports verbatim). The moderator's and participant's `facetStatus.ts` files keep their own copies of the types AND the rule-set derivation logic (Decision §2); the shell's copies serve only the FacetPill's prop typing. Type-level structural compatibility is automatic (TypeScript widens string-literal unions structurally), so the moderator's `computeFacetStatuses` output flows into `<FacetPill status={...}>` without an adapter. When a future leaf collapses the rule-set ports into shell (Decision §2's deferred task), the shell's copies become the canonical home and the moderator + participant ports retire.
  - *Rationale:* shipping the smallest type surface that lets the FacetPill stand on its own. No upstream module change to `shared-types`. No premature methodology-types-package design.

- **§2. Do we collapse `facetStatus.ts` derivation rules (the 7-rule `computeFacetStatuses` walker, ~410 lines) into shell as part of this leaf?** Alternatives:
  - *Yes — same commit.* Rejected. The brief's 0.5d budget targets the FacetPill primitive only; the derivation-rule lift is a separately-scoped 1d+ task because the rules already exist in **three** authoritative homes (server source-of-truth [`apps/server/src/projection/facet-status.ts`](../../../apps/server/src/projection/facet-status.ts), moderator client mirror [`apps/moderator/src/graph/facetStatus.ts`](../../../apps/moderator/src/graph/facetStatus.ts), participant client mirror [`apps/participant/src/graph/facetStatus.ts`](../../../apps/participant/src/graph/facetStatus.ts)); collapsing them into shell requires deciding (a) whether the shell becomes a fourth implementation or replaces the two client mirrors, (b) how the server's authoritative copy stays in lockstep with the shell client copy, (c) whether `@a-conversa/shared-types` should host the rule-set instead. None of those decisions belong in a primitive-extraction leaf.
  - *No — defer until the audience surface adds the fourth caller.* Chosen. Per the existing deferred-until-N-caller policy applied at the projector layer ([`tasks/refinements/participant-ui/part_graph_render.md`](../participant-ui/part_graph_render.md) Decision §4), the rule-set collapse waits for `audience.aud_graph_rendering` (a Cytoscape consumer like the participant) to add a fourth caller (server + moderator + participant + audience = four implementations of the same seven rules). When that fourth caller materializes, a dedicated sibling shell-extraction task lifts the rule-set into shell with the now-validated cross-Cytoscape-consumer shape. This task's scope stays the primitive only.
  - *Rationale:* keep the leaf 0.5d-sized. Honour the deferred-until-N-caller policy at the rule-set layer the same way the precedent leaves honour it at the projector layer.

- **§3. What's the scope of the `<VoteIndicator>` co-move?** Alternatives:
  - *Leave `<VoteIndicator>` in the moderator; let the shell's `<FacetPill>` import it back.* Rejected — same dependency-inversion problem (shell would depend on moderator). Hard no.
  - *Replace the `<VoteIndicator>` render inside `<FacetPill>` with a render-prop / children pattern.* The shell ships a "headless" `<FacetPill>` and each consumer (moderator, participant, future audience) passes its own `<VoteIndicator>` component. Considered — would minimize the shell's surface — but rejected because (a) the existing 4 in-pill vote-indicator row test cases in `FacetPill.test.tsx` (lines 246-321) pin the exact composition of the indicator row inside the pill (the `data-vote-indicator-row` wrapper, the `ml-1 inline-flex items-center gap-0.5` spacing, the per-vote map); a headless seam means re-pinning all four cases in each consumer + the consumer has to repeat the row-composition boilerplate every time, defeating the shared-vocabulary goal of the extraction; (b) the indicator is intrinsically tied to the pill — every site that wants the pill also wants the per-vote dot row in the same composition. Headless seams pay off when consumers want different compositions; here they want the same composition.
  - *Co-move `<VoteIndicator>` + its `axiomMarkColorFor` color-hash dependency into shell alongside `<FacetPill>`.* Chosen. The lift includes `<VoteIndicator>` + `<VoteIndicator>`'s test file + `axiomMarkColorFor` + `AXIOM_MARK_PALETTE` + `AXIOM_MARK_PALETTE_SIZE` + `AxiomMarkColor` interface + `Vote` interface + `EMPTY_VOTES` constant. The moderator's `selectors.ts` keeps its other exports (the annotation / axiom-mark / votes-by-facet projection helpers — those are moderator-graph-specific) and re-imports the lifted names from `@a-conversa/shell`. The shell-side files: `packages/shell/src/facet-pill/FacetPill.tsx`, `…/FacetPill.test.tsx`, `…/VoteIndicator.tsx`, `…/VoteIndicator.test.tsx`, `…/participant-color.ts` (hosts `axiomMarkColorFor` + the palette + the color types), `…/vote-indicator.ts` (hosts `Vote` + `EMPTY_VOTES` + the `<VoteIndicator>` re-export from the `.tsx`), `…/types.ts` (the small `FacetName` / `FacetStatus` pair per Decision §1).

Co-locating with `<FacetPill>` (rather than at a shell-top-level `colors/` directory) keeps the seam tight — every name lifts together; the future-audience facet-rendering leaf gets the whole vocabulary in one import path. The moderator's `<AxiomMarkBadge>` post-rewire imports `axiomMarkColorFor` from `@a-conversa/shell` (one-line edit at the badge's existing import; the badge itself stays in moderator).

There are no external consumers (this is a private monorepo; verified by grep — only [`apps/participant/src/detail/EntityDetailPanel.tsx:71`](../../../apps/participant/src/detail/EntityDetailPanel.tsx#L71) reaches that sub-path, and this leaf retargets that import). - **§5. The moved `FacetPill.test.tsx`'s `import { createI18nInstance } from '@a-conversa/shell'` — same package import resolution?** Alternatives:
  - *Leave the import as `from '@a-conversa/shell'` after the move.* The test file imports `createI18nInstance` from its own package via the package's public name. Workable — pnpm + Vitest's `resolve.conditions: ['source']` setup resolves a package's own name through its `source` export back to its own `src/index.ts`. This is the same pattern the moderator's tests use today (`from '@a-conversa/shell'` resolves to the shell source). Considered.
  - *Rewrite the import to a relative `from '../i18n/index.js'`.* Same physical resolution under Vitest; more explicit about the same-package-internal nature. The relative form makes it obvious at read-time that the import is shell-internal; the public-name form would suggest a cross-package boundary that doesn't exist for a same-package consumer.

- **§6. How do consumer apps' Tailwind v4 toolchains discover the `PILL_*_CLASSNAME` class literals after the move?** Alternatives:
  - *Rely on Tailwind v4's automatic `node_modules` sibling-package detection.* Tailwind v4's source autodiscovery walks the consumer app's source files (`apps/<surface>/src/**`) plus, per the v4 docs, any package imported from `node_modules` whose `package.json` declares it as a Tailwind source. The shell's `package.json` does NOT declare such a hint today, and adding it would be opaque — the shell would need a Tailwind-specific section in its `package.json`. Possible but indirect.
  - *Add a per-app Tailwind safelist for `PILL_BASE_CLASSNAME` + `PILL_STATUS_CLASSNAME` values.* Rejected — fragile (the class strings live in the shell; each app re-declaring them is the duplication the shell extraction is removing).
  - *Add a `@source` directive to each consuming app's `index.css` pointing at `packages/shell/src/**/*.{ts,tsx}`.* Chosen. Tailwind v4's `@source` directive is the canonical way to widen the JIT scanner's input set; one line per consuming app, clearly attributed in the CSS file's header comment. The consuming apps are: `apps/moderator/src/index.css` (already a `<FacetPill>` consumer today, no Tailwind change needed because the moderator's autodiscovery already covers `apps/moderator/src/graph/FacetPill.tsx`; after the move it stops covering the file → **needs the `@source` directive**); `apps/participant/src/index.css` (the participant's autodiscovery covered the participant's local source; the `PILL_*_CLASSNAME` strings now live in shell → **needs the `@source` directive**); `apps/audience/src/index.css` (the audience does not yet render facets but its skeleton consumes the shell; adding the `@source` directive now is preventive and zero-cost since the JIT only emits classes actually referenced from compiled bundles).
  - The exact CSS edit (3 apps × 1 line each):
    ```css
    /* Per tasks/refinements/shell-package/extract_facet_pill.md — Tailwind
     * scans the shared shell source so PILL_*_CLASSNAME literals get JIT'd. */
    @source '../../../packages/shell/src/**/*.{ts,tsx}';
    ```
