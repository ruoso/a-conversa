## What this task is

Add a read-only **diagnostic inspector** panel to the test-mode scrubber surface: as the operator scrubs, a panel beside the graph shows the **structural diagnostics the methodology engine surfaces for the projected state at the current position** — the cycles, contradictions, multi-warrants, dangling claims, and coherency hints, partitioned blocking vs. advisory. This lets design-iteration / debugging users see *which structural problems the engine would flag* at any point in a saved session, not just the graph shape. Concretely:

1. **A standalone `DiagnosticInspector` component** in `apps/test-mode/src/diagnostics/DiagnosticInspector.tsx`, props `{ sessionId: string; position: number }` (read-only — no `setPosition`). It fetches `GET /api/sessions/:id/diagnostics?position=N` for the current position (via a small `useDiagnosticsAtPosition(sessionId, position)` hook colocated in the same dir) and renders the returned diagnostics grouped by severity:
   - **blocking** entries (`cycle`, `contradiction`) and **advisory** entries (`multi-warrant`, `dangling-claim`, `coherency-hint`), each under stable `data-testid`s, showing the per-kind affected ids verbatim (e.g. a cycle's `nodes[]`, a contradiction's `nodeA` / `nodeB` / `edges[]`);
   - a **loading** state while the request is in flight and an **error** state if it fails (the `status` discriminant pattern of [`packages/shell/src/session-log/useSessionEventLog.ts`](../../../packages/shell/src/session-log/useSessionEventLog.ts));
   - an **empty / clean** state when the position has no diagnostics (including `position === 0`, the pre-history baseline where no event has been applied and the projection is empty — a clean, not-an-error readout; Decision §3).
2. **Mount it as a new sibling section** inside `TimelineScrubber`'s `<main>` ([`TimelineScrubber.tsx:83-151`](../../../apps/test-mode/src/scrubber/TimelineScrubber.tsx)), passing the `sessionId` + `position` it already holds — a `<section data-testid="test-mode-diagnostics">` peer of the graph / controls / snapshots / inspector / changes sections (Decision §4).
3. **i18n keys** `testMode.diagnosticInspector.*` in all three catalogs + their `.review.json` companions — the panel chrome only (aria-label, section headings, per-kind labels, severity labels, loading/error/empty strings). The raw entity ids and the `kind` discriminant strings are *data*, rendered verbatim, not translated (Decision §2).
4. **Test layers** (per ADR 0022): Vitest+RTL component tests for the panel against a mocked fetch (blocking/advisory rendering, loading, error, empty/baseline, position-change refetch with stale-response guard), and an extension of the existing scrubber Playwright spec asserting the diagnostic panel tracks position end-to-end against the real backend.

The backend endpoint this panel consumes is **separate work** (`backend.replay_endpoints.get_diagnostics_at_position`, Decision §1) and carries its own Cucumber coverage; this task is the client panel only.

## Why it needs to be done

`test_mode_timeline_scrubber` made saved sessions navigable and rendered the projected *graph* at each position; `test_mode_event_inspector` and `test_mode_changed_highlights` added the per-event readout and the per-step diff. None of them surface the methodology engine's *judgement* of the state — the structural diagnostics that, in a live session, drive the moderator's blocking/advisory cues. For design iteration and debugging (the reason test mode exists), an operator scrubbing a recorded debate needs to ask "what would the engine flag here?" at any position. This panel answers that. It is the third of the four inspector/overlay leaves hanging off the scrubber's lifted-position seam (`test_mode_event_inspector`, `test_mode_changed_highlights`, `test_mode_diagnostic_inspector`, `test_mode_export_position`) and the first that needs server-computed state at a position — establishing, alongside `test_mode_export_position`, the "test-mode panel depends on a `backend.replay_endpoints.*` read" pattern.

## Constraints / requirements

1. **Read-only; single source of position truth.** The panel consumes the lifted `position` + `sessionId` as props. It never owns or mutates position state (no `setPosition`); navigation stays single-sourced in `SessionScrubberContainer`.
2. **Diagnostics come from the server endpoint, not client recompute.** The panel fetches `GET /api/sessions/:id/diagnostics?position=N`. It does **not** import `computeAllDiagnostics` or the detectors (server-only; ADR 0044), and it does **not** reimplement them. The position it sends is the same event-sequence value the scrubber holds, which the endpoint validates against the head.
3. **Refetch on position change, guard stale responses.** The hook refetches when `(sessionId, position)` changes and ignores responses that arrive after a newer request was issued (the standard ignore-stale-result guard), so rapid scrubbing can never paint diagnostics from a superseded position. No debounce in v1 (Decision §5).
4. **Render every diagnostic kind, grouped by severity.** Show blocking (`cycle`, `contradiction`) and advisory (`multi-warrant`, `dangling-claim`, `coherency-hint`) groups, each entry rendering its affected ids verbatim. An unknown future kind must render without throwing (a generic fallback row), not crash the panel.
5. **Explicit loading / error / empty states.** While fetching: a loading affordance. On failure: an error readout with a retry, not a blank. When the endpoint returns zero diagnostics (including `position === 0`): a clean "no diagnostics" readout, not an error or an empty void (Decision §3).
6. **Mount as a sibling section in the existing layout.** Add `<section data-testid="test-mode-diagnostics">` inside `TimelineScrubber`'s `<main>`, peer to the existing sections. Do not restructure the shipped layout (Decision §4); the edit to `TimelineScrubber.tsx` is one import + one section.
7. **i18n parity.** `testMode.diagnosticInspector.*` chrome keys land in all three catalogs + both `.review.json` companions; raw entity ids and `kind` discriminants stay untranslated data; `pnpm --filter @a-conversa/i18n-catalogs run check` exits zero.
8. **No new runtime dependency.** Pure presentational React + `fetch` (the same primitive `SnapshotJumpList` uses). No data-fetching lib, no store, no new package.
9. **Accessibility.** The section carries a localized `aria-label`; the diagnostics list is keyboard-reachable; the existing scrubber spec's axe pass (ADR 0040) stays green with the new panel mounted.
10. **Build + check green.** `pnpm -F @a-conversa/test-mode build` and `pnpm run check` (lint + format + typecheck) stay green.

## Decisions

### §1 — Diagnostics are fetched from a new backend endpoint, not recomputed client-side (ADR 0044)

**Chosen:** add a backend leaf `backend.replay_endpoints.get_diagnostics_at_position` (`GET /sessions/:id/diagnostics?position=N`, sibling of `get_at_position` under [`tasks/20-backend.tji:385`](../../20-backend.tji)) that runs `projectAtPosition` + `computeAllDiagnostics` server-side and returns wire-shaped, severity-classified entries; this panel depends on it and fetches it. The diagnostics detection layer is server-only — every detector reads the `Projection` class, its proposal maps, and `deriveFacetStatus`, none of which is on the wire (`get_at_position` serializes resolved facet values, not the derivation inputs). **Rejected — port the detectors + a client `Projection` into `@a-conversa/shell`** (the way ADR 0043 ported the position primitive and shell already hosts `computeFacetStatuses` / `projectDiagnosticHighlights`): ADR 0043 accepted duplication *because the logic was four trivial integer functions pinned by a truth table*; the detection layer plus a client projection with facet-status derivation is a large surface tightly coupled to the methodology engine, an ongoing parity burden against the source of truth, and far beyond a 1d read-only panel. **Rejected — fold a `diagnostics` field into `GET /state?position=N`** so the export-position and diagnostic panels share one fetch: conflates raw projected state with derived diagnostics, bloats every state fetch, and couples two independently-evolving panels; one route per concern matches `replay_endpoints`' existing shape. **Rejected — reuse the live WS `diagnostic` broadcast:** it carries only the live head's fired/cleared diagnostics, not "what holds at past position N," which is the scrubber's whole question. Full rationale and alternatives in **ADR 0044**.

*Tech-debt registration:* the closer registers **`backend.replay_endpoints.get_diagnostics_at_position`** — "GET /sessions/:id/diagnostics?position=... — structural diagnostics at log position", **effort ~1d**, `depends data_and_methodology.replay_primitive.project_at_position, data_and_methodology.diagnostics` — in `tasks/20-backend.tji` under the `replay_endpoints` group, and adds it to this task's `depends`. It rolls up under **M8 (`m_replay_mvp`)**, which already depends on the whole `backend.replay_endpoints` group ([`tasks/99-milestones.tji:87`](../../99-milestones.tji)), so no milestone-edge edit beyond the group is needed. That leaf's refinement scopes the Cucumber scenario for the route.

### §2 — Render the raw entity ids + kind discriminants verbatim; localize only chrome

**Chosen:** display each diagnostic's affected node/edge ids and its `kind` discriminant verbatim; localize only the panel chrome (aria-label, group headings, severity/per-kind *labels*, loading/error/empty strings). This mirrors `EventInspector`'s raw-truth stance (`test_mode_event_inspector.md` Decision §2) — a debugging tool wants the actual ids the engine reasoned over. **Rejected — translate the `kind` strings or resolve ids to node wordings:** the kinds are stable data discriminants, and resolving ids to human wordings is cross-event reference resolution the inspector deliberately avoids (it would also need the projection the panel doesn't hold client-side); both add churn and obscure the raw signal. A future "human-readable diagnostics" view is speculative and not registered as work.

### §3 — `position === 0` and clean positions are an explicit "no diagnostics" state

**Chosen:** when the endpoint returns zero entries — which includes `position 0` (the pre-history baseline: empty projection, no diagnostics) and any clean mid-log stop — the panel renders a localized "no diagnostics at this position" readout. **Rejected — special-case `position 0` separately from other clean positions:** they are the same observable (empty result set); a single empty-state path is simpler and correct, and the baseline needs no distinct copy beyond what the empty state already says. **Rejected — render nothing at clean positions:** an empty panel reads as a bug or a stuck load; the panel must affirmatively say "clean here."

### §4 — Mount as a sibling section inside the scrubber's existing `<main>`, not by hoisting the layout

**Chosen:** add the `<section data-testid="test-mode-diagnostics">` directly inside `TimelineScrubber`'s `<main>`, peer to the graph / controls / snapshots / inspector / changes sections, passing the `sessionId` + `position` already in scope — a two-line change (import + section), preserving one layout owner, exactly as `test_mode_event_inspector` (Decision §4 there) and `test_mode_changed_highlights` did. **Rejected — hoist the panel layout up into `SessionScrubberContainer`:** refactors shipped, tested scrubber UI (moving `<main>`, re-homing `data-testid`s, updating tests) for tidiness not needed at this attachment. With this third sibling, the single-`<main>` layout is getting busy; if the fourth (`test_mode_export_position`) makes it genuinely strain, hoisting the layout to the container behind the same props is a contained later refactor — surfaced here as a note, **not** encoded as work now (predecessor discipline: build for today's call sites).

### §5 — A position-keyed fetch hook with a stale-response guard, reusing the shell wire shapes

**Chosen:** a small `useDiagnosticsAtPosition(sessionId, position)` hook colocated in `apps/test-mode/src/diagnostics/`, returning a `{ status, diagnostics, retry }` discriminant in the shape of `useSessionEventLog`, refetching on `(sessionId, position)` change with an ignore-stale-result guard; the response element type reuses the `Wire*Diagnostic` shapes already exported from `@a-conversa/shell`. **Rejected — promote the hook into `@a-conversa/shell` now:** there is exactly one consumer today (this panel); shell hosts cross-surface helpers, and a single-consumer hook belongs in the app until a second surface needs it. If the replay viewer (`replay_test.replay_ui`) later grows a diagnostics readout, promoting the hook to shell is a contained move — noted, not registered. **Rejected — debounce the fetch:** test mode is low-traffic and the stale-response guard already guarantees correctness under rapid scrubbing; a debounce adds latency and state for no v1 benefit. **Rejected — redeclare the diagnostic response types locally:** the shell `Wire*Diagnostic` interfaces are the maintained client mirror of the server entry shapes; redeclaring would fork them.

### §6 — Extend the existing scrubber Playwright spec; no new e2e project

**Chosen:** add diagnostic-panel assertions to [`tests/e2e/test-mode-scrubber.spec.ts`](../../../tests/e2e/test-mode-scrubber.spec.ts), reusing its `walkthrough`-synthetic-session fixture and navigation. The surface is reachable and the panel is a visible section, so the UI-stream e2e policy puts e2e in scope; sharing the scrubber's exact surface, fixture, and auth bootstrap means a new `chromium-test-mode-*` project would only re-pay setup cost. **Rejected — a dedicated `test-mode-diagnostics.spec.ts` + new Playwright project:** redundant backend round-trip and storage-state bootstrap for assertions that belong on the same page-load as the scrubber's. **Rejected — defer the e2e to a catch-all `mod_pw_*`-style task:** the deferral exception applies only when no surface renders the component; here the panel renders on a reachable route, so deferral is unavailable and none is taken (no debt forwarded). The e2e does, however, depend on the endpoint leaf shipping first (Acceptance §5).
