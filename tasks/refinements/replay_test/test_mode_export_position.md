## What this task is

Add an **export panel** to the test-mode scrubber surface that lets the operator pull the **server-authoritative projected state at the current scrubber position** out of the browser — for a fixture, a bug report, a regression baseline, or a diff against a later run. Where the scrubber *renders* the graph client-side from the in-memory event prefix (it never round-trips per step — [`test_mode_timeline_scrubber.md`](test_mode_timeline_scrubber.md) Decision §2), the **export** is the one place that wants the canonical projection the server would compute, so it calls `get_at_position` rather than re-serializing the client projection (Decision §1). Concretely, a fourth sibling `<section>` mounted in [`TimelineScrubber.tsx`](../../../apps/test-mode/src/scrubber/TimelineScrubber.tsx) beside the existing inspector panels (Decision §4), reading the lifted `position` + `sessionId`, that:

1. **An Export button.** On click, captures the current lifted `position` and fetches `GET /api/sessions/:id/state?position=N` (Decision §2 — on demand, not eagerly per step). A position-keyed fetch hook `useStateAtPosition(sessionId)` (app-local, mirroring `useDiagnosticsAtPosition`'s `fetch` + `useState` + per-effect `cancelled` stale guard) drives the four-state machine `'idle' | 'loading' | 'ready' | 'error'`.
2. **A read-only JSON readout** (`<pre data-testid="test-mode-export-readout">`) of the fetched `{ sessionId, sequence, projection }` envelope, pretty-printed via `JSON.stringify(state, null, 2)`, carrying `data-position={state.sequence}`. This is the on-screen, selectable, trivially-assertable artifact (Decision §3).
3. **A file-download affordance** that serializes the same envelope to a `Blob` and triggers a browser download named `session-<sessionId>-position-<sequence>.json` via the standard object-URL + anchor-`download` idiom (Decision §3).
4. **Loading / error+retry states**, and a **position-change reset**: when the lifted `position` prop changes after a successful export, the panel clears back to `idle` so the readout never shows a projection that disagrees with the live scrubber position (Decision §2). The clean state is "click Export to fetch the state at this position."

The panel is a pure **reader** — it consumes the lifted `position` + `sessionId`, never `setPosition`; navigation stays single-sourced in the container, exactly like the three shipped inspector siblings.

## Why it needs to be done

It is the last leaf of the `test_mode` group and the payoff of the whole replay surface: the scrubber lets the operator *find* an interesting position; export lets them *take the state away*. Test mode exists for design iteration, debugging, and demoing a recorded session without three live participants; "export the projected state at position N" is how an operator turns a replay stop into a durable artifact — a fixture to seed a regression, an attachment for a bug report, an input to a hand diff. The client-rendered scrubber graph is a *view*; it is not a serializable, server-canonical projection. `get_at_position` was built precisely so this surface can read the projection the server would compute (the same wire shape as a WS snapshot), and it was shipped with this task named as its only consumer. With both the scrubber (the position seam, the mount point) and the endpoint (the canonical state) live, the panel is a thin, fully-pinnable composition — and closing it completes the `replay_test.test_mode` group that M8 (`m_replay_mvp`) depends on.

## Constraints / requirements

1. **Pure frontend consumer; no backend change.** The endpoint, its validation, its Cucumber pin, and the wire shape already exist. This task adds an app-local hook + panel + i18n + tests only. No route, schema, or server edit.
2. **Read the server-authoritative projection; do not re-serialize the client projection.** The export fetches `GET /api/sessions/:id/state?position=N` and exports its `{ sessionId, sequence, projection }` verbatim. It must **not** reach into `@a-conversa/graph-view`'s client `projectGraph` output or hand-roll a projection→JSON serializer — the canonical state is the endpoint's, and re-deriving it client-side would let the exported artifact drift from the server's truth (Decision §1).
3. **Fetch on demand, not per step.** The hook fires only on an explicit Export click for the captured position — never eagerly on every `position` change. The scrubber deliberately avoids per-step server round-trips ([`test_mode_timeline_scrubber.md`](test_mode_timeline_scrubber.md) Decision §2); eagerly fetching the (potentially large) full projection on every drag would reintroduce exactly that cost (Decision §2).
4. **Never show a stale projection.** When the lifted `position` changes after a successful export, the panel resets to `idle` (readout cleared) so the displayed/downloadable state always matches the position it claims. The readout carries `data-position={state.sequence}`; while present it equals the live position.
5. **Reader only.** The panel takes `{ sessionId, position }` and no `setPosition`. Navigation stays single-sourced in the container (the shipped sibling-panel discipline).
6. **Opaque projection.** Type the fetched state as `{ sessionId: string; sequence: number; projection: unknown }`; serialize the envelope with `JSON.stringify(_, null, 2)` and never narrow into `projection`'s fields — mirroring the endpoint's opaque-projection contract.
7. **Graceful fetch states.** A non-OK response or a malformed body degrades to a retryable `error` readout (not a blank panel or a thrown render), reusing the diagnostics hook's `!response.ok → error` + `cancelled` guard shape.
8. **Catalog parity stays green.** `testMode.export.*` lands in all three catalogs + both `.review.json` companions; `pnpm --filter @a-conversa/i18n-catalogs run check` exits zero.
9. **a11y + build + check green.** The new section keeps the scrubber spec's axe pass (ADR 0040) green; `pnpm -F @a-conversa/test-mode build` and `pnpm run check` stay green.

## Decisions

### §1 — Export the server-authoritative projection via `get_at_position`; do not re-serialize the client projection

**Chosen:** the export fetches `GET /api/sessions/:id/state?position=N` and exports the canonical `{ sessionId, sequence, projection }` the server computes. The scrubber renders its graph client-side from the in-memory event prefix to avoid per-step round-trips ([`test_mode_timeline_scrubber.md`](test_mode_timeline_scrubber.md) Decision §2), and `@a-conversa/graph-view`'s `projectGraph` output is a *render* model, not the canonical `Projection`. The endpoint exists for exactly this consumer (its refinement names this task as the reason; the same `serializeProjectionForWire` the WS snapshot uses guarantees one wire shape). **Rejected — serialize the client `projectGraph` result:** the export artifact would be a view-model, not the server's truth, and could drift from a WS snapshot or a later server replay of the same position; this is the precise drift `get_at_position` Decision §3 (single serializer, no second wire shape) was built to prevent. **Rejected — re-implement projection-at-position in the export panel:** a full re-derivation of replay logic in the browser, duplicating the server primitive for no benefit — the endpoint already does it, validated and Cucumber-pinned. *This is a pre-settled architectural split (scrubber renders client-side; export reads server-side), so no new ADR — the question is closed by the scrubber's Decision §2 and the `get_at_position` refinement.*

### §2 — Fetch on demand on an explicit Export click; reset on position change

**Chosen:** the hook fires only when the operator clicks Export, capturing the then-current position; moving the scrubber afterward resets the panel to `idle`. **Rejected — eager fetch on every `position` change (the `useDiagnosticsAtPosition` pattern):** diagnostics eager-fetch a *small* entry list that the panel always displays; the full projection is potentially large (every node/edge/annotation/proposal/vote), and eagerly round-tripping it on every drag step reintroduces exactly the per-step server cost the scrubber's Decision §2 designed away. Export is a deliberate, occasional action, so a lazy fetch fits its usage and its payload size. **Rejected — eager fetch but show the readout only behind a toggle:** still pays the per-step network cost for state the operator usually isn't exporting; the toggle is a worse seam than a plain Export button. **Rejected — keep the last successful readout visible after the position moves (with a "stale" badge):** invites exporting a projection that disagrees with the position label; resetting to `idle` keeps the readout's `data-position` always equal to the live position — a cleaner, un-foot-gunnable invariant.

### §3 — A read-only JSON readout is the primary artifact; a file download is the export mechanism; clipboard is out of scope

**Chosen:** render the fetched envelope into a selectable `<pre data-testid="test-mode-export-readout">` (the always-present, trivially-assertable surface) and offer a file download (`session-<sessionId>-position-<sequence>.json`) via the standard `Blob` → object-URL → anchor-`download` → revoke idiom — the canonical "get the data out" action. **Rejected — download only, no readout:** browser downloads are awkward to assert and offer the operator no on-screen confirmation of *what* they exported; the readout makes the Vitest/Playwright pins direct (assert the JSON text) and gives immediate feedback. **Rejected — readout only, no download** ("select and copy the `<pre>`"): "export" connotes producing a portable file, and a deep projection is unwieldy to hand-select; the download is the load-bearing affordance, the readout its preview. **Rejected — add copy-to-clipboard now** (precedent at [`InviteParticipants.tsx:370`](../../../apps/moderator/src/routes/InviteParticipants.tsx)): a trivial future nicety, not the export's purpose, and an extra affordance + i18n + permission surface for v1; left explicitly out of scope and **not** registered as a follow-up task (it is a cheap enhancement, not blocking work — registering an "audit"-shaped or trivial-nicety leaf is the over-deferral the WBS guards against).

### §4 — Mount as a fourth reader-only sibling panel; keep the hook app-local

**Chosen:** the export panel mounts as the trailing sibling in `TimelineScrubber`'s `<main>`, taking `{ sessionId, position }`, reading the lifted position the container already owns — identical to the three shipped inspector panels — and its fetch hook lives app-local in `apps/test-mode/src/export/` (not promoted to `@a-conversa/shell`). **Rejected — a dedicated route / standalone export view:** fragments the operator's flow (scrub here, navigate elsewhere to export); the whole value is exporting *the position you're looking at*, which the sibling-panel seam delivers for free. **Rejected — promote `useStateAtPosition` to `@a-conversa/shell`:** single-consumer today (only test mode exports), so it stays in the app per ADR 0010 and the `useDiagnosticsAtPosition` Decision §5 precedent; if the replay viewer later grows its own export it is a contained promotion behind the same hook shape. **Rejected — own a second `position` in the panel:** would fork navigation; the panel reads the one lifted position, like its three siblings.

### §5 — Extend the existing scrubber spec at its existing stops; no new project or spec file

**Chosen:** add the export round-trip assertions to [`tests/e2e/test-mode-scrubber.spec.ts`](../../../tests/e2e/test-mode-scrubber.spec.ts), hooking onto its existing head and `range.fill('0')` stops, reusing its one `walkthrough` generation, its `chromium-test-mode-scrubber` project, and its `setup-auth` storage state. This continues the precedent set by every test-mode panel (`test_mode_timeline_scrubber` Decision §6, `test_mode_diagnostic_inspector_e2e_tracking` Decision §3): one spec, one fixture, no per-panel Playwright project. **Rejected — a dedicated `test-mode-export.spec.ts` + new project:** a redundant `walkthrough` generation and auth bootstrap for assertions that belong on the same page-load and the same two position stops the scrubber spec already visits. **Rejected — assert against a fresh empty session only:** position 0 over the empty baseline proves the clean path, but the head stop on the deep `walkthrough` log is what proves the export carries a real, non-trivial projection — both stops are needed.
