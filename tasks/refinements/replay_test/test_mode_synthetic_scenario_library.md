## What this task is

Pay down the reuse debt ADR 0041 deferred: make the repo's **rich `walkthrough` test-fixture** (~800 events of real debate structure — nodes, edges, annotations, proposals, votes, commits, a snapshot) available as a **synthetic scenario**, so an operator can mint a non-trivial session for design iteration with one click instead of hand-authoring it or running a live three-party debate. Concretely:

1. **A non-destructive, exhaustively-typed id re-keyer** in `apps/server/src/test-mode/synthetic/` — a pure function that takes a parsed fixture log plus a target `(sessionId, hostUserId, idFactory)` and returns `Event[]` with **every** id remapped: the fixture's session id → the fresh session id; the fixture's three role users → the operator (moderator) + the two stable synthetic debaters; every node/edge/annotation/proposal/snapshot/event-surrogate id → a fresh `idFactory()` value — **with all cross-references preserved** (an edge still points at its node, a vote still points at its proposal). Sequences are re-allocated contiguous from `1`. This is the "typed re-keyer" the task name and ADR 0041 call for.

2. **A build-time codegen step that vendors the canonical fixture into the server** as a committed typed module (new **ADR 0042**). The canonical `walkthrough` fixture lives in the `@a-conversa/test-fixtures` **devDependency**, which is pruned from the runtime image — so the data must be a compiled TS module under `apps/server/dist` to be reachable at runtime (including in the e2e/compose environment). A codegen script reads `packages/test-fixtures/src/fixtures/walkthrough/*.json` and writes a committed `*.data.ts`; a **drift-guard Vitest test** fails if the committed copy ever diverges from the canonical fixture.

3. **One new `walkthrough` scenario** registered in `SCENARIO_ENTRIES` ([`scenarios.ts:324`](../../../apps/server/src/test-mode/synthetic/scenarios.ts)): its `build` fn applies the re-keyer to the vendored data; its descriptor `{ key: 'walkthrough', title, description }` joins the list the read endpoint advertises and the gallery renders. The builder stays **synchronous** — the vendored data is an imported `const`, not an async load — so the ADR-0041 `ScenarioBuilder` signature and the route are untouched.

4. **i18n keys** `testMode.synthetic.scenario.walkthrough.{title,description}` in all three catalogs ([`packages/i18n-catalogs/src/catalogs/`](../../../packages/i18n-catalogs/src/catalogs/)), matching the existing `empty`/`structured` shape.

5. **Test layers** (per ADR 0022): Vitest re-keyer tests (validity + freshness + cross-reference integrity + user-mapping + determinism), the Vitest drift-guard, a Cucumber scenario pinning the `walkthrough` wire seam (descriptor advertised + generate persists a large owned log), and a Playwright case extending the existing synthetic-session spec to generate the walkthrough and assert a deep readout.

## Why it needs to be done

`test_mode_synthetic_session` shipped the generator but only two **small, hand-authored** scenarios — deliberately, because instantiating the rich `walkthrough` fixture non-destructively requires re-keying ids across the discriminated payload union, "meaningful, risky work the destructive loader sidesteps precisely because re-keying is hard" ([`test_mode_synthetic_session.md`](test_mode_synthetic_session.md) Decision §4). That decision registered **this task** to do exactly that work.

The downstream `test_mode_*` stream — `test_mode_timeline_scrubber` ([`tji:95`](../../60-replay-and-test-mode.tji)), `test_mode_event_inspector` ([`tji:101`](../../60-replay-and-test-mode.tji)), and the diagnostics/graph views — all drive off a loaded log and need **realistic, varied** sessions to be worth iterating against. The `structured` scenario is eleven events; the scrubber, the per-event inspector, the graph layout, and the structural diagnostics only get a real workout against a session with the `walkthrough`'s depth (decomposition, edges, annotations, meta-moves, a snapshot). This task supplies that fodder while reusing the *one canonical* rich fixture the project already maintains for projection tests, rather than minting a second divergent copy. The re-keyer is written generically ("walkthrough *and future* test-fixtures") so any later fixture becomes a scenario by registration alone.

Because generation still flows through the production write path (ADR 0041, unchanged here), the walkthrough session is a **real persisted session** read via the same `GET /api/sessions/:id/events` path as every other — one data source downstream, no second code path.

## Constraints / requirements

1. **Non-destructive and re-runnable.** The re-keyer never truncates and never replays fixed ids. Each generation mints a fresh session id and fresh entity ids; repeated calls produce disjoint sessions. No `loadFixture`/`TRUNCATE` at runtime, ever. (ADR 0041, ADR 0042.)
2. **Every id is remapped; cross-references are preserved.** A single string-keyed remap is **seeded** with the fixture session id → the fresh session id and the three fixture role users → `{ hostUserId, SYNTHETIC_DEBATER_A.id, SYNTHETIC_DEBATER_B.id }`; every **other** id string mints a fresh `idFactory()` value **on first sight and is cached**, so the same canonical id always maps to the same fresh id (preserving node↔edge, proposal↔vote, and every other reference). `actor === null` passes through. The output contains **no canonical fixture id** in any id-bearing field.
3. **User fields resolve only to the three seeded users.** Every actor/user-bearing field in the output (`actor`, `host_user_id`, `user_id`, `created_by`, `included_by`, `committed_by`, `marked_by`, `changed_by`, `withdrawn_by`, `removed_by`, `participant`) must be one of `{ hostUserId, SYNTHETIC_DEBATER_A.id, SYNTHETIC_DEBATER_B.id }`. The re-keyer **throws** if it encounters a user id not covered by the seed (a fixture with a fourth participant) rather than silently minting a dangling non-user id — fail loud, not corrupt.
4. **Exhaustively typed over the payload union.** The remapping walk is a typed `switch (event.kind)` exhaustive over `eventKinds` with a `never` default, so a future event kind (or a new id-bearing field) is a compile error, not a silent pass-through of an un-remapped id. This is the "typed" in "typed re-keyer."
5. **Real events through the production write path.** The builder returns `Event[]` from `@a-conversa/shared-types`; every event passes `validateEvent` and is appended via the unchanged route. Sequences are re-allocated contiguous ascending from `1` over the ordered event list (robust to gaps in the source). (ADR 0021, ADR 0020.)
6. **Builder stays synchronous; route + signature unchanged.** The vendored fixture data is an imported `const`, so the `walkthrough` build fn matches the existing `ScenarioBuilder` signature. No change to `ScenarioBuilder`, `routes.ts`, `withTransaction`, the gallery component, or `useSessionEventLog`.
7. **Fixture data reaches runtime as a compiled TS module (ADR 0042).** Vendor the canonical fixture via a build-time codegen into a committed `*.data.ts` under `apps/server/src/test-mode/synthetic/`, guarded by a drift-guard test against the canonical source. **No** runtime import of `@a-conversa/test-fixtures`, **no** Dockerfile change, **no** promotion of the package to a runtime dependency.
8. **`@a-conversa/test-fixtures` stays a devDependency.** Only the codegen script and the drift-guard test (build/test time) may import it; server runtime code must not.
9. **Catalog parity stays green.** `testMode.synthetic.scenario.walkthrough.*` lands in all three catalogs; `pnpm --filter @a-conversa/i18n-catalogs run check` exits zero. (Follow the predecessor's precedent on `.review.json` companions for UI-chrome keys — not required for these descriptor keys.)

## Decisions

### §1 — A uniform string-keyed remap with a seeded user/session map, minting fresh ids on first sight

**Chosen:** the re-keyer holds one `Map<string,string>` seeded with the fixture session id → fresh session id and the three fixture role users → `{ hostUserId, SYNTHETIC_DEBATER_A.id, SYNTHETIC_DEBATER_B.id }`; every other id string is minted fresh via `idFactory()` **on first sight and cached**. Because cross-references in the fixture are just repeated id *strings* (an edge's `source_node_id` is the same string as the referenced `node-created`'s `node_id`), a single string-keyed map preserves **all** references automatically — the re-keyer never needs to model reference *semantics*, only to identify which *fields* are ids. **Rejected — a per-id-kind remapper (separate node-map, edge-map, proposal-map):** more code, more ways to miss a cross-kind reference, and no benefit since the id strings are globally unique already. **Rejected — pre-scan to mint all ids then a second pass to substitute:** the lazy mint-on-first-sight is one pass and equally correct. The seeded entries are the only ids that must map to *predetermined* targets (the real users + the route's fresh session); everything else is genuinely fresh.

### §2 — Exhaustively-typed payload walk, failing loud on surprises

**Chosen:** the field-finding walk is a typed `switch (event.kind)` exhaustive over the `Event` union with a `never` default, and the user-remap **throws** on an unseeded user id. The fixture is real data the re-keyer must reproduce *faithfully*; a silent pass-through of an un-remapped id (new kind, new field, or an unexpected fourth participant) would persist a dangling reference or leak a canonical id into a live session — a corruption that's hard to spot downstream. Compile-time exhaustiveness + a loud runtime throw turn both classes of surprise into a red build or a clear error. **Rejected — a generic deep-walk that remaps any field matching `/_id$/` or a UUID-shaped string:** it would catch fields by accident (or miss `components[].node_id` nesting), couples correctness to naming/format coincidence, and can't be type-checked against the union. The explicit typed walk is the "typed re-keyer" the task asks for.

### §3 — Fixture data is vendored into the server as a build-time-generated committed TS module (ADR 0042)

**Chosen:** a codegen script reads the canonical fixture and writes a committed typed `*.data.ts` under `apps/server/src/test-mode/synthetic/`, guarded by a drift-guard test; `@a-conversa/test-fixtures` stays a devDependency. This is forced by the deployment reality: the runtime image prunes devDependencies (`Dockerfile:140`) and copies only specific `dist` trees (`Dockerfile:144–176`) — `packages/test-fixtures` is absent — and the **e2e/compose** environment runs that same image (with `NODE_ENV=development` so the gated routes register), so a runtime `import('@a-conversa/test-fixtures')` would throw `MODULE_NOT_FOUND` exactly where the feature is tested. The only artifact reliably present at runtime is compiled TS under `apps/server/dist`. **Rejected — promote `@a-conversa/test-fixtures` to a runtime dependency:** the Dockerfile would still have to copy the package's files (only enumerated `dist` trees are copied), it pulls the destructive `loadFixture` into the production runtime graph, and it ships a test-support package to prod — more coupling than vendoring a data snapshot. **Rejected — dynamic `import()` behind the `NODE_ENV` gate:** fails in the pruned compose/e2e image, making the scenario unavailable where it must work. **Rejected — copy raw JSON into `dist` + read via `fs`:** `tsc` copies no non-`.ts` assets, so it needs a bespoke copy step + a Dockerfile edit and introduces a dev-source vs `dist` path duality. A compiled TS module sidesteps all of it and keeps the builder synchronous. Full rationale + the drift-guard contract are in **ADR 0042**.

### §4 — E2e is in scope; reuse the existing synthetic-session spec + project

**Chosen:** add a Playwright **case** to the existing [`test-mode-synthetic-session.spec.ts`](../../../tests/e2e/test-mode-synthetic-session.spec.ts) in the existing `chromium-test-mode-synthetic-session` project, driving gallery → generate the `walkthrough` affordance → land on `/sessions/:id` with a **deep** readout. Per the UI-stream e2e policy the surface *is already reachable* — `test_mode_synthetic_session` shipped the gallery as the test-mode root, and a new descriptor renders an affordance with no component change — so deferral does not apply. **Rejected — a brand-new spec + a new Playwright project:** a new `chromium-*` project per scenario multiplies CI matrix cost for one more affordance that shares the same flow; extending the spec covers the new behavior at the existing project's cost. **Rejected — defer the e2e to a future `mod_pw_*`-style task:** the component is reachable and the predecessor already pays this exact e2e down; deferring would be inventing debt where none is needed. The re-keyer's correctness (freshness, cross-refs, user-mapping, no-leak) is pinned deterministically at the Vitest + Cucumber layers; the e2e proves real-backend wiring and the deep readout.

### §5 — One generic re-keyer, registered once for `walkthrough`, ready for future fixtures

**Chosen:** ship the re-keyer as a fixture-agnostic transform `(fixtureData, { sessionId, hostUserId, idFactory }) -> Event[]` and register a single `walkthrough` scenario that feeds it the vendored walkthrough data. The task name says "walkthrough (and future) test-fixtures"; writing the transform generically means a later fixture becomes a scenario by vendoring its data + adding one `SCENARIO_ENTRIES` entry — no re-keyer change. **Rejected — hand-author a bespoke walkthrough builder (like `structured`, but ~800 events):** defeats the entire point (reuse the one canonical rich fixture, don't fork it), is unreviewable at that size, and immediately drifts from the projection-test fixture it duplicates. **Rejected — build the generic multi-fixture *gallery* now (vendor `empty` + every fixture through the re-keyer):** premature; only `walkthrough` needs richer-than-`structured` reuse today, and the `empty`/`structured` hand-built scenarios already exist. One real call site, generic seam, future registration is mechanical.
