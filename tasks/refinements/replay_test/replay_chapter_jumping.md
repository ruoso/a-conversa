## What this task is

Add **chapter (snapshot) navigation** to the audience replay viewer, on top of the now-shipped seek bar. Three affordances, all driven by the session's snapshot set and all expressed in the existing event-sequence `position` vocabulary:

1. **Prev / next chapter buttons** in the controls cluster ‚Äî step to the nearest snapshot marker before / after the current `position`, disabled when there is no further chapter in that direction. This is the literal task: *"Jump to next/prev snapshot via chapter markers."*
2. **Chapter tick marks on the seek bar** ‚Äî a `<datalist>` of snapshot positions bound to the existing range input, paying down the decoration `replay_seek_bar` deferred (its Decision ¬ß3).
3. **A clickable chapter index** ‚Äî the snapshot list rendered as chapters; clicking a chapter jumps `position` to that snapshot's `logPosition`. This reuses the shipped `SnapshotList` + `resolveSnapshotPosition` and discharges the inherited `snapshot_jump_ui` list-click e2e debt.

All three share **one** snapshot source (a single lifted `useSessionSnapshots(sessionId)` read) and all funnel their writes through the existing `updatePosition`/`clampPosition` guard, so a chapter jump is just another writer of the lifted `position` ‚Äî identical to a step, a seek, or a play-loop tick. The chapter-navigation arithmetic (`snapshotPositions` / `nextSnapshotPosition` / `prevSnapshotPosition`) is **ported into `@a-conversa/shell`** from its server twin, so the container imports it rather than re-deriving it (ADR 0043).

## Why it needs to be done

It is the last leaf of `replay_ui`. The viewer can already play, pause, step ¬±1 event, seek to any position, speed up playback, and deep-link an opening position ‚Äî but it has no way to navigate by the **meaningful** stops a moderator marked. Snapshots are the chapters of a session; a replay viewer of any length is unusable without chapter navigation (the seek bar alone forces hunting frame-by-frame for the moments that matter).

It also closes two outstanding debts the WBS deliberately parked on this leaf:

- `replay_seek_bar` shipped a plain bar and deferred chapter tick marks here (its Constraint ¬ß5 / Decision ¬ß3).
- `snapshot_jump_ui` shipped the reusable jump wiring before any surface could reach it and forwarded its list-click Playwright coverage here (its ¬ß4).

Building it now turns three shipped-but-unreachable pieces (the seek bar's tick seam, the snapshot list, the jump resolver) into one reachable, e2e-covered feature.

## Constraints / requirements

1. **One snapshot source, lifted once.** The container performs a single `useSessionSnapshots(sessionId)` read; the resulting `SnapshotRecord[]` feeds all three affordances (prev/next buttons, seek-bar ticks, chapter list). Do **not** mount `SnapshotJumpList` (which fetches internally) *and* call the hook again ‚Äî that double-fetches (Decision ¬ß1).
2. **Chapter arithmetic comes from `@a-conversa/shell`, never re-derived in the component.** Port `snapshotPositions` / `nextSnapshotPosition` / `prevSnapshotPosition` into `@a-conversa/shell` (mirroring the server twin) and import them; the container must not inline the marker math, exactly as it imports `nextPosition`/`clampPosition` today (ADR 0043, Decision ¬ß2).
3. **Every chapter write funnels through `updatePosition`.** A prev/next jump, a chapter-row click, all call `updatePosition(targetPosition)` ‚Üí `clampPosition` ‚Üí `setPosition` ‚Äî the single guarded setter the step buttons, seek bar, and play loop already share. No second position writer, no bypass of the clamp.
4. **Positions only; resolve at the boundary.** Chapter affordances speak `position` (event-sequence space). The `snapshotId ‚Üí logPosition` resolution for a row click happens at the click boundary via the shipped `resolveSnapshotPosition`; `snapshotId` never leaks into `position` state.
5. **`null`-at-the-ends disables, never saturates.** `nextSnapshotPosition`/`prevSnapshotPosition` return `null` when there is no further chapter; the corresponding button is then `disabled` (and no jump fires). This differs from the event-step buttons, which saturate (Decision ¬ß4).
6. **Graceful empty / loading.** A session with no snapshots renders no ticks, disabled prev/next chapter buttons, and the list's existing no-snapshots affordance; loading and error states pass straight through `SnapshotList` (its shipped `status`/`onRetry`). The viewer never crashes on an empty or still-loading snapshot set.
7. **Localized, accessible affordances (ADR 0024, 0040).** New buttons carry localized `aria-label`s in the established `audience.replay.playback.*` namespace; new strings land in all three catalogs (`en-US`, `pt-BR`, `es-419`) **and** their `.review.json` companions. The seek-bar `<datalist>` is decorative tick marks on the existing labelled range input ‚Äî it adds no new focus trap. Disabled buttons use the same `disabled:` styling as the shipped step buttons. Keep axe-clean.
8. **No new dependency, no new architectural seam beyond the shell port.** Read-only consumer of an already-shipped, already-pinned endpoint; the only new shared code is the three pure helpers ported into the existing shell replay-navigation contract (ADR 0043). No React Query/SWR, no new store, no new route.

## Decisions

**¬ß1 ‚Äî Source the chapter markers from one lifted `useSessionSnapshots(sessionId)` read (REST), feeding all three affordances.** *Rationale:* the shipped resolution helpers and `SnapshotList` all consume the `SnapshotRecord` shape natively, and `useSessionSnapshots` is the shipped, tested, visibility-gated accessor that returns exactly that shape (with `createdAt`) plus ready/loading/error/retry states the list already renders. One lifted read feeds the prev/next buttons, the seek-bar ticks, and the clickable list with no duplication. *Alternative rejected ‚Äî derive markers from the already-loaded event log* (filter `kind === 'snapshot-created'`, map the payload + envelope into records): it saves one small fetch but needs a new `events ‚Üí SnapshotRecord[]` adapter that couples to the envelope's timestamp shape (ADR 0021) and re-implements snake‚Üícamel mapping the endpoint already does; the list endpoint is the canonical accessor and is itself just a filtered read of the same log, so the data cannot meaningfully diverge. The marginal fetch is not worth a bespoke parser. *Alternative also rejected ‚Äî mount the connected `SnapshotJumpList` for the list while calling `useSessionSnapshots` again for the buttons/ticks*: two reads of the same endpoint. Instead we lift the hook once and compose its constituents (`SnapshotList` + `resolveSnapshotPosition`) directly ‚Äî the exact parts `SnapshotJumpList` bundles ‚Äî so the single fetched array drives every affordance.

**¬ß2 ‚Äî Port `snapshotPositions` / `nextSnapshotPosition` / `prevSnapshotPosition` into `@a-conversa/shell`; the container imports them.** *Rationale:* ADR 0043 already established that the client replay-navigation contract lives in `@a-conversa/shell` and is **never** imported from `apps/server` (apps are leaf Vite bundles, not workspace libraries) and **never** re-derived inside a component ‚Äî the container imports `nextPosition`/`clampPosition`/`replayHeadSequence` from shell today. Chapter-level next/prev is the same contract one granularity coarser; it belongs beside the per-event navigation, ported from its server twin exactly as `position_navigation` was. *Alternative rejected ‚Äî inline the marker arithmetic in `ReplayPlaybackContainer`*: violates ADR 0043's "never re-derived in components," forks the semantics from their server definition (`snapshot_resolution`), and leaves the (subtle: strict `>`/`<`, `null` at ends, dedup) logic untested as a unit. **No new ADR** ‚Äî this is a direct, in-bounds extension of ADR 0043, the same way the seek bar and step buttons extended it.

**¬ß3 ‚Äî Scope includes the clickable chapter index (presentational `SnapshotList` wired to jump), not only prev/next + ticks ‚Äî paying down the inherited list-click e2e debt here.** *Rationale:* "the replay viewer's chapter index *is* the snapshot list" ([`snapshot_jump_ui.md`](snapshot_jump_ui.md) ¬ß39), and ¬ß4 of that refinement forwarded its list-render ‚Üí click ‚Üí jump Playwright coverage specifically to this leaf. Rendering the chapters as a clickable list (reusing the shipped `SnapshotList` + `resolveSnapshotPosition`, the exact pieces built for this) makes the affordance real and lets the e2e exercise the forwarded flow. *Alternative rejected ‚Äî ship only prev/next + ticks and forward the list-click debt onward*: a forward-of-a-forward (the refinement-writer brief discourages it), it would leave the list-click flow untested on the replay surface and lean entirely on `test_mode_timeline_scrubber` to cover a debt explicitly assigned to *both* inheritors. Paying it down here is the planning-debt-conscious call.

**¬ß4 ‚Äî Chapter prev/next return `null` at the ends and *disable* the button; they do not saturate like the event-step buttons.** *Rationale:* this mirrors the server helpers' contract (`null` when no further marker) and is the right chapter-UI affordance ‚Äî a disabled "next chapter" at the last chapter tells the viewer there are no more, whereas saturating would re-fire a jump to the chapter they are already on. The event-step buttons saturate (a step is always meaningful up to the boundary), but chapter stops are sparse, so absence is a first-class state. The buttons reuse the shipped `disabled:` styling. *Alternative rejected ‚Äî saturate at the nearest end marker* (clamp `null` to the first/last position): produces a button that looks enabled but jumps nowhere, a worse affordance than an honest `disabled`.

**¬ß5 ‚Äî Seek-bar ticks via a native `<datalist>` bound to the existing range input.** *Rationale:* HTML's `<input type="range" list="‚Ä¶">` + `<datalist>` is the standard, zero-JS mechanism for range tick marks; it decorates the *already-shipped* seek input (lines 202‚Äì212) without restructuring it, needs no absolute positioning math, and stays responsive and accessible for free. *Alternative rejected ‚Äî a custom absolutely-positioned tick overlay*: more code, fragile against the bar's responsive width, and an a11y surface to get right, for a worse result than the platform primitive. (`replay_seek_bar` Decision ¬ß3 already anticipated "`<datalist>` / marker decoration" as this task's shape.)

**¬ß6 ‚Äî e2e is **in scope**, not deferred; seed the `walkthrough` synthetic scenario.** *Rationale:* the UI-stream e2e policy defers only when the component is unreachable. The replay viewer is reachable at `/a/{locale}/replay/:id` and `ReplayPlaybackContainer` is mounted and shipped; this task adds affordances to the already-rendered controls cluster, so a full Playwright spec exercises real user-visible behavior. The only seeding wrinkle is that the existing helper requests `structured` (no snapshot); the chapter spec must request `walkthrough` (one snapshot marker), so the helper is parameterized by scenario. *Alternative rejected ‚Äî defer to a future `*_pw_*` task*: there is no "not yet reachable" condition to invoke, and deferring would re-park the very debt this task exists to discharge.

**¬ß7 ‚Äî No new ADR.** *Rationale:* nothing here introduces a dependency, an architectural seam, or a security trade-off. The shell-port of the snapshot navigation helpers is an in-bounds application of ADR 0043 (¬ß2); React (0003), Vitest (0006), Playwright (0008), i18n (0024), graph-view prefix render (0039), head-landing default (0045), and the snapshot-as-event / REST contracts are all already decided and shipped. The scope and affordance calls above belong in this refinement, not a new ADR.
