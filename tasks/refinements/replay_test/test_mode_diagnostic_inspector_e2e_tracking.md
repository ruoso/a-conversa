## What this task is

Land the **deferred end-to-end assertion** for the diagnostic inspector: extend the existing scrubber Playwright spec so it proves, against the **live** `get_diagnostics_at_position` endpoint, that the panel **tracks the scrubber position** and **renders the flagged diagnostic entry** at a position the walkthrough fixture is known to flag — replacing the predecessor's interim section-presence-only check. Concretely, in [`tests/e2e/test-mode-scrubber.spec.ts`](../../../tests/e2e/test-mode-scrubber.spec.ts) (project `chromium-test-mode-scrubber`, on the already-generated `walkthrough` synthetic session):

1. **At the head position** (where the spec lands after navigation; `data-head` on `test-mode-scrubber-status`), the diagnostics panel **settles to a non-empty, non-error state**: the advisory severity group (`test-mode-diagnostics-advisory`) renders, and it contains the walkthrough's known structural finding — the `non-self-referential-annotation-contradicts` **coherency-hint** that E15 (a `contradicts` edge from a node to an annotation whose anchor is a *different* node) produces once it is included in the projection (Decision §1). The flagged entry renders under its severity group as a `test-mode-diagnostics-entry` carrying `data-kind="coherency-hint"`, **not** the unknown-kind fallback row.
2. **At position 0** (the existing `range.fill('0')` baseline stop), the panel **settles to the clean empty state** (`test-mode-diagnostics-empty`) — the empty-baseline projection yields `{ diagnostics: [] }` and the panel affirmatively says "no diagnostics," not an error or a blank. The head→0 transition (non-empty → empty), driven purely by moving the scrubber position, is what demonstrates *tracking*.
3. **The panel never lands in its error readout** now that the route is live: `test-mode-diagnostics-error` has count 0 at both stops, tightening the predecessor's "degrades gracefully to error" interim behaviour into a "fetches the real endpoint and renders the real result" guarantee.

The assertions piggyback on the spec's existing head and position-0 stops and its existing `walkthrough` fixture + auth bootstrap — no new Playwright project, no new spec file, no new synthetic generation (Decision §3).

## Why it needs to be done

The predecessor's interim e2e only proved the panel *mounts* and its fetch *settles* — it could not prove the panel shows the *right* diagnostics at the *right* position, because the endpoint it reads had not shipped, so the panel degraded to its error readout end-to-end. With `get_diagnostics_at_position` now live, the whole point of the panel — "what would the methodology engine flag at this position?" — is finally observable through the real stack: real surface → `useDiagnosticsAtPosition` → live route → `projectAtPosition` + `computeAllDiagnostics` → wire-shaped entries → severity-grouped render. Pinning that round-trip with a committed Playwright assertion (ADR 0022) is the e2e the UI-stream policy requires for a reachable, rendered component; leaving it as section-presence-only would let a regression anywhere along that chain (endpoint route, projection-at-position, severity classification, or the panel's grouping) ship green. This task closes the one piece of deferred e2e debt the diagnostic-inspector leaf carried, with no debt forwarded.

## Decisions

### §1 — Anchor the flagged-entry assertion on E15's advisory coherency-hint at head; anchor "clean" on position 0

**Chosen:** assert the non-empty diagnostic at the **head** position, expecting the walkthrough's deterministic `non-self-referential-annotation-contradicts` **coherency-hint** (advisory) that E15 produces once included (log position 265 < head), and assert the **empty** state at **position 0** (the empty-baseline projection, guaranteed `{ diagnostics: [] }`). E15 — a node→annotation `contradicts` edge with a cross-anchor annotation — is the one structural finding the fixture is built to carry end-to-end ([`meta.json`](../../../packages/test-fixtures/src/fixtures/walkthrough/meta.json)); the contradiction detector deliberately skips annotation endpoints ([`contradiction-detection.ts:108-119`](../../../apps/server/src/diagnostics/contradiction-detection.ts)) and the coherency-hint detector raises the advisory instead (generalized shape unit-pinned at `coherency-hint-detection.test.ts:575`). Both anchors are sequence-stable across rekeying. **Rejected — assert "clean" on an arbitrary early/mid-log position:** intermediate positions are *not* guaranteed clean — a warrant whose two bridge edges are created in separate events transiently flags an incomplete-warrant coherency-hint between those events, so only position 0 is a provably-empty anchor. **Rejected — assert a *blocking* (cycle/contradiction) entry:** the walkthrough produces none deterministically; the contradicts edge it does carry is, by methodology design, an *advisory* coherency-hint, not a blocking contradiction. Asserting on a blocking group would pin a finding the fixture does not produce. **Rejected — author a new diagnostic-rich fixture (a clean cycle/contradiction):** new synthetic-scenario authoring is a much larger task than tightening one assertion against the deep walkthrough session that already flags; the advisory hint is a perfectly good non-empty anchor.

### §2 — Assert structurally (severity group + `data-kind` discriminant), never on a hardcoded entity id

**Chosen:** select the flagged entry via `test-mode-diagnostics-advisory` + a `test-mode-diagnostics-entry` carrying `data-kind="coherency-hint"`, and assert its rendered ids text is *non-empty* (the panel renders the affected ids verbatim) rather than equal to a specific UUID. The synthetic generator **rekeys** the fixture's ids per generation ([`scenarios.ts:328`](../../../apps/server/src/test-mode/synthetic/scenarios.ts) → `rekeyFixture(... idFactory)`), so the literal fixture UUIDs are not the runtime ids. Asserting on the kind discriminant and severity group is stable; asserting on `10000010-…-019` would fail against the rekeyed session. **Rejected — fetch the endpoint directly in the test and read back the generated id to assert verbatim:** couples the e2e to the wire shape and reimplements the panel's job; the structural assertion already proves the right entry rendered under the right group. **Rejected — assert exact id by cross-referencing the inspector payload** (the way the spec's `node-created` loop extracts `node_id`): unnecessary coupling for a coherency-hint whose ids live in a nested `hint` object — the kind + non-empty-ids check is sufficient and robust.

### §3 — Extend the existing scrubber spec at its existing stops; no new project or spec file

**Chosen:** rewrite the interim diagnostics block (lines 86–104) in place and hook the new assertions onto the spec's existing **head** (post-navigation) and **position-0** (`range.fill('0')`) stops, reusing its one `walkthrough` generation, its `chromium-test-mode-scrubber` project, and its `setup-auth` storage state. This continues the predecessor's Decision §6 (one spec, shared fixture) and matches the `.tji` note's explicit instruction to "extend `tests/e2e/test-mode-scrubber.spec.ts`." **Rejected — a dedicated `test-mode-diagnostics.spec.ts` + new Playwright project:** a redundant `walkthrough` generation and storage-state bootstrap for assertions that belong on the same page-load and the same two position stops the scrubber spec already visits; the e2e policy's catch-all-debt warning applies in reverse here — paying the debt down inline on the existing spec is the cheap, correct move. **Rejected — split head and position-0 assertions across two specs:** they are the same fixture and the same panel; one assertion block keeps the tracking transition (non-empty → empty) legible in one place.
