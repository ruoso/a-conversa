# Subscribe a client to a session's event stream

**TaskJuggler entry**: [tasks/20-backend.tji](../../20-backend.tji) — task `backend.websocket_protocol.ws_subscribe_to_session`
**Effort estimate**: 1d
**Inherited dependencies**: `backend.websocket_protocol.ws_connection_handling` (settled — the upgrade path + the per-connection context + the `__buildTestWsApp` helper), `backend.websocket_protocol.ws_message_envelope` (settled — the closed `WsMessageType` enum + `WsDispatcher.register` + `serializeWsEnvelope`), `backend.websocket_protocol.ws_auth_on_connect` (settled — `WsConnectionContext.user` is populated before the dispatcher ever sees a message), `backend.cross_session_permissions.privacy_field_enforcement` (settled — `canSeeSession(executor, sessionId, userId)`).

## Goal

Let an authenticated WebSocket client tell the server "I want to receive event broadcasts for session X" (`subscribe`) and "stop sending them" (`unsubscribe`). The server gates the request with the same `canSeeSession` predicate the HTTP routes use, records the (connection, session) tuple in a per-server-instance registry, and acks the request with `subscribed` / `unsubscribed`. The registry's `connectionsForSession(...)` snapshot is what the future `ws_event_broadcast` task will iterate to fan out frames. Subscriptions are wiped on connection close so a dropped socket can't leak entries.

## Context

The WS surface up to now is one-way (server → client `hello` on connect; client → server `dispatch` for messages whose handlers are themselves downstream). For event broadcasts to be useful, the server has to know which connections care about which sessions. The participant tablet and moderator console will both subscribe to their session immediately after `hello`; the audience surface (a future task) will subscribe to whatever session it's projecting.

The data-model already has a single source of truth for "who can see what": `canSeeSession` in [`apps/server/src/sessions/visibility.ts`](../../../apps/server/src/sessions/visibility.ts). The HTTP routes (`GET /sessions/:id`, `GET /sessions`, `POST /sessions/:id/include`, ...) all run their visibility gate through it; this task uses the same primitive over the WS surface so a client that can't see a session over HTTP can't end-run by subscribing over WS.

Storage is per-server-instance: clustering (horizontal scale-out across multiple `apps/server` processes) is out of scope. The future `ws_reconnection_handling` task gives a reconnecting client a path to re-subscribe via the same handler on a new instance; a future cluster-fanout layer would publish broadcasts onto a message bus every instance subscribes to. Neither is wired today.

Broadcast emission is NOT this task — that's `ws_event_broadcast`. This task ends at the point where the registry is queryable.

## Decisions

- **Bidirectional-map storage** (`apps/server/src/ws/subscriptions.ts`). Two `Map<string, Set<string>>` indices — `bySession: Map<sessionId, Set<connectionId>>` and `byConnection: Map<connectionId, Set<sessionId>>`. Every public-surface operation is O(1) on the relevant side: `subscribe` / `unsubscribe` are two `Set.add` / `Set.delete` calls; `connectionsForSession(s)` and `sessionsForConnection(c)` are one `Map.get`; `removeConnection(c)` iterates `byConnection.get(c)` once (typically size 1: a client usually subscribes to one session at a time). The connection-side index trades a tiny memory overhead for O(1) close-time cost — and close is the hottest path under churn. Empty sets are pruned (the map entry is deleted) so the registry doesn't leak per-session or per-connection allocations after a participant joins and leaves many sessions over a long-lived connection.

- **Visibility gate: `canSeeSession(pool, sessionId, userId)`.** The handler at [`apps/server/src/ws/handlers/subscribe.ts`](../../../apps/server/src/ws/handlers/subscribe.ts) calls the same `canSeeSession` primitive HTTP routes use. The user id comes from `connection.user.id` (populated by `ws_auth_on_connect` before the dispatcher ever sees a message). The pool is captured at handler-build time through a property-getter that resolves `getDefaultPool()` lazily on first dispatch (the same lazy-pool pattern `wsConnectionHandlingPlugin` uses). `canSeeSession` returns `false` for both "the session doesn't exist" and "the session exists but is private and the user isn't a participant" — that collapse is the existence-non-leak rule from `get_session_endpoint`'s 404-not-403 decision; the WS surface inherits it.

- **`subscribed` / `unsubscribed` ack pattern with `inResponseTo` correlation.** When the visibility check passes, the handler calls `registry.subscribe(connectionId, sessionId)` and then sends a `WsEnvelope<'subscribed'>` ack whose `inResponseTo` is the originating `subscribe` envelope's `id`. The payload echoes `sessionId` for human-readable debuggability, but `inResponseTo` is the authoritative correlation field. `unsubscribed` mirrors the shape. The four new envelope types (`subscribe`, `unsubscribe`, `subscribed`, `unsubscribed`) are added to the closed `WsMessageType` enum in [`packages/shared-types/src/ws-envelope.ts`](../../../packages/shared-types/src/ws-envelope.ts) plus the matching payload schemas + the `WsMessagePayloadMap` interface entries — TypeScript's `Record<WsMessageType, …>` exhaustiveness check enforces all three updates land together.

- **Idempotence.** `subscribe(c, s)` called twice with the same tuple is a no-op (the underlying `Set.add` is idempotent; both indices remain consistent). `unsubscribe(c, s)` called with a tuple that isn't in the registry is a no-op. The handlers always emit the ack regardless of registry state — the client's request-response correlation stays consistent. The `unsubscribe` handler does NOT run a visibility check (a client that's NOT subscribed can still ask to be removed; the ack is the contract).

- **Placeholder error path until `ws_error_message` lands.** The visibility-rejection branch in the `subscribe` handler logs at warn level (`'ws-subscribe rejected — session not visible to user (placeholder error path, ws_error_message TODO)'`) and returns. **No error envelope is sent on the wire today.** The closed `WsMessageType` discriminator does not yet contain `error`; inventing the shape here would either (a) hard-code a wire shape the `ws_error_message` task will then have to change or (b) require a forward-compatible guess. The dispatcher-seam pattern `ws_message_envelope` established is the project's chosen discipline — log + drop today, wire-format error when the owning task lands. The client's `subscribe` envelope will eventually time-out client-side waiting for the ack (no ack is the signal); when `ws_error_message` ships, that task replaces this branch with a typed error envelope carrying `code: 'not-found'` and `inResponseTo = envelope.id`.

- **Connection-close cleanup via the registry's `removeConnection`.** The connection-handler's `socket.on('close', …)` hook ([`apps/server/src/ws/connection.ts`](../../../apps/server/src/ws/connection.ts)) calls `app.wsSubscriptions.removeConnection(connectionId)` exactly once on close. The call is idempotent — a connection that never subscribed makes it a no-op. Without this teardown, a dropped socket would leak entries in both indices; calling once on close is the single teardown point. The hook runs alongside the existing `openConnections.delete(ctx)` and `ws-connection-closed` log call — no new lifecycle hook is added, just an additional line.

- **In-process / per-server-instance — clustering out of scope.** The registry lives in this server process. A horizontally-scaled deployment would NOT see broadcasts emitted on one instance reach subscribers connected to another. Two future paths recover the property: (1) `ws_reconnection_handling` lets a client reconnect (to any instance) and replay state from the data-model — the new instance reconstructs the subscription via the same `subscribe` handler, visibility is re-checked, the registry repopulates locally; (2) a future cluster-fanout layer (NOT specified yet — out of scope for M3) would publish broadcasts to a message bus every instance subscribes to. The registry's shape doesn't change; only the broadcast surface does.

- **Broadcast emission is NOT in scope — deferred to `ws_event_broadcast`.** This task ends at "registry is correctly populated + acks are sent." `ws_event_broadcast` (depends on this task) will reach for `connectionsForSession(...)` to iterate fan-out targets when a session's event is applied. No server-emitted broadcast frames land here.

- **Per-app-instance registry via `wsSubscriptionsPlugin`.** Mirrors the `wsDispatcherPlugin` pattern — `fastify-plugin`-wrapped + skip-override + a defensive `hasDecorator` guard so re-registration in a future test pattern fails loudly without crashing. Tests build a fresh app per scenario so there's no cross-test bleed. The registration order in `wsConnectionHandlingPlugin` is: `wsDispatcherPlugin` → `wsSubscriptionsPlugin` → route plugin (the connection handler reaches for `app.wsSubscriptions` in the close hook). The `wsHandlersPlugin` (a separate plugin in [`apps/server/src/ws/handlers/index.ts`](../../../apps/server/src/ws/handlers/index.ts)) registers AFTER `wsConnectionHandlingPlugin` so the dispatcher + registry decorations are already in place.

- **Builder closure, not class, for the handler.** The dispatcher's `register(type, handler)` signature takes a function; a builder that captures `(pool, registry, log)` once and returns the handler pair (`buildSubscribeHandler`, `buildUnsubscribeHandler`) keeps the handler bodies pure-function — directly unit-testable without standing up a full Fastify instance. The registry-level unit tests in `subscriptions.test.ts` exercise the bidirectional-map invariants; the handler-level tests in `handlers/subscribe.test.ts` go through the real Fastify + dispatcher path to cover the visibility gate + ack flow end-to-end.

- **Test coverage layered per ADR 0022.** Pure-logic registry behavior (idempotence, bidirectional invariants, close-cleanup) → Vitest unit tests in [`apps/server/src/ws/subscriptions.test.ts`](../../../apps/server/src/ws/subscriptions.test.ts). Handler behavior driven through the real WS upgrade path with a pglite-backed pool + a real `sessions` row → Vitest unit tests in [`apps/server/src/ws/handlers/subscribe.test.ts`](../../../apps/server/src/ws/handlers/subscribe.test.ts) (mirrors `auth.test.ts`'s `__buildTestWsApp` + memory pool pattern, with the memory pool extended to recognise the `canSeeSession` SELECT). End-to-end wire behavior → Cucumber scenarios in [`tests/behavior/backend/ws-subscribe.feature`](../../backend/ws-subscribe.feature) against pglite + a real `sessions` row.
