## What this task is

Grow the `proposal-status` broadcast listener's facet-target derivation so that pending `decompose` and `interpretive-split` proposals emit per-component `proposal-status` envelopes carrying `{ classification: 'proposed' }` for each of the N component nodes — mirroring the moderator-side `computeFacetStatuses` arm at [`apps/moderator/src/graph/facetStatus.ts:239-262`](../../../apps/moderator/src/graph/facetStatus.ts) on the server side so that participant + audience surfaces consuming the `proposal-status` broadcast (without running the client-side derivation) see the same per-component `proposed` styling the moderator already renders.

Concretely the deliverable is two files of change:

1. **Broadcast-listener extension** at [`apps/server/src/ws/broadcast/proposal-status.ts:193-346`](../../../apps/server/src/ws/broadcast/proposal-status.ts) — replace `facetTargetForProposal` (currently returns one `FacetTarget | null`) with `facetTargetsForProposal` returning `readonly FacetTarget[]` (empty array == "no broadcast"), so the four single-facet sub-kinds return a 1-element array and the two multi-component sub-kinds return an N-element array of per-component classification targets. Adjust `deriveAndFanOut` to loop over the targets array and emit one envelope per target (each with `perFacetStatus: { [target.facet]: status }`); the remaining five structural sub-kinds (`axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate`) return an empty array and the listener returns early as today. Each emitted envelope carries the SAME `proposalId` + `sequence` but a different `entityId`-keyed target (the receiver correlates per-entity via the projection load it's already running for the `event-applied` round). Per D2 the envelope shape itself stays unchanged (single-facet `perFacetStatus` per envelope) — no wire-format extension.
2. **Vitest unit coverage** at [`apps/server/src/ws/broadcast/proposal-status.test.ts`](../../../apps/server/src/ws/broadcast/proposal-status.test.ts) — four new cases (per D4):
   - **Case A (decompose-pending)** — a pending 2-component decompose proposal emits exactly 2 `proposal-status` envelopes (one per component), each with `perFacetStatus: { classification: 'proposed' }`, each addressing the same `proposalId` but reflecting the per-component derivation. The envelope IDs are server-minted UUIDs (one per envelope per the fan-out contract).
   - **Case B (interpretive-split-pending, symmetric)** — same shape but with 3 `readings` instead of 2 `components`; assert 3 envelopes emitted.
   - **Case C (withdraw)** — after a withdraw arm (N × `entity-removed` + the implicit removal of the pending proposal from `pendingProposals` per the projector), no `proposal-status` envelope is emitted because `lookupProposalPayload` returns `null` (the proposal is gone). Clients drop their per-component `proposed` styling on the `entity-removed` event-applied frames they're already consuming. (See D3 for the rationale on NOT emitting a terminal "facet went away" envelope.)
   - **Case D (commit)** — after a commit lands for the decompose proposal, the per-component facets transition to `agreed` / `committed` via the standard `deriveFacetStatus` rules; one envelope per component is emitted carrying `perFacetStatus: { classification: 'committed' }` (or `'agreed'` depending on the vote pattern in the fixture). The parent's classification facet (already-committed, since the decompose presupposes a classified parent) is unaffected.

## Why it needs to be done

Three reasons compose:

**Cross-surface consistency.** The moderator surface today consumes `computeFacetStatuses(events)` directly (per [`apps/moderator/src/graph/GraphCanvasPane.tsx:426`](../../../apps/moderator/src/graph/GraphCanvasPane.tsx)) — it runs the full client-side derivation over the raw WS event stream and gets the per-component `proposed` state for free via the arm landed in commit `166b407`. The participant (M5) and audience (M6) surfaces will land lighter-weight render paths that consume the `proposal-status` broadcast directly (per [`ws_proposal_status_broadcast.md`](./ws_proposal_status_broadcast.md) "Why it needs to be done" — "any drift between client and server logic would create silent inconsistency. The `proposal-status` broadcast lets the server be the single source of truth for the facet-status view; clients render directly from the broadcast without re-implementing the precedence rules.") Without this task, those surfaces would either have to (a) duplicate the moderator's client-side mirror for the decompose / interpretive-split case — re-introducing exactly the drift the broadcast was designed to prevent — or (b) silently under-report the component facets, leaving them with no `data-facet-status` attribute and the wrong canvas styling.

**ADR 0027 consequence.** Bullet 1 of ADR 0027 names `decompose` + `interpretive-split` explicitly alongside `classify-node` for the propose-time entity-emission contract. Bullet 4 pins "projectors / diagnostic engines / replay read facet state, not structural-event timing, to determine commit-status." The broadcast subscriber IS one of those consumers; treating the per-component classification facets as out-of-scope for the broadcast (the current behaviour) is a runtime-contract drift from the ADR that landed propose-time canvas visibility for these sub-kinds. The Status block on the source-of-debt refinement (L210) explicitly registers this task as the lockstep follow-up.

**Single-source-of-truth invariant.** The moderator's client-side mirror at [`apps/moderator/src/graph/facetStatus.ts:1-15`](../../../apps/moderator/src/graph/facetStatus.ts)'s header docblock acknowledges itself as a mirror of the server's `deriveFacetStatus` ("If a future refactor extracts a shared methodology types package, the duplication becomes the call site"). Today the mirror is the only authoritative source for the per-component decompose / interpretive-split case because the server doesn't emit a matching broadcast. This task closes the asymmetry: post-task, the server is authoritative for ALL facet status (single-target sub-kinds AND multi-component sub-kinds), and the moderator's mirror reads it from raw events only because it's the only surface that runs the full derivation locally (for the pending-proposals breakdown sidebar + the canvas styling — see D5 for why the mirror is NOT removed in this task).

Downstream consumers benefit transitively:

- `participant_ui.part_graph_view` (pending under M5) — will subscribe to `proposal-status` and apply the per-component styling without duplicating the moderator's mirror.
- `audience.aud_graph_rendering` (pending under M6) — same transitive inheritance.
- Future per-facet-status caching at the server (out of scope per [`ws_proposal_status_broadcast.md`](./ws_proposal_status_broadcast.md) "What is NOT in scope") becomes a one-swap optimization at the listener layer rather than a per-sub-kind branch.

## Constraints / requirements

- **One envelope per component, NOT one envelope with N component facets.** Per D2 the existing wire shape `{ sessionId, proposalId, sequence, perFacetStatus }` keys `perFacetStatus` by `FacetName` (`'classification' | 'substance' | 'wording'`); it does NOT key by `entityId`. The natural way to convey N components is N envelopes (each addressing one entity's classification facet) rather than a wire-shape change to introduce a per-entity nested key. This keeps the schema stable and matches the per-connection error-isolation contract (one bad envelope doesn't poison the others).
- **Each envelope reuses the same `proposalId` + `sequence`.** All N envelopes for a decompose / interpretive-split share the proposal's id + the triggering event's sequence. Receivers correlate per-entity via the per-envelope target lookup they already do (they're already correlating against their local projection / per-entity facet store). Per D2 this is the lowest-friction extension of the existing wire surface.
- **Each envelope has a unique server-minted UUID.** The envelope `id` field is a fresh `randomUUID()` per emit per the existing contract (one UUID per envelope per fan-out, mirroring `event-applied.ts`). Receivers can de-duplicate by `id` if needed; this is the standard idempotency seam.
- **Per-target iteration mirrors the proposal's `components` (or `readings`) array order.** Same convention the propose handler uses for `node-created` / `entity-included` emission (per the parent refinement's D3). Array order is the canonical iteration order; replay determinism stays consistent with the structural-event side.
- **No new envelope kind, no new payload schema, no wire-format extension.** The `'proposal-status'` envelope type stays the one already on the closed `WsMessageType` union; the `proposalStatusPayloadSchema` stays unchanged. This is purely a fan-out-multiplicity change inside the broadcast listener.
- **`facetTargetsForProposal` (plural) replaces `facetTargetForProposal` (singular).** Per D5 (alternative B rejected) we don't keep both — the call-site loop in `deriveAndFanOut` is the only consumer, and a single helper that returns an array (1-element for the single-target sub-kinds, N-element for multi-component, 0-element for purely-structural) is the cleanest expression. The four facet-targeting sub-kinds return `[{ entityKind, entityId, facet }]`; the two multi-component sub-kinds return `[{ entityKind: 'node', entityId: component.node_id, facet: 'classification' }, …]`; the five structural sub-kinds return `[]`.
- **Withdraw transition: no terminal envelope.** Per D3 the listener does NOT emit a "facet went away" envelope when the pending proposal is withdrawn. The withdraw handler emits N × `entity-removed` event-applied broadcasts; the per-component classification facets are no longer derivable (the proposal is gone from `pendingProposals`; the entity itself is invisible per `entity-removed`'s projector arm). Clients drop their per-component styling on the matching `entity-removed` frames they're already consuming. The broadcast's "no broadcast on missing proposal" path (`lookupProposalPayload === null` returns early, with the existing warn-log suppressed for this expected case per D3.b) is the natural seam.
- **Commit transition: per-component envelope per target, status reflects `deriveFacetStatus`.** When a commit lands on the decompose / interpretive-split proposal, the per-component classification facets transition to `'agreed'` / `'committed'` per the standard derivation rules. The listener loops over the same N targets and emits N envelopes with whatever `deriveFacetStatus` returns. The parent's own `classification` facet is not touched (the parent isn't the target of a decompose proposal — the components are).
- **Schema-on-write per ADR 0021.** Each emitted envelope passes `proposalStatusPayloadSchema` (already-required-shape for `{ sessionId, proposalId, sequence, perFacetStatus }`); the per-component `perFacetStatus: { classification: 'proposed' }` is a valid `z.record(z.string(), z.string())`.
- **Per-connection error isolation per the existing contract.** Each `sender(envelope)` is try-wrapped; one bad socket logs at warn level and the iteration continues. The per-target outer loop is unaffected: a send failure on target i does not abort the dispatch for target i+1.
- **No DB-cost regression for the "no subscribers" path.** The existing early-return at `proposal-status.ts:275-280` (no `connectionsForSession` → skip the projection load) stays in place. The per-target loop runs only after the projection has loaded.
- **No commit-handler / propose-handler / withdraw-handler change.** This task lives entirely inside the broadcast listener; the upstream handlers are unchanged. The propose-time `node-created` + `entity-included` fan-out from commit `166b407` is the structural-event prerequisite; this task consumes the resulting projection state.
- **Test coverage layered per ADR 0022.** Pure-logic broadcast-listener behaviour → Vitest at [`apps/server/src/ws/broadcast/proposal-status.test.ts`](../../../apps/server/src/ws/broadcast/proposal-status.test.ts). No new Cucumber / Playwright cover — the wire-path / e2e are already covered transitively by the parent refinement's lifted Scenario 3 (the moderator surface renders correctly via its client-side mirror; the participant + audience surfaces don't exist yet, so an end-to-end of the new server arm against a non-moderator client is out of reach until M5 / M6 ship).

## Decisions

- **D1 — Symmetric server-side arm in `facetTargetForProposal` (now `facetTargetsForProposal`), NOT a new broadcast envelope kind.** The existing `proposal-status` envelope already has the `perFacetStatus` shape needed to convey per-facet status. The natural extension is "emit more envelopes" (one per per-component target), not "introduce a new envelope kind" (`'component-facet-status'` or similar). Per the parent refinement's D3 ("inverse-pair invariant: the propose / withdraw handlers MUST stay in sync") and the architecture's reuse-existing-seams bias, extending the existing seam is the lowest-risk, lowest-cost change. Alternatives considered:
  - **(B) New `'component-status'` envelope kind carrying `{ proposalId, components: [{ nodeId, classification }] }` in a single envelope.** Rejected — adds a closed-union member to maintain; duplicates the wire-shape evolution work for what is structurally the same data (per-facet status); receivers have to handle two related-but-different envelope types for "the same proposal's facets just changed." The single-envelope-per-target pattern matches the existing single-facet contract and lets receivers handle them with the same code path.
  - **(C) Extend `perFacetStatus` to allow per-entity keying (`Record<entityId, Record<facetName, status>>`).** Rejected — wire-shape breaking change; existing receivers (the WS store decoders) would need a migration; the existing single-entity-per-envelope pattern is the simplest shape that conveys the data.

- **D2 — Per-component envelopes share `proposalId` + `sequence` but carry distinct per-envelope server-minted UUIDs.** Receivers can correlate envelopes back to the proposal via `proposalId`, address the per-component entity via the local projection's per-entity facet store (they're already running a projection for the `event-applied` round), and de-duplicate by envelope `id` if a reconnect replays. The wire surface gives every envelope a unique server-minted `id` per the existing fan-out contract; the `proposalId` + `sequence` are content keys, the `id` is a transport key. The alternative (re-use a single `id` across the N envelopes) would break the de-dup invariant and the per-envelope error-isolation contract; rejected.

- **D3 — Withdraw transition: NO terminal "facet went away" envelope; rely on the `entity-removed` event-applied frames.** When a decompose proposal is withdrawn, the listener observes the `entity-removed` events first (one per component), but `entity-removed` is NOT in `STATUS_AFFECTING_KINDS` so the listener returns early — no `proposal-status` envelope fires. The pending proposal is also gone from the projection (`lookupProposalPayload` would return `null` if re-triggered). Clients drop their per-component `proposed` styling on the matching `entity-removed` frames they already consume from the `event-applied` stream. Rationale:
  - **Receivers already process `entity-removed` to flip per-entity canvas visibility off** — coupling the facet-status drop to the same event is correct (no entity → no per-facet status to convey).
  - **A terminal "facet was proposed, now nothing" envelope adds wire surface for receivers to handle but conveys the same information they already have** (the entity is gone; the facet status is moot).
  - **Per-connection error isolation** — if a terminal envelope were emitted but failed to send, the receiver would be left with stale `proposed` styling on a non-existent entity, which is a worse failure mode than relying on the structural event.
  - **Alternatives considered:**
    - **(B) Emit a per-component envelope with `perFacetStatus: { classification: 'withdrawn' }` on the listener observing the `entity-removed` event.** Rejected — `'withdrawn'` is a facet-layer terminal status for a committed-then-disputed flow, NOT for an entity that has been retracted; the semantics don't match. The withdraw arm of a pending proposal removes the entity, not the facet.
    - **(C) Add `'entity-removed'` to `STATUS_AFFECTING_KINDS` and emit a per-component envelope with some sentinel status.** Rejected — the `FacetStatus` enum has no "no longer derivable" value; the listener would have to invent one or omit the envelope, leaving the receiver to special-case. Worse wire shape than option D3.

- **D4 — Vitest cover at the broadcast-listener layer; NO new Cucumber / Playwright cover in this task.** The pure-logic broadcast-listener tests at [`apps/server/src/ws/broadcast/proposal-status.test.ts`](../../../apps/server/src/ws/broadcast/proposal-status.test.ts) are the ADR 0022-compliant regression cover. The existing test infrastructure (captured-array senders + injected event-loader + captured logger + `flushMicrotasks()`) is reused; the four new cases add fixtures for the decompose / interpretive-split / withdraw / commit transitions. Rationale:
  - **The wire-path is already exercised end-to-end by the parent refinement's lifted Playwright Scenario 3** (the moderator surface renders correctly because its client-side mirror covers the case). The participant + audience surfaces that will consume the server-side arm don't exist yet (M5 / M6 are downstream).
  - **A Cucumber wire-path test against pglite would be valuable but is symmetric with the unit cover** — both pin the same per-envelope emission contract. Per the bias toward "the simpler test that pins the observable behaviour," the unit cover is the right v1 layer.
  - **When M5 lands the participant surface that consumes the broadcast, the M5 Playwright cover will exercise this code path end-to-end** — at that point the unit cover here becomes the lower-cost regression-narrow seam, with the e2e covering the broader integration.
  - **Alternative considered: Cucumber feature at `tests/behavior/backend/ws-proposal-status-decompose.feature`.** Rejected for v1 — the per-envelope emission contract is fully observable at the listener layer; the Cucumber surface would add a parallel fixture for what is effectively the same assertion, with the cost of a real pool + real socket round-trip. When the participant surface lands and an integration story emerges, that's the right time to add the Cucumber cover.

- **D5 — The moderator-side client mirror at `apps/moderator/src/graph/facetStatus.ts:239-262` is NOT removed in this commit; it stays.** The orchestrator context note ("After this task, the moderator's client-side mirror in `apps/moderator/src/graph/facetStatus.ts` may become redundant") suggested removing the moderator's per-component arm once the server-side derivation lands. **Rejected** for this task; the moderator-side mirror stays untouched. Rationale:
  - **The moderator currently consumes `computeFacetStatuses(events)` directly, NOT the `proposal-status` broadcast.** Per [`apps/moderator/src/graph/GraphCanvasPane.tsx:426`](../../../apps/moderator/src/graph/GraphCanvasPane.tsx) (`const facetStatusIndex = computeFacetStatuses(events);`) and [`apps/moderator/src/layout/PendingProposalsPane.tsx:412`](../../../apps/moderator/src/layout/PendingProposalsPane.tsx) (`const facetStatusIndex = useMemo(() => computeFacetStatuses(events ?? []), [events]);`), the moderator runs the full client-side derivation over the raw WS event stream — it does NOT subscribe to / consume `proposal-status` envelopes for its per-facet rendering. Removing the per-component arm from `facetStatus.ts` would BREAK the moderator's canvas styling for pending decompose / interpretive-split proposals.
  - **Migrating the moderator off `computeFacetStatuses` and onto `proposal-status` consumption is a separate task in its own right** (it would require: subscribing the moderator's WS store to `proposal-status` envelopes, building a per-(entity, facet) status map keyed by broadcast id + sequence, re-wiring `GraphCanvasPane` + `PendingProposalsPane` + `ProposalFacetBreakdown` to read from the broadcast-derived map instead of the client-derivation, and handling the catch-up / reconnect race where the broadcast-derived map needs to be rebuilt from `event-applied` replay). That's a substantial cross-cutting change with its own design surface (especially: how to bootstrap the broadcast-derived map on initial connect when `proposal-status` envelopes don't replay automatically).
  - **The moderator-side mirror is NOT load-bearing for cross-surface correctness once this task lands** — the participant + audience surfaces will consume the broadcast directly per their own M5 / M6 task scopes, and the mirror won't be in their dependency tree. The mirror is load-bearing ONLY for the moderator's own rendering.
  - **Alternative considered: remove the per-component arm in `facetStatus.ts` in this same commit, accepting the moderator-surface regression as a forcing function to migrate the moderator off `computeFacetStatuses` in a follow-up commit cluster.** Rejected — would land a known-broken moderator canvas styling for an unknown window of commits; violates the "ship green commits" principle. The right migration sequencing is "land the server-side arm first (this task), let the moderator continue to consume the client-side mirror, then a separate task migrates the moderator off the client-side mirror once the broadcast-driven path is wired."
  - **Alternative considered: deprecate the per-component arm with a docblock note pointing at the server-side arm.** Adopted — the docblock at [`apps/moderator/src/graph/facetStatus.ts:215-238`](../../../apps/moderator/src/graph/facetStatus.ts) (the long comment introducing the decompose arm) grows a one-line note saying "the server-side symmetric arm at `apps/server/src/ws/broadcast/proposal-status.ts` is the source of truth for non-moderator surfaces; this mirror remains the in-place derivation for the moderator's `computeFacetStatuses(events)` consumer until a future task migrates the moderator onto the broadcast path." This is the only edit to `facetStatus.ts` in this task.

- **D6 — v1 emits only the classification facet per component, NOT all three facets.** Per the moderator-side mirror (which only walks the classification facet at L239-262), this task's server-side arm emits per-component envelopes for the classification facet only. The wording + substance facets are NOT emitted in v1. Rationale:
  - **Symmetry with the moderator-side mirror.** The mirror's docblock at L215-238 documents the classification-only derivation as the v1 scope ("each component's classification facet is `proposed` while the decompose / interpretive-split proposal is pending"). Mirroring the same scope on the server side keeps the two derivations in lockstep — a future amendment that extends the moderator's mirror to wording / substance would land the server-side extension at the same time.
  - **The methodology contract for components' wording / substance facets is "each starts as `proposed` independently"** (per `docs/data-model.md` L84-87), but the propose-time `node-created` events for the components carry the `wording` directly (it's not a separate facet event); the wording facet's state at propose-time is initialized by the projector's `handleNodeCreated` arm and is observable to any consumer that has the projection loaded. The classification facet, by contrast, has no per-component event of its own — it's purely derived from the parent proposal — which is why the moderator's mirror walks it explicitly and the server-side arm needs to walk it too.
  - **Substance facet is N/A for component nodes at propose-time** — `set-node-substance` is a separate proposal sub-kind; component nodes don't carry a substance facet until a subsequent `set-node-substance` proposal targets them. The server-side arm correctly omits the substance facet for this same reason.
  - **Alternative considered: emit per-component envelopes for all three facets (`classification`, `wording`, `substance`) at v1.** Rejected — adds 3x the wire traffic for facets that either have no per-component derivation rule (substance) or are observable via the structural event already (wording, via `node-created.payload.wording`). The classification-only scope matches the moderator-side mirror and matches the actual methodology gap that prompted this task.

- **D7 — Broadcast envelope ordering: per-component envelopes follow the same iteration order as the propose-handler's component fan-out.** The propose handler walks `proposal.components` (or `.readings`) in array order; this task's broadcast listener walks the same array in the same order. Receivers observing the per-event order can rely on the per-component envelopes arriving in `components[0]`, `components[1]`, …, `components[N-1]` order. Rationale:
  - **Replay determinism is a non-negotiable invariant per ADR 0021.** The proposal's array order is the authoritative ordering; both the structural-event side (propose handler) and the facet-status side (this task's broadcast listener) walk it consistently.
  - **Per-connection error isolation does NOT depend on per-target ordering** — a send failure on envelope i does not affect envelope j; but for receivers that handle envelopes in order (e.g., a simple `useReducer` chain), the ordered iteration produces predictable behaviour.
  - **Alternative considered: emit per-component envelopes in parallel via `Promise.all`.** Rejected — the per-connection sender is synchronous (the registry's `send(envelope)` is a function call, not a Promise); parallelism would add no throughput. The per-target loop is naturally sequential.
