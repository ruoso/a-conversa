## What this task is

Land the WS handler that consumes a client→server `'withdraw-proposal'` envelope from a debater / moderator and emits the matching `entity-removed` events that retract the entities a still-pending proposal introduced at propose-time. Per ADR 0027 the entity layer and the facet layer are strictly separate; when a proposer rescinds their intent BEFORE the proposal commits, the entities the propose-time fan-out minted must leave the structure via explicit `entity-removed` events (rather than vanishing via an implicit projector derivation rule — D4 of the source-of-debt refinement). The handler is the wire-side seam that produces those events; the projector's `handleEntityRemoved` (already on `main`) consumes them.

Concretely the handler:

1. **Subscribe-before-act gate** — same `ApiError.forbidden('not subscribed ...')` as propose / vote / commit / mark-meta-disagreement (per [`apps/server/src/ws/handlers/propose.ts:175-180`](../../../apps/server/src/ws/handlers/propose.ts) and [`apps/server/src/ws/handlers/commit.ts:144-149`](../../../apps/server/src/ws/handlers/commit.ts)).
2. **Visibility re-check** — `canSeeSession(pool, sessionId, userId)` → `ApiError.notFound(...)` if the session became invisible between subscribe and withdraw (same existence-non-leak rule the propose / commit handlers use).
3. **Transactional sequence allocation + projection load + authority + state checks + INSERT(s)** — FOR UPDATE on `sessions`, MAX(sequence)+1, `projectFromLog`. Build the list of entities the proposal introduced at propose-time by re-deriving from the pending proposal record's payload (mirroring the per-sub-kind shape `buildStructuralEventsForPropose` at [`apps/server/src/methodology/handlers/propose.ts:1261-1327`](../../../apps/server/src/methodology/handlers/propose.ts) uses to emit them). Verify proposer authority + still-pending state (see D1, D2). For each retracted entity construct one `entity-removed` `EventToAppendEnvelope`; `validateEvent` + `appendSessionEvent` each in order; the events occupy `nextSeq, nextSeq+1, ...` (mirroring propose's multi-event sequence-allocator at [`apps/server/src/methodology/handlers/propose.ts:1199-1209`](../../../apps/server/src/methodology/handlers/propose.ts)).
4. **Post-commit broadcast + `proposal-withdrawn` ack** — `app.wsBroadcast.emit({ event })` per appended event AFTER the transaction's COMMIT (post-commit-emit invariant per [`apps/server/src/ws/handlers/propose.ts:329-331`](../../../apps/server/src/ws/handlers/propose.ts)), then send `proposal-withdrawn` ack frame directly on the originating socket. The proposer receives BOTH the ack (`inResponseTo` correlated) and the broadcast (the projection-update signal); non-proposer subscribed clients receive only the broadcasts.

Scope is the **WS handler + the two closed-union wire-vocabulary extensions + Vitest unit coverage + Cucumber wire-path coverage** only. **Out of scope:**

- The participant-facing gesture / UI that dispatches the `'withdraw-proposal'` envelope from a tap. That is `moderator_ui.mod_withdraw_proposal_gesture` (already registered as a tech-debt leaf in [tasks/30-moderator-ui.tji:760](../../30-moderator-ui.tji)) and will compose on top of this handler.
- The deferred Playwright Scenario 4 from the canvas-visibility refinement ("propose-then-withdraw, three browser contexts"). The UI-stream policy targets the gesture-wiring task, not this protocol task; the contract this handler establishes is the seam the gesture's e2e will drive against.
- A new engine action kind (`'withdraw-proposal'`) registered in `MethodologyAction`. The authority + state predicates this handler needs are simple enough (see D2 alternatives) that introducing a fifth engine action kind for v1 would be over-scoped. The handler enforces the predicates directly, mirroring the small set of pre-engine gates the propose handler already enforces (subscribe, visibility, optimistic-concurrency). If/when a second consumer (an HTTP endpoint, a CLI tool) needs to issue withdraw-proposal actions, the predicates move into a `withdrawProposalHandler` engine module at that point.

## Why it needs to be done

The `'entity-removed'` event kind is on the wire vocabulary, registered in the schema-on-write registry, validated by a forward-only SQL `CHECK`, and consumed by the server-side projector — but nothing emits it. The propose handler at [`apps/server/src/methodology/handlers/propose.ts:1199-1209`](../../../apps/server/src/methodology/handlers/propose.ts) emits `node-created` + `entity-included` for free-floating `classify-node` proposals (the post-ADR-0027 contract) and there is no current path to retract those entities short of committing the proposal. Without this handler:

- A proposer who taps Propose and immediately wants to rescind has no way to clear the proposed entity from the canvas. The pending proposal can be ignored / left unresolved, but the entity stays visible in `proposed` state until the moderator commits (which the proposer doesn't want) or the session ends — both worse outcomes than a clean retraction.
- The Playwright Scenario 4 ("propose-then-withdraw") from the canvas-visibility refinement is **deferred** as a tech-debt leaf precisely because this handler doesn't exist. The scenario is the eventual end-to-end cover; this task is the protocol prerequisite.
- The methodology contract (per `docs/methodology.md` L40 — "A debater who refuses to agree to (say) a `contradicts` edge against their own claims cannot make the proposal vanish — only delay or block its transition to `agreed`") draws a careful distinction: a non-proposer **cannot** make a proposal vanish (D7 of the source-of-debt refinement settled "proposer-only withdraw authority" — engine-enforced), but the proposer themselves **can** withdraw before commit. Without this handler, the asymmetry is broken in the inconvenient direction (no-one can withdraw); landing this handler restores the asymmetry the methodology contract describes.
- Downstream consumers — the projector's `handleEntityRemoved` (in `apps/server/src/projection/replay.ts`), every cross-surface canvas projector (moderator + participant + audience) that walks `node.visible` / `edge.visible`, and the `event-applied` broadcast subscribers — are all already wired for the event but never see it.

This task closes the wire-side gap. The follow-up `mod_withdraw_proposal_gesture` task wires the proposer-side UI that dispatches the envelope; this task makes the wire-side dispatch land an event the projector can consume.

## Constraints / requirements

- **Mirror propose / commit tightly.** Same `withTransaction` shape, same FOR UPDATE on `sessions`, same MAX(sequence)+1 allocator, same projection-load-per-request via `projectFromLog`, same `validateEvent` + `appendSessionEvent` per emitted event, same post-commit `broadcast.emit` + ack ordering. The six sibling handlers (propose, vote, commit, meta-disagreement, snapshot, catch-up + this) live side-by-side; structural drift would degrade reviewability.
- **Single source of truth: the projection's `findProposal`.** Authority (proposer match) + state (still-pending) decisions read the projection record returned by `findProposal`; no parallel SQL lookup of the original `proposal` event row.
- **Proposer identity from the connection, not the payload.** The wire `'withdraw-proposal'` payload carries `{ sessionId, expectedSequence, proposalEventId }` — no `proposerId` field. The handler reads `connection.user.id` and matches it against `pending.proposer`. Symmetric with propose (no `proposerId`) / vote (no `voterId`) / commit (no `moderatorId`). Pinned by a Vitest case that sends an extra `proposerId` field naming the real proposer — the schema is closed-shape so Zod strips it; even if it didn't the handler ignores it.
- **Closed-union extension** — `'withdraw-proposal'` to Group B tail; `'proposal-withdrawn'` to Group C tail of [`packages/shared-types/src/ws-envelope.ts`](../../../packages/shared-types/src/ws-envelope.ts) per the documented layout convention.
- **Proposer's dual signal** — both `proposal-withdrawn` ack (`inResponseTo` correlated) and one `event-applied` broadcast per emitted `entity-removed` event arrive at the proposer; non-proposer subscribed clients receive only the broadcasts.
- **Schema-on-write** — `validateEvent` runs on every constructed `entity-removed` envelope before `appendSessionEvent`, mirroring propose / vote / commit / meta-disagreement.
- **Optimistic-concurrency** — the client's `expectedSequence` MUST equal `MAX(sequence)` at the point of the FOR UPDATE. Mismatch synthesises the same `'sequence-mismatch'` `RejectedValidationResult` shape propose / commit use, routed through `rejectedToApiError` → wire `code: 'sequence-mismatch'` (409). Uniform client-side branching across all five write handlers.
- **Post-commit-emit invariant** — `broadcast.emit` runs AFTER the transaction's COMMIT. A broadcast that observed a rolled-back row would be lying.
- **Per-sub-kind retraction mapping mirrors `buildStructuralEventsForPropose`.** For a `classify-node` proposal whose payload carried a `wording` AND whose `node_id` matches a node minted at propose-time (the propose handler only emits `node-created` when the node didn't pre-exist), emit one `entity-removed` for that node. For other sub-kinds (`set-edge-substance` without endpoint carriage, `set-node-substance`, `edit-wording`, `axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate`, `decompose`, `interpretive-split` — see D3) v1 emits zero structural retractions; the proposal envelope itself remains in `pendingProposals` but the entity layer has nothing to retract because nothing was minted at propose-time. The `proposal-withdrawn` ack always lands (signals "your request was processed"); the `event-applied` broadcasts may be empty for the no-entity-introduced sub-kinds. See D3 for the rationale on shipping the v1 mapping inline + the follow-up that grows it.
- **The proposal envelope itself is NOT retracted by this handler.** D5: withdraw-proposal retracts the propose-time-minted ENTITIES; the proposal envelope event stays in the event log (replay-authoritative history is immutable per ADR 0020). The projector's response to the entity retractions is what removes the proposal from the canvas + from `pendingProposals` — handled by the source-of-debt refinement's audit pass (already on `main`). A future follow-up may add an explicit `'proposal-withdrawn'` EVENT kind (distinct from the `'proposal-withdrawn'` WS ENVELOPE type — overlap is intentional + namespace-distinct) if the projector needs a per-proposal terminal marker; v1 doesn't need one because the entity removals are sufficient.
- **Test coverage layered per ADR 0022.** Pure-logic handler-builder behaviour → Vitest at `apps/server/src/ws/handlers/withdraw.test.ts`. Wire-path against pglite → Cucumber at `tests/behavior/backend/ws-withdraw.feature`.

## Decisions

The propose refinement's Decisions section ([`ws_propose_message.md`](./ws_propose_message.md)) applies verbatim for the gate stack, union-layout convention, post-commit-emit invariant, dual-signal contract, projection-load strategy, optimistic-concurrency surface, and connection-derived-actor security invariant. The commit refinement's Decisions section ([`ws_commit_message.md`](./ws_commit_message.md)) applies verbatim for the "authority decision lives in one place + surfaces uniformly on the wire" pattern. Only the withdraw-proposal-specific deltas are documented below.

- **D1 — Proposer authority gate lives in the WS handler, surfaced on the wire as `forbidden`.** A subscribed non-proposer who sends `'withdraw-proposal'` against another participant's pending proposal passes the WS handler's gates 1 (subscribe) and 2 (visibility), reaches the authority check, and is rejected with `ApiError.forbidden('only the original proposer may withdraw this proposal — proposalEventId=…, you are …, the proposer is …')`. The dispatcher's `onHandlerError` seam echoes the throw as a wire `error` envelope with `code: 'forbidden'`. **Pinned in a unit test** (`HEADLINE: rejects a non-proposer attempting withdraw with 'forbidden'`). Rationale for staying at the protocol layer (`ApiError.forbidden`) rather than promoting to a new engine `RejectionReason` (`'not-proposer'`):
  - **Alternative considered (per the source-of-debt refinement note): mint `'not-proposer'` as a new `RejectionReason`.** That would require adding the variant to `RejectionReason` in [`apps/server/src/methodology/types.ts:159`](../../../apps/server/src/methodology/types.ts), extending the exhaustive `statusCodeForRejection` switch in [`apps/server/src/errors.ts:174`](../../../apps/server/src/errors.ts) (mapping to 403), AND introducing a new engine action kind `'withdraw-proposal'` registered in `MethodologyAction` so a `validateAction`-routed gate has somewhere to live. The engine work is significant and the only consumer is this one handler — the cost is higher than the benefit.
  - **Alternative considered: reuse `'not-a-participant'`.** Rejected — the requester IS a participant (the universal participant gate in `validateAction` would pass even if we routed through the engine). "Not the original proposer of this specific proposal" is a finer-grained authority condition than "not a participant in this session." Conflating the two would mask the distinction in logs and on-wire diagnostics.
  - **Alternative considered: route through the engine via a `'withdraw-proposal'` action kind.** That's the architecturally-pure mirror of commit's moderator-only gate, but commit's gate lives in the engine because the engine ALREADY uses it for the per-facet unanimity rule (rule 4 of `commitHandler`). The withdraw handler has only the authority + state predicates — both trivially expressible against the projection's `findProposal` lookup. Adding an engine action kind for v1 is over-scoped; the predicates are simpler than the engine's universal checks they would compose with.
  - **Why `'forbidden'` is the right wire code:** the requester is authenticated but lacks the standing for this specific action — exactly the canonical HTTP/WS semantics for `'forbidden'`. The same code propose / commit emit for the subscribe-before-act gate; client-side `switch (error.code) { case 'forbidden': showInsufficientAuthorityToast(); ... }` branching stays uniform. The `error.message` field carries the specific reason ("only the original proposer may withdraw …").
  - **Engine-gate promotion path stays open.** If/when a second consumer (an HTTP `DELETE /sessions/:id/proposals/:proposalId` route, a CLI tool) needs to issue withdraw-proposal actions, the predicates promote to a `withdrawProposalHandler` engine module + a new `MethodologyAction.withdrawProposal` variant; the WS handler would then route through `validateAction` like commit does. The decision to keep v1 at the protocol layer is a v1 simplification, not a permanent architectural choice. The handler's docblock cites this open promotion path so a future reader doesn't mistake the decision for a permanent assertion.

- **D2 — Proposal-state checks reuse the existing engine `RejectionReason` codes via synthesised `RejectedValidationResult`.** The handler does NOT route through `validateAction` (per D1), but it DOES route its state-check rejections through `rejectedToApiError` by synthesising `{ ok: false, reason: 'proposal-not-found' | 'proposal-already-committed' | 'proposal-already-meta-disagreement' | 'sequence-mismatch', detail: ... }`. Rationale:
  - The wire-code surface is uniform with commit / mark-meta-disagreement / vote — a client that already branches on those codes for the same proposal state-machine doesn't need a new branch.
  - `rejectedToApiError`'s exhaustive `switch` already maps every code this handler reuses (404 / 422 / 422 / 409 respectively, per `apps/server/src/errors.ts:198-228`).
  - No new `RejectionReason` variant is added (consistent with D1's "don't promote to engine" decision); the synthesis is a thin adapter at the handler level, mirroring the same pattern propose's `expectedSequence` check uses inline at [`apps/server/src/ws/handlers/propose.ts:243-249`](../../../apps/server/src/ws/handlers/propose.ts).
  - Alternative considered: throw `ApiError.notFound(...)` / `ApiError.conflict(...)` directly without the `rejectedToApiError` synthesis. Rejected — the wire `error.code` would be the generic HTTP-status-kebab (`'not-found'`, `'conflict'`) rather than the methodology-vocabulary `'proposal-not-found'` / `'proposal-already-committed'` codes the other handlers emit. The client branch would have to special-case withdraw, defeating uniformity.

- **D3 — Per-sub-kind retraction mapping mirrors `buildStructuralEventsForPropose` inline; multi-sub-kind coverage tracks the propose-emission tech-debt.** The handler derives "which entities to retract" from the pending proposal's payload by reading the same predicates `buildStructuralEventsForPropose` (at [`apps/server/src/methodology/handlers/propose.ts:1261-1327`](../../../apps/server/src/methodology/handlers/propose.ts)) uses to emit them. The two functions are inverses; they MUST stay in sync. The v1 mapping covers exactly the sub-kinds propose-time emission covers today: `classify-node` with a wording AND a node minted at propose-time → emit one `entity-removed` for that node. Every other sub-kind → emit zero structural retractions (the proposal envelope itself is the only artefact, and per D5 the envelope is not retracted by this handler). Rationale:
  - **The inverse-pair invariant is fragile if the two are far apart.** Two alternatives considered:
    - **(A) Extract a shared `proposeTimeIntroducedEntities(payload, projection): Entity[]` helper.** Rejected for v1 — `buildStructuralEventsForPropose` returns full `EventToAppend` envelopes (with sequence + UUID + actor + createdAt), not bare entity references; extracting only the references-shape pair from it would either require refactoring the propose helper or adding a parallel pure-derivation helper. The parallel helper is two duplications of the same per-sub-kind switch — same fragility risk as inlining, plus a new module. Inlining the derivation at the withdraw site, with a docblock that cites `buildStructuralEventsForPropose` as the canonical pair, keeps the coupling visible at exactly the place a future reader needs to see it.
    - **(B) Track the introduced entities on the `PendingProposal` projection record (a new `introducedEntities: Array<{ kind, id }>` field).** Rejected for v1 — adds a projection-shape change for a single read site; the projection's authoritative derivation (`handleProposal` at [`apps/server/src/projection/replay.ts:340-400`](../../../apps/server/src/projection/replay.ts)) would have to re-derive the same per-sub-kind switch on read AND store it; one or the other becomes the new source-of-truth and the other follows. Net more code than the inline derivation. If a second read site emerges, this is the right time to lift; for one read site the cost-benefit favours inlining.
  - **The follow-up tech-debt that grows the propose-time emission grows this handler in lockstep.** When the `mod_set_edge_substance_endpoint_carriage` task lands `edge-created` + `entity-included` emission for the connecting `set-edge-substance` case, this handler's per-sub-kind mapping MUST grow a `'set-edge-substance' → entity-removed(edge)` arm. When `mod_decompose_propose_time_canvas_visibility` lands N × `node-created` for decompose components, this handler's mapping MUST grow a `'decompose' → N × entity-removed(node)` arm. The docblock on `buildStructuralEventsForPropose` AND on this handler's per-sub-kind switch cross-reference each other so the inverse-pair invariant is enforced by code review (not by an automated check — same rationale as the source-of-debt refinement's D9 documented-invariant decision).
  - **Tests pin the present pairing.** The unit test for the happy path (`classify-node` propose → withdraw → one `entity-removed(node)`) is the regression cover for the current pair. The happy-path test for the no-entity-introduced sub-kind (`set-node-substance`) is the regression cover for the zero-emission case. When the per-sub-kind mapping grows, the test grows in lockstep.

- **D4 — `removed_by` and `removed_at` come from the authenticated connection + the injected clock, mirroring propose's `created_by` / `created_at` pattern.** `removed_by = connection.user.id`; `removed_at` = ISO-8601 of the injected `now()` clock (same source as `createdAt` on the event envelope). The handler does NOT take these from the request payload (which doesn't carry them — see the schema in D7). Rationale: identical to the propose handler's "actor from connection, not payload" security invariant. The injected clock is a single source for both `createdAt` (envelope-level) and `removed_at` (payload-level); they MUST be identical at the handler level to avoid drift if a future requirement separates them.

- **D5 — The proposal envelope event itself is NOT retracted by this handler.** Per ADR 0021 and ADR 0020 the event log is append-only — history is replay-authoritative and immutable. Withdrawing a proposal retracts the propose-time-minted ENTITIES (via the `entity-removed` events this handler emits); the original `proposal` event stays in the log forever. The projector's response (already on `main` per the source-of-debt refinement's audit pass) walks `node.visible` / `edge.visible` to decide what to render — once those flip false via `handleEntityRemoved`, the canvas + sidebar both stop showing the proposal. The proposal's `pendingProposals` record may stay in the projection (per `apps/server/src/projection/replay.ts`'s `handleProposal` which adds it on first-encounter and removes it on commit / meta-disagreement); v1 doesn't retract it because the entity-layer retractions are sufficient for the canvas + sidebar UX. Alternative considered:
  - **Mint a `'proposal-withdrawn'` EVENT kind (distinct from the `'proposal-withdrawn'` WS ENVELOPE type — the wire ack frame).** Rejected for v1 — the projector doesn't need a terminal marker on the proposal because the entity-layer removals already drive every visibility / sidebar / canvas reader's decision. The event-log archaeology question ("was this proposal withdrawn or just never resolved?") is answerable by walking forward from the proposal event and observing the matching `entity-removed`s; a dedicated kind would make that walk one-step instead of inferential, at the cost of a new kind to maintain. If a future consumer (a history-browser, a snapshot label, a methodology-trace renderer) needs the explicit marker, that's the right time to mint it. The decision is open in that direction; the v1 absence is documented in the handler's docblock.

- **D6 — `'withdraw-proposal'` is the WS request type; `'proposal-withdrawn'` is the ack type.** Group B / Group C per the union-layout convention. Naming chosen for verb-tense symmetry with the existing pairs (`'propose'` / `'proposed'`, `'commit'` / `'committed'`, `'vote'` / `'voted'`, `'mark-meta-disagreement'` / `'meta-disagreement-marked'`). Alternatives considered:
  - **`'withdraw'` (Group B) + `'withdrawn'` (Group C).** Rejected — `'withdraw'` is already taken as a `vote` payload arm (per [`packages/shared-types/src/ws-envelope.ts:447`](../../../packages/shared-types/src/ws-envelope.ts)'s `choice: z.enum(['agree', 'dispute', 'withdraw'])`). A reader scanning the wire-vocabulary list for "withdraw" would find a hit in two semantically-distinct surfaces (vote-withdraw vs. proposal-withdraw); the verbose `'withdraw-proposal'` disambiguates by referencing what's being withdrawn. The source-of-debt refinement's D7 + the WBS task title both use `'withdraw-proposal'` — this choice ratifies the existing usage.
  - **`'retract'` / `'retracted'`.** Rejected — `'withdraw'` is the verb the methodology + the source-of-debt refinement use everywhere ("a proposer may withdraw"); introducing a synonym would lose grep-ability.

- **D7 — Wire payload shape: `{ sessionId, expectedSequence, proposalEventId }`.** Mirrors the commit payload shape (commit takes `proposalId`; withdraw takes `proposalEventId` — same field role with the source-of-debt refinement's naming convention for "the original proposal envelope's event id"). No `proposerId` field per D1's connection-derived-actor security invariant. No `withdrawnAt` field — the server's injected clock owns the timestamp per D4. The closed Zod object schema strips any extra fields (Zod's `z.object` default behaviour).

- **D8 — Tests layered per ADR 0022.** Pure-logic handler behaviour (the gate stack, every authority + state rejection, the headline proposer-only gate, success path with the per-sub-kind retraction mapping, the security invariant pinning that a spoofed `proposerId` has zero effect, the empty-broadcasts case for zero-emission sub-kinds) → Vitest at `apps/server/src/ws/handlers/withdraw.test.ts`. Wire-path against pglite → Cucumber at `tests/behavior/backend/ws-withdraw.feature` with steps at `tests/behavior/steps/backend-ws-withdraw.steps.ts`. The Cucumber scenarios reuse the existing world/carrier pattern (auth-gated app + cookie + WS client lifecycle) from `backend-ws-auth.steps.ts` / `backend-ws-connection.steps.ts` / `backend-ws-subscribe.steps.ts` and add the withdraw-specific verbs. UI-stream e2e is NOT in scope per the task framing (the gesture-wiring task `mod_withdraw_proposal_gesture` is the UI-stream consumer that picks up Scenario 4 from the canvas-visibility refinement).
