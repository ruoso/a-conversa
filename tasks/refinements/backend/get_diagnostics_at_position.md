## What this task is

Add the fifth endpoint to the replay-routes family: `GET /api/sessions/:id/diagnostics?position=N`. It is the diagnostics-shaped sibling of `get_at_position`. Where `get_at_position` returns the **projected state** at a log position, this endpoint returns the **structural diagnostics** the methodology engine would surface for that projected state: the cycles, contradictions, multi-warrants, dangling claims, and coherency hints. It is a thin HTTP wrapper over two already-shipped primitives: read the session's full event log, replay it to position `N` with `projectAtPosition`, run `computeAllDiagnostics(projection)` over the resulting `Projection`, and return the wire-shaped entries. Position is event-sequence space (`0` = pre-history empty baseline → no diagnostics; `N` = after the event whose `sequence === N`), exactly as `projectAtPosition` defines it and as `get_at_position` already accepts.

## Why it needs to be done

[ADR 0044](../../../docs/adr/0044-replay-position-diagnostics-via-backend-endpoint.md) settled that replay-position diagnostics are served by a backend endpoint rather than recomputed client-side: the detectors read the server's `Projection` **class** (its `.nodes()` / `.edges()` / proposal maps and `deriveFacetStatus` derivation), none of which is on the wire, and porting that whole detection layer + a client `Projection` into `@a-conversa/shell` would create an ongoing parity burden against the methodology engine — the source of truth. (ADR 0043 accepted client-side duplication only because the position primitive was four trivial integer functions; that rationale does not extend to the detector layer.)

The direct downstream consumer already exists and is blocked on this endpoint: the test-mode `DiagnosticInspector` panel ([`apps/test-mode/src/diagnostics/DiagnosticInspector.tsx`](../../../apps/test-mode/src/diagnostics/DiagnosticInspector.tsx)) and its hook `useDiagnosticsAtPosition` fetch `GET /api/sessions/:id/diagnostics?position=N` on every position change. Until this endpoint returns 200, the panel renders the retryable error readout. **Shipping the endpoint flips the panel from error to live with zero client change.** It also unblocks the already-registered e2e-tightening task `replay_test.test_mode.test_mode_diagnostic_inspector_e2e_tracking` (see Acceptance, "No new Playwright task").

## Constraints / requirements

1. **Route & registration.** `app.get('/api/sessions/:id/diagnostics', {...}, handler)` inside the existing `replayRoutesPlugin` in [`apps/server/src/replay/routes.ts`](../../../apps/server/src/replay/routes.ts). No new plugin — it is the fifth sibling in the family's home.
2. **Auth.** `preHandler: app.authenticate`; the defensive `request.authUser === undefined → 500` assertion, identical to the siblings. 401 `auth-required` is middleware-owned and emitted before the handler.
3. **Path param.** `:id` is a UUID (`sessionIdParamsSchema`, already declared in the plugin). Non-UUID → 400 `validation-failed`.
4. **Query param `position`.** Required, JSON-schema `{ type: 'integer', minimum: 0 }` (ajv coerces `"5" → 5`). Missing / non-integer / negative → 400 `validation-failed` from schema validation. Reject any **unknown** query key with the same `preValidation` 400 the sibling routes use (only `position` is accepted) — identical to `get_at_position` Constraint §4.
5. **Visibility gate first.** `canSeeSession(pool, sessionId, userId)` runs before any event read. Zero visibility → 404 `not-found` (id unused or exists-but-invisible — indistinguishable from outside, per the existence-leak rule the siblings enforce). The event read never runs for an invisible session.
6. **Read the log, replay, detect.** After the gate: `const events = await readSessionEventLog(pool, { sessionId })`, then `const projection = projectAtPosition(events, sessionId, position)`, then `const diagnostics = computeAllDiagnostics(projection)`. Read the **full** log (not just up to `position`) so `projectAtPosition` validates the requested position against the *true* head sequence — same rationale as `get_at_position` Decisions §2. A `ReplayPositionError` (position `> headSequence`) is caught and re-thrown as 400 `validation-failed` carrying the valid `0..headSequence` range.
7. **No projection serialization.** Unlike `get_at_position`, this endpoint does **not** call `serializeProjectionForWire`. `computeAllDiagnostics` consumes the in-memory `Projection` object directly (it is the detectors' input type); the projection is never put on the wire here — only the diagnostics it yields. (Decisions §8.)
8. **Response (200) shape — bound by the shipped client.** `{ diagnostics: DiagnosticEntry[] }` — a JSON object with a single `diagnostics` array whose elements are the wire-shaped `DiagnosticEntry` objects `computeAllDiagnostics` returns verbatim (each has a string `kind` discriminant plus its kind-specific node/edge ids). This is exactly what `useDiagnosticsAtPosition` parses (`Array.isArray(body.diagnostics)` + per-element `kind: string`). The elements are the bare entries — **not** the WS `DiagnosticPayload` envelope, and the body is **not** a bare top-level array (Decisions §3). An empty `diagnostics: []` is the valid clean state (e.g. `position = 0`).
9. **`diagnostics` entries are opaque in the response schema** — declared `{ type: 'array', items: { type: 'object', additionalProperties: true } }`, for the same reason `get_at_position` leaves `projection` open: the entries are built by pure detectors over schema-validated events; re-validating their internal shape would couple the route to every detector's field set and duplicate the detectors' own unit coverage. The response schema keeps only the outer `{ diagnostics: [...] }` envelope honest; the detectors' Vitest tests pin the entry shapes.
10. **Read on a single connection.** Use the default pool (`ensurePool()` / `getDefaultPool()`), as the siblings do. No transaction — read-only replay.
11. **OpenAPI.** The route's `schema` carries `tags: ['replay']`, a `summary`/`description`, `security: [{ cookieAuth: [] }]`, `params`, `querystring`, and `response: { 200, '4xx': errorEnvelopeRef, '5xx': errorEnvelopeRef }`, matching the sibling routes. The generated `/docs/json` lists `GET /sessions/{id}/diagnostics`.
12. **No client changes.** The `DiagnosticInspector` panel and `useDiagnosticsAtPosition` hook are already shipped and already call this endpoint; this task touches `apps/server` (+ tests) only. Do not edit `apps/test-mode` or `@a-conversa/shell`.

## Decisions

**§1 — HTTP wrapper over `computeAllDiagnostics`; no new detection logic.** Both the replay primitive and the detector layer are shipped, refined, and unit-tested. This endpoint reads the log, calls `projectAtPosition`, calls `computeAllDiagnostics`, and returns the result — one source of truth for detection, zero parity risk. This is the decision [ADR 0044](../../../docs/adr/0044-replay-position-diagnostics-via-backend-endpoint.md) made. *Alternative — port the detectors + a client `Projection` into `@a-conversa/shell`:* rejected by ADR 0044 — the detectors need the full projection iterators, proposal maps, and `deriveFacetStatus` derivation, a large surface tightly coupled to the methodology engine; duplicating it client-side is an ongoing parity burden, and ADR 0043's "duplicate trivial logic" rationale does not extend to it.

**§2 — Reuse `readSessionEventLog` (full log) + `projectAtPosition`'s own range check.** Reading the entire log ascending lets `projectAtPosition` validate `position` against the true `replayHeadSequence`; reading only up to `position` would make an out-of-range position indistinguishable from the head and silently return a truncated projection instead of a 400. This is identical to `get_at_position` Decisions §2 — reuse the helper and the primitive's validation rather than re-deriving the range.

**§3 — Response is `{ diagnostics: DiagnosticEntry[] }` with bare entries — dictated by the already-shipped client.** The consumer `useDiagnosticsAtPosition` is on disk and binds the contract: it requires `Array.isArray(body.diagnostics)` and narrows each element on a string `kind`, then treats the element itself as a `WireDiagnostic`. So the array elements MUST be the bare `DiagnosticEntry` objects (`{ kind, ...ids }`) `computeAllDiagnostics` returns. *Alternative — return the WS `DiagnosticPayload` envelope array (`{ sessionId, kind, severity, status, sequence, diagnostic }[]`):* rejected — (a) it would break the shipped client, which narrows on `element.kind` and passes `element` as the inner diagnostic, not `element.diagnostic`; (b) `status: 'fired'|'cleared'` is a live-stream transition concept meaningless for a static positional snapshot; ADR 0044 already rejected reusing the live broadcast for exactly this reason. *Alternative — a bare top-level `DiagnosticEntry[]` body (no `{ diagnostics }` wrapper):* rejected — the client requires the `body.diagnostics` object field, and the object envelope leaves room for additive metadata (e.g. echoing `position`/`sequence`) without a breaking change, matching `get_at_position`'s `{ sessionId, sequence, projection }` enveloping instinct.

**§4 — Severity is not serialized per entry; it stays a pure function of `kind` on both ends.** ADR 0044's "classified by severity" phrasing is satisfied by classification, not by a wire field: the server has `classifyDiagnostic(entry): Severity` and the panel has its own `kind → severity` mapping, both pure functions of `kind` ∈ the five known kinds. The shipped client derives severity from `kind` and never reads a severity field off this endpoint, so adding one would be dead weight today. *Alternative — attach `severity` to each returned entry (mirroring the WS payload):* rejected for v1 as an unused field; it is a purely additive change if a future consumer ever needs server-authoritative severity, so deferring it costs nothing and keeps the response equal to `computeAllDiagnostics`'s output verbatim (one fewer transform, one fewer drift point).

**§5 — No client changes; the endpoint's existence alone makes the panel interactive.** The predecessor shipped the panel and the fetch hook pointing at this URL, deliberately degrading to an error readout until the endpoint exists (`useDiagnosticsAtPosition` lines 91–97 comment this explicitly). Once the endpoint returns 200, the panel renders live diagnostics with no client edit. This keeps the task a clean backend leaf and avoids touching the test-mode app. *Alternative — also adjust the panel/hook in this task:* rejected — there is nothing to adjust; the contract was designed against this endpoint up front (ADR 0044), and editing a working client would be churn.

**§6 — 404 (not 403) for invisible sessions, gate-before-read.** Identical to all sibling replay endpoints (existence-leak rule). The visibility gate runs before the event read, so an out-of-range-position 400 is reachable only after the session is confirmed visible and leaks nothing about a private session.

**§7 — `position` is a required query param (no implicit "head" default).** Same single-purpose contract and explicit-contract bias as `get_at_position` Decisions §5: the endpoint answers "diagnostics *at a chosen position*"; omitting `position` cleanly yields a 400 rather than a silent "current head" semantic. The position value is the same event-sequence integer the client emits via the shell position helper (ADR 0043) and that `get_at_position` accepts, so the two replay endpoints agree on every stop.

**§8 — Pass the in-memory `Projection` straight to `computeAllDiagnostics`; do not serialize it.** The detectors' input type is the server `Projection` *class* (they read its iterators), so the projection from `projectAtPosition` is fed directly to `computeAllDiagnostics` with no `serializeProjectionForWire` step — only the diagnostics cross the wire. This is the structural difference from `get_at_position`, which serializes the projection itself; here the projection is purely an intermediate.
