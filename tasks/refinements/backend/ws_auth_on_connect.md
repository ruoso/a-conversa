# Authenticate on connect using session token

**TaskJuggler entry**: [tasks/20-backend.tji](../../20-backend.tji) — task `backend.websocket_protocol.ws_auth_on_connect`
**Effort estimate**: 1d
**Inherited dependencies**: `backend.websocket_protocol.ws_connection_handling` (settled — the WS upgrade path + placeholder hello frame + the marked auth seam in `connection.ts`), `backend.auth.auth_middleware` (settled — `authenticatePlugin`, `authenticateRequest`, `AuthUser`, `AUTH_REQUIRED_CODE` / `AUTH_REQUIRED_MESSAGE`), `backend.auth.session_token_management` (settled — `signSessionToken` / `verifySessionToken` / `readSessionCookieFromHeader` / `SESSION_COOKIE_NAME`).

## What this task is

Gate the `GET /ws` upgrade with the same `aconversa-session` cookie the HTTP auth middleware reads. Three things happen together:

1. **Pre-upgrade reject via `preValidation`.** A Fastify `preValidation` hook runs on the WS upgrade `Request` BEFORE `@fastify/websocket` hijacks the socket. It reads the `Cookie` header, composes the existing `authenticateRequest` primitive from `apps/server/src/auth/middleware.ts` (cookie parse → JWT verify → users-row lookup), and either populates `request.authUser` or throws `ApiError(401, 'auth-required', AUTH_REQUIRED_MESSAGE)`. The centralized error-handler plugin renders the canonical envelope on the upgrade `Response` and the library's `onResponse` hook destroys the socket. The handshake never completes; no `connectionId` is minted; no entry lands in `openConnections`.

2. **Per-connection user shape.** When the upgrade is authenticated, the connection handler copies `request.authUser` onto `WsConnectionContext.user` so downstream tasks (`ws_event_broadcast`, `ws_subscribe_to_session`, etc.) can route messages by user id without a second users-table lookup. `WsConnectionContext.user` is typed `readonly user?: AuthUser` — optional at the type level (matching the `FastifyRequest.authUser?` augmentation in `auth/types.d.ts`) but a runtime invariant: the gate populates it before the handler runs.

3. **Shared `__buildTestWsApp` builder.** The test surface needed a single Fastify-instance builder that exercises the WS plugin's auth gate end-to-end. `connection.ts`'s `__buildTestWsApp(opts)` wires the error-handler plugin, the auth middleware (so the `request.authUser` decorator is allocated), and the WS plugin all against the same memory-backed pool + pinned session-token secret. Both Vitest (`auth.test.ts`, `connection.test.ts`) and Cucumber (`backend-ws-auth.steps.ts`) reach for it, so the WS surface under test is identical across layers.

## Why it needs to be done

The architecture (per [docs/architecture.md](../../../docs/architecture.md)) is server-authoritative and server-broadcast: every state-changing client message rides over a per-user WebSocket and every server-emitted broadcast is routed by user id. Without authentication on the upgrade, the broadcast surface cannot know which user receives a frame and the message-receive surface cannot attribute a proposal / vote / commit to a debater. Every downstream `websocket_protocol` task (`ws_subscribe_to_session`, `ws_propose_message`, `ws_vote_message`, `ws_commit_message`, `ws_meta_disagreement_message`, `ws_snapshot_message`, `ws_error_message`, `ws_event_broadcast`, `ws_proposal_status_broadcast`, `ws_diagnostic_broadcast`, `ws_reconnection_handling`) depends on the per-connection `user` field this task populates.

The HTTP auth middleware (`authenticatePlugin`) already does this for HTTP routes — but `@fastify/websocket`'s upgrade dispatch happens at the route lifecycle level, not via the `preHandler: app.authenticate` opt-in HTTP routes use. The WS surface composes the same primitive (`authenticateRequest`) through a `preValidation` hook attached to the `GET /ws` route.

## Constraints / requirements

- **Reject pre-upgrade, not accept-then-close.** The 401 fires BEFORE the WS handshake completes. The library's `injectWS` (and the browser's native `WebSocket`) sees a non-101 HTTP response and rejects/errors with "Unexpected server response: 401". No `connectionId` is minted; no entry lands in the `openConnections` set; no per-connection state is allocated. Inventing a custom 4xxx close code would diverge from the HTTP middleware's envelope for no semantic gain.
- **Reuse `authenticateRequest` from `auth/middleware.ts`.** The full chain (cookie parse → JWT verify → users-table lookup) is a single composition point. No duplicated cookie parsing, no duplicated JWT verify, no duplicated user lookup. The WS gate calls `authenticateRequest(cookieHeader, pool, secret, now?)` and either populates `request.authUser` or throws.
- **Canonical 401 envelope.** The gate throws `ApiError(401, AUTH_REQUIRED_CODE, AUTH_REQUIRED_MESSAGE)` — the same shape the HTTP middleware throws. The centralized `errorHandlerPlugin` renders the envelope. No custom WS close codes; no inline `reply.send()` in the hook.
- **Cookie-on-upgrade contract: same-origin.** The browser's native `new WebSocket(...)` API sends the session cookie on the upgrade `Request` automatically when the WS endpoint is same-origin to the page (per the WebSocket spec). The browser API does **NOT** permit setting arbitrary headers on a WS upgrade. **Loud reminder for the audience-surface frontend task**: any future cross-origin audience surface (`audience.broadcast_surface`) MUST either be same-origin to the main app or carry a different auth primitive (e.g., a short-lived query-string ticket issued by an authenticated HTTP exchange). Today's deployment is same-origin in production (and in compose dev), so the cookie reaches the upgrade as the gate expects.
- **Per-connection user shape: `{ id, screenName }`.** Identical to `AuthUser` (re-exported from `auth/middleware.ts`'s module-augmented `types.d.ts`). `WsConnectionContext.user` is `readonly user?: AuthUser` — optional at the type level (so downstream tasks' unit tests can construct a context without fabricating a user when their test isn't about auth) and runtime-guaranteed populated inside the connection handler (the gate runs first; the handler narrows defensively and throws if the field is `undefined`, which would indicate a wiring bug).
- **Lazy pool + secret resolution.** Plugin options accept `pool`, `sessionTokenSecret`, and `now` — all optional. Production passes `{}` and the plugin reaches for `getDefaultPool()` + `resolveSessionTokenSecret(process.env)` on the first upgrade. Tests inject everything up front. The no-cookie short-circuit in the `preValidation` hook fires BEFORE the pool resolves so the lazy pool resolution doesn't throw on unauth upgrades in test/dev environments where `DATABASE_URL` may be unset.
- **No ad-hoc probes.** Per ADR 0022, every empirical check is a committed test. The Vitest layer covers the rejection variants (no cookie / tampered signature / expired token) and the success path (cookie verifies, hello frame arrives, per-connection user shape exposed via `__getOpenConnectionsForTests`). The Cucumber layer covers the end-to-end wire path through a real migrated `users` table via pglite.

## Decisions

- **Pre-upgrade reject via `preValidation` (NOT accept-then-close with a 4xxx close code).** `@fastify/websocket`'s upgrade dispatch lets the full Fastify HTTP pipeline run BEFORE the socket is hijacked — `preValidation` throws `ApiError(401, AUTH_REQUIRED_CODE, AUTH_REQUIRED_MESSAGE)`, the centralized error handler writes the canonical envelope on the upgrade response, the library's `onResponse` hook destroys the socket, and the client's `injectWS` (or browser `WebSocket`) sees a non-101 response. The WS handshake never completes; no `connectionId` is minted; no per-connection state is allocated. Rationale for not picking the accept-then-close-with-4xxx path: (1) the library's README explicitly endorses `preValidation` for auth; (2) the HTTP status surface (401) is the same one HTTP routes use, so envelope monitoring + audit pipelines stay consistent; (3) inventing a custom 4401 close code would diverge from the HTTP middleware's envelope for no semantic gain; (4) the no-handshake outcome means no per-connection bookkeeping for rejected upgrades.

- **Cookie-on-upgrade contract + same-origin constraint (LOUD REMINDER FOR audience.broadcast_surface).** The browser's native `new WebSocket(...)` API sends cookies on the upgrade `Request` automatically when the WS endpoint is same-origin to the page (per the WebSocket spec) and does NOT permit setting arbitrary headers on the upgrade. Today's deployment is same-origin in production (the WS endpoint is at `/ws` on the same host the SPA loads from) and in compose dev. **The audience-surface frontend task (`audience.broadcast_surface`) MUST either remain same-origin to the main app OR carry a different auth primitive (a query-string ticket issued by an authenticated HTTP exchange, then redeemed once on the WS upgrade).** The constraint is called out here loudly so a future cross-origin audience UI doesn't silently break the auth contract.

- **Reuse `authenticateRequest` from `auth/middleware.ts` — no duplicated crypto.** The full chain (cookie parse via `readSessionCookieFromHeader` → JWT verify via `verifySessionToken` → users-row lookup with `deleted_at IS NULL`) is a single composition point. The HTTP middleware and the WS gate call the same function with the same arguments and get back the same `AuthUser` shape. The session-token primitive (HS256 via `jose`) lives in exactly one place; a future rotation / algorithm migration touches one module. The auth-middleware module's docblock already documents the WS task's dependency on this export ("WebSocket-handshake authentication — owned by `ws_auth_on_connect`. That task composes the same primitives via the `authenticateRequest` helper this module exports.").

- **Per-connection user shape: `readonly user?: AuthUser` on `WsConnectionContext`.** `AuthUser = { id, screenName }` — exactly the shape the HTTP middleware attaches to `request.authUser`. Identical because downstream WS tasks (`ws_event_broadcast`, `ws_subscribe_to_session`) need both the id (for routing) and the screen name (for human-readable broadcast attribution); a second users-table lookup at message-emit time would be wasteful. Optional at the type level (matches the `FastifyRequest.authUser?` augmentation pattern; lets downstream unit tests build contexts without fabricating a user when their test isn't about auth); runtime-guaranteed populated inside the connection handler (the gate runs first; a defensive narrow at the top of the handler throws if the field is `undefined`, surfacing wiring bugs loudly via the 1011 internal-error path).

- **401 + canonical error envelope as the rejection shape — no custom WS close codes.** `ApiError(401, AUTH_REQUIRED_CODE, AUTH_REQUIRED_MESSAGE)` rendered through the centralized `errorHandlerPlugin`. The wire shape is `{ error: { code: "auth-required", message: "..." } }` — bit-for-bit identical to the 401 a protected HTTP route emits. Operators monitor 401s in one envelope schema; audit pipelines treat all 401s as the same class of event regardless of which transport surfaced them. The accept-then-close-with-4xxx-code alternative would add a second envelope shape (and a second monitoring path) for no semantic gain.

- **`__buildTestWsApp` is the single test-app builder shared across all WS test layers.** A test-only helper (double-underscore prefix marks it test-only) that wires the error-handler plugin, the auth middleware (so `request.authUser` is decorated at the root scope), and the WS plugin all against the same memory-backed pool + pinned session-token secret + optional clock override. Both Vitest (`auth.test.ts`, `connection.test.ts`) and Cucumber (`backend-ws-auth.steps.ts`, `backend-ws-envelope.steps.ts`, `backend-ws-connection.steps.ts`) reach for it. Standardising on one builder means the WS surface under test is bit-identical across layers — no "memory-pool in unit, pglite in integration, real Postgres in e2e" drift in plugin wiring.

- **No-cookie short-circuit before pool resolution.** The `preValidation` hook checks `cookieHeader === undefined || cookieHeader === ''` BEFORE calling `auth.ensurePool()`. Rationale: `getDefaultPool()` throws when `DATABASE_URL` is unset (per `db.ts`); test/dev environments that don't set the env var would otherwise trip a confusing "DATABASE_URL is not set" error on the no-cookie path when the right error is 401. The cookie-present-but-invalid path (tampered signature, expired token, missing users row) still resolves the pool — that's the right cost-benefit (the cookie path needs the users-row lookup; the no-cookie path doesn't).

- **`hasRequestDecorator('authUser')` guard in the route plugin.** The auth middleware (`authenticatePlugin`) and this WS plugin both call `app.decorateRequest('authUser', undefined)` to pre-allocate the slot. Fastify throws on duplicate decoration; the WS plugin's defensive `if (!app.hasRequestDecorator('authUser')) app.decorateRequest(...)` keeps the two plugins compose-able in either order (production registers both; standalone WS tests may register just one). The same idempotence pattern the auth middleware itself uses.

- **WS test-app uses `'unit-test-ws-secret'`; cucumber test-app uses `'test-session-secret'`.** Two different cucumber-vs-unit secrets — neither leaks across the layer boundary because each layer's app builder is wired with its own. The Vitest tests pin `TEST_SESSION_SECRET = 'unit-test-ws-secret'` (from `apps/server/src/ws/test-helpers.ts`); the cucumber `backend-ws-auth.steps.ts` pins `'test-session-secret'` (aligning with the existing cucumber-layer secret in `backend-session-token.steps.ts` and friends). The unit tests don't share their secret with the cucumber layer; the cucumber layer doesn't share its secret with production. Both are pinned strings; neither pulls from `process.env`.

- **The 3 existing `ws-connection.feature` scenarios were migrated onto the auth-gated test app.** Pre-`ws_auth_on_connect`, those scenarios drove `createServer()` directly (no auth required to upgrade). Post-`ws_auth_on_connect`, the same scenarios would fail with HTTP 401. The minimal fix is to share the `wsAuthApp` + `wsAuthCookie` carriers with the new `backend-ws-auth.steps.ts` and gate all three ws-connection scenarios on a Background that builds the auth app + a user + a cookie. The `Then` assertion for the hello frame was also updated to read the canonical envelope shape (`parsed.payload.connectionId`) — the envelope task's hello migration was incompletely propagated into the cucumber assertions during the parallel-stream work; this task closes that gap.
