The seed contract itself does NOT change here. The server-side handler-integration tests stay the source of truth for "the server emits N per-component seed envelopes in the right order with the right payload shape"; this task pins "a real participant browser, after losing its WS connection mid-decompose, ends up with the same per-component pane styling it had before the kill."

## Why it needs to be done

The participant migration's Acceptance Criteria §137–138 explicitly scoped the reconnect sub-step as conditional on the moderator-side harness shim landing first. The harness landed on `main` 2026-05-29; the participant's reconnect-seed-visible-styling slice is now the unpinned end-to-end coverage for the participant pane's broadcast-derived rendering path.

A regression here is silent. The handler-integration tests will keep passing (the server-side seed math is unchanged). The shell-store Vitest cases will keep passing (the store's `applyProposalStatus` per-entity write is unchanged). The participant-pane Vitest cases will keep passing (`PendingProposalsPane`'s broadcast-merge-over-events logic at [`apps/participant/src/proposals/PendingProposalsPane.tsx`](../../../apps/participant/src/proposals/PendingProposalsPane.tsx) is unchanged). The visible pane styling would only break if a future change to envelope ordering, the snapshot/seed dispatch sequence, or the per-entity-cell write timing introduced a race where the first render after `snapshot-state` flushes BEFORE the seed envelopes apply. The moderator-side e2e pins this for the moderator canvas; this task pins the symmetric invariant for the participant pane — without it, the participant could drift to a regression the moderator's e2e wouldn't catch (e.g., a future surface-specific selector at the participant pane that observed the empty intermediate state while the canvas-projector recovered cleanly).

The cross-surface shape (moderator context drives the propose; participant context observes + kills) also pins **a cross-surface invariant the moderator spec cannot pin alone**: that the server's broadcast to a second-surface subscriber (the participant) carries the per-component seed envelopes on a reconnect-initiated catch-up, with the same ordering the moderator-side test pins. The moderator's Scenario 3 reconnects the SAME context that did the propose, so the seed envelopes only need to round-trip the same connection; this task pins the harder case where the propose came from a DIFFERENT connection that's still subscribed in the steady state.

Finally — the install pattern lifted here is the second of three (moderator landed first; audience is the third per the moderator's D7 "each surface installs its own when it scopes reconnect e2e"). Establishing the participant install now sets the convention for the audience-surface follow-up; the install pattern is identical across surfaces (sibling `useEffect` in the route component that owns the `useWsClient()` resolution), differing only in the route file path.

## Constraints / requirements

- **Install lives at top-level `OperateRoute`, not at a nested inner-route component.** Per D1 — the participant's `<WsClientProvider>` is at `main.tsx`, one level above `OperateRoute`; `useWsClient()` already resolves at the top-level route body. The moderator's `OperateRouteInner` split exists because the moderator's provider mounts inside its `OperateRoute`; that pattern does NOT carry over here and forcing it would add an unnecessary component split.
- **Install effect's dep array is `[client]`.** Per the moderator pattern. The effect does NOT depend on the route param (`id`); the install fires once per client identity. (`client` identity is reducer-stable per `<WsClientProvider>`'s memoization; the install runs once per route mount in practice.)
- **Cleanup deletes only the `killWebSocket` property, not the whole `__testHooks` object.** Per the moderator pattern at [`Operate.tsx:200–202`](../../../apps/moderator/src/routes/Operate.tsx). Future hooks landing under the same namespace (`forceEmitEnvelope`, `drainPending`) survive the unmount; only the property this route owns is removed.
- **Install effect is NOT gated on `import.meta.env.DEV`.** Per moderator D1 — the production-mode build tree-shakes a `DEV`-gated branch and the e2e under `make up-prod-mode` would lose its entry point. Same rationale carries over verbatim.
- **No new shell-level code.** Per D6 — the `WsClient.killWebSocket()` method is already on the interface (`packages/shell/src/ws/client.ts`); this task does NOT modify any shell-package source. The shell-side Vitest cover lives in `packages/shell/src/ws/ws-client.test.ts` and was landed by `mod_pw_reconnect_seed_visible_styling` — UNCHANGED here.
- **No audience-surface install in this task.** Per moderator D7 — each surface owns its own install when it scopes reconnect e2e. The audience surface has no pending-proposal e2e today; when it adds one, it installs the hook on its own route at that time.
- **Cross-surface spec is in a NEW file, not appended to `participant-pending-proposals.spec.ts`.** Per D2 + D3 — the existing participant spec is client-seeded via `__aConversaWsStore.applyProposalStatus`; the reconnect cycle would clear that client-only state since the server has no real proposal record. Mixing the two seeding paradigms in one test would be confusing; the new file mirrors the moderator's "one concern per spec file" convention.
- **Real-server propose is required for the reconnect sub-step.** Per D3 — `applyProposalStatus` client-side seeding survives a `__aConversaWsStore` write but not a WS reconnect (the catch-up handler re-emits state from server projection, which has no record of the client-only seed). The cross-surface test routes through the moderator's `enter-decompose-mode → fill-rows → propose-decomposition` UI flow so the server holds the actual decompose proposal at reconnect time.
- **Flash-to-undefined watcher mirrors the moderator's shape.** Per D5 — `expect.poll` at 50ms sample period, 2.5s window, against the participant pane's per-component chip testids. Reject if any sampled iteration observes `null` / `''` / `'undefined'` / `'awaiting-proposal'`.
- **Per ADR 0022 test cover.** Install lifecycle → Vitest. Browser-observable reconnect-seed contract → Playwright. No Cucumber scenario added (no wire-shape change; the broadcast-handler tests at `apps/server/src/ws/broadcast/proposal-status.test.ts` already pin the envelope shape; the handler-integration tests at `apps/server/src/ws/handlers/{snapshot,catch-up}.test.ts` already pin the seed-emit-on-reconnect contract).

## Decisions

- **D1 — Install lives at the top-level `OperateRoute` body (NOT in a nested inner component, NOT at `main.tsx`).** Unlike the moderator route's `OperateRouteInner` split, the participant's `OperateRoute` already calls `useWsClient()` at its top-level body ([`OperateRoute.tsx:139`](../../../apps/participant/src/routes/OperateRoute.tsx)) because the participant's `<WsClientProvider>` is mounted one level above the route at [`main.tsx:77`](../../../apps/participant/src/main.tsx). The install effect lands as a sibling `useEffect(() => { … }, [client])` immediately after the existing `trackSession` effect at L147–153. Rationale:
  - **Mirrors the moderator's lifecycle-correct install without unnecessary component fragmentation.** The moderator needed `OperateRouteInner` because its `<WsClientProvider>` mounts INSIDE the route; that constraint does not apply at the participant. Forcing a wrapper split here would add a component layer with no behavioral benefit and would diverge from the participant route's existing shape.
  - **Co-locates the install with the per-session lifecycle effect.** The reader sees two route-level effects side by side: one for session subscription tracking, one for the test-seam install. Both have the same lifecycle contract (mount → register, unmount → cleanup) — which is the right mental model.
  - **Cleanup contract is identical to the moderator.** `return () => { if (w.__testHooks !== undefined) delete w.__testHooks.killWebSocket; }`. The namespace object survives if other hooks land later; only the property this route owns is removed.
  - **Alternatives considered:**
    - **(B) Move the install into `OperateRouteBody` or `OperateRouteAuthenticatedBody`.** Rejected — those components run conditionally on auth status; the not-authenticated branch would skip the install. The route-level effect always fires once per mount, which is the right scope (the auth gate is a render branch, not a WS-client-availability gate; `useWsClient()` resolves at the route level whether or not the auth-guard render branch is the one that produces children).
    - **(C) Install at `apps/participant/src/main.tsx` alongside the `__aConversaWsStore` exposure.** Rejected — the client doesn't exist at mount.tsx time (it's created inside `<WsClientProvider>`'s init); the install would have to capture a lazy getter through the provider's children. Mirrors moderator D3's rejection of the same alternative.
    - **(D) Add a participant-local `<TestHooksInstaller client={…} />` component rendered inside `<OperateRoute>`.** Rejected — extra component for one `useEffect`; the inline effect is simpler.

- **D2 — Cross-surface Playwright spec lands in a NEW file (`tests/e2e/participant-reconnect-seed-visible-styling.spec.ts`), NOT appended to `participant-pending-proposals.spec.ts`.** The existing participant pending-proposals spec at [`tests/e2e/participant-pending-proposals.spec.ts`](../../../tests/e2e/participant-pending-proposals.spec.ts) is intentionally client-seeded via `__aConversaWsStore.applyProposalStatus` (per its docstring at L14–18: "this spec only pins the UI consumer"). Mixing a real-server-propose flow into the same file would either invalidate the existing client-seed approach or require two parallel seed paradigms in one test. Rationale:
  - **One concern per spec file matches the moderator convention.** The moderator side has `moderator-proposed-entity-canvas-visibility.spec.ts` (canvas styling + reconnect sub-step) as ONE file with multiple scenarios; the alternative would have been multiple smaller files. This task lands a separate file because the cross-surface setup is materially different from the existing participant spec's setup (two browser contexts, real-server propose, vs. one context with client seed) — they're not naturally the same fixture chain.
  - **Naming mirrors the moderator's pattern.** `moderator-proposed-entity-canvas-visibility.spec.ts` → `participant-reconnect-seed-visible-styling.spec.ts`. Both files concern the reconnect-seed contract at their respective surface.
  - **Future participant reconnect e2e scenarios** (e.g., vote-mid-flight reconnect, withdraw-mid-flight reconnect) compose into the same file as additional `test()`s, keeping a single home for the participant-side reconnect family.
  - **Alternatives considered:**
    - **(B) Add a new `test()` inside `participant-pending-proposals.spec.ts` with its own cross-surface fixture.** Rejected — the file's docstring explicitly scopes it to the UI-consumer path; mixing in a real-server-propose test would muddy the file's purpose and double its size. The cross-surface fixture is also heavy enough to warrant its own home.
    - **(C) Mutate the existing test to support a kill-and-reassert tail via client-only seeding (skip the real-server propose).** Rejected per D3 — a WS reconnect re-syncs from server projection; client-only seed is wiped on reconnect.
    - **(D) Land the reconnect test as a new `test.describe` block inside the moderator spec file (since the moderator does the propose).** Rejected — the file's responsibility is moderator-canvas styling; adding a participant-canvas assertion would muddle ownership and inflate runtime.

- **D3 — Cross-surface spec uses a REAL moderator-side decompose propose (NOT a client-side `applyProposalStatus` seed at the participant context).** A real WS reconnect drives the server's catch-up handler which re-emits state from server projection; a client-only seed is invisible to the server and would not be re-broadcast as a seed envelope. The cross-surface flow has the moderator browser context perform `enter-decompose-mode → fill-rows → propose-decomposition` against the live server; the resulting `proposal` event lands in the projection, and the participant's reconnect catch-up sees the per-component seed envelopes the server emits. Rationale:
  - **Pins the actual seed-on-reconnect contract.** The whole point of the test is "after the participant reconnects, do the seed envelopes from the catch-up handler rebuild the participant pane's per-component chips correctly?" That requires real seed envelopes, which require a real server-side proposal.
  - **Cross-surface shape matches existing precedent.** `tests/e2e/cross-surface-lobby-start.spec.ts` and `tests/e2e/cross-surface-debater-claim.spec.ts` establish the dual-context pattern; both run under the same Playwright project and compose-stack.
  - **Reuses the moderator route's existing real-propose path.** No new moderator-side code, no new helper. The propose chain already exists in `moderator-proposed-entity-canvas-visibility.spec.ts` Scenario 3; the participant spec can lift its setup section (extract a shared helper later if a third reconnect spec needs it; YAGNI for now).
  - **Alternatives considered:**
    - **(B) Use `applyProposalStatus` client-seed on the participant AND `applyEvent` to seed a `proposal` event mirroring what server projection would carry; bypass the kill+reconnect.** Rejected — this doesn't test reconnect at all; it just tests the merge-precedence between broadcast and events cells, which is already pinned by Vitest at `PendingProposalsPane.test.tsx`. The whole value of this task is the reconnect cycle.
    - **(C) Add a server-side test seam (`POST /api/dev/seed-proposal` style) that creates a proposal without going through the moderator UI.** Rejected — adds a new production-reachable API surface for one test; the cross-surface flow via real UI is closer to production and more discoverable. The moderator-reach-operate + propose chain is already proven in Scenario 3; reusing it adds ~30s to the new spec but no new infrastructure.
    - **(D) Run the moderator browser context in the SAME `BrowserContext` as the participant (open two pages in one context).** Rejected — the auth-cookie state would collide; one context can only be authenticated as one user at a time (per the existing `logoutAndClearAllCookies` precedent in `participant-pending-proposals.spec.ts:43–47`).

- **D4 — Pre-reconnect assertion uses the participant pane's per-facet chip testids (NOT the per-component chip testids).** The per-component chip surface is a separate concern owned by the sibling tech-debt task `part_pw_multi_component_decompose_per_component_breakdown` (per the participant migration's Status block L218). This task's headline pre-reconnect assertion is the simpler "the row's per-facet chip strip shows `data-facet-status="proposed"`", which is correct regardless of whether the per-component breakdown UI has landed. Rationale:
  - **Demarcates two distinct e2e concerns cleanly.** This task pins the reconnect-seed cycle (does the broadcast-derived cell map repopulate correctly?). The sibling pins the per-component rendering surface (does the pane render N component chips for an N-component decompose?). Both can ship independently without blocking each other.
  - **Survives if `part_pw_multi_component_decompose_per_component_breakdown` lands first OR last.** If the per-component breakdown UI has already landed when this task ships, the assertion `chips.count() ≥ 1 && all data-facet-status="proposed"` still passes (and the sibling's test catches the per-component count). If the per-component UI hasn't landed yet, the assertion still passes against the current per-facet chip strip.
  - **Picks the smallest assertion that catches the reconnect regression.** The regression we're guarding against is "the chip's `data-facet-status` flips to null / undefined during the catch-up window" — that surfaces on ANY per-facet chip, not just per-component ones. Asserting on the simpler chip set keeps the test resilient to evolution of the per-component surface.
  - **Alternatives considered:**
    - **(B) Defer this task until per-component chips land, then assert per-component.** Rejected — couples two independent concerns; the participant migration already shipped both Vitest covers (per-facet AND per-component) separately, so the e2e can split the same way.
    - **(C) Assert on both per-facet AND per-component chips, treating the per-component branch as conditional (`if locator exists`).** Rejected — conditional assertions are a code smell in Playwright; the test would silently pass even if the per-component branch returned the wrong status. Keep the assertions hard-pinned.

- **D5 — Flash-to-undefined watcher mirrors the moderator's `expect.poll` shape (50ms intervals × 40 samples = 2.5s window) verbatim.** Same rationale as moderator D6: catches a sub-100ms regression where the snapshot applies before the seed envelopes. The watcher captures the chip testids pre-kill, then samples each via `document.querySelector` + `getAttribute('data-facet-status')` and rejects if any sampled value is `null` / `''` / `'undefined'` / `'awaiting-proposal'`. Rationale:
  - **Consistency with the moderator spec keeps the watcher's failure mode interpretable.** A reader who's seen the moderator failure-mode shape (`expected 0, received N bad statuses`) reads the participant failure the same way.
  - **2.5s window is empirically sufficient.** The moderator-side experience showed the catch-up sliver is sub-100ms in steady state; a 2.5s window with 50ms samples = 40 samples is well above the threshold needed to catch even a 500ms regression. (The watcher runs AFTER the headline assertion has succeeded; we're not waiting for the chip to appear, we're confirming no future render tick regresses.)
  - **Alternatives considered:**
    - **(B) Run the watcher BEFORE the headline assertion (during the reconnect window itself).** Rejected — `expect.poll` with mid-window assertion semantics is harder to reason about; the moderator pattern runs the watcher after `waitForFunction(connectionStatus === 'open')` to confirm steady state holds.
    - **(C) Use `page.exposeBinding` + `MutationObserver` for nanosecond-precision sampling.** Rejected per moderator D6 — overkill for an attribute that mutates at React's render rate; `expect.poll` at 50ms is the Playwright-idiomatic shape and captures the failure mode at the right resolution.

- **D6 — No shell-level code changes in this task.** The `WsClient.killWebSocket()` method, the shell-side Vitest cover (`packages/shell/src/ws/ws-client.test.ts`), and the moderator-side `__testHooks` install were ALL landed by `mod_pw_reconnect_seed_visible_styling` (commit `ecb70df`). This task only adds (a) participant-route install code, (b) participant-route install Vitest cover, (c) cross-surface Playwright spec. Rationale:
  - **The shell is the shared substrate; per-surface installs are surface concerns.** The shell exposing the kill method is enough; the install at each surface is independent. (Mirrors moderator D7.)
  - **Avoids re-asserting invariants the moderator task already pinned.** The natural-reconnect-path mechanics, the rejection-of-pending-promises, the status-vs-`close()` contrast — all pinned by `packages/shell/src/ws/ws-client.test.ts`. Re-asserting them at the participant level would be a throwaway verification per ADR 0022.
  - **Alternatives considered:**
    - **(B) Add participant-surface assertions for the kill mechanics (e.g., the participant's WS-client status transitions to `'reconnecting'`).** Rejected — that's the shell's contract, not the participant's; pinning at the surface level couples the participant test to shell internals.

- **D7 — Audience-surface install is explicitly deferred to a future task (NOT scoped here).** Per moderator D7 — each surface installs its own when it scopes reconnect e2e. The audience surface has no pending-proposal e2e today; when an audience refinement scopes a reconnect test, that task gains its own install effect. No new WBS task is registered in this refinement (the audience reconnect work is downstream enough that scoping it pre-emptively would be premature; the existing convention — "wire-task inherits prior deferred e2e" per the UI-stream e2e policy — handles it organically when audience reconnect work surfaces). Rationale:
  - **Minimum production-reachable test-seam surface, scoped to actual test coverage.** Mirrors moderator D7 verbatim.
  - **Composes cleanly when the audience reconnect work eventually surfaces.** The pattern lifted to the participant is the same one the audience will lift later: sibling `useEffect` in the route component that owns `useWsClient()`.
  - **No new WBS leaf needed today.** The audience tasks already in the WBS will pick up the install requirement as part of their own reconnect e2e scope, following the inherited-debt convention.
