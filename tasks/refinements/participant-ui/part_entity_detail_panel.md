## What this task is

Render a React-driven **right sidebar panel** on the participant's `/p/sessions/:id` operate route that subscribes to `useSelectionStore((s) => s.selected)` and renders per-entity drill-down detail for whatever node or edge the debater has tapped — closing the `part_graph_view` subgroup by absorbing the per-facet / per-axiom / per-annotation / per-diagnostic / per-vote detail features the prior overlay leaves deferred to this task.

Before this leaf: tapping a node / edge writes the selection to `useSelectionStore` (per `part_pan_zoom_tap`) and the Cytoscape `:selected` branch paints a subtle in-canvas highlight + the DOM mirror stamps `data-selected="true"` — but there is NO surface that renders the per-facet status breakdown, the annotation content text, the diagnostic title prose, the per-facet own-vote breakdown, the per-voter attribution table, or the axiom-mark per-participant attribution. Every prior overlay leaf carries those features in its Out-of-scope list with this task named as the home; the methodology assumes the debater can drill into an entity to read its per-facet detail, not just scan the at-a-glance rollup signals on the canvas.

After this leaf: the participant operate route renders the `<EntityDetailPanel>` as a fixed-width right sidebar (Decision §1) alongside the existing `<GraphView>`; the panel subscribes to `useSelectionStore` and, when a node or edge is selected, renders:

1. **Identity header** — entity kind (`node` / `edge`), the wording text (or role label for edges), the entity id (for debugging / cross-reference).
2. **Rollup status badge** — the per-entity rollup (`proposed` / `agreed` / `disputed` / `meta-disagreement` / `committed` / `withdrawn`) using the SAME `PILL_BASE_CLASSNAME` + `PILL_STATUS_CLASSNAME` palette the moderator's `<FacetPill>` carries (imported from [`apps/moderator/src/graph/FacetPill.tsx`](../../../apps/moderator/src/graph/FacetPill.tsx) per Decision §6).
3. **Per-facet pill row** — one `<FacetPill>` per applicable facet (nodes carry `classification`, `substance`, `wording`; edges carry `substance`) — directly mirroring the moderator's in-card pill row that the participant Cytoscape canvas could not host. This satisfies the `part_per_facet_state_styling` Decision §6 deferral.
4. **Axiom-mark attribution** — when `nodeHasAxiomMark(...)` is true for a node, render a "Bedrock for: [participant-A], [participant-B]" row using `methodology.axiomMark.label` + per-participant screen-name resolution. Per Decision §5 — the bucketed `Map<nodeId, readonly AxiomMark[]>` is already exposed by `axiomMarks.ts`; this leaf reads it.
5. **Annotations list** — when `hasAnnotation` is true, render one row per annotation with the per-kind localized label (`t('methodology.annotationKind.<kind>')`) + the verbatim `content` string + the per-author screen name (resolved via the same roster) + a relative `createdAt`. Mirrors the moderator's `<AnnotationBadge>` row vocabulary — see Decision §6 on the import path.
6. **Diagnostic messages** — when `diagnosticHighlight` is non-null, render one row per active kind with the per-kind localized title + description using the existing `diagnostics.<kind>.title` / `.description` catalog keys (populated for en-US / pt-BR / es-419 per `part_diagnostic_highlights` Inputs). A "(blocking)" / "(advisory)" badge inline so the debater sees severity without scanning the canvas.
7. **Own vote summary** — when the debater has voted on any facet of the selected entity, render a per-facet own-vote breakdown ("Classification: you voted agree; Wording: you voted dispute"). Reads from the `OwnVoteIndex` already derived by `<GraphView>` (Decision §2 — the same index hoists from `<GraphView>` to `<OperateRoute>` so both surfaces share the memo).
8. **Other voters' summary** — render a per-voter table (one row per `OtherVote` in `data.otherVotes`) with the voter's resolved screen name + their per-entity rollup arm (`agree` / `dispute`). Decision §5 explains the screen-name resolution.
9. **Empty / unknown state** — when `selected === null`, render an empty-state body with a localized "Tap a node or edge to see its detail" message. When the selected id no longer corresponds to a current element (e.g. the entity has been removed by a downstream wire event, or the wording rolled back to dangling-edge filter), render an "(element no longer present)" message AND auto-clear the selection on the next render via a `useEffect`. Decision §10 explains.

The panel is **read-only**. The methodology layer's editing affordances (vote-cast, annotation-create, axiom-mark, propose, commit, withdraw) are deliberately NOT in this leaf — every action is owned by a downstream `participant_ui.*` task (`part_voting.*`, `part_axiom_mark_from_tablet.*`, etc.). The panel surfaces the data; the future-leaves attach actions to the same panel surface (Decision §11 explains the action-slot stub).

## Why it needs to be done

`m_participant_mvp` ([`tasks/99-milestones.tji`](../../99-milestones.tji)) is the milestone at which a debater can see AND ENGAGE with the live graph from their tablet. The `part_graph_view` subgroup has, through 11 prior leaves, lit up every visual vocabulary the canvas can host: baseline rendering, per-facet status, per-facet per-facet styling, axiom marks, annotations, diagnostic highlights, own votes, other votes, pan/zoom, tap-to-select. Each prior leaf explicitly deferred its drill-down detail surface to "the future entity detail panel" — this task IS that panel; closing it closes the entire `part_graph_view` subgroup.

The methodology cannot complete its agreement loop on the participant surface without the detail panel:

- [`docs/methodology.md`](../../../docs/methodology.md) — "Agreement is per-facet and per-participant"; the debater needs to see EACH facet's state independently, not just the per-entity rollup the canvas paints. Without the per-facet pill row, the debater can read "this entity is disputed" but cannot answer "WHICH facet is disputed?" — a load-bearing methodological question that drives the next vote/amend/withdraw gesture.
- [`docs/methodology.md`](../../../docs/methodology.md) "Structural diagnostics" + "Annotations" — both the diagnostic kind titles AND the annotation content text are first-class methodological information. The canvas at-a-glance signal carries presence/severity/count; the localized prose lives in the panel (the i18n keys are already populated by the moderator-side leaves for exactly this surface).
- [`DESIGN.md`](../../../DESIGN.md) "Agreement-driven progress" — the debater's per-facet decisions (vote / withdraw / amend) are what move the methodology forward; the panel is the debater's only surface that exposes the per-facet state without context-switching to a future pending-proposals pane.

Architecturally, closing `part_graph_view` is the gating step for the rest of `m_participant_mvp`. The next milestone-critical groups (`part_pending_proposals.*`, `part_voting.*`, `part_axiom_mark_from_tablet.*`, `part_session_recovery.*`) all assume the operate route's read-mostly surface is complete; this leaf signs that completion.

Downstream concretely:

- **`part_voting.part_vote_button_per_facet`** (future P2 leaf) attaches per-facet vote buttons to this panel's per-facet pill row — the `<FacetPill>` row is the natural anchor for the per-facet vote action (the moderator's surface uses the same pattern). Decision §11 reserves an action-slot prop on the panel for the future leaf to fill without re-shaping the panel.
- **`part_axiom_mark_from_tablet.part_mark_axiom_action`** (future leaf) attaches an "axiom-mark this node" action to the panel's identity header (similar to how the moderator's axiom-mark submenu attaches to its selection store). Same action-slot mechanism.
- **`audience.aud_entity_detail_panel`** (future, sibling to `aud_graph_render`) becomes the third Cytoscape consumer if it also needs a per-entity detail surface — the panel component this leaf ships becomes a natural shell-extraction candidate at that point. Two-callers-is-YAGNI today (`part_per_facet_state_styling` Decision §2 precedent).
- **`part_session_recovery.*`** (future) needs the panel to handle entity ids that no longer match (stale selection after a snapshot reload) — Decision §10's stale-entity branch already handles that path; no refactor needed when recovery lands.

## Decisions

### §1 — Fixed-width right sidebar layout (Tailwind `w-80`); panel always present with empty-state body when nothing is selected

Four layout options were named in the orchestrator brief; settled against this leaf's constraints:

- **(A) Right sidebar (fixed-width panel docked to the right of the graph).** Chosen. Tailwind `w-80` (320px) — wide enough to host the per-facet pill row + the annotations list + the per-voter table without horizontal scrolling, narrow enough to leave the canvas region as the dominant surface (the participant's primary mode is "read the graph"). Mirrors the moderator's `<RightSidebar>` posture for cross-surface familiarity (the moderator has a stacked pending-proposals / diagnostic-flags / change-history sidebar; the participant has a per-entity detail sidebar — different content, same screen-side anchor). The sidebar scrolls vertically within its column when content exceeds viewport height (`overflow-y-auto` on the panel root).
- **(B) Bottom drawer (slides up from the bottom).** Rejected for v0: the participant is desktop-first AND tablet-second (`docs/participant-ui.md` P1 + P2 — view the graph + single-tap votes; no mobile-phone profile in the WBS); a bottom drawer optimizes for narrow viewports the participant deliberately doesn't target. If a future leaf adds a mobile-phone form factor, swapping the sidebar for a drawer at a Tailwind breakpoint is a cosmetic change that does NOT reshape the panel's data flow.
- **(C) Floating popover near the selected element.** Rejected: would compose poorly with the canvas-dot overlay from `part_other_vote_indicators_canvas_dots` (Decision §3 of that leaf placed the dots below the node bounding box; a popover near the same element would either occlude the dots or push them further off-canvas). The fixed sidebar keeps the canvas region uncluttered.
- **(D) Modal.** Rejected: disruptive — the debater would lose the at-a-glance view while reading the per-facet detail. The methodology assumes panoramic + per-entity drill-down composed, not modal-context-switched.

**Sub-decision (1c) — always-present-with-empty-state vs slide-in-on-select:** chosen always-present-with-empty-state for two reasons: (i) layout stability — the canvas region's width doesn't reflow when the debater taps an element; (ii) discoverability — a debater who hasn't yet tapped sees the empty-state body explaining the gesture, instead of needing to discover the panel by trial. The simpler default; slide-in animation is a future polish leaf if real usage shows the always-present surface is too noisy.

### §2 — Hoist the projection chain from `<GraphView>` to `<OperateRoute>`; both children receive the projection outputs as props

The prior 11 leaves accumulated the projection chain inside `<GraphView>` (each leaf added one or two `useMemo`s + threaded one or two new indexes into `projectGraph`). The panel needs access to the SAME projection outputs (per-facet status, axiom marks, annotations, diagnostic highlights, own votes, other votes, plus the projected node + edge lists). Three options:

- **(a) Hoist the projection chain to `<OperateRoute>`; both `<GraphView>` and `<EntityDetailPanel>` receive the projection outputs as props.** Chosen. The route runs each projection ONCE per `events` change; both children re-render with the same outputs. The projection helpers are pure (per the prior leaves' acceptance criteria); calling them at the route level instead of the component level changes WHERE the memos live but not WHAT they compute. The hoist is mechanical — the existing `useMemo` calls move up one component layer; the `<GraphView>` consumes prop reads where it previously consumed internal memo reads.
- **(b) Keep the projection chain in `<GraphView>`; re-derive in `<EntityDetailPanel>`.** Rejected: the projection helpers walk the entire events log; re-deriving in the panel DOUBLES the per-`events`-change computation cost. Pure functions are cheap but not free — the prior leaves' acceptance criteria list ~10 projections × O(n) events; running each twice burns N memo calls per `events` change for no behavioural benefit. The hoist option computes once.
- **(c) Create a shared `useEntityProjection` hook that memoizes the chain externally to both components.** Mechanically equivalent to (a) but adds a new abstraction for two call sites. Rejected: two callers is YAGNI; the hoist option uses the existing route-level `useMemo` idiom without inventing a new shape. If a third caller materializes (audience-side or a future debug panel), promoting the hoist's body to a custom hook is one mechanical refactor away.

Decision §2: ship (a). The `<OperateRoute>` body grows from one component-render line to a memo-then-render block; the children's APIs widen with the projection-output props. The Cytoscape mount + sync flow inside `<GraphView>` is unchanged (it consumes the same shape, just from a prop instead of an internal memo).

A consequence of (a): `<GraphView>`'s public API widens by ~9 props. The widening is mechanical and aligned with the existing prop-shape conventions (the component already takes `sessionId` + `currentParticipantId`; adding the projection outputs follows the same "explicit dependencies, no hidden derivations" posture per `part_graph_render` Decision §5).

### §3 — Panel subscribes to `useSelectionStore` directly (bottom-up state read); selection-related data is store-state, not prop-thread

The panel needs to know the currently-selected entity. Three options:

- **(a) Subscribe to `useSelectionStore((s) => s.selected)` directly inside `<EntityDetailPanel>`.** Chosen. Mirrors the moderator's `<CaptureTargetChip>` precedent (the moderator component also subscribes directly via `useSelectionStore`; see [`apps/moderator/src/layout/CaptureTargetChip.tsx`](../../../apps/moderator/src/layout/CaptureTargetChip.tsx)). The selection store is a global Zustand slice — every reader subscribes to it directly without thread-passing.
- **(b) Thread the `selected` value as a prop from `<OperateRoute>`.** Rejected: `<OperateRoute>` would need to subscribe to the store, then re-render the panel on every selection change. Same React-render cost as (a) but with the prop-thread overhead AND the loss of cross-component subscription clarity (the panel is the panel; the route doesn't need to know).
- **(c) Compute the selected ENTITY (not just the selection) at the route level and thread it as a prop.** Rejected: same issue as (b) — the route gains a responsibility (resolving the selection to an entity) that lives more naturally in the panel (the panel is the entity consumer; the route is the layout composer).

Decision §3: ship (a). The panel imports `useSelectionStore` from `../stores/selectionStore`; the route stays out of the selection-state plumbing.

A sub-decision: the panel's discrimination on `selection.kind`. The store's `EntityKind` is `'node' | 'edge' | 'annotation'` (per `entityKindSchema`), but the tap handler in `part_pan_zoom_tap` only writes `'node'` and `'edge'` today (annotations are not tap-selectable in the current canvas). The panel renders the empty-state body when `selection.kind === 'annotation'` (the reserved-for-future arm). The future annotation-tap surface (whenever it lands) extends the panel to render annotation detail; this leaf reserves the arm without implementing it.

### §4 — `<FacetPill>` import from the moderator workspace; reuses `PILL_BASE_CLASSNAME` + `PILL_STATUS_CLASSNAME` + the `<FacetPill>` component itself

The per-facet pill row + the rollup-status badge both need the moderator's pill vocabulary (border style, color, opacity per status). Three options:

- **(a) Import `<FacetPill>` + `PILL_*_CLASSNAME` from the moderator workspace.** Chosen. Precedent: [`apps/moderator/src/layout/ProposalFacetBreakdown.tsx:43`](../../../apps/moderator/src/layout/ProposalFacetBreakdown.tsx#L43) already does the same import within the moderator. The participant-side import crosses workspace boundaries, but the workspaces share the build (pnpm monorepo, single TypeScript project per ADR 0013); the cross-workspace import is mechanically a normal module import.
- **(b) Duplicate the `<FacetPill>` source into the participant workspace.** Rejected: same YAGNI / duplication-cost argument the prior leaves vetted for the projection helpers. The pill component is small (~50 lines); duplicating burns no obvious cost, but the cross-workspace import is precedented + simpler.
- **(c) Extract `<FacetPill>` into `@a-conversa/shell` before this leaf ships.** Rejected for the same trigger-on-the-third-caller reason: the participant is the second caller; the audience surface is the natural third trigger. Extraction now would shape the shell-export around the participant's needs which may differ from the audience's.

Decision §4: ship (a) — cross-workspace import. The participant's `EntityDetailPanel.tsx` imports `import { FacetPill, PILL_BASE_CLASSNAME, PILL_STATUS_CLASSNAME } from '@a-conversa/moderator/graph/FacetPill'` (or the equivalent relative path the build resolves). A note in the panel's header documents the cross-workspace import + cites the moderator's `ProposalFacetBreakdown.tsx:43` precedent.

If real usage shows the cross-workspace import causes build issues (it shouldn't — the moderator is a peer workspace, not a downstream consumer), the duplication option (b) is the fallback; the swap is mechanical.

### §5 — `participantRoster.ts`: walk `participant-joined` / `participant-left` from the events slice; remove left participants from the resolver (no historical retention)

The panel needs to resolve UUIDs to display names for the per-axiom-mark attribution, the per-annotation author, the per-other-voter table, AND (for completeness) any future per-vote / per-action surface. Three options for where the resolver lives:

- **(a) Walk `participant-joined` / `participant-left` from the per-session events slice; build a `Map<userId, screenName>` resolver.** Chosen. The events slice is already loaded for every other projection helper; reading two more event kinds is incremental. The pattern is precedented by [`apps/participant/src/routes/LobbyRoute.tsx:117-165`](../../../apps/participant/src/routes/LobbyRoute.tsx#L117) (the lobby's slot reducer walks the same events).
- **(b) Fetch user profile per UUID via `/api/users/:id`.** Rejected: HTTP roundtrip per voter; UI freezes during fetch; no methodology benefit over the local derivation.
- **(c) Display UUIDs (no resolver).** Rejected: methodologically silent — the per-voter attribution surface needs to read as "Alice voted dispute on the classification" not as "550e8400-e29b voted dispute" (the UUID surface is for debugging only; the v0 user surface needs the screen name).

Sub-decision: when a participant leaves, the resolver REMOVES the entry (not retains historically). Two options:

- **Remove on left:** the per-voter table won't surface their screen name if they leave; the fallback `(unknown user)` (from `screenNameFor`'s third arg) renders instead. Cleaner cross-time semantic: the panel reflects the CURRENT session's participants. Mirrors the lobby's "current debater" semantic.
- **Retain on left:** the per-voter table would always show the screen name even if the participant has left. Could be useful for historical attribution but couples the resolver's behaviour to the per-vote-history posture which `part_other_vote_indicators` Decision §1 + the `'withdraw'` arm specifically REMOVES from the per-entity list. Inconsistent: the per-voter table removes left voters' entries (by `withdraw` semantics) but would retain their names — confusing.

Decision §5: ship (a) with remove-on-left. The fallback to `(unknown user)` for an unresolved UUID is the safety net for any edge case (server-emitted vote events for a participant whose `participant-joined` event hasn't yet been ingested locally, for instance — the per-event arrival order isn't strict in the WS slice).

### §6 — Annotation row + diagnostic row + axiom-mark attribution: reimplement in participant workspace (do NOT import moderator's `<AnnotationBadge>` / `<AxiomMarkBadge>`)

The moderator ships `<AnnotationBadge>` ([`apps/moderator/src/graph/AnnotationBadge.tsx`](../../../apps/moderator/src/graph/AnnotationBadge.tsx)) and `<AxiomMarkBadge>` for the per-annotation / per-axiom-mark React row. Two options for the participant panel:

- **(a) Reimplement plain `<li>` rows in the participant workspace using existing primitives (no badge component).** Chosen for v0. The badge components carry moderator-specific affordances (inline action menus, hover-to-detail popovers via the moderator's `<HoverPopover>`) that the participant doesn't want — the participant's panel is read-only at v0, no actions, no hover-to-detail (the panel IS the detail). Reimplementing as plain `<li>` rows keeps the participant's surface free of moderator-specific UX vocabulary.
- **(b) Import `<AnnotationBadge>` / `<AxiomMarkBadge>` from the moderator workspace.** Rejected for v0: the components carry editing affordances; the participant would need to either suppress them (defeating the purpose of import-vs-reimplement) or render them and have the buttons do nothing (a UX trap — disabled-looking active buttons). The `<FacetPill>` is different because it's a pure-display component with no action affordances; the badge components couple display + action in one tree.

Sub-decision: the per-participant axiom-mark attribution row uses textual comma-separated screen names ("Bedrock for: alice, ben"), NOT the moderator's per-participant chromatic badge. Per Decision §1 of `part_axiom_mark_decoration` ("Per-participant chromatic identity deferred to the entity detail panel") — the chromatic badge is the eventual surface; for v0 the textual surface satisfies the methodology requirement (the debater needs to know WHO has marked it as bedrock; the chromatic identity is a polish layer). The textual surface is sub-day work; the chromatic surface would consume a meaningful chunk of the 1d budget.

Decision §6: reimplement annotation row + diagnostic row + axiom-mark attribution as plain `<li>` rows in the participant workspace; import only `<FacetPill>` + `PILL_*_CLASSNAME` from the moderator (Decision §4). A future polish leaf can swap the textual axiom-mark attribution for the chromatic badge if real usage shows the textual surface insufficient.

### §9 — Action-slot prop on the panel reserves the surface for future leaves (voting, axiom-mark, propose) without breaking the panel's API

The methodology layer's per-entity actions (vote on a facet, axiom-mark, propose-amend, propose-decompose) all anchor naturally on the per-entity detail panel — the moderator does the same on its in-card surface. - **(a) Reserve an `actionSlot?: ReactNode` prop on the panel; renders the children at the bottom of the panel if provided, nothing if absent.** Chosen. - **(b) Wait for the future leaves to extend the panel's API.** Rejected: would force the future leaves to refactor the panel signature (and every test factory that constructs the panel with mock props). Reserving the slot now is one new prop today, zero refactor later.

Decision §9: ship (a). The slot prop is unconsumed at v0; the panel renders a placeholder when the slot is absent — actually, the panel renders NOTHING when the slot is absent (no placeholder; the absence is silent). The future leaves attach actions via the slot prop without re-shaping the panel.

### §10 — Stale entity branch: when `selected` references an unknown id, render the localized stale-entity body AND auto-clear the selection on the next tick

The selection store carries an id; the projection may not. Mismatch cases: (i) a deleted entity (rare — the methodology has no entity-deletion event today, but a future leaf might add one); (ii) a dangling-edge filter dropping the edge from the projection (`<GraphView>`'s projection filters edges whose endpoints aren't both present — per `part_graph_render`); (iii) a session-reload race where the selection persists across a snapshot reload but the snapshot doesn't contain the entity.

Three options:

- **(a) Render a localized "(element no longer present)" body AND auto-clear the selection on the next tick via a `useEffect`.** Chosen. The body explains the situation to the debater; the auto-clear restores the panel to the empty-state on the next render (the `useEffect` triggers `useSelectionStore.getState().clear()` which fires the store's `set` which re-renders the panel which now sees `selected === null`). The two-tick cycle is intentional: tick 1 shows the explanatory body so the debater notices the staleness; tick 2 returns to the empty-state.
- **(b) Render the empty-state body immediately (skip the explanation).** Rejected: the debater wouldn't know WHY the panel went empty after they tapped a node. Confusing UX.
- **(c) Keep rendering the LAST selected entity's snapshot (cache the data).** Rejected: would surface stale per-facet data that doesn't match the current event log. The methodology requires accurate per-facet state; staleness is worse than absence.

Decision §10: ship (a). The Vitest case (b) pins both the body AND the auto-clear; the `useEffect` cleanup is straightforward.

### §11 — Per-facet per-other-voter breakdown is OUT of v0 scope; the per-entity per-voter table is the only "other voters" surface; the per-facet drill-down is registered as informal tech-debt

The current `OthersVoteIndex` carries `readonly OtherVote[]` per entity (with the per-`(entity, facet)` → per-entity dispute-wins rollup baked into `projectOtherVotes` per its Decision §1); reaching the per-facet detail requires either widening the projection's data shape OR re-walking the events log inside the panel for the selected entity. Both are sub-day moves, neither fits in the v0 budget.

Decision §11: ship the per-entity per-voter table (Section 8 of the panel) as the v0 "other voters" surface; defer the per-facet per-voter breakdown to a follow-up task. Registering as informal tech-debt in the orchestrator notes — explicit follow-up registration is the orchestrator's call (this refinement names the deferral; whether to file a `.tji` task today is upstream).

Sub-decision: the panel reserves NO additional API surface for the per-facet per-voter breakdown. When the follow-up task lands, it either widens the projection (adding a per-facet per-voter dictionary to `OthersVoteIndex`) and re-renders the panel's Section 8, OR adds a new section underneath Section 8 with the per-facet drill-down. Either path is a future-leaf decision; this leaf doesn't preempt.
