## What this task is

Extend the participant's read-mostly `<GraphView>` so every node OR edge whose facet-targeting pending proposals carry votes by OTHER participants (not the current one) surfaces an at-a-glance per-other-voter row on the node/edge — the seventh visual-vocabulary layer on top of `part_graph_render` (baseline), `part_per_facet_state_styling` (per-facet rollup status), `part_axiom_mark_decoration` (axiom-mark boolean overlay), `part_annotation_render` (annotation amber overlay), `part_diagnostic_highlights` (structural-diagnostic amber ring), and `part_own_vote_indicators` (the current participant's own-vote label-outline ring — the sibling that just landed at commit `69bc92c`).

Before this leaf: a debater can see THEIR OWN vote disposition on the entity via the label-outline ring; they can see the aggregate per-facet rollup status (`proposed` / `disputed` / `agreed`); but they cannot see WHICH OTHER PARTICIPANT voted which way on the entity. The methodology asks the debater questions like "am I the holdout on this entity, or is someone else disagreeing too?" and "did the third debater weigh in yet?" — questions the prior visual layers can't answer without drilling into the (future) pending-proposals pane to scan per-proposal vote records.

After this leaf: each node and edge whose facet-targeting proposals carry per-other-participant votes surfaces a per-voter row of attribution data (one entry per OTHER voter who has cast a recordable vote on any facet-targeting proposal of the entity, color-coded by the voter's latest arm — `agree` / `dispute` — plus stable per-voter identity in the DOM mirror).

## Why it needs to be done

`m_participant_mvp` ([`tasks/99-milestones.tji`](../../99-milestones.tji)) is the milestone at which a debater can see and engage with the live graph from their tablet. `part_graph_render` lit up the rendering surface; the FIVE prior overlay leaves (per-facet status / axiom / annotation / diagnostic / own-vote) painted the per-aggregate + per-self vocabularies; this leaf paints the **per-other-participant** vocabulary — the at-a-glance "where do the other debaters stand on the live questions right now?" surface that closes the methodology's primary feedback loop on the social-coordination side.

The methodology assumes the debater can answer questions like "am I the holdout, or is someone else disagreeing too?" and "did the third participant weigh in yet?" without context-switching to the (future) pending-proposals pane:

- [`docs/methodology.md`](../../../docs/methodology.md) § "Facets" + "Votes / Withdrawal" — the per-(participant, proposal) vote IS the agreement primitive. The current participant's own vote is part of the picture (settled by `part_own_vote_indicators`); the other participants' votes are the OTHER part — together they reconstruct the methodology's per-`(entity, facet)` per-participant decision graph that drives forward progress. Without the other-vote indicator, the debater sees only their own contribution + the aggregate rollup, which is opaque ("the facet is `disputed` — but is it because I disputed it, or because someone else did?").
- [`docs/participant-ui.md`](../../../docs/participant-ui.md) P1 + P2 (view the graph + single-tap votes) — P1 is the "read the live state" surface. The per-(other-participant, target) vote disposition is on the read side because the debater reads how the room is voting before tapping their own vote.
- [`DESIGN.md`](../../../DESIGN.md) "Agreement-driven progress" — the methodology engine treats `agreed` as the green-light state for forward progress. The debater seeing "I voted agree on this; one other participant voted agree; we're one vote away from `agreed`" is the load-bearing social-coordination signal for "should I be pushing back on a holdout right now, or are we converging?". Without the other-vote indicator, the debater has to drill into a future pane to reconstruct who agreed every time the question comes up.

Downstream concretely:

- **`part_entity_detail_panel`** (the React-driven tap-to-detail panel, future) is the natural home for the per-facet per-voter breakdown. When that leaf lands, it imports `projectOtherVotes` (alongside the already-shipped `projectOwnVotes`) so the panel can render one row per other voter per facet alongside the participant's own per-facet vote.
- **`part_voting.part_vote_single_tap`** (the canvas-side vote-cast gesture, future) uses the other-vote count + mix as one of the inputs to its tap affordance (e.g. "your vote will tip this facet to `agreed`" — a function of (own-vote + others-votes) the projection already carries).
- **`part_other_vote_indicators_canvas_dots`** (the v1 canvas-rendering polish leaf, NOT YET REGISTERED in `tasks/40-participant-ui.tji`) consumes the SAME `projectOtherVotes` projection shape this leaf ships — no projection refactor needed; the future leaf wires a Cytoscape overlay or compound-node rendering onto the same per-entity `otherVotes: readonly OtherVote[]` array `projectGraph` now emits.
- **`audience.aud_other_vote_indicators`** (future, sibling to `aud_annotation_rendering`) becomes the third Cytoscape consumer of the per-other-voter projection family. When it lands, the natural extraction trigger lifts the shared helper (moderator's `projectVotesByFacet` + the participant's `projectOwnVotes` + this leaf's `projectOtherVotes`) into `@a-conversa/shell`.
- The participant's `<GraphView>` becomes the **second concrete adoption of the moderator's per-participant vote-attribution data shape** (now with the own/other complement split on the participant side, where the moderator's surface didn't need it because all moderator views are "spectator-of-debate" views, not "I-am-a-debater" views).

## Decisions

### §1 — Vote vocabulary at the indicator layer: per-voter `{ participantId, choice: 'agree' | 'dispute' }`; `'withdraw'` removes the entry; symmetric across node AND edge targets; per-entity rollup with dispute-wins tie-break

The wire `vote` enum has three arms (`'agree' | 'dispute' | 'withdraw'`) per [`votePayloadSchema`](../../../packages/shared-types/src/events.ts#L364). The participant's at-a-glance "where do the other debaters stand?" surface needs a narrowed per-voter shape.

Three options for the per-voter sentinel:

- **(a) Three-state `'agree' | 'dispute' | 'withdraw'` per voter entry; the voter stays in the per-entity list with their latest arm.** Rejected: same rationale as `part_own_vote_indicators` Decision §1(a). The participant's at-a-glance pre-commit surface treats withdrawal as the explicit retraction back to the un-voted baseline; surfacing withdrawn voters at the at-a-glance layer over-states the post-commit signal's prominence and inflates the per-entity row width without adding load-bearing signal at the dot-count layer. The explicit "this participant withdrew" detail is the entity-detail-panel's future job.
- **(b) Two-state `'agree' | 'dispute'`; the `'withdraw'` arm REMOVES the voter's entry from the per-entity list.** Chosen. The per-entity list IS the at-a-glance "who's still actively voting agree or dispute on this entity?" record; a withdrawal collapses to the un-voted state at that layer (the entry vanishes). This mirrors the own-vote leaf's Decision §1 posture verbatim (withdrawal collapses to `'none'` for own-vote; the symmetric move for the per-other-voter list is to remove the entry).
- **(c) Three-state with the third arm rendered as a third visual encoding (e.g. a slate-gray dot).** Rejected: would require either (i) a Cytoscape stylesheet extension (which Decision §3 explicitly defers) or (ii) a more complex DOM-mirror attribute encoding. The two-state list keeps the v0 surface minimal; the third-arm signal can land in the future entity-detail-panel.

Decision §1 (b): ship (b). `OtherVote = { readonly participantId: string; readonly choice: 'agree' | 'dispute' }`. The closed two-arm `choice` covers the at-a-glance "did this voter agree or dispute?" question; withdrawal is encoded as "no entry in the list".

**Symmetry across node AND edge targets**: same posture as `part_own_vote_indicators` Decision §1, `part_annotation_render` Decision §1, `part_diagnostic_highlights` Decision §1. The wire `proposal` family targets both node entities (`classify-node`, `set-node-substance`, `edit-wording`, `amend-node`, `axiom-mark`) AND edge entities (`set-edge-substance`); the moderator's `projectVotesByFacet` buckets both per its `mod_vote_indicators_in_sidebar` Decision §4 extension. The participant's `OthersVoteIndex` mirrors that bucketing with `{ nodes: Map, edges: Map }`.

**Per-entity rollup rule** (collapsing per-`(entity, facet)` votes per other voter to one entry per `(entity, voter)`): a participant who voted on multiple facets of the same entity appears ONCE in the per-entity list with a `choice` rolled up by the same dispute-wins rule as `part_own_vote_indicators` Decision §1:

- If the voter cast a `'dispute'` on ANY tracked facet of the entity → entry's `choice` is `'dispute'`.
- Else if the voter cast an `'agree'` on ANY tracked facet → entry's `choice` is `'agree'`.
- Else (only `'withdraw'` arms, or no arms) → voter is NOT in the list.

Rationale: same "ratchet to the conservative state" methodology posture the own-vote leaf adopted. If voter X is actively disputing one facet of the entity, the at-a-glance signal is "X disagrees with this entity right now" rather than "X half-agrees" — the per-facet breakdown is the future entity-detail-panel's surface.

### §2 — Project per-entity other-votes in the participant workspace; do NOT port the moderator's full `projectVotesByFacet`; do NOT extract into the shell yet

The moderator's `projectVotesByFacet(events): Map<entityId, Map<FacetName, Vote[]>>` is the canonical client-side full vote projection. The participant's at-a-glance other-votes layer needs a narrower output. Three options:

- **(a) Port `projectVotesByFacet` verbatim into the participant workspace; narrow at the consumer (`<GraphView>`) via a `useMemo`.** Rejected: same rationale as `part_own_vote_indicators` Decision §2(a). The per-facet per-participant detail isn't load-bearing for the at-a-glance layer; carrying the full structure through inflates memory + recomputation cost for data the surface doesn't read at v0. The future entity-detail-panel WILL need the per-facet detail; that consumer can read the moderator's full projection (which is already extractable on the participant side via the trigger-on-the-third-caller policy) when it lands.
- **(b) Write a narrowed `projectOtherVotes(events, currentParticipantId): OthersVoteIndex` in the participant workspace; single-pass walk that filters to non-current-participant inline and rolls up per-entity per-voter with dispute-wins tie-break.** Chosen. Mirrors the `ownVotes.ts` posture exactly — same proposal-target walk shape (drift-parity note in the header), same `EMPTY_*` shared-reference discipline, same single-pass-with-incremental-rollup structure. The output shape is what the projector needs without further narrowing.
- **(c) Compute the other-votes at the canvas layer by re-using `ownVotes.ts`'s walk with an inverted filter.** Rejected: would couple the two projections (own + other) to a single module with two related-but-distinct outputs, complicating the test surface. The clean separation today (two parallel modules, two parallel projections, two parallel projector arguments) keeps each projection independently testable + the extraction trigger for the future shell helper clean (the future helper lifts both at once with the moderator's `projectVotesByFacet`).

Decision §2 (b): ship (b). The narrowed projection lives at `apps/participant/src/graph/otherVotes.ts`. Shell extraction waits for the third caller (audience surface) per the same trigger-on-the-third-caller policy `axiomMarks.ts`, `annotations.ts`, `diagnosticHighlights.ts`, and `ownVotes.ts` already followed.

The participant's `otherVotes.ts` carries the same drift-risk header note as `ownVotes.ts`: a future `proposal-target` schema change touches `facetStatus.ts:132-164`'s walk, `ownVotes.ts:108-126`'s walk, this projection's walk, AND the moderator's `selectors.ts:736-754`'s walk. Four co-shaped locations today; the natural unification is `@a-conversa/shell` once a third Cytoscape consumer (audience) materializes.

### §3 — DOM-mirror-only for v0; defer canvas-side rendering to a future polish leaf

The own-vote leaf claimed the `text-outline-*` Cytoscape stylesheet primitive family for the at-a-glance "this is YOUR vote" ring (Decision §3 of `part_own_vote_indicators`). The other-vote signal needs a DIFFERENT visual encoding — a per-voter row of attribution (count of voters + arm-per-voter mix), not a single per-entity sentinel. Three options for the canvas-side rendering:

- **(A) Extend Cytoscape with per-voter dot elements via the compound-node mechanism** (each parent node gets a small ring of child "dot" nodes, one per other-voter). Rejected: forces Cytoscape's compound-node mechanism — a load-bearing layout change that ripples into `cy.layout({ name: 'cose' })` (the children inherit the parent's layout group; the layout pass treats the compound parent as a single unit but the children's positions are derived). Complicates the projection significantly (the projector would emit per-voter child nodes interleaved with the per-entity parent nodes); breaks the symmetric `nodes.length === per-node-created-event count` invariant the existing tests rely on. High implementation cost; high test churn.
- **(B) Layer a React DOM overlay above the Cytoscape canvas** (a React component reads `cy.elements().forEach(ele => ele.renderedPosition())` to position per-entity dot rows in screen-coordinate space; rerenders on Cytoscape's `pan` / `zoom` / `render` events). Rejected for v0: introduces a new rendering paradigm (React DOM positioned above a Canvas-rendered Cytoscape; the existing surface is single-paradigm Cytoscape-only). Real engineering value but high in-leaf scope — needs a positioning helper, a Cytoscape-event-subscription hook, a viewport-relative coordinate transform, and re-render correctness under pan/zoom/element-change. A future leaf can land this incrementally.
- **(C) DOM-mirror-only for v0; defer canvas-side rendering to a future polish leaf `part_other_vote_indicators_canvas_dots`.** Chosen. The DOM mirror is the testability seam (already `aria-hidden` + `sr-only`); extending it with a nested `<ul data-other-votes>` per `<li>` row carries the per-voter attribution data losslessly + composably for Playwright assertion. The at-a-glance VISUAL signal on the canvas is deferred to a future leaf — orchestrator's call whether to register it now. The deferral is principled: the data layer (projection + projector + mirror) is the load-bearing seam; the canvas overlay is a polish on top of an already-correct data pipeline. A future canvas-overlay leaf reads the same `otherVotes: readonly OtherVote[]` field this leaf stamps on each Cytoscape element data record (the `...node.data` / `...edge.data` spread in the localized memo carries the field through onto the Cytoscape element record as inert data — Decision §6 documents this explicitly).

Decision §3 (C): ship (C). No new Cytoscape stylesheet selectors; no new React overlay component; the DOM mirror nested `<ul>` is the load-bearing surface for v0. A future leaf `part_other_vote_indicators_canvas_dots` (not yet registered in `tasks/40-participant-ui.tji` — orchestrator's call) lands the canvas-side visual signal on top of the already-shipped data pipeline.

**Stylesheet primitive collision check** (still relevant for future readers of the canvas-overlay leaf): the prior overlays own `border-*`, `background-*`, `opacity`, `outline-*` (rollup); `border-style: 'double'` + `border-width: 3` (axiom); `overlay-*` / `underlay-*` (annotation); `border-*` (override) / `underlay-*` (diagnostic); `text-outline-*` (own-vote). Still unclaimed Cytoscape stylesheet primitive families: `text-*` minus `text-outline-*` (label color, font, halign, valign); `shape` (used by baseline as `round-rectangle`); `padding` (used by baseline); `curve-style` (edge — used by baseline as `bezier`); plus all the per-element `cy` event subscriptions for React-DOM-overlay positioning. The future canvas-overlay leaf has room.

### §4 — Per-voter list rather than per-entity rolled-up sentinel

The own-vote leaf's projection emits `OwnVoteIndex = { nodes: Map<string, OwnVote>; edges: Map<string, OwnVote> }` — a single per-entity sentinel (one voter, one arm-or-`none` per entity). The other-vote projection emits `OthersVoteIndex = { nodes: Map<string, readonly OtherVote[]>; edges: Map<string, readonly OtherVote[]> }` — a per-entity LIST (many voters, each with an arm).

Three options for the per-entity output shape:

- **(a) Per-entity rolled-up sentinel** (e.g. `'all-agree' | 'mostly-agree' | 'split' | 'mostly-dispute' | 'all-dispute' | 'none'`). Rejected: lossy — the at-a-glance "how many others, and which way?" question needs the per-voter count + arm. A rolled-up sentinel hides "X agree and Y dispute" behind a single aggregate; the moderator's own surface (the canonical reference) explicitly carries per-voter detail per `mod_vote_indicators_on_graph` Decision §1.
- **(b) Per-entity COUNT pair** (`{ agreeCount: number; disputeCount: number }`). Rejected: drops per-voter identity. The Playwright assertion "the SAME voter switched arms and didn't double-count" + the future entity-detail-panel's "show me which voter agreed" both need the per-voter UUID. The count pair is a derived view; emitting the per-voter list as the canonical form is more flexible.
- **(c) Per-entity per-voter list** (`readonly OtherVote[]` per entity, each entry `{ participantId, choice }`). Chosen. Carries the load-bearing per-voter UUID + arm; the count pair is `list.length` + a per-arm group-by; the rolled-up sentinel is a `list`-derived function. The future entity-detail-panel reads the list directly. Memoization-stable per Decision §5's insertion-order rule.

Decision §4 (c): ship (c). The per-entity list IS the canonical projection output; all downstream consumers (this leaf's mirror, the future canvas overlay, the future entity-detail-panel) derive their views from the list.

### §5 — Insertion-order preservation per voter: first-vote arrival order, in-place overwrite on arm-switch (mirrors the moderator's `positionIndex` posture)

The per-entity list needs a deterministic order for stable rendering across re-projection. Three options:

- **(a) Alphabetical by `participantId`.** Rejected: alphabetical-by-UUID is functionally arbitrary (UUIDs are random hex) and provides no semantic value; it also produces a deterministic order that's hard for humans to predict (the moderator's at-a-glance dot row would jump around as new voters join).
- **(b) Vote-time order (insertion order by latest vote arrival).** Rejected: a voter who switches arms would jump to the end of the list, disrupting the visual stability the moderator's same surface preserves.
- **(c) First-vote-arrival order; subsequent votes by the same voter overwrite in-place at the original position.** Chosen. Same posture as the moderator's `projectVotesByFacet` (Decision §1 of `mod_vote_indicators_on_graph` + the `positionIndex` map at [`selectors.ts:787`](../../../apps/moderator/src/graph/selectors.ts#L787) — "Insertion order in the per-facet list: order of each participant's FIRST vote on that facet (subsequent votes by the same participant overwrite the choice in-place, preserving the original position). This keeps the rendered dot order stable across the agree↔dispute switch."). Mirrors the moderator's behaviour verbatim at the data layer; the future canvas-overlay leaf inherits the same stability guarantee.

Decision §5 (c): ship (c). `projectOtherVotes` maintains a per-`(entityId, facet)` `Map<participantId, position-in-list>` accumulator AND the per-entity per-voter list; arm-switching writes to the indexed position; first-vote-arrival pushes. The per-entity rollup (Decision §1) collapses per-facet votes by the same voter to one entry per entity (which is added at the FIRST `(any-facet-of-this-entity, this-voter)` vote arrival; subsequent votes on other facets re-evaluate the voter's rolled-up `choice` per dispute-wins, in-place at the original position).

The withdraw arm (Decision §1) REMOVES the entry — when a voter's latest vote on every tracked facet of an entity is `'withdraw'`, the voter's entry is removed from the list. To preserve insertion-order semantics for the NON-removed voters, removal closes the gap (shifts later entries left by one); a future re-add by the same voter pushes to the end (per first-vote-arrival).

### §6 — DOM mirror surface: nested `<ul data-other-votes>` inside each per-entity `<li>`, with per-voter `<li data-other-vote data-voter-id="<uuid>" data-vote="agree|dispute">`

The DOM mirror is `aria-hidden="true"` + `sr-only` and serves as the canvas-blind testability seam. The other-votes signal extends both `<li>` row kinds.

Three options for the mirror attribute shape:

- **(a) Comma-separated attribute string** (e.g. `data-other-votes="<uuid1>:agree,<uuid2>:dispute"`). Rejected: needs custom parsing in Playwright tests + visual-inspection in devtools; loses the per-voter `<li>` as a queryable seam; loses the per-voter Playwright probe (e.g. "find the `<li>` with this voter id under this entity's mirror row").
- **(b) Flat attributes on the parent `<li>` per voter** (e.g. `data-other-vote-1="<uuid>:agree"` / `data-other-vote-2="<uuid>:dispute"`). Rejected: variable-arity attribute family is awkward to assert against (Playwright probes can't enumerate dynamically-named attributes cleanly); shapes the mirror non-uniformly across rows with different vote counts.
- **(c) Nested `<ul data-other-votes>` inside the parent `<li>`, with per-voter `<li data-other-vote data-voter-id="<uuid>" data-vote="agree|dispute">`.** Chosen. The DOM structure mirrors the data shape (entity-then-list); per-voter `<li>` is a queryable seam (Playwright `locator('[data-node-id="..."] ul[data-other-votes] li[data-voter-id="..."]')` resolves directly); the empty-list case renders an empty `<ul>` so the absent-children assertion is `locator('[data-node-id="..."] ul[data-other-votes] li').count() === 0` (cleanly distinct from "the `<ul>` is missing entirely", which would be a projector bug).

Decision §6 (c): ship (c). The parent `<li>` keeps all its existing flat attributes (`data-rollup-status`, `data-facet-*`, `data-is-axiom`, `data-has-annotation`, `data-annotation-count`, `data-diagnostic-severity`, `data-diagnostic-kinds`, `data-own-vote`) — they're a fixed-arity attribute family. The variable-arity per-voter information is the nested `<ul>`. The two information layers nest cleanly; the existing tests on the parent `<li>` attributes are unaffected.

Per-voter `<li>` attribute names:
- `data-other-vote` — sentinel (empty value) for the "is this a vote row?" probe.
- `data-voter-id="<uuid>"` — the per-voter identity for the per-voter probe.
- `data-vote="agree|dispute"` — the per-voter arm sentinel.

The future canvas-overlay leaf reads the same data off the Cytoscape element record (via `...node.data` / `...edge.data` spread carrying `otherVotes` through to the localized memo's data record). Two surfaces, one data source.

### §7 — Seventh `test()` block REUSES the alice+ben pair; synthesizes other-voter UUIDs at the event-seed layer (the 12-user pool is exhausted)

The existing `tests/e2e/participant-graph-render.spec.ts` describe has SIX `test()` blocks running under `test.describe(...)` (parallel). The 12-user pool is exhausted: alice+ben, maria+dave, frank+erin, grace+henry, ivan+julia, kate+leo (per `part_e2e_user_pool_expansion.md` Decision §2 — the pool was sized for 6 blocks max). The 7th block this leaf adds has no fresh pair available.

Four options:

- **(a) Use a synthetic-user injection: REUSE an existing pair (e.g. alice+ben from block 1) AND seed the OTHER-voter UUIDs at the event-seed layer with synthetic UUIDs that don't correspond to logged-in dev users.** Chosen. The projection compares `voter.id !== currentParticipantId` as a string-equality check; the per-`vote` `participantId` is a UUID string and the projection does NOT validate the UUID against the `participant-joined` event log (the wire's vote envelope is independent of the participant-presence envelope). Synthetic UUIDs at the seed layer behave indistinguishably from real per-user UUIDs at the projection layer — the seam is below the OIDC dance entirely. Reuses block 1's `freshContext` + per-block-isolated session; no race because each block creates its own session (the session-creation flow is per-block under `createSession`). Wall-clock for the spec file is unchanged. NEW TECH DEBT: a future block that needs MORE THAN ONE login-real user (e.g. observing the OTHER participant's tab in real-time) would need either a per-block-distinct pair (the pool is gone) or a pool re-expansion — that's a future concern, not this leaf's.
- **(b) Expand the dev-user pool again (12 → 14)** — same posture as `part_e2e_user_pool_expansion`. Rejected: the pool was expanded once already; expanding again for one fresh pair is overkill when the synthetic-UUID-at-the-seed-layer approach (option a) covers the use case. The pool expansion costs an Authelia restart + a `users.yml` edit + a `DEV_USER_POOL` constant bump for ONE block of value. Defer pool re-expansion to a future leaf that genuinely needs multi-OIDC-login per block.
- **(c) Split this leaf's e2e into a new spec file `tests/e2e/participant-other-vote-indicators.spec.ts`** with its own setup. Rejected: would duplicate ~80 lines of fixture composition for one block; the spec file is the natural home for "participant graph render" coverage. Same posture as `part_own_vote_indicators` Decision §6 picked block 6 in the existing file.
- **(d) Flip the describe to `.serial` and reuse alice+ben.** Rejected: the pool-expansion refinement specifically reverted the `.serial` regression; re-introducing it for one block would undo the wall-clock improvement.
