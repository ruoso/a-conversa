## What this task is

Light up annotation-of-annotation overlay propagation on the participant's read-mostly Cytoscape canvas. When an annotation A1 carries another annotation A2's UUID in its `target_node_id` field (the polymorphic-entity-id slot of the wire `annotation-created` payload), and A2 is itself materialized as an annotation graph-node (because some `edge-created` references it as an endpoint per the predecessor task), A2's graph-node renders with the existing amber overlay + a `data-annotation-count` ≥ 1 in the mirror — exactly the same `hasAnnotation` / `annotationCount` vocabulary the predecessor `part_annotation_render` established for statement nodes.

The work is small. The participant's annotation graph-node materialization pass at [`apps/participant/src/graph/projectGraph.ts:707-708`](../../../apps/participant/src/graph/projectGraph.ts) already routes `hasAnnotation: nodeHasAnnotation(nodeAnnotationIndex, payload.id)` and `annotationCount: annotationCountFor(nodeAnnotationIndex, payload.id)` through the existing `nodeAnnotationIndex` (a `Map<string, readonly Annotation[]>` keyed by string id). The bucketer's lookup is target-kind-agnostic — it doesn't care whether `payload.id` is a statement-node id or an annotation id, because the UUID space is shared. So mechanically, the propagation already happens any time an `annotation-created` event arrives with `target_node_id = <another-annotation-uuid>`.

What's missing is two things:

1. **Bucketer-side honesty.** [`packages/shell/src/annotations/annotations.ts:110-124`](../../../packages/shell/src/annotations/annotations.ts) is named `groupAnnotationsByNode` and its docstring talks about "node target." The name and prose imply the bucket key is a statement-node id, which under-specifies the polymorphic-entity-id convention (the key is whatever UUID lives in `targetNodeId`, and that UUID may name a statement node OR an annotation). This task adds `groupAnnotationsByEntityId` as the renamed export — same body, same return shape — with a docstring that documents the polymorphic-entity-id semantic and the annotation-of-annotation propagation it enables. The old name `groupAnnotationsByNode` stays as a backward-compat re-export so the audience + moderator surfaces don't get drawn into this task's scope.

2. **Committed test cover.** [`apps/participant/src/graph/projectGraph.test.ts`](../../../apps/participant/src/graph/projectGraph.test.ts) gains three new cases (per Acceptance criteria below) pinning the propagation end-to-end — seeding an `annotation-created` whose `target_node_id` carries another annotation's UUID and asserting the materialized annotation graph-node carries `hasAnnotation: true` + `annotationCount` ≥ 1. The shell-level bucketer test gains one explicit case showing the bucketer is target-kind-agnostic. The existing `annotations.test.ts` continues to pin the statement-node-targeting case; the new cases are additive.

Concretely the deliverable is:

1. **`groupAnnotationsByEntityId(annotations)` export at the shell layer.** [`packages/shell/src/annotations/annotations.ts`](../../../packages/shell/src/annotations/annotations.ts) gains the new export. Implementation is verbatim today's `groupAnnotationsByNode` body (skip annotations with `targetNodeId === null`; bucket by `targetNodeId`). The docstring documents the polymorphic-entity-id convention: the bucket key is whatever UUID lives in `targetNodeId`, which the wire schema allows to be any UUID (statement node id or annotation id); consumers look up by the entity id they have, regardless of kind.
2. **`groupAnnotationsByNode` becomes a thin re-export alias.** The old name continues to work for audience + moderator + any external consumers; this task's scope stays bounded to the participant surface.
3. **Participant migration to the new name.** [`apps/participant/src/graph/annotations.ts:25-31`](../../../apps/participant/src/graph/annotations.ts) (the participant's annotation re-export shim) adds `groupAnnotationsByEntityId` to its re-export list; the participant's GraphView + OperateRoute use sites adopt the new name. The variable identifier `nodeAnnotationIndex` stays — D3 explains why a `entityAnnotationIndex` rename is out of scope.
4. **Updated comment in `projectGraph.ts`.** The breadcrumb at L688-692 naming "annotation-of-annotation overlay propagation (Decision §7 — future task)" is updated to point at THIS refinement's Status block, since the propagation lights up here.
5. **Vitest cover.** Three new cases in `projectGraph.test.ts` (Acceptance criteria below); one new case in the shell-layer `annotations.test.ts` (or the participant shim's test, depending on which file already pins shell-layer behavior). No new files; all cases additive.

Out of scope:

Their adoption of the new bucketer name + the propagation behavior is registered under Tech-debt as `aud_annotation_of_annotation_overlay_chain` and `mod_annotation_of_annotation_overlay_chain` follow-ups (named-future-tasks under Tech-debt registration).

## Why it needs to be done

**The methodology surfaces annotation-on-annotation as a structural reality.** Per [`docs/methodology.md`](../../../docs/methodology.md) §"Annotations" and the coherency-hint catalogue extension landed by `coherency_annotation_of_annotation_chain_rule` (2026-05-30), annotations can themselves be commentary on other annotations — a participant clarifying or contradicting another participant's annotation is part of the methodology's deliberation loop. The participant canvas should surface this commentary at the same at-a-glance card layer it surfaces commentary on statement nodes.

**The visual surface is already half-built.** The predecessor task `part_render_annotation_endpoint_edges` (done 2026-05-30) materializes annotation graph-nodes as Cytoscape entities with their own `hasAnnotation` / `annotationCount` fields. Today those fields read `false` / `0` from the existing `nodeAnnotationIndex` for any real-world session because no annotation-created event has yet been observed carrying another annotation's UUID in `target_node_id`. When the methodology engine (or a moderator authoring an annotation-on-annotation) starts minting such events, the predecessor's pipeline will silently propagate — but with no committed test pinning that behavior, a future refactor could regress it without anyone noticing. This task pins the behavior at the Vitest layer with explicit cases.

**The bucketer's name is misleading.** [`packages/shell/src/annotations/annotations.ts:101-108`](../../../packages/shell/src/annotations/annotations.ts) — "Bucket annotations by their node target." The prose implies "node = statement node," but the wire schema's `target_node_id` is just a `z.string().uuid().nullable()` (see [`packages/shared-types/src/events.ts:345-362`](../../../packages/shared-types/src/events.ts) — quoted in the predecessor's Inputs) — it accepts any UUID. The bucketer is already polymorphic-entity-id at the implementation layer; renaming to `groupAnnotationsByEntityId` makes the contract honest. Future contributors reading "node" stop assuming "statement node only."

**The bucketer is shared infrastructure.** All three UI surfaces (participant, audience, moderator) consume the same shell-level `groupAnnotationsByNode`. Adding `groupAnnotationsByEntityId` at the shell layer means the audience + moderator follow-ups can adopt the same propagation without a second extraction pass — the seam is in place.

**Pinning the convention closes a documentation debt.** The predecessor's D7 explicitly identified this as a "small but distinct refactor that touches the `part_annotation_render` contract" — the contract being "annotations target node/edge entities and overlay on them." This task ratifies the conceptual widening to "annotations target ANY entity that has a UUID id, including other annotations" and commits Vitest cover that pins the convention.

## Decisions

- **D1 — Add `groupAnnotationsByEntityId` as a renamed export at the shell layer; keep `groupAnnotationsByNode` as a thin backward-compat alias.** Rationale:
  - **The bucketer is already polymorphic-entity-id at the implementation layer.** [`packages/shell/src/annotations/annotations.ts:110-124`](../../../packages/shell/src/annotations/annotations.ts) buckets by `annotation.targetNodeId` — a string lookup that doesn't care whether the UUID names a statement node or an annotation. The rename makes the contract honest; the implementation needs no semantic change.
  - **Audience + moderator stay out of scope.** Renaming `groupAnnotationsByNode` outright would force the audience + moderator's import sites to migrate in this commit — drawing them into a task they don't otherwise overlap. The backward-compat alias means they can adopt the new name in their own follow-up refinements at their own pace.
  - **The shell layer is the architecturally correct home.** Per ADR 0026 + `extract_cytoscape_projectors`, cross-surface vocabulary lives in `@a-conversa/shell`; adding the new export there means audience + moderator can adopt it later without a second extraction pass.
  - **Alternative considered: rename `groupAnnotationsByNode` outright, no backward-compat alias.** Rejected — forces audience + moderator migration in this commit (out of scope for 0.5d) and creates churn in passes-through-import files that don't otherwise change.
  - **Alternative considered: add `groupAnnotationsByEntityId` only in the participant workspace, leave shell untouched.** Rejected — duplicates the shell-level bucketer in the participant workspace, exactly the kind of cross-surface duplication `extract_cytoscape_projectors` lifted in the first place. Adding at the shell layer is the right altitude.
  - **Alternative considered: deprecate `groupAnnotationsByNode` with a `@deprecated` JSDoc tag.** Considered, NOT rejected — the implementer may add the tag at their discretion. The deprecation is mild (the old name is still functionally correct; "node target" is just under-specified) and a JSDoc tag would surface a soft signal in the IDE. Acceptance does not require the tag.

- **D2 — The new bucketer's body is verbatim today's `groupAnnotationsByNode`; the rename is conceptual, not functional. No XOR-3 or composite-key restructuring at this task.** Rationale:
  - **The 0.5d budget binds.** A functional restructure — composite `${kind}:${id}` keys, XOR-3 across (`targetNodeId`, `targetEdgeId`, hypothetical `targetAnnotationId`), or a unified-merge-of-node-and-edge-indexes — each lands a multi-day refactor with cross-surface implications. None is required to light up the annotation-of-annotation propagation today (the existing string-keyed bucket already accommodates annotation-ids in the `targetNodeId` slot).
  - **The UUID space is shared; collision risk is statistically zero.** A unified node+edge+annotation entity index keyed by raw UUID has no realistic ambiguity (UUID v4 collisions across the populations are not a real-world concern). The wire schema's `targetNodeId` slot already accommodates polymorphic UUIDs; the consumer-side lookup is by id alone.
  - **Future work can revisit if a real-world need surfaces.** If the methodology-engine adopts the chain rule structurally and emits annotation-on-annotation events at scale, OR if a future ADR-level decision adds an explicit `target_annotation_id` wire field, the bucketer can be restructured then. Today's rename keeps the door open without committing to a specific shape.
  - **Alternative considered: introduce composite `${kind}:${id}` keys.** Rejected — forces every consumer to construct prefixed keys; creates a fan-out of touch points in audience + moderator + shell tests; out of budget at 0.5d.
  - **Alternative considered: merge `nodeAnnotationIndex` and `edgeAnnotationIndex` into a single unified index.** Rejected — the merge is a real semantic restructure (touches the edge-rendering path, which has different consumer needs) and is out of budget. The participant's annotation graph-node materialization already works correctly with the two-index pattern; this task ratifies that, not refactors it.

- **D3 — The participant's variable identifier `nodeAnnotationIndex` is preserved across the migration; we do NOT rename to `entityAnnotationIndex`.** Rationale:
  - **Cosmetic-only churn is out of budget at 0.5d.** Renaming the local variable touches every reader of [`apps/participant/src/graph/GraphView.tsx`](../../../apps/participant/src/graph/GraphView.tsx) + [`apps/participant/src/routes/OperateRoute.tsx`](../../../apps/participant/src/routes/OperateRoute.tsx) + every `projectGraph` test that constructs the index by hand; the runtime change is zero.
  - **The local name is implementation-level; the shell-level rename is the contract-level honesty.** Reader confusion about "is this a node-only index?" lives at the shell-level export (where the rename lands) — the participant's local variable is one level down and reads less as a contract.
  - **Future task can rename if cost-justified.** A standalone cosmetic refactor renaming the participant-side local identifiers is cheap and discoverable when the next participant-graph touch lands; deferring is low-risk.
  - **Alternative considered: rename `nodeAnnotationIndex` → `entityAnnotationIndex` participant-wide.** Rejected — out of budget; the cost-benefit is poor since the local readers all already understand the polymorphic semantics from the shell-level docstring.

- **D4 — No wire schema change. The annotation-created payload's `target_node_id` field (`z.string().uuid().nullable()`) continues to carry the polymorphic entity-id; no explicit `target_annotation_id` field added; no XOR-3 refine.** Rationale:
  - **The wire schema already accommodates the convention.** `target_node_id` is unconstrained as to which entity kind the UUID names; the bucketer is type-agnostic. End-to-end propagation works today with no schema touch.
  - **Adding `target_annotation_id` is an ADR-level decision.** A wire-schema 3-way XOR (`(target_node_id, target_edge_id, target_annotation_id)` exactly one set) is an architectural commitment with backend + projection + validator + replay consequences; it deserves its own ADR if/when the methodology-engine adoption surfaces the need. Today, no observed real-world adoption demands it — the chain rule is advisory-only.
  - **Forward compat is preserved.** If a future ADR adds `target_annotation_id`, the bucketer migrates from `groupAnnotationsByEntityId(annotations) → buckets-by-targetNodeId-only` to a three-way merge — a contained migration at that point.
  - **Alternative considered: add `target_annotation_id` to `annotation-created` payload in this task.** Rejected — wire-schema change, backend touch, projection touch, replay touch; vastly exceeds 0.5d and warrants an ADR.
  - **Alternative considered: enforce at the validator layer that `target_node_id` references a known statement node.** Rejected — this would FORECLOSE the annotation-of-annotation convention before it lands. The polymorphic-entity-id interpretation REQUIRES the validator to accept any UUID; that's the existing behavior.

- **D5 — No Playwright spec. The propagation is pinned at the Vitest layer (projector + bucketer cases); the existing predecessor Playwright block already exercises annotation graph-node mirror rendering at the e2e seam.** Rationale:
  - **The WBS author's explicit directive.** The task's `.tji` note says "Vitest cover (no Playwright — observable via the mirror)." Honoring that directive is the default unless the UI-stream e2e policy plainly says otherwise — it doesn't here, because the cited unit+component coverage IS load-bearing.
  - **The change is a thin Map-keying semantic.** What flips between "today" and "after this task" is: an annotation-created with `target_node_id` carrying another annotation's UUID gets bucketed under that UUID. The bucketer is target-kind-agnostic by construction; the data-flow from bucketer → projector → graph-node data is fully pinned by Vitest at the function-call layer. Playwright would re-seed the same wire payloads and assert the same mirror attributes — duplicating coverage without unique behavioral signal.
  - **The predecessor's Playwright block already exercises annotation graph-node mirror rendering at the e2e seam.** Block-12 of [`tests/e2e/participant-graph-render.spec.ts`](../../../tests/e2e/participant-graph-render.spec.ts) seeds an annotation-endpoint edge + materialized annotation node and asserts the mirror surfaces it. Adding a parallel block for the annotation-on-annotation case would pile on the file with very similar coverage. The UI-stream policy explicitly warns "Watch the inherited-debt count on `*_pw_*` catch-all e2e tasks" — and `participant-graph-render.spec.ts` is the proximal catch-all; adding to it without unique signal is exactly the pile-on the policy warns against.
  - **The component IS reachable.** Per the UI-stream e2e policy, the deferred-e2e exception applies only when the component is "not yet reachable." Annotation graph-nodes ARE reachable (predecessor task wired the route). But the policy's default-to-e2e posture explicitly notes the alternative "a thin Playwright spec that asserts component-presence + affordance-state-from-route is better than full deferral" — and that thin spec is exactly what the predecessor's block-12 already provides; the new behavior (the propagation) is a data-flow internal to projectGraph that's better pinned at the unit layer.
  - **ADR 0022 is satisfied.** Every empirical check is a committed Vitest case; no throwaway verification. The "no Playwright" choice is documented, defended, and aligned with the WBS author's intent.
  - **Alternative considered: add a 5th block to `participant-graph-render.spec.ts`.** Rejected — duplicates predecessor coverage; pile-on the proximal catch-all file; no unique behavioral signal beyond what Vitest pins.
  - **Alternative considered: defer Playwright to a future `part_pw_annotation_of_annotation_overlay` task.** Rejected — registers a thin debt item with no real coverage delta; the Vitest layer is the right pin for a data-flow Map-keying change.

- **D6 — Audience + moderator surface propagation is registered as Tech-debt named-future-tasks (`aud_annotation_of_annotation_overlay_chain`, `mod_annotation_of_annotation_overlay_chain`); not bundled here.** Rationale:
  - **Scope locality.** Each UI surface has its own annotation rendering vocabulary (audience: same shell bucketer + similar Cytoscape; moderator: ReactFlow with `groupAnnotationsByEdge` only today). Migrating each is a parallel surface task that doesn't share commit cycles with the participant.
  - **The shell-level export is in place.** This task's `groupAnnotationsByEntityId` addition at the shell layer means audience + moderator follow-ups can adopt the new name + propagation without a second extraction pass. The seam is ready.
  - **0.5d budget.** Bundling audience + moderator migration would 3× the touch surface + 3× the testing scope.
  - **Alternative considered: migrate all three surfaces in one commit.** Rejected — out of budget; muddies the per-surface task boundary the WBS deliberately maintains.

- **D7 — No new ADR. The polymorphic-entity-id convention is a Refinement-level decision documented here; the seams it touches (Cytoscape per ADR 0004, event envelope per ADR 0021, shell per ADR 0026, entity vs facet per ADR 0027) all exist and are honored.** Rationale:
  - **The convention is not an architectural commitment.** It's a clarification of an already-existing interpretation of the wire schema's `target_node_id` field as a polymorphic UUID slot. The wire schema doesn't change; the bucketer's runtime behavior doesn't change; only the export name + docstring make the convention legible.
  - **An ADR for wire-schema 3-way XOR (`target_annotation_id` explicit) is the natural ADR-level decision if/when adoption demands it.** That ADR would justify a real schema change; today's task doesn't make that commitment.
  - **Alternative considered: write an ADR for "annotation-of-annotation overlay propagation."** Rejected — over-specifies a convention that's already implicit in the wire schema and the shell bucketer; would set precedent for ADR'ing every consumer-side interpretation.
