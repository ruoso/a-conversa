## What this task is

This task closes the root-app side of ADR 0026's stack-validation contract ([docs/adr/0026-micro-frontend-root-app.md#L115](../../../docs/adr/0026-micro-frontend-root-app.md#L115)): "Vitest unit tests for the dispatcher, manifest loader, and providers; a Playwright smoke for `/login` → `/screen-name` → `/` for a new user." The dispatcher (`SurfaceHost`) and manifest loader (`apps/root/src/surfaces/manifest.ts`) already have Vitest coverage that landed under `root_moderator_cutover`, and `LandingRoute` is pinned at the four auth states under `root_landing`. What is missing — and is in scope here — is (a) Vitest coverage for the three auth-chrome routes whose branching is still implicit (`LoginRoute`, `ScreenNameRoute`, `AuthCallbackRoute`), and (b) a Playwright scenario that drives `/login` → SSO → `/screen-name` → `/` for a new user and asserts the user lands on the bare authenticated landing — the canonical happy path that no existing spec covers end-to-end (the closest, `auth-flow.spec.ts`'s `landing-to-lobby`, exits onto `/m/sessions/new` instead of `/`).

The Playwright scenario lands as an additional serial scenario in the existing [tests/e2e/auth-flow.spec.ts](../../../tests/e2e/auth-flow.spec.ts), not as a new top-level spec, so the dev Authelia rate-limit envelope around the OIDC dances stays intact. The Vitest additions land alongside the existing root-app suite at `apps/root/src/App.test.tsx` (or in route-co-located `*.test.tsx` files at the implementer's preference).

## Why it needs to be done

Three loops close here.

First, the ADR 0026 contract: the root app's `mount`-time public surface is the dispatcher, the manifest loader, the providers, and the auth-chrome routes, and ADR 0026 explicitly pinned "Vitest unit tests for the dispatcher, manifest loader, and providers; a Playwright smoke for `/login` → `/screen-name` → `/` for a new user" as the per-leaf test contract. The Vitest half has accumulated in pieces under `root_moderator_cutover` and `root_landing` but was never audited as a single set against the ADR's checklist; the Playwright half has never landed at all. This task is the audit and the smoke.

Second, the deferred-e2e debt from `root_landing`. [root_landing.md#L49](root_landing.md#L49) explicitly defers "Cross-route Playwright coverage of the landing (the `/login` → `/screen-name` → `/` new-user flow and the deep-link return-to round-trip ending on `/`) ... to `root_app.root_tests` (the sibling 1d task already in the WBS)." The refinement's Status block at [root_landing.md#L90](root_landing.md#L90) reaffirms this. The closer for `root_landing` did not register a separate follow-up because the inheritance points at this task by name; that promise is paid here.

Third, the regression class. The auth-chrome routes' branching is small but load-bearing — `LoginRoute` reads `useAuth().status` and dispatches four ways (loading → frame; unauthenticated → `window.location.replace('/api/auth/login')`; needs-screen-name → `<Navigate to="/screen-name">`; authenticated → `<Navigate to={resolvePostAuthTarget()}>`), and the same pattern repeats in `ScreenNameRoute`. Only one of the four `LoginRoute` arms is pinned today (the unauthenticated → SSO branch, in [App.test.tsx#L20](../../../apps/root/src/App.test.tsx#L20)); the other three are observable only by reading source. The two-consumer invariant on the screen-name flow's success transition — `ScreenNameForm`'s `onSuccess` callback deliberately does NOT navigate, leaving the route's auth-status re-render to be the sole consumer of the remembered return-to ([App.tsx#L81–L89](../../../apps/root/src/App.tsx#L81)) — is exactly the kind of contract that breaks silently when an unrelated edit "tidies up" the callback to do something useful. A Vitest pin is the right layer to lock it.

## Constraints / requirements

The existing `data-testid` values (`route-screen-name`, `route-logout`, `route-title`, `root-start-session`, `root-open-moderator`, `auth-login-button`, etc.) are unchanged; the new cases assert against them.

It uses a fresh `browser.newContext({ ignoreHTTPSErrors: true })` so cookies and sessionStorage from earlier scenarios in the block do not leak in.
- The Playwright scenario must drive the navigation client-side — visit `/` unauthenticated, assert the unauthenticated landing renders (eyebrow + body + start-session link + `LoginButton`), click the `LoginButton`, follow the redirect chain through Authelia via `loginAs`, fill the screen-name form (the new-user branch's `/screen-name?from=callback` arrival), and assert the final URL is `/` and the authenticated landing card renders with `maria` interpolated into the welcome title and the `root-open-moderator` affordance present. - The serial-scenario header comment at [auth-flow.spec.ts#L33–L62](../../../tests/e2e/auth-flow.spec.ts#L33) lists the five existing scenarios; the new scenario must be appended to that list (as #6) with the same level of detail, so future readers see the serial dependency at a glance.

- Per [ADR 0022](../../../docs/adr/0022-no-throwaway-verifications.md#L25), every probe in this work lands as a committed test in the same commit as the analysis; no throwaway scripts.

## Acceptance criteria

- Vitest coverage in `apps/root/src/` pins, at minimum:
  1. `LoginRoute` loading arm: status=`loading` renders a frame (no `auth-login-button`, no SSO redirect).
  2. `LoginRoute` unauthenticated arm: status=`unauthenticated` invokes `window.location.replace('/api/auth/login')`. (Already pinned at [App.test.tsx#L20](../../../apps/root/src/App.test.tsx#L20) — kept and not duplicated.)
  3. `LoginRoute` needs-screen-name arm: status=`needs-screen-name` `Navigate`s to `/screen-name`.
  4. `LoginRoute` authenticated arm with no remembered return-to: status=`authenticated` `Navigate`s to `/` (the `resolvePostAuthTarget` default) and the sessionStorage `'a-conversa:return-to'` slot remains empty.
  5. `LoginRoute` authenticated arm with a remembered return-to: with sessionStorage `'a-conversa:return-to'` pre-set to `/m/sessions/abc`, status=`authenticated` `Navigate`s to `/m/sessions/abc` and the sessionStorage slot is cleared.
  6. `ScreenNameRoute` loading arm: status=`loading` renders a frame (no `route-screen-name`).
  7. `ScreenNameRoute` unauthenticated-without-`?from=callback` arm: status=`unauthenticated`, no query param, `Navigate`s to `/login`. No form rendered.
  8. `ScreenNameRoute` unauthenticated-with-`?from=callback` arm: status=`unauthenticated`, `?from=callback`, the `route-screen-name` `<main>` renders and the `ScreenNameForm` is mounted (assert against a form element or the existing form's `data-testid`).
  9. `ScreenNameRoute` authenticated arm with a remembered return-to: with sessionStorage pre-set to `/p/sessions/xyz/invite`, status=`authenticated` `Navigate`s to that target and the sessionStorage slot is cleared.
  10. `AuthCallbackRoute` writes `?return_to=/m/sessions/abc` into the `'a-conversa:return-to'` sessionStorage slot and `Navigate`s to `/login`.
  11. `AuthCallbackRoute` without `?return_to`: leaves the sessionStorage slot untouched and `Navigate`s to `/login`. (Pins the no-op path so a regression that always calls `rememberReturnTo(null as any)` surfaces.)
- A Playwright scenario is appended to `test.describe.serial(...)` at [tests/e2e/auth-flow.spec.ts](../../../tests/e2e/auth-flow.spec.ts), using `maria`, that:
  1. Allocates a fresh `browser.newContext({ ignoreHTTPSErrors: true })`.
  2. Visits `/`, asserts the unauthenticated landing renders (`root-start-session` link visible, `auth-login-button` visible, the `root.landing.eyebrow` and `root.landing.body` translated strings present — assertion against the rendered text in the project's default locale is acceptable, the catalog parity check covers the other locales).
  3. Clicks the `auth-login-button`, waits for the URL to reach `authelia.aconversa.local`, then drives `loginAs(page, { username: 'maria' })`. `loginAs` fills the `/screen-name` form (new-user branch) and polls `/api/auth/me` until 200.

  5. Updates the serial-block header comment at [auth-flow.spec.ts#L33–L62](../../../tests/e2e/auth-flow.spec.ts#L33) to list the new scenario as #6 with the same level of detail as scenarios #1–#5.

## Decisions

- **Decision: append the Playwright scenario to the existing `tests/e2e/auth-flow.spec.ts` serial block rather than authoring a new top-level spec under `tests/e2e/root-app-smoke.spec.ts`.**
  - Alternative: new spec file `tests/e2e/root-app-smoke.spec.ts` matched by a new `chromium-root-smoke` project in `playwright.config.ts`, with its own per-test OIDC dance.
  - Alternative: new spec file matched by the existing `chromium-create-session`-style project, reusing the `setup-auth` bootstrap auth jar to short-circuit the OIDC dance.
  - Chosen because the new-user branch — the one this task specifically pins — cannot be exercised via the `setup-auth` bootstrap jar (the jar is `alice`'s post-login state, which short-circuits the dance, including the screen-name detour we are trying to assert). A spec file with its own per-test dance would add an OIDC handshake to the dev Authelia outside the existing serial-block rate-limit envelope, exactly the failure mode the `setup-auth` project was created to fix (see the serial-block comment at [auth-flow.spec.ts#L33–L62](../../../tests/e2e/auth-flow.spec.ts#L33) and the `setup-auth` rationale at [playwright.config.ts#L185–L215](../../../playwright.config.ts#L185)). Appending to the existing serial block reuses the existing rate-limit envelope, costs one new dance against the serial budget, and keeps the spec layout the same shape as the `landing-to-lobby` scenario that the `mod_session_lobby` work already added to the block under the same logic. The bookkeeping cost — that the file is nominally owned by `backend.backend_tests.be_e2e_tests.auth_flow_integration` per its header — is paid by leaving the existing refinement back-link in the file header intact and adding the new scenario's refinement back-link inline above the test function.

- **Decision: do not author a separate Playwright scenario for the deep-link-return-to-round-trip-ending-on-`/` case.**
  - Alternative: a second new serial scenario that pre-seeds `'a-conversa:return-to' = '/'` (or visits `/auth/callback?return_to=/`), drives the OIDC chain for `dave`, and asserts the landing on `/`.
  - Alternative: extend the chosen Playwright scenario to also drive the round-trip step before the OIDC dance.
  - Chosen because the round-trip-ending-on-`/` case is the same code path as the no-remembered-target default (`resolvePostAuthTarget()` returns `'/'` either way), and the per-route consumption of the remembered value is pinned at the Vitest layer by criterion #5 (`LoginRoute`-authenticated-with-remembered) and #9 (`ScreenNameRoute`-authenticated-with-remembered), plus the existing [LandingRoute.test.tsx](../../../apps/root/src/routes/LandingRoute.test.tsx) authenticated-with-remembered arm. The cross-route Playwright pin already discharged by `auth-flow.spec.ts`'s existing `landing-to-lobby` scenario (which proves the round-trip works end-to-end with a non-`/` target — `/m/sessions/new`) is what makes the round-trip mechanism itself trustworthy; a duplicate scenario that varies only in the remembered target's value would add wall-clock and rate-limit cost without pinning a new behavior. A regression specific to the `'/'`-target path would have to hide inside `sanitizeReturnTo`'s string handling, and that helper is already pinned at the unit layer in the SurfaceHost test surface.

- **Decision: pick `maria` as the new-scenario user.**
  - Alternative: `dave`, `erin`, or `frank`.
  - Alternative: reuse `ben` or `alice` (the users already in the serial block).
  - Chosen because the OIDC callback's new-user branch only fires for an unseen subject — `alice` is consumed by scenarios #1–#3 and `ben` by scenario #5, so both already have a `users` row by the time scenario #6 runs in CI's serial order. `maria` is the next-in-list per [tests/e2e/fixtures/dev-users.ts#L41–L46](../../../tests/e2e/fixtures/dev-users.ts#L41), so picking her preserves the pattern of "consume seeded users in declaration order" the existing serial block established and leaves `dave`/`erin`/`frank` for future serial-block additions.

- **Decision: scope Vitest coverage to LoginRoute / ScreenNameRoute / AuthCallbackRoute and skip LogoutRoute.**
  - Alternative: add LogoutRoute cases for completeness.
  - Rejected because LogoutRoute is already pinned end-to-end at [App.test.tsx#L105](../../../apps/root/src/App.test.tsx#L105) — the POST `/api/auth/logout`, the `window.location.replace('/login')`, and the sessionStorage clear are all asserted there. Adding a duplicate case under a different file would add maintenance cost without pinning anything new; the LogoutRoute branch coverage is complete.

- **Decision: do not test the providers (`BrowserRouter` + `AuthProvider` + `I18nextProvider`) as a separate Vitest unit.**
  - Alternative: a unit test that mounts `main.tsx`'s provider stack in isolation and asserts each context is reachable.
  - Rejected because the provider stack is three lines of composition (one `BrowserRouter`, one `AuthProvider` from the shell, one `I18nextProvider` from the shell), and every existing test that uses `renderWithProviders` ([apps/root/src/testing/renderWithProviders.tsx](../../../apps/root/src/testing/renderWithProviders.tsx)) exercises the same composition transitively. The shell's own Vitest suite already pins each provider's contract in isolation. A dedicated test of the composition would add bookkeeping cost without pinning a behavior that is not already pinned by the next layer up.

- **Decision: no new ADR.**
  - Alternative: a new ADR for the test-suite shape of the root app.
  - Rejected because the test-suite shape was already set by ADR 0026 ("Vitest unit tests for the dispatcher, manifest loader, and providers; a Playwright smoke for `/login` → `/screen-name` → `/` for a new user") and the layer-routing rules by ADRs 0006/0007/0008/0022. Nothing here introduces a new dependency, a new architectural seam, or a security-relevant trade-off — every decision above is either implementing ADR 0026's test contract or a tactical choice about where the assertions live. A new ADR would record nothing new.
