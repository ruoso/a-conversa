# root_landing

- **TaskJuggler entry**: [tasks/45-root-app.tji](../../45-root-app.tji#L41) — task `root_app.root_landing`
- **Effort estimate**: 0.5d
- **Inherited dependencies**:
  - `root_app.root_moderator_cutover` (settled — the root host already serves `/`, owns the top-level `BrowserRouter` + shell-provided `AuthProvider`/`I18nProvider`, dispatches `/m/*` via `SurfaceHost`, and the moderator now mounts as a region rather than a standalone SPA; see [tasks/refinements/root-app/root_moderator_cutover.md](root_moderator_cutover.md#L94) and [apps/root/src/App.tsx](../../../apps/root/src/App.tsx#L233)).
  - As a consequence of the cutover absorbing this slot's wiring early, a working `LandingRoute` already exists at [apps/root/src/App.tsx](../../../apps/root/src/App.tsx#L27) covering both auth states plus the remembered return-to consumption. This task's job is to make that landing's contract explicit, close the i18n + extraction gaps left by the cutover, and pin the landing-specific behavior in committed coverage. It is not a from-scratch implementation.

## What this task is

This task formalizes the `/` landing route owned by `apps/root/`. The route renders differently for the four `useAuth()` states the shell exposes (`loading`, `needs-screen-name`, `unauthenticated`, `authenticated`), participates in the host's deep-link round-trip by consuming the `sessionStorage`-stored return-to that `SurfaceHost` writes when an unauthenticated visitor hits a protected surface route, and offers the only landing-page affordance present in the product today — a `Create Session` handoff into `/m/sessions/new` plus the auth-chrome edges (`LoginButton` when unauthenticated, `Sign out` when authenticated).

The scope is intentionally small. The cutover already wired the route, the providers, the surface dispatcher, and the moderator handoff into place; what is missing — and is in scope here — is (a) extracting `LandingRoute` from its inline definition in `apps/root/src/App.tsx` into its own module so that landing-only changes do not perturb the auth-chrome route definitions, (b) routing the two remaining hardcoded copy strings (`a-conversa` eyebrow, `Micro-frontend host app` body) through `@a-conversa/i18n-catalogs` so the landing is internally consistent with the rest of the surface, and (c) explicit Vitest coverage for the four-state branching and the remembered return-to behavior so the landing has a stand-alone behavioral pin and is not only indirectly exercised by the broader auth-chrome and dispatcher tests.

## Why it needs to be done

The landing is the only route the root app exposes that has actual content of its own. Every other root-owned route is either auth chrome that delegates to shell (`ScreenNameForm`, `LoginButton`, the SSO redirect) or a `SurfaceHost`-mounted region. The landing therefore carries the entire surface area for "what does an authenticated user see when they navigate to the bare host?" and "what does an unauthenticated visitor see when they land on the domain without a deep link?" Today both of those answers exist in code but neither is captured anywhere as the intended product surface, so the next person who edits `apps/root/src/App.tsx` has no signal about which parts of `LandingRoute` are load-bearing.

The remembered return-to consumption on `/` is the third arm of a three-way pact with `LoginRoute` (App.tsx:142) and `ScreenNameRoute` (App.tsx:172): all three must consume the sessionStorage value via `takeRememberedReturnTo()` when they observe `status === 'authenticated'`, and only one of the three may consume it for a given visit. The cutover refinement called this out in a code comment on `LandingRoute` (App.tsx:39–47) but did not promote the invariant into a refinement or a regression test. Locking it down here is cheap insurance against the next round of edits accidentally re-introducing a path where the user lands on the welcome card instead of the deep link they originally requested.

The hardcoded eyebrow and body strings predate the moderator's broader i18n discipline (see the `auth.login.*` and `moderator.createSession.*` namespaces this same component already consumes) and are visible enough — they are literally the first paint a fresh visitor sees — that leaving them untranslated reads as an oversight. Closing the gap now keeps `/` from being the only screen in the host that ships English-only copy.

## Inputs / context

- The existing landing implementation, in full, lives in [apps/root/src/App.tsx](../../../apps/root/src/App.tsx#L27) as the `LandingRoute` component (lines 27–114), the surrounding routes table (lines 233–247), and the `LoadingFrame` helper (lines 15–25).
- The remembered-return-to plumbing — `rememberReturnTo()`, `takeRememberedReturnTo()`, and the `'a-conversa:return-to'` sessionStorage key — is defined in [apps/root/src/surfaces/SurfaceHost.tsx](../../../apps/root/src/surfaces/SurfaceHost.tsx#L9) and re-exported alongside `SurfaceHost`. `LandingRoute`, `LoginRoute`, and `ScreenNameRoute` all consume it; `SurfaceHost`'s unauthenticated deflection at [apps/root/src/surfaces/SurfaceHost.tsx](../../../apps/root/src/surfaces/SurfaceHost.tsx#L149) is the only writer.
- The shell auth contract is fixed by [packages/shell/src/auth/types.ts](../../../packages/shell/src/auth/types.ts#L33): `status` is the discriminated union `'loading' | 'unauthenticated' | 'needs-screen-name' | 'authenticated'`; `user` (a `{ userId, screenName }` pair, by deliberate minimum-disclosure design — no role field, no email) is present only in the `authenticated` arm.
- The existing i18n keys the landing already consumes — `auth.login.title`, `auth.login.welcome`, `auth.login.checking`, `auth.login.logout`, and `moderator.createSession.title` — are defined in [packages/i18n-catalogs/src/catalogs/en-US.json](../../../packages/i18n-catalogs/src/catalogs/en-US.json#L6) and ship with their `pt-BR` and `es-419` siblings in the same directory. The `auth.login.welcome` value carries an ICU `{name}` placeholder which the landing already feeds with `auth.user.screenName`.
- ADR continuity: [docs/adr/0026-micro-frontend-root-app.md](../../../docs/adr/0026-micro-frontend-root-app.md#L60) fixes "Landing route at `/` (authenticated + unauthenticated states)" as a root-app responsibility, and the ADR's roadmap note (line 13) leaves the door open for "a marketing-thin landing page that loads quickly" later — this refinement scopes only the minimum that has to exist today.
- Sibling refinement [tasks/refinements/root-app/root_tests.md](root_tests.md) (pending) carries the cross-route Playwright smoke that exercises `/login` → `/screen-name` → `/` for a new user. Landing-only Vitest coverage lands here; the in-browser end-to-end pin for the same flow stays with `root_tests` rather than being duplicated.
- The existing root-app Vitest suite at [apps/root/src/App.test.tsx](../../../apps/root/src/App.test.tsx#L30) already pins the unauthenticated landing's `root-start-session` link target and the authenticated landing's `root-open-moderator` affordance. The gap this task closes is the `loading` and `needs-screen-name` branches and the remembered-return-to consumption on `/`.

## Constraints / requirements

- The landing must keep handling all four `useAuth()` states. `loading` renders the shell-styled `LoadingFrame` (no flash of the welcome card before auth resolves); `needs-screen-name` redirects to `/screen-name` (mid-onboarding users never see the welcome card); `authenticated` either consumes a remembered return-to (deep-link round-trip) or renders the welcome card; `unauthenticated` renders the unauthenticated landing.
- The remembered-return-to consumption must run on the `/` route as well as on `/login` and `/screen-name`. A user who clicked a protected deep link, completed SSO, and was 302'd back to the root by the OIDC callback (which has no awareness of the original target — that lives only in sessionStorage) must end up at the deep link, not at the welcome card. The current single-consumer invariant (only one of the three auth-chrome routes consumes the value for a given visit, since `takeRememberedReturnTo()` deletes it) must be preserved.
- The landing must not introduce client-side role-gating. The auth shape is `userId + screenName` only, by deliberate design; both authenticated and unauthenticated users continue to see the same `Create Session` handoff, and the moderator surface continues to enforce its own `RequireAuth` gate on `/m/sessions/new`. This is not a regression — it is the established product contract from the cutover and from the OIDC minimum-disclosure design.
- All user-visible strings on the landing must resolve through `react-i18next` against the shared catalogs. The two remaining hardcoded strings — the `a-conversa` eyebrow and the `Micro-frontend host app` body caption — must be replaced with translated keys. New keys land in `en-US.json` plus their `pt-BR` and `es-419` siblings; the parity script `packages/i18n-catalogs/scripts/check-parity.ts` must stay green.
- `LandingRoute` must move into its own module (suggested: `apps/root/src/routes/LandingRoute.tsx`) so it is testable in isolation and so the App.tsx file shrinks back toward "routes table + auth chrome handlers." The existing `data-testid` values (`route-home`, `route-title`, `root-authenticated-eyebrow`, `root-open-moderator`, `root-logout-link`, `root-start-session`, `auth-login-button`) must keep their current spellings so the existing App.test.tsx assertions and any downstream Playwright selectors keep working.
- No new dependency, no new ADR-level architectural choice. This task lives strictly inside ADR 0026's already-accepted root-app responsibilities; if implementation surfaces a new architectural question (e.g. moving from a stand-alone landing to a separately-bundled landing surface as ADR 0026's line-13 roadmap hints at) that becomes a stop-and-write-an-ADR event, not part of this 0.5d slot.

## Acceptance criteria

- `LandingRoute` is its own module under `apps/root/src/routes/` and is consumed by `apps/root/src/App.tsx`'s routes table. `apps/root/src/App.tsx` no longer defines the landing UI inline. The existing `data-testid` values listed above are unchanged.
- The eyebrow and body strings on both the authenticated and unauthenticated landings resolve through `react-i18next`. New keys are added to `packages/i18n-catalogs/src/catalogs/en-US.json` plus the `pt-BR` and `es-419` catalogs; the catalog parity check (`packages/i18n-catalogs/scripts/check-parity.ts`) stays silent.
- Vitest coverage in `apps/root/src/routes/LandingRoute.test.tsx` (or alongside the existing `App.test.tsx`, at the implementer's preference) pins, at minimum: (1) the `loading` arm renders `auth-checking` and does not render `route-home`; (2) the `needs-screen-name` arm `Navigate`s to `/screen-name`; (3) on `/` with `status === 'authenticated'` and a sessionStorage `'a-conversa:return-to'` value set, the route `Navigate`s to that target and clears the storage entry; (4) on `/` with `status === 'authenticated'` and no remembered value, the welcome card renders with the user's screen name interpolated into the title; (5) the unauthenticated arm renders the start-session link and the `LoginButton` (already partially pinned by App.test.tsx:30 — extended to also assert the eyebrow + body strings now resolve through i18n, not literal text).
- Per [ADR 0022](../../../docs/adr/0022-no-throwaway-verifications.md#L31), all the above coverage lands as committed tests in the same commit as the refactor; no throwaway scripts.
- `pnpm run check` and `pnpm run test:smoke` stay green after the changes.
- Cross-route Playwright coverage of the landing (the `/login` → `/screen-name` → `/` new-user flow and the deep-link return-to round-trip ending on `/`) is **explicitly out of scope here and deferred to** `root_app.root_tests` (the sibling 1d task already in the WBS). The closer for this task does not need to register a new follow-up — `root_tests` already inherits the responsibility from its own dependency chain.

## Decisions

- **Decision: extract `LandingRoute` to its own module rather than leaving it inline in `App.tsx`.**
  - Alternative: keep all root-app routes (LandingRoute, LoginRoute, ScreenNameRoute, LogoutRoute, AuthCallbackRoute) inline in `App.tsx`.
  - Alternative: extract every route to its own module in the same commit ("if we extract one, extract all").
  - Chosen because the landing is the only one of the five with real product UI to maintain (the other four are mostly imperative redirects + small wrappers around shell components), and because the next round of edits to the landing — translated strings, expanded affordances if the ADR-0026 roadmap-line-13 marketing landing later happens — should not have to touch `App.tsx`. Extracting the auth-chrome routes as well is a larger refactor than this slot's 0.5d budget should pay for, and would dilute the test surface of a small focused change; leaving them in App.tsx until they accumulate their own reason to move is the cheaper call.

- **Decision: i18n the two remaining hardcoded strings in this same task.**
  - Alternative: defer the i18n work to a separate `frontend_i18n` follow-up.
  - Alternative: leave the strings untranslated indefinitely.
  - Chosen because the landing already routes five other strings through `react-i18next` (lines 20, 66, 75, 82, 98, 107 in App.tsx), so the two outliers are an internal inconsistency rather than a missing capability. The catalog edit is small, the parity script enforces shape correctness, and bundling the change with the extract avoids two separate commits touching the same component. Indefinite untranslated strings are unacceptable because they are visible above the fold to every visitor.

- **Decision: pin the remembered-return-to consumption on `/` with explicit Vitest coverage.**
  - Alternative: rely on the existing App.test.tsx coverage of the `/login` → SSO flow and the manifest-level deep-link deflection at line 75–103.
  - Alternative: defer this assertion to the cross-route Playwright smoke in `root_tests`.
  - Chosen because the three-way pact (LoginRoute, ScreenNameRoute, LandingRoute all consuming the same single-use sessionStorage entry) is exactly the shape of invariant that Cucumber-or-Playwright-only coverage misses — a regression in `LandingRoute`'s consumption arm would still let the end-to-end flow pass through `LoginRoute` in the common case and only surface when the OIDC callback's specific 302-back-to-`/` branch fires. A Vitest assertion on the `/`-arm consumption is the right layer to pin the per-route contract; the e2e in `root_tests` then validates the integrated flow.

- **Decision: keep `Create Session` visible to unauthenticated visitors instead of hiding it behind auth.**
  - Alternative: only show the create-session handoff to authenticated users; show only the login button to unauthenticated visitors.
  - Alternative: rewrite the landing's affordance set entirely (e.g. "Join a session" vs "Create a session" with role-style ambiguity).
  - Chosen because it preserves the cutover's established surface and the moderator's `RequireAuth` gate on `/m/sessions/new` is the canonical authorization point — the landing's link is a convenience handoff, not the gate. Hiding it from unauthenticated visitors would force them to click through `Sign in with SSO` and then a separate "create session" affordance, which is a worse first-visit flow. Rewriting the affordance set is outside this 0.5d slot's scope; if and when the ADR 0026 line-13 marketing landing happens, that will be its own task with its own ADR-tier decisions.

- **Decision: no new ADR.**
  - Alternative: write a new ADR for the landing's surface and affordance shape.
  - Rejected because every decision above is either implementing ADR 0026 (the route exists at `/`, both auth states, owned by the root) or is a tactical implementation choice (extract vs inline, where the i18n keys live, what `data-testid`s to keep). Nothing here introduces a new dependency, a new architectural seam, or a security-relevant trade-off. The ADR-0026-line-13 future marketing landing, when it lands, will need its own ADR; this task does not pre-empt it.
