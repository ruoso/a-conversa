## What this task is

Re-source the moderator's per-entity, per-facet status from the server-broadcast `proposal-status` stream rather than the client-side `computeFacetStatuses(events)` derivation that runs on every WS event today.

Each `proposal-status` envelope already addresses exactly one `(entity, facet)` per the per-target loop landed in `facet_status_server_decompose_component_facets`; the new fields make the entity identity explicit on the wire so receivers can build a per-`(entityKind, entityId, facet)` cell without re-walking the proposal payload.

## Why it needs to be done

Three reasons compose:

**Single-source-of-truth invariant.** Per the predecessor's "Why it needs to be done" and the canonical [`ws_proposal_status_broadcast`](../backend/ws_proposal_status_broadcast.md) rationale, the `proposal-status` broadcast lets the server be the single source of truth for facet status, and clients render directly without re-implementing the precedence rules. The moderator surface today violates this — it consumes `computeFacetStatuses(events)` and runs the full derivation locally — which is exactly the drift the broadcast was designed to prevent. The participant (M5) and audience (M6) surfaces will consume the broadcast directly; landing this migration before those surfaces ship means the broadcast-derivation path is exercised end-to-end on the moderator first (where the v1 cover is densest), validating the contract for the lighter-weight downstream surfaces.

**Predecessor commit's deferred lockstep.** The predecessor's D5 explicitly left the client mirror in place because "removing the per-component arm without first migrating the moderator off `computeFacetStatuses` would break the moderator's canvas styling for pending decompose / interpretive-split." This task is the migration the predecessor anticipated. Without it, the per-component arm at L418–443 stays in `packages/shell/src/facet-status/facet-status.ts` as duplicated derivation logic, with the server-side authoritative arm in `apps/server/src/ws/broadcast/proposal-status.ts` as a parallel implementation that the moderator never reads. The lockstep maintenance burden (any extension to facet derivation has to land in both places, with no compiler-level enforcement they stay in sync) is real cost per commit.

**Wire-shape gap exposed by multi-envelope-per-proposal.** The current `proposal-status` envelope shape `{ sessionId, proposalId, sequence, perFacetStatus: { [facet]: status } }` does NOT carry `entityKind` + `entityId`. For single-target sub-kinds (`classify-node`, `set-node-substance`, etc.) the target entity is implicit in the proposal payload and the receiver can resolve it via the events stream. For multi-component sub-kinds (`decompose`, `interpretive-split`) the predecessor emits N envelopes that all share `proposalId` + `sequence` and differ only by server-minted UUID — the receiver CANNOT disambiguate which component each envelope is about without joining against the proposal payload's `components[i]` / `readings[i]` array AND assuming the iteration order matches the predecessor's D7 contract. The current shell store keyed by `proposalId` (last-write-wins) loses N − 1 of the N envelopes silently. This task closes the wire-shape gap by making each envelope carry its target entity explicitly — a small additive extension that the predecessor's D1(C) discussion ("extend `perFacetStatus` to allow per-entity keying — rejected") considered the nested-keying variant of but rejected for shape-evolution reasons; the top-level-fields variant adopted here keeps `perFacetStatus` shape stable AND adds explicit entity identity.

Downstream consumers benefit transitively:

- `participant_ui.part_graph_view` (M5) — inherits a broadcast-derived rendering path validated by the moderator's denser e2e cover before participant tablets ever wire in.
- `audience.aud_graph_rendering` (M6) — same transitive inheritance.
- The shell mirror's six facet-targeting arms become deletable in a future cleanup once the remaining moderator-side callers (`apps/moderator/src/graph/selectors.ts`, methodology engine) migrate too — but that's downstream of this task per the `.tji` note.

## Constraints / requirements

- **No new wire envelope kind.** The `'proposal-status'` envelope type stays on the closed `WsMessageType` union; the schema extension is additive (two required top-level fields) — receivers running on the older schema would fail validation, but every receiver lives in the same monorepo and ships in lockstep with the server per [`docs/dev-environment.md`](../../../docs/dev-environment.md)'s compose-stack model.
- **`perFacetStatus` shape stays unchanged.** Each envelope continues to carry `{ [facetName]: status }` with exactly one entry; the per-entity dimension lives in the new top-level fields, not nested inside `perFacetStatus`. Preserves the predecessor's D1 rejection of nested per-entity keying while still making the entity identity explicit.
- **Reconnect-seed envelopes are sent to the requesting connection ONLY, NOT broadcast.** The snapshot / catch-up case-2 paths are request/response (one client asks for state; one client receives it). The seed envelopes follow the same single-recipient model — they are NOT additional broadcasts to the session's other connections. Implementation: call `connection.socket.send(serializeWsEnvelope(envelope))` per seed envelope, NOT the broadcast bus.
- **Seed envelopes follow `snapshot-state` in send order on the wire.** The receiver applies `snapshot-state` first (advancing `lastAppliedSequence`), then the seed envelopes populate the per-entity facet-status map. Order matters because the seed envelopes' `sequence` field equals the projection's `lastAppliedSequence` at snapshot-build-time; receivers MUST NOT discard them as already-applied.
- **Seed envelopes are NOT registered in any persistence layer.** They are derived state, not events; replay (per ADR 0027 and the projector contract) ignores them. The seed exists only to populate the receiver's per-entity map on initial connect because `proposal-status` broadcasts do not replay automatically (the source-of-debt observation from the `.tji` note).
- **Per-entity map shape: `Map<string, FacetStatus>` keyed by `${entityKind}:${entityId}:${facetName}`.** Per D2 — `Map<>` rather than `Record<>` so the contract reads "cell-or-absent" cleanly and the missing-key path is the universal "no pending proposal affects this facet" signal. Plain composite-string key (not a tuple) so the map is JSON-serializable for any future snapshot / persistence consumer.
- **`clearProposalFacetStatusForEntity` removes all matching cells.** When an `entity-removed` event lands, every key with the matching `${entityKind}:${entityId}:` prefix is deleted from the map. Linear-scan removal is acceptable — the map is small (one cell per pending facet target; bounded by the session's pending-proposal count × facets-per-proposal). Per D3.
- **Decompose / interpretive-split arm at `facet-status.ts:418-443` is deleted in this commit, not deprecated.** Per the predecessor's D5 — the arm exists ONLY to serve the moderator's `computeFacetStatuses(events)` consumer, and this task removes the consumers. The six remaining facet-targeting arms in the same function stay (other callers; see `selectors.ts`, methodology engine).
- **`ProposalFacetBreakdown`'s `resolveStatus` precedence collapses to "server frame → default `'proposed'`".** The client-mirror branch is dead code for facets the broadcast covers (all six facet-targeting + two multi-component sub-kinds — full coverage for the proposals this component renders); deleted in this commit. The default `'proposed'` fallback stays as a defensive guard during the snapshot-seed → first-render window and for the brief edge case where a proposal lands but its `proposal-status` envelope hasn't yet (per the post-`event-applied` dispatch order from `ws_proposal_status_broadcast`).
- **Existing `pendingProposals` slot is removed entirely.** The only consumer today is `PendingProposalsPane.tsx:690`'s `pendingProposals?.[row.proposalEventId]?.perFacetStatus` lookup; that lookup is rewritten to resolve the proposal's target entity (from the proposal payload, available in `events`) and query the new per-entity map. No deprecation window; the migration is atomic per commit.
- **Schema-on-write per ADR 0021.** Every emitted envelope (live broadcast + reconnect-seed) passes the extended `proposalStatusPayloadSchema`; the receiver's WS client validates on decode per the existing pipeline.
- **Per-connection error isolation per the existing contract.** Reconnect-seed sends are wrapped in try/catch with warn-log on failure; one failed seed envelope does not abort the remaining envelopes (mirrors the broadcast listener's per-target isolation from the predecessor's D7 / per-connection isolation from `ws_proposal_status_broadcast`).
- **Test coverage layered per ADR 0022.** Pure-logic shell-store behaviour → Vitest. Server-side seed emission → Vitest at the handler layer. Cross-surface end-to-end (the user-visible behaviour the migration preserves) → Playwright re-pin of `moderator-proposed-entity-canvas-visibility.spec.ts` + the new reconnect sub-step. ## Decisions

- **D1 — Extend `proposalStatusPayloadSchema` with top-level `entityKind` + `entityId` (NOT nested inside `perFacetStatus`).** The predecessor's D1(C) rejected `perFacetStatus: Record<entityId, Record<facetName, status>>` (nested per-entity keying) because it's a wire-shape breaking change AND the existing single-entity-per-envelope pattern is simpler. This task adopts a DIFFERENT shape: keep `perFacetStatus` keyed by facet name only (one entry per envelope, unchanged), AND add `entityKind` + `entityId` as required top-level fields alongside `proposalId` + `sequence`. Rationale:
  - **Preserves the predecessor's single-facet-per-envelope contract.** `perFacetStatus` shape stays — receivers that already read `.classification` / `.wording` / `.substance` from it don't change.
  - **Makes entity identity explicit on the wire.** The predecessor's per-component fan-out (D2: per-envelope distinct UUIDs, shared `proposalId` + `sequence`) currently leaves the receiver to disambiguate components via the proposal payload (which requires joining against the events stream and assuming D7 iteration-order matches). Top-level entity fields make the disambiguation trivial — every envelope is self-describing.
  - **Schema-on-write per ADR 0021.** `z.enum(['node', 'edge'])` + `z.string().uuid()` are tight constraints; receivers fail validation on malformed envelopes per the existing pipeline.
  - **Alternatives considered:**
    - **(B) Keep envelope shape; key shell store by envelope `id`.** Rejected — receivers still cannot tell which entity each envelope refers to. The disambiguation gap doesn't move; it just shifts from "client lookup" to "client cannot lookup at all."
    - **(C) Keep envelope shape; require receivers to join against the proposal payload + D7 iteration order.** Rejected — fragile (depends on undocumented client-side order tracking), couples the receiver to the per-target iteration contract, and requires every consumer to re-implement the join.
    - **(D) Introduce a new envelope kind `'component-facet-status'` for the multi-component case only.** Rejected per the predecessor's D1(B) — adds a closed-union member to maintain, splits the receiver's per-facet handling into two parallel code paths. The additive top-level-fields extension keeps the handling unified.
    - **(E) Defer the wire change; pin the migration to the single-target sub-kinds only (skip the decompose / interpretive-split arm deletion).** Rejected — the `.tji` task title + scope item (4) explicitly require the per-component arm removal, and that removal is impossible without per-envelope entity identity (the predecessor's D5 made exactly this point in deferring the migration).

- **D2 — Shell-store slot: `pendingProposalFacetStatus: Map<string, FacetStatus>` keyed by `${entityKind}:${entityId}:${facetName}`.** The receiver-side natural shape is "given an entity + facet, what's the pending-proposal-derived status?" Per-entity-keyed access is what every consumer wants (`GraphCanvasPane` reads per-node-id; `PendingProposalsPane` reads per-proposal-target-id; `ProposalFacetBreakdown` reads per-(proposal-target, facet)). Rationale:
  - **`Map<>` rather than `Record<>`** — the "absent" path is the universal signal for "no pending proposal affects this facet"; `Map.has(key)` reads cleanly; iteration for the `clearProposalFacetStatusForEntity` walk is natural.
  - **Composite-string key** — JSON-serializable for any future snapshot / persistence consumer; trivially constructible at write- and read-sites with a small `cellKey(entityKind, entityId, facetName)` helper.
  - **Last-write-wins per cell** — per-(entity, facet) is the natural granularity. The live broadcast emits one envelope per cell per status-affecting event; each write replaces the prior status (transitions `'proposed' → 'committed'` per Case D in the predecessor naturally).
  - **Alternatives considered:**
    - **(B) Keep `Record<proposalId, ProposalStatusPayload>` and accumulate per-proposal arrays.** Rejected — receivers still need a per-entity index to render; the array would be walked + filtered on every render. Per-entity map collapses the lookup to O(1) directly.
    - **(C) Nested `Map<entityKind, Map<entityId, Record<facetName, FacetStatus>>>`.** Rejected — three-level nesting for what is naturally a one-level cell-lookup; the cellKey-as-composite-string is simpler with no observable difference for the receiver.

- **D3 — `entity-removed` events trigger `clearProposalFacetStatusForEntity` in the store.** Per the predecessor's D3 — "withdraw transition: no terminal envelope; rely on the `entity-removed` event-applied frames." The shell store's `applyEvent` extension is where this contract executes for the moderator's per-entity map: on each `entity-removed`, walk the map and delete every `${entityKind}:${entityId}:*` cell. Rationale:
  - **Mirrors the broadcast listener's per-target isolation.** The server-side listener returns early on `entity-removed` (not in `STATUS_AFFECTING_KINDS`); the store side completes the contract on the client.
  - **Per-event idempotent.** Deleting absent cells is a no-op; replaying the same `entity-removed` (catch-up edge case) is safe.
  - **No new event kind, no new envelope.** The migration uses the existing `entity-removed` event-applied broadcast as its clear-signal; the predecessor's D3 explicitly designed this seam.
  - **Alternatives considered:**
    - **(B) Server emits a terminal `proposal-status` envelope with `perFacetStatus: { classification: 'withdrawn' }` on withdraw.** Rejected per the predecessor's D3(B) — `'withdrawn'` is a facet-layer terminal status for committed-then-disputed flow, NOT for retracted-from-pending; the semantics don't match.
    - **(C) Time-based eviction (TTL on cells).** Rejected — adds complexity; the `entity-removed` event is already the deterministic signal.

- **D4 — `ProposalFacetBreakdown.resolveStatus` collapses to "server frame → default `'proposed'`"; client-mirror branch is dead-code deleted.** The current precedence is server frame → client mirror → default `'proposed'`. Post-migration: the per-entity map covers all facet-affecting cases the broadcast handles (the six single-target sub-kinds + the two multi-component sub-kinds, per `facetTargetsForProposal`). The client-mirror branch becomes unreachable for every proposal kind this component renders. Rationale:
  - **Keeps dead-code rot out.** A defensive fallback to a code path that can never fire is worse than no fallback — it lies to readers about which path is reachable.
  - **The default `'proposed'` fallback stays.** It still fires during the post-`event-applied` / pre-`proposal-status` window (the predecessor's `ws_proposal_status_broadcast` dispatch order pins `event-applied` first, then `proposal-status`; a render between the two would otherwise show undefined styling).
  - **Alternative considered: keep the client-mirror branch as a "double-source" defensive guard.** Rejected — defeats the single-source-of-truth invariant this task is designed to enforce; the broadcast IS the source of truth.

- **D5 — Selector pattern for the per-entity map: `useProposalFacetStatusMap()` returning a `FacetStatusIndex`-shaped object.** The current `computeFacetStatuses(events)` returns a `FacetStatusIndex` (`{ nodes: Map<entityId, Record<facet, status>>, edges: Map<entityId, Record<facet, status>> }`) that consumers (`GraphCanvasPane`, `PendingProposalsPane`) use via `.nodes.get(id)?.[facet]`. The new selector preserves this shape so consumers' inner lookup code stays unchanged — only the source of the map changes. Rationale:
  - **Minimizes consumer-side churn.** The `data-facet-status` plumbing in `StatementNodeData` + the per-row breakdown in `PendingProposalsPane` keep working against the same `.nodes.get(id)?.[facet]` shape.
  - **The selector is a thin adapter** — walks the per-`${entityKind}:${entityId}:${facetName}` map and rebuilds the two-level shape on each render. With Zustand's selector pattern, the adapter is `useStore(s => buildFacetStatusIndex(s.pendingProposalFacetStatus))` with a `shallow` equality if needed.
  - **Memoization is at the consumer level.** `GraphCanvasPane` already wraps `facetStatusIndex` derivation in `useMemo`; the new selector slots in the same way. `PendingProposalsPane` does the same.
  - **Alternative considered: expose the raw `Map<string, FacetStatus>` and rewrite every consumer to use the cellKey helper.** Rejected — touches every consumer's inner lookup loop for no behavioural win; the adapter shape is cheaper to land + safer to review.

- **D6 — Tolerate the predecessor's "Case D divergence" gap (post-commit-decompose components render `'proposed'` instead of `'committed'`) for now; migration preserves observable behaviour.** Per the predecessor's Status block L179, `applyCommittedProposal`'s decompose arm at `apps/server/src/projection/replay.ts` only flips `parent.visible = false` post-commit — it does NOT mark per-component facets committed. So `deriveFacetStatus` returns `'proposed'` for committed-decompose components today. The client-side mirror at `facet-status.ts:418-443` has the SAME gap (it walks the proposal kind only, not the commit transition; the components stay in the `proposed`-derived branch). The migration swaps the SOURCE of the `'proposed'` value (server broadcast instead of client mirror) but preserves the displayed value. Rationale:
  - **Migration is behaviour-preserving by design.** A user looking at the moderator canvas before and after the migration sees the same `data-facet-status="proposed"` on a committed-decompose component. The divergence is a projection-arm gap, not a migration concern.
  - **Closing the gap is registered as a separate task.** Per the `.tji` block: `data_and_methodology.methodology_engine.replay_decompose_commit_marks_component_classification_committed` — landing that task will cause BOTH the broadcast-derived map AND (if anyone still consumes it) the client mirror to flip the value together.
  - **NOT a hard dependency.** The migration ships before the projection fix is fine — observable behaviour is preserved.
  - **Alternative considered: block this migration on the projection-arm fix.** Rejected — the gap doesn't exist in the migration's user-observable surface; sequencing the projection fix first adds no value to the moderator UX and just delays the source-of-truth swap.

- **D7 — Reconnect-seed envelopes are sent on the requesting connection only (NOT broadcast), and after `snapshot-state` (NOT before).** The seed exists to populate the requesting client's per-entity map; the other clients in the session already have populated maps from the live broadcasts that have been firing. Sending the seed as a broadcast would be a duplicate write for those clients (idempotent but wasteful) AND would conflict with the per-connection error-isolation invariant. After `snapshot-state` because the receiver's map-population should reflect the projection state AT the snapshot's `lastAppliedSequence`; sending before would race the snapshot's `applySnapshot` store write. Rationale:
  - **Mirrors `snapshot-state`'s own request/response shape.** That envelope is `inResponseTo`-correlated and sent on the requesting connection only; the seed envelopes follow the same lifecycle.
  - **Per-connection error isolation extends naturally.** Per-seed try/catch; one failed seed doesn't poison the rest of the seed round or the rest of the connection.
  - **Alternative considered: include the per-entity facet status in the `snapshot-state` envelope itself.** Rejected — `snapshot-state.payload.projection` is `z.unknown()` per the schema's "wire-shape-loose, serialization-helper-tight" design (per `ws-envelope.ts` L861–870); cramming derived facet-status into it tightens the coupling between the snapshot wire shape and the broadcast wire shape that the schema-on-write convention explicitly avoids. Separate envelopes preserve the separation.
  - **Alternative considered: client derives the seed itself from the snapshot's projection (running `deriveFacetStatus` equivalent in JS).** Rejected — duplicates the server-side derivation logic on the client (the exact failure mode the broadcast was designed to prevent); see the predecessor's D5 for the rationale on keeping facet status server-authoritative.
