## What this task is

Two deliverables, in lockstep:

1. **Participant dispute affordance.** Add the user-visible gesture that lets a debater dispute a committed annotation's `substance` facet. Concretely: (a) make the annotation rows in the participant detail panel's annotations list *clickable* so selecting one opens that annotation's own detail (`selection.kind === 'annotation'`); and (b) mount a **dispute** affordance in the annotation-detail `actionSlot` that calls the already-built seam `useVoteAction({ entity_kind: 'annotation', entity_id, facet: 'substance' }).castVote('dispute')`. No backend, schema, hook-type, or projector change — the seam landed with `annotation_facet_vote_seam`; this is purely the UI surface that drives it.

2. **Moderator Playwright round-trip.** Land the e2e spec that drives the full cross-surface flow against the compose stack: moderator captures a node and commits a reframe meta-move (→ `annotation-created`), a debater disputes the resulting annotation through the new affordance, and the moderator's graph badge gains `data-facet-status="disputed"` live over WS. This is the round-trip the `.tji` note scopes ("commit + participant-dispute round-trip") and the disputed-annotation e2e that three prior refinements deferred to this wiring task.

## Why it needs to be done

`mod_meta_move_disputed_visibility` built the rose badge but deferred its Playwright cover (Decision §4) because "the end-to-end conditions that produce a disputed meta-move annotation are not yet reachable." `annotation_facet_status_logic` lit the agreed/committed status but again deferred the disputed e2e (its AC: "deferred to `mod_annotation_dispute_e2e`") because the unanimity gate blocks any disputed state without a vote seam. `annotation_facet_vote_seam` then supplied the vote path and confirmed (its AC Playwright bullet) that the `[data-facet-status="disputed"]` moderator round-trip belongs here, and that "all three point at the *same* single disputed-annotation round-trip, so the catch is one coherent scenario." This task is the terminus of that chain: it makes the dispute gesture reachable and pins the one round-trip that discharges all three deferrals.

Because the dispute affordance is now reachable (a route renders the annotation detail; the event surface drives it; the participant can select and dispute), the UI-stream e2e policy's default applies — full Playwright is in scope, not deferred.

## Constraints / requirements

- **Drive the already-built seam — no backend/schema/hook-type/projector change.** The affordance calls `useVoteAction({ entity_kind:'annotation', entity_id, facet:'substance' }).castVote('dispute')`. The wire message, persisted event, engine accept-arm, replay, and shell projector all landed with `annotation_facet_vote_seam`; touching them here is out of scope. `substance` is the only disputable annotation facet (ADR 0038 §4 / vote-seam Decision §4).
- **Reachable selection path.** The participant must be able to select the annotation without store-poking: an annotation row in the target entity's annotations list is made clickable (→ `select({ kind:'annotation', id })`), which opens the annotation detail branch that hosts the dispute affordance. (Promoted/endpoint annotations are already selectable via graph tap; the clickable row covers non-promoted annotations.)
- **Affordance lives in the annotation-detail `actionSlot`, not in `ParticipantVoteButtons`.** `ParticipantVoteButtons` is built around the node/edge facet catalog; annotations are substance-only and entity-layer. Mount the dispute affordance in the `actionSlot` (OperateRoute-owned, mirroring `ParticipantAxiomMarkButton`) — see Decision §3.
- **Testid + status convention mirrors existing vote buttons.** Expose a stable testid (e.g. `participant-annotation-dispute-button`) and an in-flight/resolved status data-attribute (analogous to `data-vote-state` / `data-facet-status`) so the spec asserts the click → in-flight → settled transition and the resolved `disputed` substance state. Respect the per-participant already-voted guard surfaced by the seam (a second identical vote is rejected server-side).
- **Real cross-surface round-trip; no store-seeding for the dispute.** The precondition annotation and the dispute vote must be real server state so the moderator badge updates via genuine WS broadcast (Decision §1). `wsStoreSeed` is not used in this spec.
- **The moderator badge is the assertion surface, untouched.** Assert against `annotation-badge-<id>` gaining `data-facet-status="disputed"`; do not alter `AnnotationBadge`, `selectors.ts`, or the `FacetStatusIndex` shape.
- **Locale + scrollbar fixtures.** Use the `no-scrollbars` re-exported `test`/`expect`; rely on the `setup-auth` bootstrap and `authedContext` jars (no per-test OIDC dance).
- Verifications per ADR 0022: Vitest component test for the new affordance; Playwright for the cross-surface round-trip. No throwaway scripts; no forced-state/seeded fake e2e.

- **Affordance — dispatch.** Clicking the dispute affordance calls `useVoteAction({ entity_kind:'annotation', entity_id, facet:'substance' }).castVote('dispute')`, dispatching the facet-arm wire payload with `entity_kind:'annotation'`, `facet:'substance'`, `choice:'dispute'`; the affordance reflects in-flight → settled via a stable data-attribute, and reflects the resolved `disputed` substance status.

## Decisions

- **§1 — Real cross-surface round-trip; no store-seeding for the dispute.** The dispute vote and the precondition annotation are produced through the real moderator capture → meta-move propose → debater agreement → commit flow, then a real participant dispute click; the moderator badge update is observed over live WS. **Rejected:** seeding the dispute (or the committed annotation) via `wsStoreSeed`. `wsStoreSeed` injects only into one browser's Zustand store, never real server state — so a participant's real dispute vote would hit a server with no such annotation and be rejected, and no server-side event-seed endpoint exists (verified). A store-seeded "render-only" spec would also re-cover what `mod_meta_move_disputed_visibility`'s `AnnotationBadge` component tests already pin, exercising neither the new affordance nor real WS propagation — i.e. a fake e2e the UI policy forbids.
- **§2 — Drive the existing facet-keyed seam; add only the UI gesture.** `annotation_facet_vote_seam` (ADR 0038, commit `047bd8be`) already encodes annotation disputes as facet-keyed votes (`entity_kind:'annotation'`, `facet:'substance'`) end-to-end. The affordance simply calls `useVoteAction(...).castVote('dispute')`. **Rejected:** introducing a new `target:'annotation'` wire arm or a bespoke dispatch — ADR 0038 settled the facet-keyed encoding and the seam is built; a parallel arm would contradict the shipped contract and duplicate the engine/replay/projector handling.
- **§3 — Affordance in the annotation-detail `actionSlot`, not in `ParticipantVoteButtons`.** The annotation detail branch already exists and exposes an `actionSlot` (EntityDetailPanel 522–524) that OperateRoute populates today for the node-scoped axiom-mark button — the dispute affordance follows the same composition. **Rejected:** extending `ParticipantVoteButtons` to a third entity kind. That component is built around the node/edge facet *catalog* (wording/classification/substance/shape, NODE_FACETS/EDGE_FACETS); an annotation is substance-only and entity-layer, so threading it through the catalog machinery is more code than a focused affordance and risks regressing the node/edge rows. The clickable annotation row + actionSlot affordance is the minimal reachable path.
- **§4 — Precondition annotation via a real committed reframe meta-move.** A committed meta-move is the canonical producer of `annotation-created` (`meta_move_commit_logic`), and `methodology-full-flow` Phase 9.2 already demonstrates driving a proposal commit to a rendered `annotation-badge`. Reuse that path. **Rejected:** any lighter precondition — impossible given §1 (no server-side seed path), and the meta-move propose/commit UI is reachable today (moderator-capture covers propose; PendingProposalsPane covers commit). The initial substance status of the fresh annotation is irrelevant to the test: the dispute is a *direct* facet vote that folds to `disputed` via the shell's Rule 5 regardless of the pre-dispute state.
- **§5 — One spec discharges all three inherited deferrals.** `mod_meta_move_disputed_visibility`, `annotation_facet_status_logic`, and `annotation_facet_vote_seam` all point at the *same* disputed-annotation round-trip (vote-seam AC made this explicit). A single coherent scenario pays the whole debt down — consistent with the brief's "watch the inherited-debt count" guidance; there is no `mod_pw_*` catch-all accumulating here. **Rejected:** splitting into separate specs per deferral — they describe one flow.
- **§6 — Reuse existing testid/status conventions; no new ADR.** Assert on the existing `annotation-badge-<id>` + `data-facet-status="disputed"` (moderator) and expose a `participant-vote-button-*`-style testid + status data-attribute (participant). The affordance placement is a UI choice documented here, not an architectural decision — ADR 0038 already governs the semantics, so no new ADR is warranted.
