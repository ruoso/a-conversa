## What this task is

Extend the [`decomposeProposalSchema`](../../../packages/shared-types/src/events/proposals.ts) + [`interpretiveSplitProposalSchema`](../../../packages/shared-types/src/events/proposals.ts) (and the shared [`proposalComponentSchema`](../../../packages/shared-types/src/events/proposals.ts) they both consume) with a single new required field — `node_id: UUID` — on each component / reading object, and grow the propose handler's `decompose` and `interpretive-split` arms inside `buildStructuralEventsForPropose` to emit N × (`node-created` + `entity-included`) at propose-time alongside the `proposal` envelope. Per the methodology (`docs/methodology.md` L84 — "agreeing to decompose N into A + B agrees to the structural restructuring (N is removed; A and B exist as entities)") and per `docs/data-model.md` L84-87 + L211-213, a decompose proposal produces **N component nodes** (where 2 ≤ N ≤ 10); there is no source-decomposition edge minted by the propose-time fan-out — the parent stays visible during the proposed window and flips invisible only on commit per the existing `handleCommit` arm at [`apps/server/src/projection/replay.ts:691-711`](../../../apps/server/src/projection/replay.ts). `interpretive-split` follows the identical topology with the `readings` array (R27 bounds + `proposalComponentSchema` shape both shared).

Concretely the deliverable is five files of change:

1. **Schema extension** at [`packages/shared-types/src/events/proposals.ts:206-211`](../../../packages/shared-types/src/events/proposals.ts) — add `node_id: z.string().uuid()` to `proposalComponentSchema` as a **REQUIRED** field (see Decision D1 for the rationale on required-vs-optional + why this is unlike the `set-edge-substance` precedent). The schema-extension docblock cites ADR 0027 + this refinement, mirroring the `classifyNodeProposalSchema` evolution paragraph at L79-88. Both `decomposeProposalSchema` (L219-223) and `interpretiveSplitProposalSchema` (L233-237) consume the extended shape automatically — no per-sub-kind schema change at the outer level.
2. **Client builder update** at [`apps/moderator/src/layout/useProposeProposalAction.ts:138-159`](../../../apps/moderator/src/layout/useProposeProposalAction.ts) — `buildProposal()` mints a fresh UUID per component / reading row inside the `wireRows.map` and threads it onto each element as `node_id` (alongside `wording` + `classification`). Per D2 the IDs are minted CLIENT-side (not server-side) so the moderator's optimistic-clear path (the `exitDecomposeMode()` + `setProposing(true)` block at L260-266) can later observe the matching `node-created` events arrive by the IDs it already knows, and so the snapshot-restore path on `WsRequestError` (L286-315) restores the per-row state without needing to round-trip the IDs through the error envelope.
3. **Propose-handler arms** at [`apps/server/src/methodology/handlers/propose.ts:1460-1568`](../../../apps/server/src/methodology/handlers/propose.ts) — the `decompose` and `interpretive-split` arms of `buildStructuralEventsForPropose` (currently `default: break;` no-op fall-throughs with the long header docblock at L1449-1457 naming this task as the follow-up) get the real emission. Each arm walks `action.proposal.components` (or `.readings`) in array order and emits per element: one `node-created` envelope carrying `{ node_id, wording, created_by, created_at }`, followed by one `entity-included` envelope carrying `{ entity_kind: 'node', entity_id: node_id, included_by, included_at }`. Each event takes the next sequence slot via the existing `seq(events.length)` helper. For a 2-component decompose, the emitted events array contains 5 envelopes in order: `node-created(c1)`, `entity-included(c1)`, `node-created(c2)`, `entity-included(c2)`, `proposal`. For N components the count is 2N+1.
4. **Lockstep withdraw arms** at [`apps/server/src/ws/handlers/withdraw.ts:511-526`](../../../apps/server/src/ws/handlers/withdraw.ts)'s `entitiesToRetractForWithdraw` switch — `'decompose'` and `'interpretive-split'` lift out of the no-op fall-through batch (L518-519) and gain their own arms: for each `component` in `payload.components` (or `reading` in `payload.readings`) where `projection.getNode(component.node_id) !== undefined`, push `{ entityKind: 'node', entityId: component.node_id }`. The visibility-flip via `handleEntityRemoved` is what removes each component from the canvas; the parent node's visibility is unaffected (it never flipped — the propose-time emission doesn't touch the parent per the methodology contract). The docblock at L417-440 grows two new bullets describing the new arms and removes both task names from the open-debt closing line.
5. **Lift Playwright Scenario 3** at [`tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts:210-236`](../../../tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts) — the deferred-block comment removes the Scenario 3 paragraph (L216-220) and the spec adds the real scenario. Per the parent refinement's Acceptance criteria L103 the scenario asserts: parent node remains visible (per ADR 0027 L54 — "The original target node remains visible alongside the proposed components during the proposed window"); both child nodes render with `data-facet-status="proposed"`; the canvas shows three `statement-node-<id>` testids (parent + 2 children). The scenario reuses `moderatorReachOperate` + `proposeStatement` for the parent + the decompose-mode gesture for the children.

## Why it needs to be done

Three reasons compose:

**Methodology contract — `docs/methodology.md` L57 + L84.** "A proposed change appears on the graph in `proposed` state from the moment it is made." Today the decompose case half-honors this contract: the moderator captures a parent statement, captures the N component wordings + classifications, clicks Propose-decomposition, and gets a sidebar entry but no canvas children. The methodology's evaluation premise — "what this would become" alongside "what it is now" — fails: the participants can see the parent but not the proposed components, so they have no canvas surface to vote against, draw edges from, or counter-propose against. The decompose flow is the single most important multi-entity proposal in the methodology (per `docs/methodology.md` L108 it's the resolution path for every disputation test that surfaces a compound structure); leaving it canvas-invisible at propose-time is a usability regression worse than the original single-statement bug because the moderator and participants are now staring at empty canvas after typing N component wordings.

**ADR 0027 consequence — bullet 1, L20-32 + L54.** "When a proposal introduces new entities (a `classify-node` for a fresh node, an edge to an existing target, a `decompose` with N components + M edges, an `interpretive-split` with the analogous shape), the propose handler emits the corresponding `node-created` / `entity-included` / `edge-created` events alongside the `proposal` envelope." The ADR named the decompose + interpretive-split cases explicitly; commit `17822e1` landed only the `classify-node` arm because the propose payload didn't carry per-component IDs. The ADR's "Consequences" section (L54) explicitly anticipates this work: "Multi-entity proposals (`decompose`, `interpretive-split`) work without special-casing. Under this ADR the propose handler walks the proposal's child specs and emits one structural event per child; projectors already handle arbitrary numbers of structural events." Leaving the arms unimplemented after the ADR + the precedent `set-edge-substance` arm landed means the runtime contract drifts from the architectural decision — exactly the failure mode the ADR was supposed to prevent.

**Inverse-pair invariant.** Per [`ws_withdraw_proposal_message.md`](../backend/ws_withdraw_proposal_message.md) D3 the `entitiesToRetractForWithdraw` switch and `buildStructuralEventsForPropose` are inverses that "MUST stay in sync." Growing the propose-time emission without growing the withdraw arms leaves a proposer who withdraws an N-component decompose proposal with the N proposed-component nodes orphaned on the canvas forever (the parent's visibility was never touched at propose-time; the only entity-layer effect of the withdraw should be retracting the components). The withdraw refinement explicitly names this task as the trigger for the lockstep update, and commit `e89cf06` amended this task's `.tji` note to call it out as a hard requirement.

Downstream consumers benefit transitively:

- `participant_ui.part_graph_view` (pending under M5) — inherits the projector seam; no per-surface fix needed.
- `audience.aud_graph_rendering` (pending under M6) — same transitive inheritance.
- `mod_interpretive_split_mode` (already shipping the matching UI mode + the shared `useProposeProposalAction({ mode: 'interpretive-split' })` consumer per [`mod_propose_decomposition.md`](./mod_propose_decomposition.md) extraction) — the interpretive-split client gesture flows through the SAME `buildProposal()` helper this task extends; landing the schema change + the propose-handler arm gives both sub-kinds canvas-visibility on propose-time at once.
- The lifted Scenario 3 in the Playwright e2e becomes the regression cover that catches any future drift in either direction (a propose-time arm that stops emitting; a wire-shape change that loses the per-component ID carriage; a projector regression that fails to render proposed components).

## Constraints / requirements

- **`node_id` is REQUIRED on `proposalComponentSchema`, not `.optional()`.** Per D1: unlike the `set-edge-substance` precedent (where `.optional()` was forced by the two-shape requirement of the sub-kind — connecting case carries endpoints, substance-only re-vote doesn't), the `decompose` / `interpretive-split` sub-kinds have a single shape: every propose envelope mints fresh component nodes. There is no "decompose re-vote" use-case; the methodology contract is unambiguous that a decompose proposal IS a structural restructuring that introduces N component entities. Required is the simpler, drift-resistant choice. The migration moment is one-shot: the field lands required from the first emit. No existing call site emits a `decompose` proposal that would lack the field, because the only call site is the client builder this task updates in the same commit.
- **Client-side UUID minting (not server-side).** Per D2 the IDs are minted on the client (in `buildProposal()` inside `useProposeProposalAction.ts`) BEFORE the envelope leaves the moderator. This mirrors the `classify-node` pattern (the moderator's capture flow mints the `node_id` for free-floating statements client-side; the propose handler reads it from the payload). Server-side minting would force a round-trip before the proposer's own canvas could resolve the component IDs (the moderator would have to wait for the `event-applied` broadcast before knowing what IDs to draw); client-side minting lets the moderator's optimistic-clear path observe the component IDs immediately and lets the snapshot-restore path on `WsRequestError` not depend on the error envelope carrying server-allocated IDs.
- **Per-component emission order: `node-created` then `entity-included`, repeated per component in array order.** Per D3 the per-component pair is the same shape `classify-node` already uses; the array order of `proposal.components` is the canonical iteration order (the proposal payload's array IS the source of truth for the components' relative ordering). This is the same iteration order the methodology engine's `handleCommit/decompose` arm would walk if it were the structural seam (it isn't — under this task the decompose-commit's responsibility narrows to the parent-visibility flip only).
- **Parent node visibility is UNCHANGED by the propose-time emission.** Per the methodology contract (`docs/methodology.md` L84 + `docs/data-model.md` L282-285) the parent stays visible during the proposed window and flips invisible only on commit of `decompose` / `interpretive-split`. The propose-time fan-out emits ONLY the per-component `node-created` + `entity-included` events — no `entity-removed` for the parent, no `visibility-changed` for the parent. The commit handler's existing parent-flip-off arm at `replay.ts:691-711` runs at commit-time as before.
- **No edge events emitted at propose-time.** Per the methodology + data-model citations there is NO source-decomposition edge from the parent to the components — decompose is a structural restructuring, not an additive edge minting. The propose-time fan-out per component is exactly two events (`node-created` + `entity-included`); the total per N-component decompose is `2N + 1` events (the `+1` is the `proposal` envelope at the end).
- **Fresh-component predicate symmetry with `classify-node`.** The `classify-node` arm uses `projection.getNode(nodeId) === undefined && wording !== undefined` (both branches must hold). The `decompose` / `interpretive-split` arms could mirror this with a per-component `projection.getNode(component.node_id) === undefined` check, but per D6 it is OMITTED in v1: the moderator's client mints fresh UUIDs every envelope-build (so a collision would mean a UUID-v4 collision, which is effectively impossible). The server-side `validateEvent` schema-on-write pass on each emitted `node-created` envelope catches any payload-shape issue; the projector's `handleNodeCreated` would reject a duplicate node insertion if one ever occurred. Adding the defensive predicate adds runtime work for a condition that cannot arise from the documented client path.
- **Schema-on-write per ADR 0021.** The extended `proposalComponentSchema` validates at `parseWsEnvelope` time; payloads with non-UUID `node_id` strings (or with the field missing) are rejected before the propose handler runs. Each emitted `node-created` envelope runs through `validateEvent` (per the propose handler's existing schema-on-write pass) — its payload schema requires `node_id` + `wording` + `created_by` + `created_at`, all of which the per-component loop provides.
- **Sequence allocation mirrors `classify-node` / `set-edge-substance`.** Each emitted event takes the next slot via `seq(events.length)`; the `proposal` envelope takes the last slot at `action.sequence + structuralEvents.length`. For a 2-component decompose this produces sequences `n, n+1, n+2, n+3, n+4` (where `n = action.sequence`) for the events `[node-created(c1), entity-included(c1), node-created(c2), entity-included(c2), proposal]`. For N components: `2N + 1` sequence slots consumed.
- **Lockstep withdraw arms in the same commit cluster.** Per the source-of-debt withdraw refinement's D3 + the `.tji` note amendment (commit `e89cf06`), the inverse-pair invariant between `buildStructuralEventsForPropose` and `entitiesToRetractForWithdraw` is enforced by code review. Landing the propose-time emission without the matching retraction would leave a proposer who withdraws an N-component decompose proposal with N orphaned component nodes on the canvas in `proposed` state forever. Same-commit-cluster landing makes the inverse-pair locality a code-review fact.
- **No new event kind, no new payload schema, no SQL migration.** All wire-format primitives are already on `main` per commit `17822e1` + the `set-edge-substance` commit cluster: `entity-removed` event kind, `entityRemovedPayloadSchema`, the SQL `CHECK` constraint widening, the projector's `handleEntityRemoved` arm. This task threads existing primitives through new code paths; the schema addition is scoped to the proposal payload (`proposalComponentSchema`'s new `node_id` field) only.

## Decisions

- **D1 — `node_id` is REQUIRED on `proposalComponentSchema`, not `.optional()`.** Unlike the `set-edge-substance` precedent (where `.optional()` was forced by the two-shape requirement of that sub-kind — connecting case carries endpoints, substance-only re-vote doesn't), the `decompose` / `interpretive-split` sub-kinds have a SINGLE shape on the wire: every envelope mints fresh component nodes. There is no "decompose re-vote" or "interpretive-split re-vote" use-case in the methodology — the entire premise of these sub-kinds is structural restructuring of the parent into new components. Making the field required:
  - **Catches missing-ID bugs at envelope-parse time** rather than letting them silently fall through to the propose handler's emission step (where a missing ID would either crash inside the loop or — worse — silently emit a `node-created` with `undefined` for the `node_id` payload, which would then crash inside `validateEvent`).
  - **Aligns with the methodology contract.** The methodology says decompose introduces N components; the wire shape now says the same thing structurally (every component MUST have an identity). Optional would let a future buggy client send a half-formed payload that the server would reject at runtime; required rejects it at envelope-parse-time, with a clearer error path.
  - **Alternatives considered:**
    - **(B) `.optional()` mirroring the `set-edge-substance` precedent.** Rejected — the precedent's `.optional()` was forced by the two-shape requirement, not chosen for its own sake. Decompose / interpretive-split have no second shape; copying the precedent for surface-uniformity reasons would import a constraint that doesn't apply here.
    - **(C) Server-mint the IDs (no schema change).** Rejected — see D2.
    - **(D) `refine()` guard enforcing "all components present OR all components absent" (so an N-element array with missing IDs on some elements is structurally rejected).** Rejected — required-on-the-element schema covers this without needing the refine pass.

- **D2 — Per-component UUIDs are minted CLIENT-side (in the propose hook's `buildProposal()`), not server-side.** The client mints fresh UUIDs at envelope-build-time inside `useProposeProposalAction.ts`'s `buildProposal()`; the server's propose handler reads them from the payload and emits them directly on the `node-created` envelopes. Mirrors the `classify-node` pattern (the moderator's capture flow mints the `node_id` for free-floating statements client-side; the propose handler reads it from the payload — see `apps/moderator/src/layout/useProposeAction.ts`'s `buildClassifyNodeProposal` for the established pattern). Rationale:
  - **The proposer's own canvas can resolve the component IDs immediately.** With client-side minting the moderator knows the component IDs at envelope-build-time; the optimistic-clear path at L260-266 can flip out of decompose mode AND the proposer's own canvas can render the proposed components from the locally-broadcast structural events (via the `event-applied` round-trip) by the IDs it already knows. Server-side minting would require either (a) round-tripping the IDs back to the client via the `proposed` ack frame (which today doesn't carry server-allocated IDs — that's a wire-shape addition), or (b) the client waiting for the `event-applied` broadcasts before it can confirm the component IDs — both add latency to a flow that should be immediate.
  - **The snapshot-restore-on-error path doesn't depend on server IDs.** On `WsRequestError` the propose hook restores the per-row state via the snapshot taken at L254-257; the per-row state doesn't carry the IDs (the IDs are derived at envelope-build-time, not stored on the row). On retry, `buildProposal()` mints fresh IDs — that's the right behavior because a retry of a failed propose is structurally a fresh propose envelope with fresh entity IDs. If the IDs were server-minted and a previous round had landed partial state on the server, retry would have to either (a) reuse the previously-allocated IDs (requiring server-side dedup) or (b) leak the previous IDs forever — both worse than fresh-mint-per-attempt.
  - **Alternatives considered:**
    - **(B) Server-mint inside `buildStructuralEventsForPropose`.** Rejected — see "the proposer's own canvas" paragraph above; the latency cost is non-trivial for a flow whose whole point is propose-time canvas visibility.
    - **(C) Mint at row-add time (store IDs on the per-row state `decomposeComponents[i].node_id`).** Rejected — couples the store shape to the propose envelope shape; a retry after error then has stable IDs across attempts, which sounds like a feature but is actually a correctness hazard (the server may have appended events with the first attempt's IDs that became orphans; the second attempt then collides on those IDs). Mint-at-envelope-build-time gives each attempt a clean ID space.
    - **(D) Hybrid: client mints, server validates uniqueness against the projection.** Rejected for v1 — adds runtime work for a condition (UUID-v4 collision) that effectively cannot occur. See D6 for the defensive-predicate-omission rationale.

- **D3 — Per-component emission order: `node-created` THEN `entity-included`, repeated per component in proposal-payload array order.** Each component takes 2 sequence slots in the order `[node-created, entity-included]`; the per-array-element pair is then concatenated in the proposal payload's array order. For a 2-component decompose: sequences `[c1.node-created, c1.entity-included, c2.node-created, c2.entity-included, proposal]`. Rationale:
  - **Mirrors the `classify-node` arm's pair order at `propose.ts:1479-1508`** — every event consumer that handles `node-created` followed by `entity-included` for the single-node case extends naturally to N pairs without per-arm special-casing.
  - **Proposal-payload array order is the canonical iteration order.** The proposal's `components` array IS the authoritative ordering (the order the moderator captured the components in the UI). The methodology engine's commit-time iteration (when the parent-flip-off lands) doesn't depend on a particular order, but the propose-time emission should be deterministic for replay correctness — array order is the natural choice.
  - **Alternatives considered:**
    - **(B) All `node-created`s first, then all `entity-included`s.** Rejected — would require the projector to handle a `node-created` whose subsequent `entity-included` has not yet been seen (the events are processed in sequence order during replay; a node without an `entity-included` in this session would be invisible). The per-component pair groups the structural events for one entity into adjacent sequence slots, which is the same pattern `classify-node` already established.
    - **(C) Random order.** Rejected — replay determinism is a non-negotiable invariant per ADR 0021.

- **D4 — Withdraw arm grows in lockstep in the same commit cluster.** Per the source-of-debt withdraw refinement's D3 + the `.tji` note amendment (commit `e89cf06`), the inverse-pair invariant between `buildStructuralEventsForPropose` and `entitiesToRetractForWithdraw` is enforced by code review. Landing the propose-time emission without the matching retraction would leave a proposer who withdraws an N-component decompose proposal with N orphaned component nodes on the canvas in `proposed` state forever. Same-commit-cluster landing makes the inverse-pair locality a code-review fact. Alternative considered: ship the propose-time emission first, ship the withdraw arms in a follow-up commit. Rejected — same rationale as the `set-edge-substance` precedent's D3.

- **D5 — Two separate `case` arms (`decompose` + `interpretive-split`), NOT a shared loop over a per-sub-kind discriminator.** The two arms are near-identical (the only field-level difference is `action.proposal.components` vs `action.proposal.readings`). Could be collapsed into a shared local helper `emitComponentNodes(components: ReadonlyArray<ProposalComponent>): EventToAppend[]` called from both arms. Rejected for v1 in favor of keeping two separate `case` blocks. Rationale:
  - **Mirrors the per-sub-kind switch shape at `propose.ts:1470-1565`.** Every existing arm (`classify-node`, `set-edge-substance`) is a self-contained `case` block; reading the switch top-to-bottom gives a per-sub-kind picture of "what does propose-time emit for this sub-kind." Extracting a shared helper would force readers to traverse the helper definition to understand either arm.
  - **The duplication is shallow (one field name).** A 10-line per-arm body with one differing field name is the simplest expression of "these two sub-kinds do the same thing on different array fields"; the shared helper would add 5 lines (helper definition + per-arm one-line call site) for net-zero line savings and one extra symbol.
  - **If a third sub-kind ever needs the same pattern, the right time to extract the helper is then** — at that point the abstraction has three call sites and the reuse case is real. Two call sites is the borderline; one-field-name duplication is on the keep-inline side of the borderline.

- **D6 — No defensive per-component fresh-node predicate (no `projection.getNode(component.node_id) === undefined` check).** The `classify-node` arm uses `projection.getNode(nodeId) === undefined && wording !== undefined` (both branches must hold) because that sub-kind serves two shapes (free-floating mint vs re-classify of an extant node). The decompose / interpretive-split sub-kinds have a single shape (always mint), and per D2 the client mints fresh UUIDs every envelope-build (so a collision would mean a UUID-v4 collision, which is effectively impossible). Adding the defensive predicate would add runtime work for a condition that cannot arise from the documented client path. Rationale:
  - **Schema-on-write per ADR 0021 catches malformed payloads.** Each emitted `node-created` envelope runs through `validateEvent`; a payload-shape issue fails loudly before `appendSessionEvent`.
  - **The projector's `handleNodeCreated` would reject a duplicate insertion if one ever occurred.** The defense-in-depth is already there; adding the propose-handler-level check is redundant.
  - **Alternative considered: defensive predicate with skip-on-duplicate.** Rejected — would silently drop a component the moderator captured, producing a degenerate proposal that the moderator can't see (the per-component fan-out wouldn't fire for the dropped component). A loud rejection is better UX than a silent drop.
  - **Alternative considered: defensive predicate with rejection.** Rejected — would surface a `RejectedValidationResult` for what is effectively impossible; the wire-error code surface would have to include a new `'component-id-collision'` reason that no real client path can trigger. Net more code for net less correctness.

- **D7 — Commit handler is NOT modified by this task.** The existing `handleCommit` arm for `decompose` + `interpretive-split` at `replay.ts:691-731` (the parent-visibility flip) stays unchanged. Before this task, the commit handler's responsibility was implicitly "create the component nodes too" (per the TODO comment at L692-699). With this task, the component nodes are already on the projection at commit-time (they entered at propose-time per this task), so the commit handler's responsibility narrows naturally to "flip the parent off" — which is exactly what the existing arm already does. No code change in commit-side; the TODO comment at L692-699 becomes obsolete and can be removed in a cleanup pass (out of scope for this task per the surgical-change principle, but flagged as a follow-up).
  - **Alternative considered: amend the TODO comment in the same commit cluster.** Rejected — the TODO comment removal is a docs-only change in a different file; the task's focus is the propose-time fan-out + the lockstep withdraw arms + the e2e activation. The TODO is informational, not load-bearing.

- **D8 — Lift Playwright Scenario 3 in place, not in a new spec file.** The parent refinement's deferred-scenarios list explicitly names this task as the activator for Scenario 3. The scenario stays in `tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts` because the spec file IS the regression cover for ADR 0027's stack-validation tests (per the ADR's "Stack-validation tests" section at L58-60). Splitting the decompose scenario into a new file would orphan it from the scenario set the ADR named. Same rationale as the `set-edge-substance` precedent's D6.
