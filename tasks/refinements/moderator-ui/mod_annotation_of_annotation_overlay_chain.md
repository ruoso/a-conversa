## What this task is

Light up annotation-of-annotation overlay propagation on the moderator console's ReactFlow canvas. When an annotation A1 is promoted to a `<AnnotationNode>` (because some `edge-created` references its id via `source_annotation_id` or `target_annotation_id`, per the predecessor `mod_render_annotation_endpoint_edges`), and another annotation A2 carries A1's UUID in its `target_node_id` field, A1's materialized graph-node carries A2 in its `data.annotations` array — and the `<AnnotationNode>` component renders A2 as an `<AnnotationBadge>` pill on the annotation card itself. The moderator surface sees the same "annotation on annotation" visual vocabulary the participant + audience already pinned end-to-end.

The mechanical change is three-step:

1. **Rename the bucketer call site to `groupAnnotationsByEntityId`** — runtime byte-for-byte unchanged; the rename makes the polymorphic-entity-id convention legible at the moderator surface (today's import name `groupAnnotationsByNode` is misleading once the bucket key is an annotation id).
2. **Thread `annotationsByNode` into `projectAnnotationNodes` so the materialization pass stamps `data.annotations`** instead of leaving it absent. The bucketer (post-`allAnnotations.filter(!promotedAnnotationIds.has(annotation.id))`) already buckets A2 under key A1 (A2 is non-promoted; A2's `targetNodeId` is A1's UUID; the bucketer is target-id-keyed); the missing seam is the read inside `projectAnnotationNodes` — adding it lights up the data flow end-to-end.
3. **Widen `<AnnotationNode>` to render `<AnnotationBadge>` for each stamped annotation** — a small flex-row below the existing content row, reusing the moderator's existing badge vocabulary. The card's fixed-height container becomes `minHeight`-bounded so the row grows the card vertically when present and is absent (with no allocated row at all) when `annotations.length === 0`.

The moderator's surface differs from the audience's at one observable point: the audience uses a separate DOM-positioned overlay (`<AudienceAnnotationOverlay>`) that reads `node.data('annotations')` for ALL nodes regardless of kind, so stamping data on the annotation node was sufficient — the overlay rendered the badge for free. The moderator's annotation badge rendering lives INSIDE the per-node component (`<StatementNode>` renders its own badge list at [L526-532](../../../apps/moderator/src/graph/StatementNode.tsx)); `<AnnotationNode>` does NOT currently have a badge-list render — so the propagation data lands on `data.annotations` AND the component must be widened to render it. This is the moderator-specific delta beyond what the audience refinement absorbed.

Concretely the deliverable is:

1. **`groupAnnotationsByNode` → `groupAnnotationsByEntityId` rename** at [`apps/moderator/src/graph/GraphCanvasPane.tsx:136`](../../../apps/moderator/src/graph/GraphCanvasPane.tsx) (named import) and [L545](../../../apps/moderator/src/graph/GraphCanvasPane.tsx) (call site inside `projectNodes`). The local variable identifier `annotationsByNode` is preserved (D2).

The per-promoted-annotation arm reads `nodeAnnotationIndex.get(annotation.id) ?? EMPTY_ANNOTATIONS` and stamps the result on `data.annotations`.
4. **The call site in `projectNodes` threads `annotationsByNode` through** at [`apps/moderator/src/graph/GraphCanvasPane.tsx:719`](../../../apps/moderator/src/graph/GraphCanvasPane.tsx) (the existing call passes `(allAnnotations, promotedAnnotationIds, events)`; it gains a fourth argument).

No filter-side refactor.

## Why it needs to be done

**The participant + audience have pinned this convention at their surfaces; the moderator is the third and last UI-stream surface in the natural rollout.** The participant's `part_annotation_of_annotation_overlay_chain` (done 2026-05-30) established the convention and the structural test template; the audience's `aud_annotation_of_annotation_overlay_chain` (done 2026-05-30) lit up the same propagation at the broadcast surface; both explicitly registered THIS task as the moderator follow-up (participant D6; audience tech-debt §). A moderator looking at a session canvas where the participant + audience surfaces both render meta-annotations consistently — and the moderator does NOT — exhibits cross-surface inconsistency for a methodology pattern the engine already recognizes (`coherency_annotation_of_annotation_chain_rule` is done and advisory-only).

**The visual surface is already half-built.** The predecessor `mod_render_annotation_endpoint_edges` (done 2026-05-30) materializes annotation graph-nodes via the hybrid-promotion pattern. The promoted annotation graph-node already renders with `kind` + `content`; the missing surface is the per-annotation badge row layered on top of the card. The `<AnnotationBadge>` component is already proved by `<StatementNode>` + `<StatementEdge>` — reusing it inside `<AnnotationNode>` is a small additive render with no new component, no new icon vocabulary, no new color choice.

**The bucketer's name is misleading on the moderator side too.** The `extract_cytoscape_projectors` lift put `groupAnnotationsByNode` in `@a-conversa/shell`; the moderator imports it via [`GraphCanvasPane.tsx:136`](../../../apps/moderator/src/graph/GraphCanvasPane.tsx). The participant's task renamed to `groupAnnotationsByEntityId` with polymorphic-entity-id docstring; the moderator continuing to import the legacy name implies a "node-only" reading that's now demonstrably wrong (the bucket key can be an annotation id). Migrating the moderator site makes the convention honest at the third and final consumer.

**Pinning the convention at Vitest closes a documentation debt.** Without committed cases, a future refactor of `projectAnnotationNodes` or the `<AnnotationNode>` component could regress the annotation-of-annotation propagation silently — the methodology-engine's `coherency_annotation_of_annotation_chain_rule` (done 2026-05-30, advisory-only) names the structural pattern; the moderator's coverage should pin the visual surfacing for the same reason the participant + audience did.

## Decisions

- **D1 — Migrate the call site from `groupAnnotationsByNode` to `groupAnnotationsByEntityId` (no semantic change); pair the rename with the propagation wire-up at `projectAnnotationNodes` AND the component-side badge-row render.** Rationale:
  - **The rename alone is insufficient.** A pure import-rename would land the docstring honesty but leave the moderator's annotation-of-annotation propagation broken (`projectAnnotationNodes` doesn't read any per-target index) AND invisible (`<AnnotationNode>` has no badge-list render). The audience could land a rename + projection-stamp without component changes because its DOM overlay was already node-kind-agnostic; the moderator's per-node-component badge rendering means the third leg (component widening) is part of the same cohesive change.
  - **Bundling rename + stamp + render in one commit lights up the propagation end-to-end** — parallel to the participant + audience outcomes. Splitting into three micro-commits adds three review surfaces for no separable behavioral signal.
  - **The shell-level export is already in place.** The participant's task added `groupAnnotationsByEntityId` to `@a-conversa/shell` with the polymorphic-entity-id docstring; the moderator consumes the shell directly. The import-name swap is one-line.
  - **Alternative considered: ship the rename only; defer the projection + render to separate tasks.** Rejected — splits a 0.5d cohesive change into three micro-commits; the rename alone is a no-op runtime change; readers can't tell from the rename commit what the convention is for.
  - **Alternative considered: ship the projection-stamp only; defer the component render to a follow-up.** Rejected — the data lands on `data.annotations` but the user never sees the propagation. The task's name ("overlay propagation on materialized moderator annotation graph-nodes") commits to the visible overlay, not just the data-flow. A follow-up render-only task would land an asymmetric in-flight state where the data flows but doesn't surface; cleaner to bundle.

- **D2 — Preserve the local variable identifier `annotationsByNode` across the migration; do NOT rename to `annotationsByEntityId`.** Rationale (mirrors participant D3 / audience D2 verbatim):
  - **Cosmetic-only churn is out of budget at 0.5d.** Renaming the local variable in `projectNodes` would touch every reader of the index inside the loop body; the runtime change is zero.
  - **The local name is implementation-level; the shell-level rename is the contract-level honesty.** The polymorphic-entity-id semantic lives in the shell-level export's docstring (where the moderator-side reader looking for "what does this bucket mean?" lands); the moderator-side local name is one level down.
  - **Future task can rename if cost-justified.** A standalone cosmetic refactor renaming the moderator-side local identifiers is cheap and discoverable when the next moderator-graph touch lands; deferring is low-risk.
  - **Cross-surface consistency.** Participant + audience both preserved their local names per their D3/D2; the moderator preserving `annotationsByNode` keeps the three surfaces' local-name posture aligned (consistency on "the local name doesn't matter; the shell-level export does").
  - **Alternative considered: rename `annotationsByNode` → `annotationsByEntityId` moderator-wide.** Rejected — out of budget; cross-surface inconsistency for zero behavioral benefit.

- **D3 — Thread `annotationsByNode` into `projectAnnotationNodes` as a fourth parameter rather than computing it inline OR inlining the materialization into `projectNodes`.** Rationale (mirrors audience D3 with moderator-specific seam names):
  - **`projectAnnotationNodes` already takes context as parameters.** Its current signature is `(annotations, promotedSet, events)` — `events` is threaded through for `buildAnnotationHostIndex`. Adding `nodeAnnotationIndex` as a fourth parameter keeps the function pure and its dependencies explicit; it's symmetric with the existing parameter-passing posture.
  - **Computing `annotationsByNode` inside `projectAnnotationNodes` would duplicate the bucket-build work.** `projectNodes` already builds the (filtered) `annotationsByNode` for the per-statement-node-arm read at L586; passing it through avoids a second `groupAnnotationsByEntityId + filter` pass.
  - **Inlining the materialization into `projectNodes` would break the existing helper-layer Vitest seam.** `projectAnnotationNodes` is independently tested in `selectors.test.ts`; folding it into `projectNodes` would force all annotation-projection coverage through the `GraphCanvasPane` integration test surface, undermining the predecessor task's deliberate per-helper split.
  - **Alternative considered: compute `annotationsByNode` inside `projectAnnotationNodes`.** Rejected — duplicates work + adds a `groupAnnotationsByEntityId` import to the per-helper boundary for a value `projectNodes` already holds.
  - **Alternative considered: inline `projectAnnotationNodes` into `projectNodes`.** Rejected — undermines the predecessor's testability seam.

- **D4 — Widen `<AnnotationNode>` to render an `<AnnotationBadge>` list row, reusing the existing badge component; relax the card's fixed `height: 56` to `minHeight: 56` so the row grows the card vertically; defer dagre dimensions reconciliation to the standing `mod_layout_measured_dimensions` post-mount measurement pass.** Rationale (moderator-specific delta beyond what audience D1–D6 cover):
  - **The audience could absorb the propagation without component changes because its DOM overlay is node-kind-blind.** The moderator's badge rendering is per-component (inside `<StatementNode>` / `<StatementEdge>`). Without a parallel render inside `<AnnotationNode>`, the propagation data lands but the user sees nothing. The task name commits to the visible overlay, so the render is in scope.
  - **Reuse `<AnnotationBadge>` rather than build a new variant.** The badge already encodes the moderator's per-annotation vocabulary (localized kind label, content hover, amber palette, `data-annotation-kind` / `data-testid` conventions). Adding a third call site (after `<StatementNode>` L526-532 and `<StatementEdge>`'s edge-label-renderer arm) is consistent with the existing pattern and requires no new component, no new icon, no new color choice.
  - **Switch fixed `height` → `minHeight` so the badge row grows the card.** The card's content area is fixed at 56px today (header + truncated content row). A flex-wrap badge row underneath would clip if `height: 56` stays; relaxing to `minHeight: 56` lets the card grow vertically while preserving the existing baseline for empty-meta-annotation cards. The exported `ANNOTATION_NODE_HEIGHT` constant (read by dagre as the placement footprint) stays at 56 — dagre's initial placement is unchanged.
  - **Defer dagre dimensions reconciliation to `mod_layout_measured_dimensions`.** That task's post-mount measurement pass reads each rendered node's actual DOM height and feeds it back into dagre on the next layout cycle. The badge-row growth is reconciled there naturally; no per-task layout-engine wiring is required. Acknowledged consequence: a brand-new annotation-node-with-meta-annotations renders for one paint cycle with the 56px-allocated footprint before the measurement pass triggers a re-layout. That's the same behavior the existing `<StatementNode>` has when its content grows — no regression vs. the established convention.
  - **Alternative considered: render only a count indicator (`+N annotations`) rather than the full badge list.** Rejected — diverges from the moderator's existing per-annotation badge vocabulary; loses the kind discrimination (which is the load-bearing semantic for a moderator reading the canvas). The participant uses a count because Cytoscape can't host React components inside nodes; the moderator can, so reusing `<AnnotationBadge>` is the consistent move.
  - **Alternative considered: keep `height: 56` fixed and clip overflow.** Rejected — the badge row would render clipped or absent; the user can't see the propagation, which defeats the task's purpose.
  - **Alternative considered: bump `ANNOTATION_NODE_HEIGHT` to a larger constant (e.g. 88) to accommodate one row of badges.** Rejected — the new height applies even to annotation nodes with no meta-annotations, wasting layout space for the common case. `minHeight` + dagre measurement reconciliation handles the variable-height case naturally.
  - **Alternative considered: split the render-side change into a follow-up `mod_annotation_node_meta_badge_render` task.** Rejected — leaves the propagation data invisible after this task lands; the task name commits to the visible overlay. Bundling keeps the in-flight state coherent.

- **D5 — No Playwright spec. The propagation is pinned at the Vitest layer (selectors + component); the predecessor's Playwright block already exercises annotation graph-node rendering at the e2e seam.** Rationale (mirrors participant D5 / audience D5 with moderator-specific seam names):
  - **The change is a thin Map-keying semantic plus a small in-component render.** What flips between "today" and "after this task" is: `projectAnnotationNodes` reads `nodeAnnotationIndex.get(annotation.id)` and stamps the result on `data.annotations`; `<AnnotationNode>` renders one `<AnnotationBadge>` per entry. The data-flow + render are fully pinned by Vitest at the selector layer + the component layer; the moderator's existing Playwright suite exercises the surrounding `/sessions/:id/operate` flow at the e2e seam.
  - **The predecessor's Playwright block already exercises annotation graph-node rendering at the e2e seam.** Per `mod_render_annotation_endpoint_edges` Acceptance criteria — "Includes Playwright cover asserting an annotation-endpoint edge appears on the canvas." Adding a parallel block for the annotation-on-annotation badge would pile on the file with very similar setup (annotation-endpoint edge seeding + canvas-node assertion + a per-badge assertion). The UI-stream policy explicitly warns about pile-on of catch-all e2e files; the badge-presence signal is what the component test pins.
  - **The component IS reachable** (the moderator console renders the canvas at `/sessions/:id/operate`; the predecessor task wired the annotation node). The deferral is justified by coverage duplication, not by unreachability — matching the participant + audience reasoning verbatim.
  - **ADR 0022 is satisfied.** Every empirical check is a committed Vitest case; no throwaway verification.
  - **Inherited-debt count check.** No catch-all `mod_pw_*` task is in flight that this refinement should pay debt down on by inlining a spec — the predecessor `mod_render_annotation_endpoint_edges` itself owns the proximate Playwright coverage for annotation-node rendering. No new debt registered (per Acceptance criteria's Tech-debt registration row).
  - **Alternative considered: add a parallel block to the predecessor's Playwright spec.** Rejected — duplicates predecessor coverage; pile-on the proximal catch-all file; the unique signal ("badge present on the annotation card") is fully captured by the component test.
  - **Alternative considered: defer Playwright to a future `mod_pw_annotation_of_annotation_overlay` task.** Rejected — registers a thin debt item with no real coverage delta; the Vitest layer is the right pin for a data-flow Map-keying change + a small in-component render. The audience + participant both rejected the same option on the same grounds.

- **D6 — No new ADR. The polymorphic-entity-id convention is the participant predecessor's already-documented contract; this task is the moderator-side consumer-migration that adopts it, plus a per-component render reuse that's a refinement-level choice.** Rationale:
  - **The convention's docstring lives at the shell layer.** [`packages/shell/src/annotations/annotations.ts:100-120`](../../../packages/shell/src/annotations/annotations.ts) already documents the polymorphic-entity-id semantic; no new architectural commitment lands here.
  - **The moderator surface's wire-up is a Refinement-level decision, not an architectural one.** Threading `nodeAnnotationIndex` into `projectAnnotationNodes`, widening `AnnotationNodeData` by one field, and adding a third `<AnnotationBadge>` call site inside `<AnnotationNode>` — all of these are function-signature + component-level changes; no new dependency, no new architectural seam.
  - **Alternative considered: write an ADR for "moderator surface annotation-of-annotation propagation."** Rejected — over-specifies a consumer-side wiring that's already constrained by the shell-layer contract; would set precedent for ADR'ing every per-surface consumer migration.
