## What this task is

Extend [`setEdgeSubstanceProposalSchema`](../../../packages/shared-types/src/events/proposals.ts) (currently `{ kind, edge_id, value }` only) with three new required-but-conditional fields — `source_node_id`, `target_node_id`, `role` — that the propose handler's `set-edge-substance` arm consumes to emit `edge-created` + `entity-included` events at propose-time alongside the `proposal` envelope, parallel to the free-floating `classify-node` fan-out commit `17822e1` landed for nodes. The change closes the second half of the canvas-visibility contract per ADR 0027 ("propose-time emits structural events for new entities") for the connecting-edge case: today the source node enters the canvas at propose-time (via the moderator's two-envelope `classify-node` → `set-edge-substance` chain — the first envelope mints the source node) but the edge itself does not surface until commit, leaving the canvas in an inconsistent half-proposed state during the proposed window.

Concretely the deliverable is four files of change:

1. **Schema extension** at [`packages/shared-types/src/events/proposals.ts:116-122`](../../../packages/shared-types/src/events/proposals.ts) — three new fields on `setEdgeSubstanceProposalSchema`, ALL marked OPTIONAL (`.optional()` — see Decision D1 for the rationale on optional-vs-required + the "new-edge predicate" the propose handler uses to discriminate the connecting case from substance-only re-votes against an existing edge).
2. **Client builder update** at [`apps/moderator/src/layout/useProposeAction.ts:192-198`](../../../apps/moderator/src/layout/useProposeAction.ts) — `buildSetEdgeSubstanceProposal` grows three new parameters (`sourceNodeId`, `targetNodeId`, `role`) and threads them onto the payload. The two-envelope `classify-node` → `set-edge-substance` chain in `propose()` (L398-404) passes `nodeId` (the just-minted source), `targetEntityIdNow` (the existing target the moderator clicked), and `edgeRoleNow` (the role the moderator selected via `<EdgeRoleSelector>`) into the builder.
3. **Propose handler arm** at [`apps/server/src/methodology/handlers/propose.ts:1313-1322`](../../../apps/server/src/methodology/handlers/propose.ts) — the `set-edge-substance` arm of `buildStructuralEventsForPropose` (currently a no-op placeholder with a long header comment naming this task as the follow-up) gets the real emission. When all three new fields are present AND `projection.getEdge(edge_id)` is undefined (the fresh-edge predicate — symmetric with `classify-node`'s `projection.getNode(nodeId) === undefined && wording !== undefined` shape), emit one `edge-created` followed by one `entity-included` per the established sequence-allocator pattern.
4. **Lockstep withdraw arm** at [`apps/server/src/ws/handlers/withdraw.ts`](../../../apps/server/src/ws/handlers/withdraw.ts)'s `entitiesToRetractForWithdraw` switch — add a `'set-edge-substance'` case mirroring the `'classify-node'` case: when the pending proposal's payload carries the three new fields AND `projection.getEdge(payload.edge_id) !== undefined` (the edge was minted at propose-time so the projector has it; the visibility-flip via `handleEntityRemoved` is what removes it from the canvas), push `{ entityKind: 'edge', entityId: payload.edge_id }` onto the retraction list.

Plus the test-coverage tighten in `tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts:143-204` — scenario 2's deferred "edge visibility is OUT OF SCOPE" note becomes an active assertion on the rendered `data-facet-status="proposed"` edge per Acceptance criteria below.

## Why it needs to be done

Three reasons compose:

**Methodology contract — `docs/methodology.md` L57.** "A proposed change appears on the graph in `proposed` state from the moment it is made." Today the connecting-edge case half-honors this contract: the source node appears, the edge does not. A debater whose tablet shows a `proposed` node hanging in mid-air with no visible connection to the existing target cannot evaluate the proposal's argumentative load — the edge IS the load-bearing structural claim ("X supports Y", "X rebuts Y", "X qualifies Y"). The half-rendered state is a usability regression against the prior commit-time-only contract (which at least showed nothing until commit, signalling honestly that the structural relationship was still pending).

**ADR 0027 consequence — bullet 1, L20-32.** "When a proposal introduces new entities (a `classify-node` for a fresh node, an edge to an existing target, a `decompose` with N components + M edges, an `interpretive-split` with the analogous shape), the propose handler emits the corresponding `node-created` / `entity-included` / `edge-created` events alongside the `proposal` envelope." The ADR named the edge case explicitly; commit `17822e1` landed only the node arm because the `set-edge-substance` payload didn't carry the endpoints. Leaving the edge arm unimplemented after the ADR landed means the runtime contract drifts from the architectural decision — exactly the failure mode the ADR was supposed to prevent.

**Inverse-pair invariant.** Per [`ws_withdraw_proposal_message.md`](../backend/ws_withdraw_proposal_message.md) D3 the `entitiesToRetractForWithdraw` switch and `buildStructuralEventsForPropose` are inverses that "MUST stay in sync." Growing the propose-time emission without growing the withdraw arm leaves a proposer who withdraws a connecting `set-edge-substance` proposal with the source node retracted (via the first envelope's withdraw, when the gesture-wiring task lands) but the edge orphaned on the canvas in `proposed` state forever. The withdraw refinement explicitly names this task as the trigger for the lockstep update, and commit `e89cf06` amended this task's `.tji` note to call it out as a hard requirement.

Downstream consumers benefit transitively:

- `participant_ui.part_graph_view` (pending under M5) — inherits the projector seam; no per-surface fix needed.
- `audience.aud_graph_rendering` (pending under M6) — same transitive inheritance.
- `mod_draw_edge_flow` (the dedicated drag-to-create UI path, not yet implemented) — when it lands, it will mint a connecting edge proposal via the same client builder this task extends; the propose-handler arm is the seam it reuses.
- The lifted scenario 2 in the Playwright e2e becomes the regression cover that catches any future drift in either direction (a propose-time arm that stops emitting; a wire-shape change that loses the endpoint carriage; a projector regression that fails to render the proposed edge).

## Constraints / requirements

- **Three new fields are `.optional()` on the schema, not `.required()`.** Per D1: the `set-edge-substance` proposal sub-kind serves two distinct use-cases on the wire — (a) proposing the substance for a freshly-minted edge (the connecting case this task targets), and (b) proposing a substance re-vote against an extant edge (e.g. the defeater-precommit flow at `apps/server/src/methodology/handlers/proposeDefeaterPreCommit.test.ts:175-205`). Case (b) has no source / target / role to carry — the edge already exists; the proposal is a facet-layer transition only. Making the fields required would break (b) and force every existing call site to thread fake values. Making them optional keeps both shapes valid; the propose handler discriminates via the fresh-edge predicate `projection.getEdge(edge_id) === undefined`.
- **Fresh-edge predicate symmetry with `classify-node`.** The `classify-node` arm uses `projection.getNode(nodeId) === undefined && wording !== undefined` (both branches must hold) — node-absence AND wording-presence together signal "the client is minting this entity." The `set-edge-substance` arm mirrors: `projection.getEdge(edge_id) === undefined && source_node_id !== undefined && target_node_id !== undefined && role !== undefined`. All four branches must hold. The predicate's `===` / `!==` shape is intentional: a future `set-edge-substance` proposal that carries the endpoints AGAINST an existing edge (e.g. some future re-classify flow) emits zero structural events; the projector already has the edge.
- **Schema-on-write per ADR 0021.** The extended `setEdgeSubstanceProposalSchema` validates at `parseWsEnvelope` time; payloads with unknown fields are stripped per Zod's default behaviour. The emitted `edge-created` envelope runs through `validateEvent` (per the propose handler's existing schema-on-write pass) — its payload schema already requires `source_node_id` / `target_node_id` / `role`, so a propose handler bug that drops one of the three fields fails loudly at `validateEvent` before `appendSessionEvent`.
- **Reuse `edgeRoleSchema` for the new `role` field.** The proposal payload's `role: edgeRoleSchema.optional()` reuses the existing enum at [`packages/shared-types/src/events/enums.ts:21-31`](../../../packages/shared-types/src/events/enums.ts) (re-exported from `events.ts`). Don't redeclare the enum inline; the seven-value vocabulary stays single-source-of-truth shared with `edgeCreatedPayloadSchema` and the moderator's `<EdgeRoleSelector>`.
- **Sequence allocation mirrors `classify-node`.** Each emitted event takes the next slot via `seq(events.length)` (the existing helper at `propose.ts:1269`); the `proposal` envelope takes the last slot at `action.sequence + structuralEvents.length` (no change to the L1200-1209 envelope construction). For a connecting `set-edge-substance` with the three new fields, the emission produces three events total: `edge-created` at `action.sequence + 0`, `entity-included` at `action.sequence + 1`, `proposal` at `action.sequence + 2`.
- **Connecting-edge UX assumes the source node was minted by the prior `classify-node` envelope.** The client's two-envelope chain (`useProposeAction.ts:380-404`) fires the `classify-node` envelope first; its broadcast advances `lastAppliedSequence` before the second envelope's `expectedSequence` token is read. By the time the propose handler processes the `set-edge-substance` envelope, the projection's `getNode(sourceNodeId)` resolves the just-minted node — the predicate `projection.getNode(source_node_id) !== undefined` could be added as a defensive validator-check (rejecting `'target-entity-not-found'` if violated), but per D2 the v1 handler does NOT add this check (the engine's existing universal-pass placeholder already passes through and the methodology validator for `set-edge-substance` is the right home for a cross-field referential check, not the structural-event builder). If a future task adds the methodology validator, the check belongs there.
- **Lockstep withdraw arm is in the same commit cluster.** Per the source-of-debt refinement's withdraw-handler invariant and the `.tji` note amendment (commit `e89cf06`), the withdraw handler's `entitiesToRetractForWithdraw` switch grows the matching `'set-edge-substance' → entity-removed(edge)` arm in the same commit (or tight commit cluster) as the propose-time emission. The two are an inverse pair; landing one without the other leaves the bug half-fixed (the proposer can never retract the proposed edge).
- **No new event kind, no new payload schema, no SQL migration.** All wire-format primitives are already on `main` per commit `17822e1`: `entity-removed` event kind, `entityRemovedPayloadSchema`, the SQL `CHECK` constraint widening, the projector's `handleEntityRemoved` arm. This task threads existing primitives through new code paths; the schema additions are scoped to the proposal payload (`setEdgeSubstanceProposalSchema`) only.
- **The Playwright scenario 2 lift is the UI-stream e2e cover.** Per ORCHESTRATOR.md "UI-stream e2e policy" + the parent refinement's deferred-scenarios list, this task activates the already-shipping scenario 2 by removing the "OUT OF SCOPE" comment block + tightening the assertions. The scenario stays in `tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts` (rather than a new spec file) because it's the same canvas-visibility contract; splitting into a new file would orphan the contract from the scenario set ADR 0027's stack-validation tests live in.
- **Test coverage layered per ADR 0022.** Wire-shape pin → Vitest at `apps/server/src/methodology/handlers/propose.test.ts` (the new arm emits the three events in order) + `apps/server/src/ws/handlers/withdraw.test.ts` (the new arm retracts the edge) + `packages/shared-types/src/events/proposals.test.ts` (the three optional fields round-trip cleanly) + `apps/moderator/src/layout/useProposeAction.test.tsx` (the client's second envelope carries the three fields). UI-stream end-to-end → the lifted scenario 2 in `tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts`.

## Decisions

- **D1 — Three new fields are `.optional()` on `setEdgeSubstanceProposalSchema`, with a fresh-edge predicate in the propose handler.** The `set-edge-substance` proposal sub-kind serves two distinct use-cases on the wire: (a) proposing the substance for a freshly-minted edge (the connecting case this task targets); (b) proposing a substance re-vote against an extant edge (e.g. the defeater-precommit flow at `apps/server/src/methodology/handlers/proposeDefeaterPreCommit.test.ts:175-205`, which sends `{ kind: 'set-edge-substance', edge_id: <existing>, value: 'disputed' }` with no endpoints). The propose handler's `set-edge-substance` arm discriminates via `projection.getEdge(edge_id) === undefined && source_node_id !== undefined && target_node_id !== undefined && role !== undefined` — case (a) emits two structural events, case (b) emits zero. Alternatives considered:
  - **Required fields on the schema.** Rejected — would break the defeater-precommit test + any future substance-only re-vote flow. Forces every call site to thread fake endpoints which would then have to be ignored by the handler — extra wire bytes for no gain, and "thread fake fields then ignore them" is a recipe for the fake field becoming load-bearing through misuse.
  - **Two separate sub-kinds (`mint-edge-substance` and `revote-edge-substance`).** Rejected — proliferates sub-kinds for what is structurally one operation (proposing a substance value for an edge). The methodology engine's `facetTargetForProposal` switch at `apps/server/src/methodology/handlers/commit.ts:95` would need both arms; vote / commit / mark-meta-disagreement would all branch on the split; the WBS + design docs would need updating. The schema-level `.optional()` + propose-handler predicate is a strictly smaller change.
  - **Refine() guard on the schema enforcing "all-three-present OR all-three-absent".** Rejected — adds runtime validation work for a constraint the propose handler already discriminates on. A wire payload with two-of-three present (which the refine would reject as malformed) is structurally a client bug; rejecting it via the propose handler's "predicate doesn't hold, emit no structural events" path AND the projector's eventual "edge_id doesn't exist post-commit" rejection at the commit handler is sufficient. The simpler schema is easier to extend if/when a future flow needs a different mix of fields.

- **D2 — No cross-field referential check at the propose-handler level for source / target / role.** The propose handler's `set-edge-substance` arm does NOT check that `source_node_id` and `target_node_id` reference existing visible nodes on the projection, nor that the proposed `role` is valid for the source / target kinds. Per the file's current header comment (L1106-1112) the `set-edge-substance` arm falls through to the universal-pass placeholder; a future methodology validator (sibling task) will tighten it. The structural-event builder's job is wire-shape emission, not methodology validation — keeping the two concerns separate matches the pattern the file's other arms already follow (the methodology validators are at the top of the file; the structural-event builder is at the bottom; each arm is one of the two, never both). Alternative considered: add a defensive `projection.getNode(source_node_id) !== undefined` check in the structural-event builder. Rejected — the check belongs in the methodology validator alongside the role-source-target-compatibility check the methodology framework eventually needs; landing one without the other puts cross-field validation in two places (the structural-event builder's defensive check + the future methodology validator) and creates a maintenance hazard. The current commit's `validateEvent` schema-on-write pass on the emitted `edge-created` envelope catches malformed UUIDs / enum values; the deeper referential check is deferred to the methodology-validator task.

- **D3 — Lockstep withdraw arm grows in the same commit cluster.** Per the source-of-debt withdraw refinement's D3 + the `.tji` note amendment (commit `e89cf06`), the inverse-pair invariant between `buildStructuralEventsForPropose` and `entitiesToRetractForWithdraw` is enforced by code review. Landing the propose-time emission without the matching retraction would leave a proposer who withdraws a connecting `set-edge-substance` proposal with an orphaned edge on the canvas in `proposed` state forever (the proposal envelope itself stays in the log per ADR 0021's immutability; the entity layer needs an explicit retraction to flip `edge.visible = false`). Alternative considered: ship the propose-time emission first, ship the withdraw arm in a follow-up commit. Rejected — the inverse-pair invariant is fragile precisely because the two functions live in different files; a deferred follow-up risks the same drift the original gap caused. Same-commit-cluster landing makes the inverse-pair locality a code-review fact.

- **D4 — Reuse `edgeRoleSchema` for the new `role` field.** The proposal payload's `role` field is typed `edgeRoleSchema.optional()` — the same enum [`packages/shared-types/src/events/enums.ts:21-31`](../../../packages/shared-types/src/events/enums.ts) exports for `edgeCreatedPayloadSchema` and the moderator's `<EdgeRoleSelector>`. Single-source-of-truth on the seven-value vocabulary. Alternative considered: define a new `setEdgeSubstanceRoleSchema` enum local to `proposals.ts`. Rejected — the methodology contract is that the role on the proposal becomes the role on the structural event; using two different enums would invite drift if a new role lands in the structural-event enum without landing in the proposal enum (or vice versa). The single enum is the simpler, drift-resistant choice.

- **D5 — Client builder grows three required parameters (not optional ones) for the connecting case.** The client-side `buildSetEdgeSubstanceProposal` helper in `useProposeAction.ts` takes the three new parameters as required (not optional) in the connecting branch: the caller (the `propose()` function at L313-423) already knows it's connecting via the `isConnecting` flag at L352, and it has all three values in lexical scope. The schema-level `.optional()` is for wire-shape consumers who legitimately omit the endpoints (the defeater-precommit flow + any future substance-only re-vote flow); the client-side helper is a constructor for one specific shape (the connecting case) and benefits from required parameters that fail loudly at the call site if a refactor breaks the wiring. Alternative considered: optional parameters on the builder too. Rejected — would let a buggy call site forget to pass one of the three and silently produce a malformed payload that the propose handler's predicate would treat as the substance-only re-vote shape, emitting no structural events and producing the exact regression this task closes. Required parameters make the bug a TypeScript error at compile time.

- **D6 — Lift Playwright scenario 2 in place, not in a new spec file.** The parent refinement's deferred-scenarios list explicitly names this task as the activator for scenario 2's edge-visibility assertion. The scenario stays in `tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts` because the spec file IS the regression cover for ADR 0027's stack-validation tests (per the ADR's "Stack-validation tests" section at L58-60). Splitting the edge assertion into a new file would orphan it from the scenario set the ADR named. Alternative considered: new spec file `moderator-proposed-edge-canvas-visibility.spec.ts`. Rejected — fragments the contract across two files for what is structurally one scenario (the same canvas, the same propose chain, the same `data-facet-status="proposed"` contract; the lift only adds one locator to an already-running test).
