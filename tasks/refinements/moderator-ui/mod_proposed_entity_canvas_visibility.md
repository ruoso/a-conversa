## What this task is

Close a regression where proposed statements appeared **only in the pending-proposals sidebar** and were NOT actually rendered on the graph canvas, violating the design contract in `docs/methodology.md` L57 ("A proposed change appears on the graph in `proposed` state from the moment it is made") and `docs/moderator-ui.md` L46 ("The graph shows the new node and edge in `proposed` state. The pending-proposals pane fills in"). The regression surfaced 2026-05-16 from a manual-browser observation by daniel@ruoso.com and was diagnosed in `tasks/99-milestones.tji` L52: the propose handler today emits exactly one `proposal` event per envelope (per [`tasks/refinements/backend/ws_propose_message.md`](../backend/ws_propose_message.md) L13 + the corrected wire-shape narration from [`mod_propose_action_refinement_amendment`](./mod_propose_action_refinement_amendment.md)); structural creation (`node-created` / `entity-included` / `edge-created`) is a commit-time fan-out. The canvas projector at `apps/moderator/src/graph/GraphCanvasPane.tsx:434` walks `node-created` events to render nodes, so a propose without a commit produces a sidebar entry but no canvas node.

This task lands the architecturally-correct fix per [ADR 0027](../../../docs/adr/0027-entity-and-facet-layers-strict-separation.md). It ships in three layered stages:

1. **Failing-first Playwright e2e** under `tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts` covering four scenarios (free-floating single-node propose, propose-with-new-edge, 2-component decomposition propose, propose-then-withdraw). Each asserts canvas state via `page.getByTestId('statement-node-<nodeId>')` + `data-facet-status="proposed"`. Authored + committed **red** against the pre-fix runtime; the red commit IS the verification artifact per ADR 0022.
2. **Protocol + handler + migration fix** per ADR 0027: the propose handler grows a structural-event emission step per sub-kind that introduces new entities; the commit handler stops re-emitting structural events for entities created at propose-time (except `edit-wording.restructure`'s genuine commit-time mint); a new `withdraw-proposal` WS handler emits `entity-removed` events; a new event kind `entity-removed` is added to the SQL `CHECK` + payload-schema registry + projector audit pass; every consumer of `node-created` / `edge-created` / `entity-included` is migrated from event-presence to facet-status reads.
3. **Re-run the e2e green; update unit tests + amend refinements.** `apps/moderator/src/graph/GraphCanvasPane.test.tsx`'s `node-created` seeding stays valid (the projector contract on `node-created` is unchanged); the amendments to [`tasks/refinements/backend/ws_propose_message.md`](../backend/ws_propose_message.md) and [`tasks/refinements/backend/ws_commit_message.md`](../backend/ws_commit_message.md) land in the same commit cluster.

## Why it needs to be done

The methodology contract is unambiguous: a proposed change is **visible** on the canvas from the moment of proposal. The visibility is the property that prevents stonewalling (per `docs/methodology.md` L40 — "A debater who refuses to agree to (say) a `contradicts` edge against their own claims cannot make the proposal vanish — only delay or block its transition to `agreed`"). Without this fix:

- The moderator's at-a-glance canvas scan misses every in-flight proposal. The sibling `mod_proposed_state_styling`'s dashed-border + opacity-60 visual contract is undermined because no entity actually reaches the canvas to receive it.
- The participant tablet (which mounts the same projection model per ADR 0026) and the audience surface inherit the same gap. A debater whose tablet shows no canvas reflection of a proposal cannot vote against it via a canvas gesture.
- The pending-proposals sidebar is the **listing** of in-flight proposals; the canvas is the **structural view** of the graph. Conflating "no commit yet" with "not on the canvas" loses the structural-view-of-proposals semantics that diagnostics, axiom-marks, and edge proposals all rely on.

The deeper rationale comes from the test pyramid: the unit tests at `apps/moderator/src/graph/GraphCanvasPane.test.tsx` (per `tasks/99-milestones.tji` L52, lines L713 / L887 / L689) pass today because they seed `node-created` events directly into the WS store, short-circuiting the live event flow. The tests assert exactly the contract this task pins, but they don't exercise the propose→canvas path because they synthesize the structural events themselves. No existing Playwright spec asserts canvas state after a propose (`tests/e2e/moderator-capture.spec.ts` only checks the sidebar). The failing-first Playwright e2e in stage 1 closes that pyramid gap; the underlying protocol fix in stage 2 closes the runtime gap. This is exactly the contract-drift class that the test layering in ADRs 0006 / 0007 / 0008 is designed to catch.

Downstream consumers benefit transitively:

- `moderator_ui.mod_graph_rendering.mod_per_facet_state_visualization` (done) — once entities reach the canvas at propose-time, the per-facet visualization fires for proposed facets immediately.
- `participant_ui.part_graph_view` (pending under M5) — inherits the projector seam; no per-surface fix needed.
- `audience.aud_graph_rendering` (pending under M6) — same transitive inheritance.
- `moderator_ui.mod_tests.mod_e2e_playwright.mod_pw_full_session_run` (M7 dependency) — the walkthrough scenario covers propose-time visibility; landing this task unblocks the visual cells.

## Decisions

- **D1 — Protocol resolution path: propose-time emits structural events; commit becomes a pure facet-state transition.** Per ADR 0027 L20-32 the entity layer and the facet layer are architecturally separate. The facet layer exists precisely so the entity layer doesn't have to carry agreement state. `node-created` was always meant to be "entity entered the structure"; the facets (proposed / agreed / disputed / committed / withdrawn / meta-disagreement) were always the layer that knows the negotiation state. The current implementation collapsed the two by gating structural events on commit; the fix removes that gate. Alternatives (per ADR 0027 L34-40):
  - **(B) Client-side ghost synthesis** (each projector composes a virtual node from the `proposal` event when no `node-created` has landed). Rejected — duplicates a branch across three projector implementations (moderator + participant + audience); introduces id-reconciliation at commit-time (ghost id vs server-allocated real id); keeps the entity-layer-as-commit-record confusion alive in the protocol while papering over it in the client.
  - **(C) New `proposed-entity-placeholder` event kind** that the commit handler consumes and replaces with a real `node-created`. Rejected — adds an event kind to maintain when the existing `node-created` already means what we need it to mean once the propose-vs-commit timing constraint is removed; net more code than firing the existing event earlier.
  - **(D) Refinement-doc-only amendment** without changing the runtime. Rejected — the original implementation got the timing wrong because the principle wasn't visible enough; burying the correction in a refinement risks the same drift recurring.

- **D2 — Edges are in scope for this task.** `edge-created` follows the same commit-time-gated fan-out as `node-created` and has the same gap. The propose handler change in D1 extends to `edge-created`; the audit pass extends to every `edge-created` consumer; the failing-first Playwright e2e includes at least one edge-visibility assertion (scenario 2). Even though `mod_draw_edge_flow` (the dedicated drag-to-create UI path) is not yet implemented, the e2e drives an edge-creating proposal via the implicit edge that `set-edge-substance` proposals stage when a new statement connects to an existing target. Alternative considered: split into a follow-up `mod_proposed_edge_canvas_visibility` leaf. Rejected — would force the same Q&A round again and risk drift if the follow-up is deprioritized; the bug is one bug, not two.

- **D3 — Multi-entity proposals (`decompose`, `interpretive-split`) are in scope.** Under D1 the propose handler emits N × (`node-created` + `entity-included`) + M × (`edge-created` + `entity-included`) for the proposal's child specs alongside the `proposal` envelope. The original target node stays visible during the proposed window (per `docs/data-model.md`'s Node visibility — target visibility flips off only on commit of `edit-wording.restructure` / `decompose` / `interpretive-split`); the canvas shows target AND proposed components AND proposed edges simultaneously. That simultaneous visibility is the intended UX (moderator and participants need to see "what this would become" alongside "what it is now" to evaluate the proposal). Commit then atomically flips the target's visibility off and transitions the components' facets. Rationale for inclusion in this task: the multi-entity behavior was meant to be part of the original implementation but missed by the same commit-time-gating bug; landing the fix without exercising the multi-entity path leaves the bug half-fixed. The e2e scenario 3 (2-component decomposition) is the regression cover.

- **D4 — Withdraw / rejection lifecycle uses explicit `entity-removed` events.** When a proposal is withdrawn before commit, the propose-time-created entities leave the structure via explicit `entity-removed` events. Sub-shapes for node vs edge vs annotation via the `entity_kind` discriminator (mirroring the `entity-included` payload). Alternative considered:
  - **Implicit derivation** from "proposal withdrawn AND entity has no committed facets". Rejected (per ADR 0027 L30) — pushes a derivation rule into every projector (moderator + participant + audience), makes "why isn't this entity visible?" a multi-step query instead of a single-event lookup, and creates permanent projector-complexity tax for a condition cheaper to represent explicitly.

- **D5 — Architectural principle captured in [ADR 0027](../../../docs/adr/0027-entity-and-facet-layers-strict-separation.md), landed ahead of this task.** ADR 0027 (accepted 2026-05-16) is the architectural source of truth for the entity-vs-facet separation. Landing it ahead of the implementation means the principle is visible to anyone reading the WBS or design docs before the runtime change lands. ADR 0021 received an amendment-pass note pointing forward to ADR 0027 for the timing semantics of structural events. This task lands the runtime changes ADR 0027 mandates; no new ADR is required (per `tasks/refinements/README.md:46-51` the refinement is the work-shaping document; ADRs capture architectural choices, and this task is implementation of an already-accepted architectural choice).

- **D6 — Failing-first e2e is a real failing test, not a `test.fixme` or `test.fail()` annotation.** Per ADR 0022's "no throwaway verifications" framing, the failing-first commit produces real test failures the orchestrator can verify discriminate the regression. A `test.fixme` would silently no-op and provide no evidence; a `test.fail()` annotation would let the red commit pass CI but has to be removed in the fix commit anyway (same workflow as flipping the assertions, but without the explicit evidence in the red commit's test output). Plain-`expect` assertions on red, captured to a file in the orchestrator's test-output-handling convention, are the strongest artifact. Alternative considered: a pre-fix smoke-only commit that stages the file without running it. Rejected — gives the orchestrator no way to verify the test discriminates, and leaves a commit on `main` where the spec is silently broken.

- **D7 — Proposer-only withdraw authority, engine-enforced.** Only the original proposer (identified by `action.actor` on the original `proposal` event) may withdraw the proposal. A non-proposer participant attempting withdraw is rejected with a new engine `RejectionReason` `'not-proposer'` mapped to wire 403 via `rejectedToApiError`. Parallels the moderator-only commit gate (`'not-a-moderator'` per `ws_commit_message`'s Decisions); per-action authority lives in the engine, surfaced uniformly on the wire. Alternative considered: let the moderator also withdraw. Rejected for v1 — the moderator already has meta-disagreement-mark + structural-resolution surfaces; "the moderator cancels someone else's proposal" is a heavier action that should be designed explicitly (with an explicit `cancel-proposal` surface) if/when needed, not silently accreted on the withdraw path. The narrower v1 authority gate is the more honest default; widening it later is non-breaking.

- **D8 — `entity-removed` payload schema discriminates entity kind via `entity_kind: 'node' | 'edge' | 'annotation'`.** Same shape as `entity-included`'s payload so the two are parallel and a projector that handles inclusion can mechanically extend with the matching removal. Alternative considered: three separate event kinds (`node-removed`, `edge-removed`, `annotation-removed`). Rejected — the entity-included precedent uses a single payload-discriminated kind; mirroring it keeps the registry's append-only growth bounded.

- **D9 — Audit pass is a documented invariant, not an automated check.** The audit pass migrates "if I see `node-created` then the entity is committed" call sites to consult `facetStatus.ts` / `facet-status.ts` instead. Documented in inline docblock comments on projector entry points. Alternative considered: a custom ESLint rule flagging "structural-event-presence as commit-status" patterns. Rejected for v1 — the pattern is too contextual for a syntactic rule (the same code shape can mean either "I'm projecting structural state" or "I'm checking commit-status"); the documented-invariant approach is honest about the contextual nature, and future drift is caught by the unit tests that exercise the projector against propose-time-only events.

- **D10 — Forward-only migration adds `entity-removed` to the `CHECK` constraint.** Per ADR 0020 the migration is `apps/server/migrations/0012_session_events_entity_removed.sql` (or next-numbered) and does a `ALTER TABLE session_events DROP CONSTRAINT ...; ALTER TABLE session_events ADD CONSTRAINT ... CHECK (kind IN (..., 'entity-removed'));` sequence. No down migration. The migration's docblock cites this refinement + ADR 0027.
