## What this task is

Concrete, mechanical work:

- **New library at the moderator workspace**: `@dagrejs/dagre@1.1.4` — pinned, no caret, in `apps/moderator/package.json` under `dependencies`. The package is the actively-maintained MIT fork of the abandoned upstream `dagre`. The library choice is pre-decided by [ADR 0025](../../../docs/adr/0025-graph-layout-engine-dagre.md); this task imports it. (No other moderator-workspace dependency provides a layout algorithm: `reactflow` is interactive-edit-first by design and ships zero auto-layout; `react` / `react-dom` / `zustand` / `i18next` are irrelevant; the repo-root `cytoscape@^3.30.0` devDependency is for a smoke script and ADR 0004 forbids using Cytoscape on the moderator surface.)
- **New module**: `apps/moderator/src/graph/layoutEngine.ts` exporting:
  - `applyLayout(nodes, edges, options?)` — pure function. Walks `nodes`; any node whose id is present in `options.cache` gets its cached `{ x, y }` carried through unchanged. Any node NOT in the cache is fed to a dagre graph along with every edge whose source AND target are also being laid out (or already in the cache, so dagre can place the new node relative to its neighbours). The function returns a new `Node[]` whose `position` is the cached-or-dagre-derived coordinate; the rest of each node object passes through unchanged (no reallocation of `data`, `type`, `id` — referential identity of those fields is preserved per node so React / ReactFlow memoization downstream still benefits).
  - `relayoutAll(nodes, edges, options?)` — pure function. Ignores any cache and feeds every node + every edge to dagre for a fresh pass. Used by the future "tidy up" action; this task exports the function and tests it but does not wire any UI affordance to call it.
  - A `LayoutOptions` interface carrying `rankdir: 'TB' | 'BT' | 'LR' | 'RL'` (default `'TB'`), `nodeWidth: number` (default `220`), `nodeHeight: number` (default `90`), `rankSep: number` (default `60`), `nodeSep: number` (default `40`). Constants exported alongside (e.g. `DEFAULT_LAYOUT_OPTIONS`) so the test suite asserts against the same source of truth the production code reads.
- **Wire the new module into `<GraphCanvasPane>`**: at the call-site of `useMemo(() => projectNodes(events, diagnosticHighlights), [events, diagnosticHighlights])`, compose the layout pass:
  ```ts
  const positionCacheRef = useRef<Map<string, { x: number; y: number }>>(new Map());
  const nodes = useMemo(() => {
    const projected = projectNodes(events, diagnosticHighlights);
    const laidOut = applyLayout(projected, edges, { cache: positionCacheRef.current });
    // Update the cache with every position emitted, so the NEXT
    // memoization tick reuses these positions for already-laid-out nodes.
    for (const node of laidOut) {
      positionCacheRef.current.set(node.id, node.position);
    }
    return laidOut;
  }, [events, diagnosticHighlights, edges]);
  ```
  The `edges` dep is new in this `useMemo` (previously `nodes` was memoized on `[events, diagnosticHighlights]` alone) because the layout result depends on the edge set — a new `edge-created` event between two existing nodes does NOT move either node (both are in the cache and pass through unchanged), but a new `edge-created` event that connects a NEW node to an existing one feeds both into the dagre pass and the dagre placement of the new node depends on the edge being there.
- **Remove the placeholder grid from `projectNodes`**: the `GRID_COLS` / `GRID_DX` / `GRID_DY` constants and the `position: { x: (i % GRID_COLS) * GRID_DX, y: Math.floor(i / GRID_COLS) * GRID_DY }` block in `projectNodes` are replaced with a constant default position (`{ x: 0, y: 0 }` is fine — `applyLayout` will overwrite it for every node). Keeping the grid would be dead code; deleting it is the simpler path and the test cases that asserted against the grid layout (`projectNodes` "grid layout" case in `GraphCanvasPane.test.tsx`) are rewritten to assert against `applyLayout`'s output via the new `layoutEngine.test.ts` instead.
- **Preserve every other existing seam**: `data-testid="graph-canvas-root"`, the `NODE_TYPES` / `edgeTypes` module-scope refs, the selection / context-menu wiring, the `<Background />` grid, the empty `events` array fallback, the `EMPTY_EVENTS` / `EMPTY_ACTIVE_DIAGNOSTICS` stable references, every test id stamped by `<StatementNode>` / `<StatementEdge>`. The layout-engine wiring is purely a position-assignment seam.

The algorithm runs:

- On every `useMemo` tick triggered by `events` / `diagnosticHighlights` / `edges` reference changes — i.e. after each event the store applies that touches the per-session record. For already-laid-out nodes the path is "look up in cache, return"; only new nodes go through dagre.
- Implicitly on initial mount (the first tick with non-empty `events`).
- On user-triggered re-layout via `relayoutAll()` — the function is exported and tested, but the affordance is a future task (`mod_layout_tidy_action`).

## Why it needs to be done

`mod_node_rendering` left the canvas with a deterministic 4-column grid keyed by `node-created` arrival order (`x = (i % 4) * 240`, `y = floor(i / 4) * 140`). The grid keeps nodes from stacking but carries **none** of the methodology structure: a `supports` edge from `data` to `claim` is visualized identically regardless of where the two nodes sit on the grid, and the moderator cannot read "claim at the top, evidence underneath" from the canvas. Per `docs/methodology.md`, the methodology surfaces a top-down causal / inferential flow at the modal case (claim → warrant → data); the canvas should make that the at-a-glance reading order.

Without auto-layout the canvas is effectively unreadable beyond ~6 nodes — the grid's left-to-right wrap places ranks 1 and 2 (a claim and its first piece of evidence) on the same horizontal row, so the supports/rebuts hierarchy gets lost in the wrapping. The moderator can drag nodes by hand (ReactFlow defaults), but that's a per-session manual cost the methodology doesn't impose anywhere else — every other view (sidebar, mode banner, diagnostic flag pane) reads the structure automatically. This task closes that gap.

Downstream: `mod_capture_flow` (M4 milestone dependency) depends on `!mod_graph_rendering` as a whole. After this task and the one remaining sibling (`mod_pan_zoom`) close, the capture-flow stream unblocks.

## Constraints / requirements

- **Library**: `@dagrejs/dagre@1.1.4`, pinned, no caret. Added to `apps/moderator/package.json` `dependencies` (matches the `reactflow: 11.11.4` pin convention from `mod_graph_canvas_pane`). The package bundles its own `graphlib` peer so no second dependency line is needed.
- **Default orientation**: `rankdir: 'TB'` (top-to-bottom). The methodology's primary reading order is "claim at the top, evidence underneath"; placing source nodes (in the supports/rebuts/qualifies graph sense) above their targets matches that vertical reading. The function accepts other directions (`BT` / `LR` / `RL`) via the `LayoutOptions` parameter; the default is `'TB'`. Per-debate orientation switch is a future task if real moderation sessions show TB is wrong for some shapes.
- **Stability strategy**: incremental positions are cached in a `useRef<Map<string, { x: number; y: number }>>` inside `<GraphCanvasPane>`. The cache is consulted on every memoization tick: a node whose id is in the cache reuses the cached `{ x, y }`; a node whose id is NOT in the cache goes through dagre alongside every edge connecting it to either another new node or an already-cached node. Existing nodes never move on incremental events. `relayoutAll()` (separately exported) re-runs dagre over every node, intentionally moving everything — that's the user-triggered "tidy up" semantics; not wired to a UI affordance in this task.
- **Re-layout trigger**: auto on every `useMemo([events, diagnosticHighlights, edges])` tick. The `edges` dep is added in this task (the existing `useMemo` keyed only on `[events, diagnosticHighlights]`) because the layout result depends on the edge set, not just the node count.
- **Dimensions fed to dagre**: constant `width: 220, height: 90` per node. The width matches the typical rendered card (`min-w-[12rem] max-w-[18rem]` → ~192–288 px including padding; 220 is the empirical mid-point). The height is the typical two-line wording + kind label + one decoration row. Variable-height cards (per-facet pill row + axiom-mark badges + vote indicators) WILL overflow the 90 px slot in worst cases; the DOM-measured improvement is the follow-up `mod_layout_measured_dimensions` task, NOT in scope here.
- **Determinism**: `applyLayout` produces the same output for the same input. Dagre's algorithm is deterministic given a fixed graph + node ordering; the projection feeds nodes in `node-created` arrival order, which is the wire-event order the store preserves. Test suite asserts determinism by running `applyLayout` twice on the same input and comparing positions.
- **Performance**: ≤ 100 nodes is the realistic upper bound for a single debate. The algorithm must run in <100 ms on that size. Empirically dagre handles 100 nodes in well under 10 ms; the budget is generous so a regression is caught by the test rather than by user-visible canvas jank. One Vitest case constructs a 100-node graph and asserts `applyLayout` completes within a deliberately loose 100 ms wall-clock budget on the CI runner (and the test logs the actual time so a future tightening pass has the data).
- **Variable-height tolerance**: the constant 90 px height pin means very tall cards (multiple facet pills + multiple axiom-marks + multiple vote indicators + a diagnostic halo) will visually overflow the row below them. Accepted v1 trade-off pinned by ADR 0025; the `mod_layout_measured_dimensions` follow-up task lifts the limitation.
- **Meta-disagreement node compatibility**: `border-double` resolves to two thin parallel lines drawn within the card's box-sizing border. The double border does NOT widen the rendered footprint relative to a single border — Tailwind's `border` utility uses `box-sizing: border-box` defaults — so the dagre constant 220 px width remains accurate for meta-disagreement nodes.
- **Diagnostic-halo ring compatibility**: the `ring-4` / `ring-2` halo from `mod_diagnostic_highlighting` is rendered via CSS `box-shadow` (Tailwind's `ring-*` utility), which is `outline`-like and does NOT change the element's layout footprint. The dagre constant width is accurate for haloed cards.
- **Layout is locale-independent**: the algorithm itself does not read text. The constant 220 px width pin handles the localized-text width axis — the card's `max-w-[18rem]` (288 px) bounds the rendered width regardless of locale, so the dagre footprint is identical across en-US / pt-BR / es-419.
- **Compose with existing layers**: layout produces positions only; rendering layers (state-styling, halo, popover, decoration badges) stay separate. No edits to `<StatementNode>` / `<StatementEdge>` / `<HoverPopover>` / the state-styling branches / the diagnostic-halo branch. The layout pass writes `node.position`, every other field passes through unchanged.
- **`<GraphCanvasPane>` re-mount clears the cache**: the cache lives in a `useRef` scoped to one `<GraphCanvasPane>` instance, so route navigation that unmounts and remounts the component starts fresh. This is the right semantics for "the moderator left and came back" — a fresh arrangement is fine in that case.
- **Tests** (committed, per ADR 0022):

## Decisions

- **Layout library: `@dagrejs/dagre`.** Pre-decided by [ADR 0025](../../../docs/adr/0025-graph-layout-engine-dagre.md). Alternatives surveyed and rejected there: `elkjs` (20× the bundle weight; async API complicates the projection's pure-function story; layered algorithm gives the same result dagre does for the argument-graph use case); `d3-force` (force-directed produces non-hierarchical layouts and is non-deterministic, both wrong for the methodology's claim-at-top reading); manual positioning (a future escape valve, wrong as the default). The `@types/d3-force` reference in the existing lockfile is a transitive type stub, not a runtime library.
- **Direction: `rankdir: 'TB'`.** The methodology's primary reading order is "claim at the top, evidence underneath" (see `docs/methodology.md`). Other directions (`BT`, `LR`, `RL`) are accepted via the `LayoutOptions` parameter but `TB` is the default; per-debate orientation switching is deferred.
- **Stability strategy: position-cached.** Cached `(id → {x, y})` in a `useRef` inside `<GraphCanvasPane>`. Existing nodes never move on incremental events; only new nodes (and edges among them or to the cache) feed dagre. The alternative — relayout-from-scratch on every event — is rejected because it visually moves every node on every event, which is disorienting for the moderator. The third option (relayout-from-scratch only on user-triggered "tidy up", auto-layout-only-new-nodes otherwise — the strategy ADR 0025 settles on) is what we implement. The user-triggered button itself is deferred to `mod_layout_tidy_action`; the function `relayoutAll()` is exported in this task so the future button has something to call.
- **Re-layout trigger: event-driven auto.** Every `useMemo([events, diagnosticHighlights, edges])` tick re-runs the projection + layout. For already-cached nodes the cost is a Map lookup; only new nodes go through dagre. This keeps the canvas responsive without adding any user-facing affordance.
- **Module home: `apps/moderator/src/graph/layoutEngine.ts`.** Pure module; no React; no store coupling; no DOM access. Mirrors the established pattern of `facetStatus.ts` / `diagnosticHighlights.ts` (pure projections invoked from `<GraphCanvasPane>`'s `useMemo` blocks). The caller decides when to invoke. Locating the file under `apps/moderator/src/graph/` keeps every ReactFlow-touching module under one folder.
- **Constant per-node dimensions: 220×90 px.** Matches the typical rendered card. Worst-case overflow (many decoration rows) is the v1 trade-off documented under `mod_layout_measured_dimensions` (future task). Alternative — DOM-measured dimensions via `getBoundingClientRect` after the first render — is rejected for v1 because it adds a measure → re-layout cycle that has its own subtle bugs (layout thrash, observer ordering) and the constant-dimension path is simpler and good enough for the common case.
- **Edges added to the `nodes` `useMemo` deps.** Previously the memo keyed on `[events, diagnosticHighlights]` because `projectNodes` does not read edges. The layout pass DOES read edges (dagre needs them to compute layered placement), so the dep is widened. The cost is one more reference comparison per render; the benefit is correctness — without it, adding an edge between two existing nodes would not retrigger the layout and a new node connected by a new edge would be placed without considering that edge.
- **`projectNodes` placeholder grid deleted, not preserved.** The grid was always a placeholder pinned in `mod_node_rendering` with a forward reference to this task. Keeping it would be dead code. Tests asserting against the grid (one case in `GraphCanvasPane.test.tsx`) are rewritten to assert against `applyLayout` output.
- **`useRef` for the cache, not `useState`.** The cache is a side-effect store, not a render-driving piece of state. Writing to it should NOT cause a re-render (the `useMemo`'s `events` / `edges` deps drive the re-render; the cache write is a downstream effect). `useRef` gives exactly that semantics. `useState` would re-render on every write, defeating the purpose.
- **Cache cleared on component re-mount.** The cache lives in a `useRef` scoped to one component instance, so route navigation that unmounts and remounts `<GraphCanvasPane>` starts fresh. This is the right semantics for "the moderator left and came back" — the cache is not persisted to localStorage, not serialized to the store, not coordinated with the server.
- **No new ADR for this task.** [ADR 0025](../../../docs/adr/0025-graph-layout-engine-dagre.md) (Accepted 2026-05-15) already pinned every architectural choice: library, version, orientation, stability strategy, dimensions, module home. This refinement is the task-scope pin for the implementation; the ADR is the architectural pin for the choice. The Refinement-Writer brief says "Adding a new graph-rendering dependency is ADR-worthy" — the ADR was already written; this refinement references it.
- **Playwright spec scoped under acceptance criteria, NOT deferred.** Per the UI-stream e2e policy (ORCHESTRATOR.md commit `28a71f9`), every UI-stream task must scope a Playwright spec. Layout behavior is observable in-browser (node positions, no overlap, edges visible) and is exactly the kind of test Playwright is for. The spec follows the established `tests/e2e/moderator-hover-details.spec.ts` shape: `loginAs` → POST /sessions → goto operate → `seedWsStore` → assertions.
- **Stability tolerance: 2 px.** Sub-pixel rounding in CSS transforms + ReactFlow's internal viewport math can produce ±1 px drift across renders for the same logical position. The 2 px tolerance absorbs that noise while still catching a real "all nodes moved" regression (which would shift by tens of px or more).
- **Test idempotency on cycles.** Dagre breaks cycles by reversing the offending edge for the layering pass. We include a unit test for the `A → B`, `B → A` cycle case to pin that the engine doesn't throw and produces finite positions — the cycle case is rare in real debate graphs but the `methodology.md` text explicitly allows it.
- **No new i18n keys.** Layout produces positions, not text. The new spec asserts position semantics, not catalog content. No new keys land in `packages/i18n-catalogs/`.
