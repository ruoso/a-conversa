# Commit logic — moderator commits when all current participants vote agree

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.commit_logic`
**Effort estimate**: 1d
**Inherited dependencies**: depends on `methodology_engine.agreement_state_machine` (the framework — `MethodologyAction`, `validateAction`, `RejectionReason`, the primitives, the `registerActionHandler` registry, and the placeholder commit handler). Concretely settled: the structural validator (`event_validation`, ADR 0021), the in-memory `Projection` with its `pendingProposals` / `committedProposals` / `unresolvedMetaDisagreements` maps, `deriveFacetStatus`, and `replay.ts`'s `handleCommit` (which applies the structural commit effect on the read side after this validator runs on the write side).

## What this task is

Replace the placeholder `commit` handler that ships with `agreement_state_machine` with the real methodology-engine validator. A `commit` action is legal only when **(a)** the requester is the session's moderator, **(b)** the proposal exists in `pending` state, and **(c)** every current participant (`left_at IS NULL`) has voted `agree` on that proposal. On `Valid` the handler emits a single `EventToAppend` of kind `commit` whose payload is the matching `CommitPayload` from `@a-conversa/shared-types` (`proposal_id`, `moderator`, `committed_at`). On any rule failure the handler returns a typed `Rejected` with a `RejectionReason` already minted by the prior task's union.

## Why it needs to be done

`docs/methodology.md` (lines 15–25) is unambiguous: only the moderator commits; the commit is the structural-not-interpretive enactment of agreement that every participant has expressed. The placeholder handler from `agreement_state_machine` returns `Valid` for *any* commit action that passes the universal checks (session match / sequence match / participant gate) — including commits from debaters, commits of nonexistent proposals, commits of proposals nobody has voted on, and commits of proposals that are already committed or in meta-disagreement. Without this task, the API layer would accept all those illegal commits. The downstream `withdrawal_logic` task also depends on this: `withdrawal_logic` needs to assume `commit_logic` has gated which proposals can transition out of `pending`, so the write-side state machine is well-defined when withdrawals start arriving.

The boundary is precise: this task validates the *intent* to commit on the **write** side (does the request pass methodology rules?). The projection's `replay.ts/handleCommit` applies the *structural effect* on the **read** side (set facet value, mark facet `agreed`, move the proposal from `pendingProposals` to `committedProposals`, set `committedProposalEventId` + `committedAt` on the facet so `deriveFacetStatus` can identify it as committed). Both must run in order: `validateAction` returns `Valid`, the API layer appends the event, then `applyEventIncremental` calls `replay.ts/handleCommit`. The two layers are designed to stay in sync — rule 4 of this task (unanimity over current participants) is the same predicate that `deriveFacetStatus` evaluates on the read side (rule 6 of `deriveFacetStatus`: "every current participant has voted agree → agreed"). They should agree by construction.

## Constraints / requirements

- The handler **does not write events**; it returns a `ValidationResult`. On `Valid` it emits exactly one `EventToAppend` of kind `commit`.
- The handler **does not mint timestamps or ids**; the API layer mints `eventId`, `createdAt`, and `committedAt` before calling the engine.
- The handler **does not call `validateEvent`**; the API layer runs the structural validator separately (ADR 0021). Methodology validation is on top of structural validation.
- No new event payloads, `ProjectionChange` discriminators, or shared-types schemas. No new `RejectionReason` values — the prior task's union covers every case.
- Don't modify `apps/server/src/projection/*`. The projection's `handleCommit` and `deriveFacetStatus` are stable; this task validates *before* they run.
- Don't pre-empt the sibling tasks: `withdrawal_logic` (uses the committed-proposal record this validator creates), other handlers (`vote_logic`, `meta_disagreement_logic`, etc.).
- Verifications per ADR 0022: Vitest at `apps/server/src/methodology/handlers/commit.test.ts`; Cucumber + pglite scenarios at `tests/behavior/methodology/commit.feature` with step defs in `tests/behavior/steps/methodology-commit.steps.ts`.

## Acceptance criteria

- `apps/server/src/methodology/handlers/commit.ts` exports a default `Validator<CommitAction>` that:
  1. Calls `requireModerator(projection, action.requester)`. On failure returns the rejection (reason `'not-a-moderator'` for joined-but-wrong-role; `'not-a-participant'` for not-joined-at-all — the latter is technically unreachable because `validateAction` already ran the universal participant gate, but the helper's contract still distinguishes the two).
  2. Calls `findProposal(projection, action.proposalEventId)`. Branches:
     - `null` → reject `'proposal-not-found'`.
     - `state: 'committed'` → reject `'proposal-already-committed'`.
     - `state: 'meta-disagreement'` → reject `'proposal-already-meta-disagreement'`.
     - `state: 'pending'` → continue.
  3. For the four facet-targeting proposal sub-kinds (`classify-node`, `set-node-substance`, `set-edge-substance`, `edit-wording`), look up the affected facet's `perParticipant` map and assert every current participant has voted `'agree'`. For the other (structural / multi-entity) sub-kinds — `decompose`, `interpretive-split`, `axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate` — those sub-kinds' agreement state isn't tracked on a single facet's `perParticipant` map; their per-participant vote state is the methodology engine's downstream responsibility. For this task we apply rule 4 only to the four facet-targeting sub-kinds; for the structural sub-kinds the rule is *deferred* with an honest `'illegal-state-transition'` rejection carrying a `detail` that names the unsupported sub-kind. Sibling tasks (`decomposition_logic`, etc.) will tighten by registering their own per-sub-kind branches as they land. (See Decisions below — this is the only design call where commit_logic has to draw a boundary inside its rule set.)
  4. On rule 4 failure (not every current participant has voted agree) → reject `'unanimous-agree-required'` with a `detail` listing which participants still need to vote or have voted dispute / withdraw.
  5. On `Valid` emit one `EventToAppendEnvelope<'commit'>` whose payload is `{ proposal_id: action.proposalEventId, moderator: action.requester, committed_at: action.committedAt }`. Envelope fields are mirror-copied from the action.
- `apps/server/src/methodology/handlers/index.ts` (new barrel) and `apps/server/src/methodology/engine.ts` wire registration so the commit handler replaces the placeholder. The wiring uses module-init side effects analogous to `installDefaultHandlers` — the engine's own init still installs placeholders first, then the handlers module replaces commit specifically. To avoid double-registration ordering issues with siblings that follow, the existing `installDefaultHandlers` call is replaced with a single `installHandlers` that installs the real commit handler plus the remaining placeholders for the other three kinds.
- `apps/server/src/methodology/handlers/commit.test.ts` covers:
  - Reject when requester isn't the moderator (debater requests commit; reason `'not-a-moderator'`).
  - Reject when proposal doesn't exist (unknown id; reason `'proposal-not-found'`).
  - Reject when proposal is already committed (reason `'proposal-already-committed'`).
  - Reject when proposal is in meta-disagreement (reason `'proposal-already-meta-disagreement'`).
  - Reject when no participant has voted yet (reason `'unanimous-agree-required'`).
  - Reject when one participant has voted `agree` but another hasn't voted (reason `'unanimous-agree-required'`).
  - Reject when one participant has voted `dispute` (reason `'unanimous-agree-required'`).
  - Reject when one participant has voted `withdraw` without a prior commit (reason `'unanimous-agree-required'`).
  - Accept when every current participant has voted `agree` — assert `Valid.events` is one `commit` event with `payload.proposal_id`, `payload.moderator`, `payload.committed_at` correctly set; envelope `id` / `sequence` / `actor` / `createdAt` / `sessionId` mirror the action.
  - Accept when a participant left after agreeing — the remaining current participants are unanimous; the left participant's `agree` vote does not count and is not required. Per the chosen semantics (see Decisions).
  - Reject when the proposal sub-kind is structural (`decompose`, etc.) with the deferred-handling rejection (reason `'illegal-state-transition'`, detail names the sub-kind).
- `tests/behavior/methodology/commit.feature` covers 2–3 DB-driven scenarios:
  1. Successful commit — three participants joined, classify-node proposal pending, all three voted agree (votes inserted as events), moderator commits → `Valid` with one commit event.
  2. Reject by non-moderator — same setup, but a debater requests the commit → `Rejected` with reason `'not-a-moderator'`.
  3. Reject when not all current participants have agreed — same setup but only two votes recorded; moderator requests commit → `Rejected` with reason `'unanimous-agree-required'`.

## Decisions

- **Where the handler lives.** A new `apps/server/src/methodology/handlers/` directory mirroring the per-action-module shape the prior refinement suggested. `commit.ts` exports a default `Validator<CommitAction>`. Wired into the engine via a barrel (`handlers/index.ts`) imported at engine module-init.
- **Factor handlers now vs. wait.** **Factored now.** The prior task left this open ("Pick the cleaner of 'factor now' vs 'factor when a sibling forces it.'"). Reasons to factor now: (1) commit_logic is the first sibling and the handler module is non-trivial — branching on found-state, walking the facet, formatting unanimity-required details. Inlining all four placeholders plus this real handler in `engine.ts` would push the file past 300 lines and bury the dispatcher. (2) Future siblings (`withdrawal_logic`, `vote_logic`, `meta_disagreement_logic`, etc.) get a clear template — one file per action handler, registered from a barrel. (3) The factoring is mechanical: move the three remaining placeholders to `handlers/{propose,vote,markMetaDisagreement}.ts` so the engine's `installHandlers` step is just four imports + four `registerActionHandler` calls. The placeholders stay placeholders; they're not in this task's scope.
- **Rule set (numbered, in evaluation order).**
  1. **Moderator gate.** `requireModerator(projection, action.requester)` must succeed. → `'not-a-moderator'` (or `'not-a-participant'` if not joined; unreachable in practice because the universal gate already filtered).
  2. **Proposal exists.** `findProposal(projection, action.proposalEventId)` must return non-null. → `'proposal-not-found'`.
  3. **Proposal is pending.** The found state must be `'pending'`, not `'committed'` or `'meta-disagreement'`. → `'proposal-already-committed'` / `'proposal-already-meta-disagreement'`.
  4. **Unanimous agree across current participants.** For the four facet-targeting sub-kinds, the affected facet's `perParticipant` map must contain a record for every current participant and every record's `vote` must be `'agree'`. For the structural sub-kinds the rule is deferred (see below). → `'unanimous-agree-required'` (or `'illegal-state-transition'` for the structural sub-kinds).
- **Rule 4 implementation — direct walk vs. `deriveFacetStatus`.** Implementation walks the `perParticipant` map directly rather than calling `deriveFacetStatus`. Three reasons: (a) we need the **detail string** to enumerate the specific participants who still need to vote, which `deriveFacetStatus` doesn't surface; (b) the direct walk is one Map lookup + a Set comparison — same big-O as `deriveFacetStatus` rule 6 anyway; (c) calling `deriveFacetStatus` would couple the write-side validator to a read-side derivation that may grow new statuses (e.g. the existing `'withdrawn'` status which doesn't apply pre-commit). The two layers compute the same predicate independently and must stay in sync; that's enforced by a Vitest assertion that on the accept path `deriveFacetStatus` would return `'agreed'` (read-side cross-check).
- **Structural-sub-kind boundary.** `commit_logic` doesn't know how to validate unanimity for `decompose` / `interpretive-split` / `axiom-mark` / `meta-move` / `break-edge` / `amend-node` / `annotate` because those sub-kinds don't have per-participant vote state on the projection — `handleVote` only writes to `perParticipant` for the four facet-targeting sub-kinds. The methodology engine's per-sub-kind sibling tasks (`decomposition_logic`, `axiom_mark_logic`, etc.) will tighten this when they land: each can decide whether to record per-participant votes on a different projection structure, or whether to short-circuit because the sub-kind is structural-only. For now, commit_logic returns `'illegal-state-transition'` with a `detail` like `"commit of proposal sub-kind 'decompose' is deferred to decomposition_logic"`. The committee of sub-kind handlers will replace this branch via `registerSubKindHandler` later if the design calls for it; for v1 the rejection is honest.
- **Participant-leaves-after-voting semantics.** **A participant who left no longer counts toward rule 4 — neither in the unanimity requirement nor in the agreement tally.** Concretely: rule 4 walks `currentParticipants(projection)` (which excludes left participants by construction) and asserts each has a `perParticipant` record with vote `'agree'`. If a participant agreed and then left, the remaining current set may be unanimous and the commit succeeds. Rationale: (a) `deriveFacetStatus` already filters `perParticipant` by `currentIds` (line 102 of `facet-status.ts`); the write-side validator stays consistent with the read-side derivation. (b) `docs/methodology.md` says "every current participant must agree before any change lands" — a participant who has left the session is no longer current, so they neither contribute to the agreement nor block it. (c) The methodology rule is "we agree among the people in the room." If someone leaves the room, they didn't agree to whatever happens next, but they also can't filibuster from outside. The left participant's prior `agree` *does* sit in the `perParticipant` map (left votes are historical state per `handleParticipantLeft`); we just don't read it when checking unanimity. The change history records who was present at commit time, so post-hoc auditing can ask "did the participant who left agree to this?" — but the methodology doesn't gate the commit on that.
- **No new `RejectionReason` values.** The prior task's union already includes every reason this handler needs: `'not-a-moderator'`, `'proposal-not-found'`, `'proposal-already-committed'`, `'proposal-already-meta-disagreement'`, `'unanimous-agree-required'`, `'illegal-state-transition'`.
- **`EventToAppend` shape mirrors the placeholder.** Same payload, same envelope fields. The only thing this handler adds is the four-rule write-side gate before emitting.
- **No fan-out of `vote` events.** A commit doesn't synthesize the participants' votes; those events already exist in `session_events` (the API layer appended them as the participants voted). The commit event references the proposal id; the projection's `handleCommit` resolves the affected facet from the pending-proposal record.
- **Test layout.** Vitest at `apps/server/src/methodology/handlers/commit.test.ts` so the file sits next to the handler. The existing `apps/server/src/methodology/engine.test.ts` keeps covering the framework-level behavior; the new test file owns the commit-specific rule set.
- **`installHandlers` shape after factoring.** `engine.ts` keeps `installHandlers()` as the module-init step. After this task, `installHandlers` imports the real `commitHandler` from `./handlers/commit.js` and the three placeholder handlers from `./handlers/{propose,vote,markMetaDisagreement}.js`, then calls `registerActionHandler` four times. The handler files don't self-register — they export the validator and the central `installHandlers` does the registration. This sidesteps circular-import hazards: `engine.ts` imports the handlers; handlers import only `types.ts` and `primitives.ts`. The barrel `handlers/index.ts` re-exports each handler for downstream consumers (tests, future API integration).
