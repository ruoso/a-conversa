# Reword vs. restructure — propose-side validator for the `edit-wording` proposal sub-kind

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.reword_vs_restructure`
**Effort estimate**: 2d
**Inherited dependencies**: depends on `methodology_engine.agreement_state_machine` (the framework — `MethodologyAction`, `validateAction`, `RejectionReason`, `requireParticipant`, etc.). Indirectly: `methodology_engine.decomposition_logic`, `interpretive_split_logic`, `axiom_mark_logic`, and `meta_move_logic` (the prior four propose-arm tightenings — their factoring (`validateXProposal(projection, action)` locals dispatched from `propose.ts`'s switch), their `nodeIsVisible` reuse, and their structural-shape-is-upstream's-concern layering decision are the templates this task extends). Concretely settled: the Zod `editWordingProposalSchema` (nested `z.discriminatedUnion('edit_kind', [rewordEditProposalSchema, restructureEditProposalSchema])`) in `packages/shared-types/src/events/proposals.ts` enforces structural shape — `kind: 'edit-wording'` outer + `edit_kind: 'reword' | 'restructure'` inner — per ADR 0021; the projection's read-side `replay.ts/applyCommittedProposal` `edit-wording` arm (currently flips `old.visible = false` on restructure but does not add the new node — same Open Question shape as decompose / interpretive-split / meta-move / axiom-mark).

## What this task is

Tighten the `propose` handler's `edit-wording` arm with the real methodology-engine validator. The propose handler currently dispatches `decompose`, `interpretive-split`, `axiom-mark`, and `meta-move` to their real validators and lets the other seven sub-kinds — including `edit-wording` — fall through to the universal-pass placeholder path. This task replaces the fall-through for `edit-wording` with a sub-switch on `edit_kind`. The two inner kinds share three propose-side rules and `restructure` adds a fourth (the `new_node_id` collision check):

For **both** `reword` and `restructure`:

1. `node_id` references a node that exists in this session's projection.
2. The node is currently visible in the projection (a not-visible node — already superseded by a prior decompose / interpretive-split / restructure per the visible-graph derivation in `docs/data-model.md` lines 273–285 — can't be re-edited; the edit would render on a node nobody can see).
3. No conflicting `edit-wording` / `decompose` / `interpretive-split` proposal is currently pending against the same node. The three structural sub-kinds are **mutually exclusive on the same parent**: each flips `parent.visible = false` on commit (`decompose` and `interpretive-split` directly; `edit-wording` with `edit_kind: restructure` via supersession; `reword` updates in place but still owns the wording facet and races with the others). Only one of the three may be pending against a given node at a time.

For **restructure only**:

4. `new_node_id` does NOT collide with any existing node id in the projection. A restructure creates a new node carrying `new_wording`; the id is minted by the proposer and threaded into a paired `node-created` event at commit time. If `new_node_id` already names a node in the projection, the propose is a structural conflict — committing it would overwrite an existing entity.

The structural payload shape — `kind: 'edit-wording'`, `edit_kind` enum, `node_id: UUID`, `new_wording: string.min(1)`, and (for restructure) `new_node_id: UUID` — is already enforced upstream by the Zod schema (`editWordingProposalSchema`'s nested discriminated union, ADR 0021). The validator relies on that layering and does not re-check.

On `Valid` the handler emits exactly one `EventToAppend` of kind `proposal` whose payload is `{ proposal: action.proposal }`. On any rule failure the handler returns a typed `Rejected` with a `RejectionReason` that names the specific failure.

Scope is **propose-side only**. The commit-time multi-event emission for restructure — the paired `node-created` for `new_node_id` so the replacement node is materialized when the commit lands — is **not** in scope and is the same Open Question shape as decompose / interpretive-split / meta-move / axiom-mark. See Open Questions.

## Why it needs to be done

`docs/methodology.md` lines 157–166 ("Editing wording: reword vs. restructure") and `docs/data-model.md` lines 247–249 + 281–285 + 302 together settle the two inner edit-wording kinds and their distinct projection semantics:

- **Reword** (C3): "the edit produces a clearer phrasing of the same statement. The node ID is preserved; the wording facet's value changes (the prior wording is recorded in the change history). All edges into or out of the node remain attached."
- **Restructure** (C4): "the edit produces a meaningfully different statement, not the same node in better words. A new node is created (with its own facets in `proposed`) and the original node is superseded — removed from the visible graph. Edges incident to the original become invisible by virtue of the missing endpoint and don't auto-follow to the new node; if participants want analogous edges on the replacement, they propose them explicitly."

The choice between reword and restructure is itself the proposal's `edit_kind` discriminator — participants negotiate it at the proposal layer (the moderator can vote `dispute` if they think a "reword" is really a restructure, etc.). This task validates the propose-side rules that bound the operation regardless of which inner kind the proposer chose.

The placeholder propose handler currently accepts edit-wording proposals against:
- non-existent nodes (the projection would later throw `ReplayError` at commit time — see `replay.ts` lines 572–575 for reword, lines 601–605 for restructure);
- already-superseded nodes (the wording edit would attach to a node nobody can see);
- nodes that already have another pending decompose / interpretive-split / edit-wording (the structural race ends in one operation overwriting the other's effect at commit);
- and most importantly: for restructure, it accepts proposals where `new_node_id` collides with an existing node — committing such a proposal would silently overwrite the colliding entity.

Without this task the API layer would write all those illegal proposals as if they were live; downstream consumers would surface confused state.

The conflict-walker extension is the new design constraint that emerges with this task. `decomposition_logic` introduced the conflict-walker against the `{decompose}` set; `interpretive_split_logic` expanded it symmetrically to `{decompose, interpretive-split}`. This task expands it once more to `{decompose, interpretive-split, edit-wording}`. The argument is the same in all three cases: each sub-kind flips `parent.visible = false` on commit (directly or via supersession). Even reword — which doesn't flip visibility — owns the same wording facet and would race with a pending restructure on the same node; treating both inner kinds uniformly under the conflict-walker is the simpler and safer design.

## Constraints / requirements

- The handler **does not write events**; it returns a `ValidationResult`. On `Valid` it emits exactly one `EventToAppend` of kind `proposal`.
- The handler **does not mint timestamps or ids**; the API layer mints `eventId` and `createdAt` before calling the engine. `new_node_id` for restructure is part of the proposal payload — the proposer mints it; the propose-side validator checks it doesn't collide with an existing node id.
- The handler **does not call `validateEvent`**; the API layer runs the structural validator (Zod) separately (ADR 0021). Methodology validation is on top of structural validation. The handler can rely on `node_id` being a valid UUID, `new_wording` being non-empty, `edit_kind` being one of `'reword' | 'restructure'`, and `new_node_id` (when present) being a valid UUID.
- **No new `RejectionReason` value.** Rule 1 reuses `'target-entity-not-found'`; rules 2, 3, and 4 reuse `'illegal-state-transition'` (matching the decompose / interpretive-split / axiom-mark / meta-move arms). The `detail` string carries the per-rule specificity.
- Extend `CONFLICTING_PARENT_KINDS` in `propose.ts` to include `'edit-wording'`. The `ConflictingParentKind` union in `primitives.ts` widens additively (to `'decompose' | 'interpretive-split' | 'edit-wording'`).
- Extend `findConflictingProposalAgainst` to read the right field per conflicting sub-kind: `decompose` / `interpretive-split` read `payload.parent_node_id`; `edit-wording` reads `payload.node_id`. The walker takes the target-node id (whatever the caller is asking about) and compares against whichever field the candidate proposal carries. This is a single localized change in `findConflictingProposalAgainst`.
- Update the decompose + interpretive-split arms' Vitest tests to add a symmetric case each: reject when an `edit-wording` proposal is pending against the same parent. The arms themselves require no edit — they already invoke the walker with `CONFLICTING_PARENT_KINDS`; once the set widens, the cross-kind rejection works automatically. The test addition is the regression pin.
- Don't pre-empt the other proposal sub-kinds. `edit-wording` is the fifth arm tightened; the remaining six (`classify-node`, `set-node-substance`, `set-edge-substance`, `break-edge`, `amend-node`, `annotate`) stay on the placeholder path until their own sibling tasks land.
- Don't pre-empt commit-time multi-event emission for restructure. The current `applyCommittedProposal` restructure branch only flips `oldNode.visible = false` and asserts the replacement node already exists; emitting the paired `node-created` is the Open Question shape that decompose / interpretive-split / meta-move / axiom-mark all also flagged.
- Verifications per ADR 0022: Vitest at `apps/server/src/methodology/handlers/proposeEditWording.test.ts`; Cucumber + pglite scenarios at `tests/behavior/methodology/propose-edit-wording.feature` with step defs in `tests/behavior/steps/methodology-propose-edit-wording.steps.ts`.

## Acceptance criteria

- `apps/server/src/methodology/handlers/propose.ts` gains an `edit-wording` arm in its sub-kind switch, dispatching to a new local function `validateEditWordingProposal(projection, action)`. The function mirrors the sibling validators in shape, applies the three shared rules in evaluation order, then sub-switches on `edit_kind` for the restructure-only rule 4, and returns a `RejectedValidationResult | null`.
- `apps/server/src/methodology/handlers/propose.ts` extends `CONFLICTING_PARENT_KINDS` to `{'decompose', 'interpretive-split', 'edit-wording'}`. The decompose + interpretive-split arms pick up the cross-kind rejection automatically through the shared walker.
- `apps/server/src/methodology/primitives.ts` widens `ConflictingParentKind` to `'decompose' | 'interpretive-split' | 'edit-wording'` and extends `findConflictingProposalAgainst` to read `payload.node_id` when the candidate's kind is `edit-wording` (versus `payload.parent_node_id` for decompose / interpretive-split). The walker returns the first matching pending proposal.
- `validateEditWordingProposal` enforces in evaluation order:
  1. **Node-exists.** `projection.getNode(node_id)` non-undefined → else `'target-entity-not-found'`, detail naming the missing id and the session.
  2. **Node-visible.** `nodeIsVisible(projection, node_id) === true` → else `'illegal-state-transition'`, detail including the phrase "not currently visible".
  3. **No conflicting edit-wording / decompose / interpretive-split pending.** → else `'illegal-state-transition'`, detail naming the conflicting proposal's id, kind, and the node id.
  4. (restructure only) **`new_node_id` does not collide.** `projection.getNode(new_node_id) === undefined` → else `'illegal-state-transition'`, detail naming the colliding id and the phrase "already names an existing node".
- On `Valid` emit one `EventToAppendEnvelope<'proposal'>` whose payload is `{ proposal: action.proposal }`. Envelope fields mirror-copied from the action — same shape decompose's / interpretive-split's / axiom-mark's / meta-move's accept path uses.

  - **Reword**: reject when node doesn't exist; reject when node is not visible; reject when a decompose against the same node is pending; reject when an interpretive-split against the same node is pending; reject when another edit-wording (reword OR restructure) against the same node is pending; accept a well-formed reword.

## Decisions

- **Scope: propose-side only.** This task validates `propose edit-wording`. Commit-time multi-event emission for restructure (the paired `node-created` for `new_node_id` so the replacement is materialized at commit) is **not** in scope — see Open Questions.
- **Rule set (numbered, in evaluation order).**
  1. **Node-exists.** → `'target-entity-not-found'`.
  2. **Node-visible.** → `'illegal-state-transition'`.
  3. **No conflicting edit-wording / decompose / interpretive-split pending against the same node.** → `'illegal-state-transition'`.
  4. (restructure only) **`new_node_id` does not collide with an existing node id.** → `'illegal-state-transition'`.
- **No new RejectionReason value.** Rules 1–4 reuse `'target-entity-not-found'` and `'illegal-state-transition'` exactly as the decompose / interpretive-split / axiom-mark / meta-move arms use them. The `detail` string carries the kind-specific specificity (which node, which conflicting proposal, which colliding new id, which session).
- **Sub-switch on `edit_kind` inside `validateEditWordingProposal`.** The three shared rules run first; the function returns early on rejection. The `new_node_id` collision check runs only when `edit_kind === 'restructure'`. Alternative considered: two top-level functions (`validateRewordProposal`, `validateRestructureProposal`) dispatched from the propose-handler's switch via the inner discriminator. *Rejected.* The outer `kind: 'edit-wording'` is the dispatch axis the methodology engine cares about; the inner `edit_kind` is a wording-edit-internal distinction. A single function with a sub-switch keeps the symmetry visible and matches the shape Zod uses (one outer kind + nested discriminated union).
- **`CONFLICTING_PARENT_KINDS` extension to include `'edit-wording'`.** Treating reword and restructure uniformly under the conflict-walker is the simpler design. Restructure flips `oldNode.visible = false` on commit (same shape as decompose / interpretive-split); reword updates the wording facet in place but still owns that facet exclusively while it's pending. Even a pending reword should block a new decompose on the same node (the decompose would commit second and supersede the just-rewored node, wasting the moderator's commit-time agreement). The walker rejects symmetrically across all three sub-kinds.
- **`findConflictingProposalAgainst` extension to read per-kind fields.** Decompose's and interpretive-split's payloads carry `parent_node_id`; edit-wording's carries `node_id`. The walker switches on the candidate's `payload.kind` and reads the right field — the caller passes a target-node id (whatever the caller is calling that node) and the walker normalizes the lookup. The function signature is unchanged.
- **`new_node_id` collision check (restructure only).** Rule 4 checks `projection.getNode(new_node_id) === undefined`. The Zod schema enforces `new_node_id` is a valid UUID; the methodology validator enforces it doesn't already name a node. Alternative considered: defer the check to commit-time, when the paired `node-created` would surface the collision. *Rejected.* The collision is a propose-side configuration error — the proposer minted the wrong id. Catching it at propose-time gives the proposer immediate feedback rather than letting a doomed proposal accumulate votes.
- **Cross-kind rejection symmetry pins in decompose + interpretive-split tests.** The walker change is one line; the regression pin is two test cases (one in each sibling test file). Same pattern interpretive_split_logic.md used when extending the walker from `{decompose}` to `{decompose, interpretive-split}` — it added a symmetric case to `proposeDecompose.test.ts`. This task does the same for the `edit-wording` extension.
- **Where the handler lives.** `propose.ts` itself, factored locally as `validateEditWordingProposal(projection, action)`. Mirrors the prior four sub-kinds' factoring choice. If `propose.ts` grows past ~600 lines as more sub-kinds tighten, factor into per-sub-kind files.
- **Structural shape is upstream's concern.** The Zod schema (`editWordingProposalSchema` — nested `z.discriminatedUnion('edit_kind', ...)`) enforces all structural constraints. Per ADR 0021 the validator runs at the API ingress before the methodology engine sees the action. The methodology validator relies on this layering and does not re-check.
- **Test layout.** Vitest at `apps/server/src/methodology/handlers/proposeEditWording.test.ts` (new file — same naming pattern as the sibling `proposeXxx.test.ts` files). Cucumber + pglite at `tests/behavior/methodology/propose-edit-wording.feature` with step defs in `tests/behavior/steps/methodology-propose-edit-wording.steps.ts`. Reuses `event-rows.ts` and the shared Then steps. The step file uses a distinct UUID prefix (`c3...`) to avoid collisions with the prior propose-XXX step files.

## Open questions

- **Commit-time multi-event emission for restructure.** `replay.ts`'s `applyCommittedProposal` edit-wording / restructure arm (lines 595–623) flips `oldNode.visible = false` and asserts the replacement node already exists via `projection.getNode(new_node_id)`. There is currently no path that emits the paired `node-created` for `new_node_id` ahead of the commit — the propose-side does not emit it; `commit_logic`'s rule 4 rejects structural-sub-kind commits with `'illegal-state-transition'`; and no separate methodology-engine entry point exists to issue the paired event. Resolving the gap is the same Open Question shape that decompose / interpretive-split / meta-move / axiom-mark all also flagged — cross-reference those refinements' Open Questions. Settling will likely fold into a unified follow-up alongside `commit_logic`'s structural-sub-kind support (probably a `commit_logic` extension that, on commit of a structural sub-kind, emits the entity-creation events in addition to the commit event). The current task does not settle this; the integration tests cover propose-side only.
