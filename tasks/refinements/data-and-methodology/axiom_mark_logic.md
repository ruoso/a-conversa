# Axiom-mark logic — propose-side validator for the `axiom-mark` proposal sub-kind

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.axiom_mark_logic`
**Effort estimate**: 1d
**Inherited dependencies**: depends on `methodology_engine.agreement_state_machine` (the framework — `MethodologyAction`, `validateAction`, `RejectionReason`, `requireParticipant`, `findProposal`, etc.). Indirectly: `methodology_engine.decomposition_logic` and `methodology_engine.interpretive_split_logic` (the prior two propose-arm tightenings — their factoring (`validateXProposal(projection, action)` locals dispatched from `propose.ts`'s switch), their `nodeIsVisible` reuse, and their structural-shape-is-upstream's-concern layering decision are the templates this task extends). Concretely settled: the Zod `axiomMarkProposalSchema` in `packages/shared-types/src/events/proposals.ts` (enforces structural shape — `{ kind: 'axiom-mark', node_id: UUID, participant: UUID }`); the projection's read-side `replay.ts/applyCommittedProposal` `axiom-mark` arm which records the (node, participant) pair on `node.axiomMarks` when an axiom-mark proposal commits.

## What this task is

Tighten the `propose` handler's `axiom-mark` arm with the real methodology-engine validator. The propose handler currently dispatches `decompose` and `interpretive-split` to their real validators and lets the other nine sub-kinds — including `axiom-mark` — fall through to the universal-pass placeholder path. This task replaces the fall-through for `axiom-mark` with four methodology-specific rules, in evaluation order:

1. The `node_id` references a node that exists in this session's projection.
2. The node is currently visible in the projection (a not-visible node — already superseded by decompose / interpretive-split / restructure per the visible-graph derivation in `docs/data-model.md` lines 273–285 — can't be axiom-marked: the axiom-mark would render on a node nobody can see).
3. The proposal's `participant` field equals the action's `requester`. Axiom-marks are **per-participant** and **personal**: the speaker declares their own bedrock. You cannot propose an axiom-mark on behalf of someone else. The other participants vote on whether to record your axiom-mark afterwards (they may dispute if they think you're being unreasonable), but the proposal itself must come from the participant whose bedrock is being recorded.
4. The participant has not already had an axiom-mark for this node committed. A second axiom-mark from the same participant on the same node would be a no-op duplicate — the per-(node, participant) entry already exists in `projection.getNode(nodeId).axiomMarks`.
5. The structural payload shape — `kind: 'axiom-mark'`, `node_id: UUID`, `participant: UUID` — is already enforced upstream by the Zod schema (`axiomMarkProposalSchema`, ADR 0021). The validator relies on that layering and does not re-check.

On `Valid` the handler emits exactly one `EventToAppend` of kind `proposal` whose payload is `{ proposal: action.proposal }`. On any rule failure the handler returns a typed `Rejected` with a `RejectionReason` that names the specific failure.

**No conflict with decompose / interpretive-split.** An axiom-mark is **per-participant**, not structural. It does not flip `node.visible = false` on commit; it adds an entry to `node.axiomMarks[participant]`. A pending decompose or interpretive-split against the same node does NOT conflict with an axiom-mark proposal against that node — both can coexist and resolve independently. The `findConflictingProposalAgainst` walker is **not** extended; the `CONFLICTING_PARENT_KINDS` set stays at `{'decompose', 'interpretive-split'}`.

Scope is **propose-side only**. The commit-time structural fan-out — recording the (node, participant) entry in `projection.axiomMarks` with the proposal-event id and timestamp — already exists in `replay.ts/applyCommittedProposal`'s `axiom-mark` arm (lines 667–686) and is not in scope. Per the existing TODO comment there, the commit handler currently writes `proposalEventId: ''` and `markedAt: ''` placeholders into the record because those values aren't carried in the proposal payload — that's a separate refinement to land alongside `commit_logic`'s axiom-mark sub-kind support, not this task. The integration tests in this task cover only the propose-side path and assert nothing about commit.

## Why it needs to be done

`docs/methodology.md` lines 192–200 ("Axioms / terminal values") and `docs/data-model.md` lines 36–38 ("Axiom marks") together settle the axiom-mark as the methodology's primary tool for surfacing per-participant bedrock: "Ben's axiom mark on N9 records Ben's bedrock; Anna may add her own axiom mark to the same node, or hold a different node as her axiom, or hold no axiom at all in this debate." Surfacing axioms is "a primary success state" of the format — the debate dead-ends at "A holds X as bedrock, B holds Y as bedrock, and that is the real disagreement."

The placeholder propose handler currently accepts axiom-mark proposals against:
- non-existent nodes (the projection would later throw `ReplayError` at commit time);
- already-superseded nodes (the axiom-mark would render on an invisible node — a UX defect);
- nodes the participant has already axiom-marked (silent no-op duplicate);
- and most importantly: it accepts axiom-marks where one participant proposes a mark on behalf of another, violating the personal-bedrock invariant.

Without this task the API layer would write all those illegal proposals as if they were live; downstream consumers would surface confused state.

The participant-equals-requester rule is the new design constraint that emerges with this task. Decomposition and interpretive-split are *structural* operations: anyone may propose them because they target the graph itself, and unanimity (per the methodology's universal agreement rule) governs whether the change lands. Axiom-marks are *personal*: anyone may dispute another participant's axiom-mark (per `docs/methodology.md` line 196 — "agreement on an axiom mark is roughly: 'we all agree that this participant holds this node as bedrock for this debate' — not 'we all agree the node is true'"), but the *proposal* must originate with the participant declaring the bedrock. Surfacing someone else's bedrock on their behalf would be a category error: bedrock is what *this person* refuses to retract from, and only they can declare it.

## Constraints / requirements

- The handler **does not write events**; it returns a `ValidationResult`. On `Valid` it emits exactly one `EventToAppend` of kind `proposal`.
- The handler **does not mint timestamps or ids**; the API layer mints `eventId` and `createdAt` before calling the engine.
- The handler **does not call `validateEvent`**; the API layer runs the structural validator (Zod) separately (ADR 0021). Methodology validation is on top of structural validation. The handler can rely on `node_id` and `participant` being valid UUIDs.
- One new `RejectionReason` value is added — `'axiom-mark-not-self'` for rule 3. The closest existing reason, `'self-vote-not-allowed'`, has the *opposite* semantic shape ("you cannot vote on something that is yours") and is misleading here ("you can only mark your *own* axiom"). A new typed code is the cleanest pin. Rule 4 reuses `'illegal-state-transition'` (the participant's state on this node already includes the axiom-mark; the proposed transition is illegal). Rules 1 and 2 reuse the existing `'target-entity-not-found'` and `'illegal-state-transition'` exactly as the decompose / interpretive-split arms use them.
- Don't modify the projection layer beyond adding a thin getter primitive `hasAxiomMark(projection, nodeId, participantId)` in `primitives.ts`. The projection's `axiomMarks` map already exposes everything needed via `getNode()`.
- Don't extend the conflict-walker (`findConflictingProposalAgainst` / `CONFLICTING_PARENT_KINDS`). Axiom-mark is non-structural — it does not flip `parent.visible = false` and does not compete with decompose / interpretive-split. The mutual-exclusion set stays at `{'decompose', 'interpretive-split'}`.
- Don't pre-empt the other proposal sub-kinds. `axiom-mark` is the third arm tightened; the remaining eight (`classify-node`, `set-node-substance`, `set-edge-substance`, `edit-wording`, `meta-move`, `break-edge`, `amend-node`, `annotate`) stay on the placeholder path until their own sibling tasks land.
- Don't pre-empt commit-time semantics. The propose handler emits one `proposal` event; the commit handler's existing axiom-mark arm (placeholder `proposalEventId: ''`, `markedAt: ''` fields per the read-side TODO) is unchanged. A follow-up will tighten the commit-time path.
- Verifications per ADR 0022: Vitest at `apps/server/src/methodology/handlers/proposeAxiomMark.test.ts`; Cucumber + pglite scenarios at `tests/behavior/methodology/propose-axiom-mark.feature` with step defs in `tests/behavior/steps/methodology-propose-axiom-mark.steps.ts`.

## Decisions

- **Scope: propose-side only.** This task validates `propose axiom-mark`. Commit-time semantics (the `proposalEventId`/`markedAt` payload-threading currently stubbed in `replay.ts`'s axiom-mark arm) is **not** in scope — see Open Questions.
- **Rule set (numbered, in evaluation order).**
  1. **Node-exists.** → `'target-entity-not-found'`.
  2. **Node-visible.** → `'illegal-state-transition'`.
  3. **Participant-equals-requester.** → `'axiom-mark-not-self'` (new RejectionReason value).
  4. **No duplicate axiom-mark.** → `'illegal-state-transition'`.
  5. (Structural shape — enforced upstream by Zod.)
- **Rule 3 RejectionReason: `'axiom-mark-not-self'` (new), not `'self-vote-not-allowed'`.** Considered alternatives:
  - **Reuse `'self-vote-not-allowed'`.** *Rejected.* The existing code's literal name says "you cannot vote on something that is yours" (the original intent was to reserve it for a possible "the proposer can't vote on their own proposal" rule — see `withdrawal_logic.md` line 98 — which the vote handler later decided NOT to enforce). The axiom-mark rule is the OPPOSITE: "you can only mark your OWN axiom; you cannot mark someone else's." Reusing a same-letters-different-meaning code would mislead readers and create a false signal at the type level. The names should reflect the rules.
  - **Reuse `'illegal-state-transition'`.** *Rejected.* Functional — every rule failure is, in a sense, an illegal state transition — but the type-level loss of fidelity matters: a UI surfacing "your propose was rejected" needs to distinguish "you tried to mark someone else's axiom" from "the node is already invisible" from "you've already marked this node." Three different remediations.
  - **Add `'axiom-mark-not-self'`.** *Chosen.* Reads at the call site exactly as the rule reads in the docs. Pairs naturally with a UI message like "axiom-marks are personal — propose this from {participant}'s account instead." The union grows by one — additive change, no impact on existing code paths.
- **Rule 4 RejectionReason: `'illegal-state-transition'` (existing), not a new `'already-axiom-marked'`.** Considered alternatives:
  - **Add `'already-axiom-marked'`.** *Rejected.* A new value pays its way only when callers need to distinguish it from the umbrella. Here the remediation is the same as the not-visible case ("the node is in a state that blocks this action"): there's nothing else to do — the mark is already on record. The umbrella's `detail: string` carries the specific message. Mirrors the decompose handler's rule 2/3 grouping under `'illegal-state-transition'`.
  - **Reuse `'illegal-state-transition'`.** *Chosen.* Consistent with the sibling-arm grouping (decompose's rule 2 "parent not visible" and rule 3 "conflicting proposal pending" both use it).
- **No conflict with decompose / interpretive-split.** An axiom-mark is per-participant and does NOT flip `parent.visible = false` on commit. It can coexist with a pending decompose or interpretive-split against the same node. The `CONFLICTING_PARENT_KINDS` set stays at `{'decompose', 'interpretive-split'}`. Rationale: `docs/methodology.md` line 175 even notes that axiom-marking is one of the resolution paths for a cycle (alongside breaking a `supports` edge or decomposing a node in the cycle) — these are alternative resolution moves, not mutually exclusive proposals racing to flip the same state.
- **Per-participant uniqueness model.** The duplicate check is keyed on `(node, participant)`, not on `node` alone. Anna's axiom-mark on N9 does not block Ben from also marking N9 — `docs/data-model.md` line 38 calls this out explicitly: "Ben may hold N9 as an axiom while Anna does not, or both may hold the same node as axiomatic from their respective frames." The test suite covers the per-participant case with an explicit "second participant marks same node — accepted" scenario.
- **Duplicate-check source: `projection.axiomMarks` map (committed marks only), not pending proposals.** Considered alternatives:
  - **Check pending proposals AND committed marks.** *Rejected* for this task. A second propose for an already-pending axiom-mark by the same participant on the same node is harmless — the duplicate vote/commit cycle would no-op. The right place to short-circuit it is at the propose-side, but doing so requires walking pending proposals; the simpler "have you already committed this?" check matches the methodology's structural invariant ("each axiom-mark is one entry on record"). A future tightening can add the pending-proposal check; the current rule is necessary, not sufficient.
  - **Check only committed marks.** *Chosen.* The `axiomMarks` map is populated on commit (`replay.ts` line 676). A `hasAxiomMark` primitive thinly wraps the map lookup. The rule reads: "if this participant already has an axiom-mark on this node committed, a second proposal is illegal."
- **Self-vote question (reaffirmed).** `docs/methodology.md` line 196 ("agreement on an axiom mark is roughly: 'we all agree that this participant holds this node as bedrock for this debate'") confirms the vote round runs normally after the propose lands. The propose enforces `participant === requester` (only the bedrock-holder may declare it); the subsequent vote round lets the *other* participants dispute (per `docs/methodology.md` lines 198–200, axiom-marks aren't always uncontested — sometimes other participants think the marker is being unreasonable). This task settles the propose-side rule; vote-side semantics for axiom-marks remain on the placeholder path until a sibling task tightens them (the existing vote handler treats structural sub-kinds as no-prior-vote — `'agree'` / `'dispute'` accept, `'withdraw'` rejects with `'no-prior-agree'` — which is fine for axiom-marks too).
- **Where the handler lives.** `propose.ts` itself, factored locally as `validateAxiomMarkProposal(projection, action)`. Mirrors decomposition_logic's and interpretive_split_logic's choices — see their refinements for the factoring rationale. If `propose.ts` grows past ~400 lines as more sub-kinds tighten, factor into per-sub-kind files.
- **Structural shape is upstream's concern.** The Zod schema in `axiomMarkProposalSchema` enforces `node_id: z.string().uuid()` and `participant: z.string().uuid()`. Per ADR 0021 the validator runs at the API ingress before the methodology engine sees the action. The methodology validator relies on this layering and does not re-check.
- **Test layout.** Vitest at `apps/server/src/methodology/handlers/proposeAxiomMark.test.ts` (new file — same naming pattern as `proposeDecompose.test.ts` / `proposeInterpretiveSplit.test.ts`). Cucumber + pglite at `tests/behavior/methodology/propose-axiom-mark.feature` with step defs in `tests/behavior/steps/methodology-propose-axiom-mark.steps.ts`. Reuses `event-rows.ts` and the shared Then steps. The step file uses a distinct UUID prefix (`a1...` series) to avoid collisions with the propose-decompose (`e0...`) and propose-interpretive-split (`f0...`) step files.
