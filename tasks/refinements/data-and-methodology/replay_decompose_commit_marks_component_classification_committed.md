## What this task is

Extend the projector's commit handler so that committing a `decompose` or `interpretive-split` proposal stamps each component / reading node's `classificationFacet.committedProposalEventId` (and `committedAt`) with the parent proposal event's id and the commit event's `committed_at`. Today only the single-target sub-kinds (`classify-node`, `set-node-substance`, `set-edge-substance`, `edit-wording`) stamp the marker; the two multi-component sub-kinds leave each component's marker as `null`, so `deriveFacetStatus` returns `'proposed'` (rule 7) rather than `'committed'` (rule 5).

Concretely the deliverable is one small read-side change in the projector plus its test cover:

1. **Projector extension** in [`apps/server/src/projection/replay.ts`](../../../apps/server/src/projection/replay.ts) — extend the per-facet stamping that `handleCommit` already runs at L479-486 to handle the two multi-component sub-kinds. Per D1 below the cleanest expression is to rename / replace the single-target `facetTargetForProposal(payload)` helper at L371-384 with a multi-target `facetTargetsForProposal(payload)` returning `readonly FacetTarget[]` (mirroring the broadcast-side helper renamed in commit `30f890d`), with the four single-target sub-kinds returning a 1-element array, the two multi-component sub-kinds returning N per-component classification targets walking `proposal.components` / `proposal.readings` in array order, and the five purely-structural sub-kinds (`axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate`) returning `[]`. The `handleCommit` call-site then loops over the array and stamps each target with the same `(committedProposalEventId, committedAt)` pair. The `handleVote` and `handleMetaDisagreementMarked` call-sites at L446-459 + L527-533 — which today use the single-target return — stay structurally correct (votes against a multi-component proposal are votes-on-the-parent-proposal, not per-component; meta-disagreement on a decompose marks the proposal itself, not per-component facets); D2 below addresses the call-site shape.

2. **Vitest cover** — extend [`apps/server/src/projection/replay.test.ts`](../../../apps/server/src/projection/replay.test.ts) (cases at L695-734 + L736+ already cover the parent-becomes-invisible structural arm; add sibling cases asserting that each component's `classificationFacet.committedProposalEventId === proposalEventId` and `deriveFacetStatus(projection, 'node', component.node_id, 'classification') === 'committed'` post-commit). Tighten [`apps/server/src/ws/broadcast/proposal-status.test.ts`](../../../apps/server/src/ws/broadcast/proposal-status.test.ts) Case D (lines 989-1070) to assert `expect(env.payload.perFacetStatus).toEqual({ classification: 'committed' })` (replacing the shape-only `Object.keys(...).toEqual(['classification'])` assertion and removing the docblock note about the divergence — see D3 below for why we tighten in the same commit).

## Why it needs to be done

Three reasons compose:

**Closes the Case D status-value gap registered in the source-of-debt commit.** The Status block of [`facet_status_server_decompose_component_facets.md`](../backend/facet_status_server_decompose_component_facets.md) at L179 names this gap explicitly: "the refinement's Case D acceptance criterion asserts `perFacetStatus: { classification: 'committed' }` post-commit, but production code only flips `parent.visible = false` and does NOT mark per-component facets committed." The landed Case D test asserts multiplicity + sequence + facet-key shape but NOT the status value, with an in-line docblock noting that the status-value gap is registered as this task. Closing the projector's stamping gap lets Case D be tightened to its full intended assertion in the same commit.

**ADR 0027 single-source-of-truth invariant.** Bullet 4 of ADR 0027 pins "projectors / diagnostic engines / replay read facet state, not structural-event timing, to determine commit-status." Today the per-component classification facets' commit-state is observable only via the structural side-channel "the parent proposal is in `committedProposals`" — anyone reading `deriveFacetStatus` for the component sees `'proposed'`, because the read-side rule 5 demands a stamped `committedProposalEventId` and the projector's write-side never stamps one. Post-task, `deriveFacetStatus` returns `'committed'` directly for the per-component classification facets, and all four facet-state consumers (the broadcast listener, the moderator-side client mirror, downstream diagnostic engines, future replay-at-position queries) see the same answer without having to special-case the multi-component sub-kinds.

**Cross-surface consistency follow-through.** The broadcast listener's `deriveAndFanOut` (per the source-of-debt commit) now fans out one `proposal-status` envelope per component on commit, calling `deriveFacetStatus` for each. Without this task's projector stamping, every one of those envelopes carries `perFacetStatus: { classification: 'proposed' }` — silently wrong from the participant + audience surfaces' perspective, because the proposal HAS committed and the components' facets ARE in `'committed'` state by the methodology contract. Per `docs/methodology.md` L84 ("agreeing to decompose N into A + B agrees to the structural restructuring (N is removed; A and B exist as entities)") and `docs/data-model.md` L84-87, the act of committing a decompose / interpretive-split commits the structural change including each component's emergence as a committed entity with its classification facet in `'committed'` state — that's the whole point of the operation. The broadcast surface needs the projector to express that fact.

Downstream consumers benefit transitively:

- `participant_ui.part_graph_view` (pending under M5) — will subscribe to `proposal-status` and apply the per-component styling; without this task the per-component facets would render as `'proposed'` post-commit, contradicting the moderator's canvas styling.
- `audience.aud_graph_rendering` (pending under M6) — same transitive inheritance.
- The `data_methodology_tests.dm_e2e_tests.walkthrough_replay_e2e` task (pending) — the example-walkthrough's decompose / interpretive-split scenarios need the per-component facets to reach `'committed'` for the post-commit assertions to be meaningful.

## Constraints / requirements

- **Per-component stamping uses the SAME `(committedProposalEventId, committedAt)` pair across all N components.** The parent decompose / interpretive-split proposal commits ONCE; the per-component facets each carry the SAME commit fact (the parent proposal's event id, the commit event's `committed_at`). Receivers correlate per-component via `node.id`, not via distinct per-component commit ids.
- **Replay determinism preserved.** Iteration order over `proposal.components` / `proposal.readings` is array-position order — same convention the propose handler uses for `node-created` / `entity-included` emission. A re-projection of the same log produces identical projection state.
- **Strictly additive on the projection model.** No new fields on `FacetState<T>`, no new fields on `ProjectedNode`, no new event kinds, no new payload schema. The `committedProposalEventId: string | null` field is the existing seam — this task populates it for N additional facets per multi-component commit.
- **`applyCommittedProposal`'s structural arms are NOT touched.** Per D1 the per-component facet stamping happens at the `handleCommit` call-site, not inside `applyCommittedProposal`'s decompose / interpretive-split arms. Rationale in D1: keeps `applyCommittedProposal` strictly structural (per its existing docblock at L46-54 + the TODO comments at L692-699 + L714-717), avoids passing the `proposalEventId` + `committedAt` through the `applyCommittedProposal` signature (which today takes only the `ProposalPayload`), and co-locates all per-facet stamping in `handleCommit` next to the existing single-target stamping at L479-486.
- **`handleVote` / `handleMetaDisagreementMarked` call-sites do not iterate per-component.** Votes target the parent decompose proposal as a whole, not the per-component facets (per `docs/methodology.md` — participants vote on the proposed decomposition, not on each component independently). Same for meta-disagreement. Per D2 those two call-sites consume only the first / nothing of the multi-target helper's return — see D2 for the call-site shape that preserves this.
- **`deriveFacetStatus` is NOT touched.** Rule 5 already returns `'committed'` for a facet with stamped `committedProposalEventId` and no current dispute / withdraw. This task populates the marker; the read-side is unchanged.
- **No DB migration, no event-schema change, no API-layer change.** Pure projector-internal change.
- **Per ADR 0021 schema-on-write invariant.** No envelope shape changes; the existing `proposalEventId` (a `z.string().uuid()`) is the same id stamped on `committedProposalEventId`.
- **Test coverage per ADR 0022.** Pure-logic projector behaviour → Vitest at `apps/server/src/projection/replay.test.ts` + small assertions in `apps/server/src/projection/facet-status.test.ts` if the existing decompose-projection fixture isn't already in place there. The broadcast-listener Case D tightening at `apps/server/src/ws/broadcast/proposal-status.test.ts` is the cross-layer regression that pins the end-to-end "commit → broadcast emits `'committed'`" contract.
- **No new Cucumber / Playwright cover.** This is a projector-internal change; the wire path is already exercised by the broadcast-listener Case D (post-tightening) and the existing `tests/e2e/moderator-proposed-entity-canvas-visibility.spec.ts` (which exercises the moderator surface end-to-end). When the participant / audience surfaces land (M5 / M6), their own e2e cover will exercise the post-commit `'committed'` styling transitively.

## Decisions

- **D1 — Stamping happens at the `handleCommit` call-site, NOT inside `applyCommittedProposal`'s decompose / interpretive-split arms.** The orchestrator context note framed the fix as "extend the `applyCommittedProposal` decompose / interpretive-split arms ... to set each `component.classificationFacet.committedProposalEventId = parentProposalEventId`." Either location works structurally; this refinement chooses `handleCommit`'s existing per-facet stamping block (L479-486) over `applyCommittedProposal`'s arms. Rationale:
  - **Co-location with existing seam.** `handleCommit` already runs `facetTargetForProposal(pending.payload)` + `facetStateForTarget(...)` + stamp at L479-486 for the four single-target sub-kinds. Extending the helper to return an array and looping over it is a minimal, local change at the same seam — one helper renamed plural, one `if` becomes a `for`, no signature change to `applyCommittedProposal`.
  - **`applyCommittedProposal` stays strictly structural.** Its docblock at L46-54 + the TODO comments at L692-699 + L714-717 explicitly carve out "structural effect only — methodology semantics (per-participant agreement-state, axiom-mark invariants, decompose component-edge rebinding) live downstream." Per-component facet stamping is a facet-state mutation, not a structural one; landing it inside the structural arms would muddy the separation and the comments would no longer be honest.
  - **No signature change to `applyCommittedProposal`.** The function takes `(projection, proposal, changes)` — it does NOT have access to the parent proposal event id or the commit event's `committed_at` (both of which are needed for the stamp). Threading those through the signature would touch every arm (twelve sub-kinds) for a write-side concern that only two sub-kinds need; the `handleCommit` seam already has both values in scope (`payload.proposal_id` and `payload.committed_at`).
  - **Alternatives considered:**
    - **(B) Stamp inside the `applyCommittedProposal` `decompose` / `interpretive-split` arms.** Rejected — would require either (i) extending `applyCommittedProposal`'s signature to take the proposal event id + committed_at, touching every arm, or (ii) refactoring the function to take the full `CommittedProposalRecord` shape (still a signature change). Both options propagate write-side concerns into the structural-effect layer for negligible gain (the helper-loop at `handleCommit` is one line).
    - **(C) Add a separate per-component facet-stamping helper invoked alongside `applyCommittedProposal` in `handleCommit`.** Rejected — that's effectively D1 with an extra layer of indirection; the helper-rename approach (`facetTargetsForProposal` plural) is the same cost with less code.

- **D2 — `facetTargetsForProposal` (plural) replaces `facetTargetForProposal` (singular).** The helper grows from returning `FacetTarget | null` to returning `readonly FacetTarget[]`. The four single-target sub-kinds return a 1-element array; the two multi-component sub-kinds return N per-component classification targets; the five purely-structural sub-kinds return `[]`. Call-sites adapt:
  - **`handleCommit`** — loops over the array, stamping each target with the SAME commit pair.
  - **`handleVote`** — consumes only the FIRST element of the array (per the methodology contract: votes target the parent proposal as a whole, not per-component facets; per-component vote-recording would be a category error). For multi-component sub-kinds the array's "first element" is the first component's classification facet, but `handleVote` should NOT touch it — votes against decompose / interpretive-split proposals are not facet-vote events. The cleanest expression is to keep `handleVote`'s call shape as "early-return if the array is empty OR the proposal is multi-component"; in practice the existing branch structure (the "four facet-targeting sub-kinds" comment at L440-444 already enumerates that votes only apply to those four) naturally translates to "consume only the first element when the array has exactly one element; otherwise no-op." A small `firstFacetTargetForVote(payload)` wrapper (returning `FacetTarget | null` for the four single-target sub-kinds and `null` for everything else) is the cleanest reading; this task adds that wrapper alongside the plural rename.
  - **`handleMetaDisagreementMarked`** — same wrapper as `handleVote`; meta-disagreement marks the proposal as a whole, not per-component facets.
  - **Alternatives considered:**
    - **(B) Keep both `facetTargetForProposal` (singular) AND `facetTargetsForProposal` (plural).** Rejected — duplicates the per-sub-kind switch; future maintenance has to update both whenever a sub-kind changes shape. The wrapper pattern (D2 chosen) has the same property without the duplication.
    - **(C) Pass a callback to a unified iterator.** Rejected — over-abstracted for two call-sites; the wrapper is simpler.

- **D3 — Tighten Case D in the SAME commit as the projector fix.** The orchestrator context note left this as an explicit decision point. This refinement chooses to land the tightening in the same commit. Rationale:
  - **The tightening is the cross-layer regression check for this task.** Case D's existing shape-only assertion was a placeholder pending exactly this task — its docblock names the gap explicitly. Landing the projector fix without tightening would leave a known-stale assertion (a passing test that asserts less than the projector now produces); landing the tightening separately would briefly create the inverse (a passing projector but a not-yet-tightened test).
  - **Same commit avoids a between-commit window where the broadcast contract is under-asserted.** If the projector fix lands alone, Case D continues to pass but the test no longer asserts the actual production behaviour — a regression in the projector that re-introduces the bug would not be caught by Case D until the tightening lands. Same-commit landing closes that window.
  - **Cross-layer atomicity.** The projector and broadcast layers are coupled by the commit fact (commit → projector stamps → broadcast derives → envelope carries `'committed'`). Tightening Case D in the same commit pins the full chain atomically.
  - **Alternative considered: split the tightening into a separate follow-up commit.** Rejected — adds a known-stale-test window for no benefit. The tightening is a one-line `expect` change plus a docblock update; it's not a meaningful commit-size reduction to split it off.

- **D4 — Per-component stamping uses the parent proposal event id as `committedProposalEventId`, NOT a synthesized per-component event id.** The methodology contract says the parent decompose / interpretive-split proposal commits ONCE; the per-component facets each carry the SAME commit fact. The natural id to stamp is the parent proposal event's id (which is `payload.proposal_id` in `handleCommit`'s scope, matching what's already stamped for single-target sub-kinds at L483). Alternative considered: mint a per-component commit-marker id (e.g. `${proposalEventId}:${componentIndex}`). Rejected — adds a new id-minting concern, breaks the "stamping field references an existing event id" invariant other code paths rely on (e.g. the `CommittedProposalRecord` lookup by proposal event id), and does not aid de-duplication on the receiving side (receivers correlate per-component via `node.id`, not via the marker).

- **D5 — `committedAt` is stamped with the COMMIT event's `committed_at`, NOT the parent proposal event's `createdAt`.** The committed-at field semantically captures "when did this facet's commit fact land," which is the commit event's `committed_at` (the methodology contract for commits per `docs/data-model.md`). The parent proposal event's `createdAt` is when the proposal was raised, which is not the commit fact. Single-target sub-kinds already follow this convention at L484 (`facet.committedAt = payload.committed_at` where `payload` is the `CommitPayload`); this task carries the same convention to per-component stamping for consistency.

- **D6 — Only the classification facet is stamped per component; wording + substance are NOT stamped.** Mirrors the broadcast listener's D6 from the source-of-debt refinement (v1 emits only the classification facet per component). Rationale:
  - **Classification is the only facet whose state is derived purely from the parent proposal.** Per the data-model contract, each component's wording is set at `node-created` time directly (the component carries its wording in the `node-created` payload, not via a separate facet event); each component's substance facet is N/A at propose-time (substance is set by a subsequent `set-node-substance` proposal targeting the component). The classification facet, by contrast, has no per-component event of its own — its state is purely derived from the parent decompose / interpretive-split commit.
  - **Symmetry with the source-of-debt commit's D6.** The broadcast listener emits only classification envelopes per component; the projector stamping must match — otherwise the projector would mark wording / substance committed but the broadcast would never surface them (or vice versa).
  - **Alternative considered: stamp all three facets per component.** Rejected — wording is already initialized to its final value at `node-created` time (no separate commit fact); substance has no value until a future `set-node-substance` proposal commits against the component. Stamping them would mean "marker is set, but the facet was never proposed via this proposal" — semantically incorrect.
