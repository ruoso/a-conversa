# Interpretive-split logic — propose-side validator for the `interpretive-split` proposal sub-kind

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.interpretive_split_logic`
**Effort estimate**: 1d
**Inherited dependencies**: depends on `methodology_engine.decomposition_logic` (the prior sibling — its propose-side validator, the `nodeIsVisible` primitive, the conflict-walker primitive `decomposeConflictsWith`, the `'target-entity-not-found'` `RejectionReason`, and the per-sub-kind dispatch shape in `propose.ts` are all directly extended here). Transitively: `methodology_engine.agreement_state_machine` (the framework — `MethodologyAction`, `validateAction`, `RejectionReason`, `requireParticipant`, `findProposal`, etc.). Concretely settled: the Zod `interpretiveSplitProposalSchema` in `packages/shared-types/src/events/proposals.ts` (enforces structural shape — 2..10 readings, each with non-empty wording and a statement-kind classification — identical shape to decompose's `components`); the projection's read-side `replay.ts/applyCommittedProposal` `interpretive-split` arm which flips `parent.visible = false` on commit, mirroring the decompose arm.

## What this task is

Tighten the `propose` handler's `interpretive-split` arm with the real methodology-engine validator, mirroring decomposition_logic's tightening of the `decompose` arm. The propose handler currently dispatches `decompose` to its real validator and lets the other ten sub-kinds — including `interpretive-split` — fall through to the universal-pass placeholder path. This task replaces that fall-through for `interpretive-split` with three methodology-specific rules, in evaluation order:

1. The `parent_node_id` references a node that exists in this session's projection.
2. The parent node is currently visible in the projection (a not-visible parent — already decomposed, interpretively-split, restructured, or otherwise superseded per the visible-graph derivation in `docs/data-model.md` — can't be re-split).
3. **No other pending decompose OR interpretive-split proposal references the same `parent_node_id`** (mutual-exclusion conflict). The two operations are mutually exclusive on the same parent because both flip `parent.visible = false` on commit; a pending one of either kind blocks a new one of either kind.
4. The structural payload shape — `readings.length ∈ [2, 10]`, each reading's `wording` non-empty, each `classification` a valid `StatementKind` — is already enforced upstream by the Zod schema (`interpretiveSplitProposalSchema`, ADR 0021). The validator relies on that layering and does not re-check.

On `Valid` the handler emits exactly one `EventToAppend` of kind `proposal` whose payload is the matching `ProposalEnvelopePayload` (`{ proposal: action.proposal }`). On any rule failure the handler returns a typed `Rejected` with a `RejectionReason` that names the specific failure.

Scope is **propose-side only** — same scoping discipline as decomposition_logic. Interpretive-split's commit-time multi-event fan-out (creating the reading nodes and including them into the session) is a separate concern that mirrors decompose's commit-time gap; see Open Questions and cross-reference decomposition_logic.md.

The conflict-walker primitive `decomposeConflictsWith` in `primitives.ts` is refactored: it becomes a more general `findConflictingProposalAgainst(projection, parentNodeId, conflictingKinds: Set<'decompose' | 'interpretive-split'>)` that walks `projection.pendingProposals()` and returns the first pending proposal whose payload's `kind` is in the `conflictingKinds` set AND whose `parent_node_id` matches. Both the `decompose` arm and the `interpretive-split` arm call it with the same two-element set `{'decompose', 'interpretive-split'}`. The decompose arm's existing rule 3 ("no conflicting decompose pending") is upgraded to "no conflicting decompose OR interpretive-split pending" — the mutual exclusion is symmetric, and decomposition_logic's existing test expectations are preserved (a decompose-vs-decompose conflict is still rejected) while gaining the symmetric case (a decompose-vs-interpretive-split conflict is also rejected).

## Why it needs to be done

`docs/methodology.md` lines 168–181 settle the interpretive-split as the methodology's primary tool for "we agree this statement is real but read it as different things" — produces N alternative readings of the same parent. The structural treatment is identical to decompose ("the parent node is replaced by component nodes in the current view; the change history records the original wording and the operation" — line 180), but the methodology semantics differ (decompose: "the speaker intended multiple claims and bundled them"; interpretive-split: "the speaker may have intended one claim, but the wording admits multiple readings"). The placeholder propose handler currently accepts interpretive-split proposals against nonexistent parents, already-superseded parents, and parents that already have a competing decompose or interpretive-split pending. Without this task, the API layer would write all those illegal proposals as if they were live; downstream consumers would throw `ReplayError`s at commit time rather than the propose author hearing a clean methodology rejection at submission time.

The boundary is precise: this task validates the *intent* to propose an interpretive-split on the **write** side (does the request pass methodology rules?). The projection's `replay.ts/applyCommittedProposal` for `interpretive-split` is the *structural-effect* path that runs at commit time (it flips `parent.visible = false`, same as the decompose arm). The two layers don't overlap: this validator runs at propose time, and `applyCommittedProposal` runs at commit time after `commit_logic` has gated the commit.

The mutual-exclusion extension is the new design constraint that emerges with this task. Decomposition_logic's rule 3 was "no other decompose pending against this parent" because at the time only decompose targeted `parent.visible`. With interpretive-split landing, both operations target `parent.visible` and committing either one races against committing the other; the propose-side validator must reject the second of either kind. The symmetry is natural: a pending interpretive-split blocks a new decompose just as a pending decompose blocks a new interpretive-split.

Downstream consumers — the eventual commit-time component/reading-node creation for both decompose and interpretive-split — depend on this rule set as the template they extend.

## Constraints / requirements

- The handler **does not write events**; it returns a `ValidationResult`. On `Valid` it emits exactly one `EventToAppend` of kind `proposal`.
- The handler **does not mint timestamps or ids**; the API layer mints `eventId` and `createdAt` before calling the engine.
- The handler **does not call `validateEvent`**; the API layer runs the structural validator (Zod) separately (ADR 0021). Methodology validation is on top of structural validation. The handler can rely on `readings.length ∈ [2, 10]` and `wording: non-empty` being already-true.
- No new `RejectionReason` values are added — all three rules reuse the existing values (`'target-entity-not-found'` from decomposition_logic, `'illegal-state-transition'` umbrella for rules 2 and 3).
- Don't modify `apps/server/src/projection/*`. The projection's `applyCommittedProposal` interpretive-split arm is stable; this task validates *before* commit.
- The conflict-walker refactor MUST keep decomposition_logic's existing test expectations green. Specifically, the symmetric mutual exclusion means: (1) a decompose still rejects when another decompose is pending against the same parent (unchanged); (2) a decompose now ALSO rejects when an interpretive-split is pending against the same parent (newly symmetric); (3) an interpretive-split rejects when either a decompose or another interpretive-split is pending (the new case). Decomposition_logic's existing tests don't exercise case (2) — that's a new assertion added to `proposeDecompose.test.ts` by this task.
- Don't pre-empt the other proposal sub-kinds. `interpretive-split` is the second arm tightened; the remaining nine (`classify-node`, `set-node-substance`, `set-edge-substance`, `edit-wording`, `axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate`) stay on the placeholder path until their own sibling tasks land.
- Don't pre-empt commit-time multi-event emission for interpretive-split. The propose handler emits one `proposal` event; reading-node creation (N `node-created` + N `entity-included` + commit) is the same Open Question decomposition_logic flagged. Cross-reference its refinement.
- Verifications per ADR 0022: Vitest at `apps/server/src/methodology/handlers/proposeInterpretiveSplit.test.ts`; Cucumber + pglite scenarios at `tests/behavior/methodology/propose-interpretive-split.feature` with step defs in `tests/behavior/steps/methodology-propose-interpretive-split.steps.ts`.

## Decisions

- **Scope: propose-side only.** This task validates `propose interpretive-split`. Commit-time multi-event emission is **not** in scope — see Open Questions, which cross-reference decomposition_logic.md's identical open question.
- **Rule set (numbered, in evaluation order).** Same shape as decompose's rule set:
  1. **Parent-node-exists.** → `'target-entity-not-found'`.
  2. **Parent-node-visible.** → `'illegal-state-transition'`.
  3. **No conflicting decompose OR interpretive-split pending.** → `'illegal-state-transition'`.
  4. (Structural shape — enforced upstream by Zod.)
- **Conflict-walker refactor: `decomposeConflictsWith` → `findConflictingProposalAgainst`.** Considered alternatives:
  - **Add a second primitive `interpretiveSplitConflictsWith` and call both from each arm.** *Rejected.* Two walks, two predicates, two call-sites per arm — the surface doubles and the symmetry is implicit (only enforceable by reading both call-sites). A single parameterized walk makes the symmetry explicit (both arms pass the same conflicting-kinds set).
  - **Generalize the walker to take a payload-predicate function.** *Rejected* as premature. The two parent-targeting sub-kinds share the exact same payload shape (`parent_node_id` field); a `(payload) => boolean` predicate is more flexible than needed and harder to read. The set-of-kinds parameter is the minimum sufficient generalization.
  - **Take a single `'decompose' | 'interpretive-split'` discriminator (one kind at a time) and let each arm pass its own set.** *Rejected.* The two arms need the SAME set (mutual exclusion is symmetric — both reject against either kind). A set parameter that callers happen to populate the same way is the right shape; a per-call discriminator would invite the bug where one arm forgets the other kind.
  - **Set parameter with the `'decompose' | 'interpretive-split'` union type.** *Chosen.* The narrow type matches the only two sub-kinds whose payloads carry `parent_node_id` in v1; future parent-targeting sub-kinds widen the union additively. The constant `CONFLICTING_PARENT_KINDS` lives at module scope in `propose.ts` and is the single source of truth for "what blocks a new decompose or interpretive-split"; both arms reference it.
- **Rule 3 detail: name the conflicting sub-kind in the rejection detail.** The decompose arm's existing detail said "another decompose proposal (id) against parent is already pending"; with mutual exclusion the message becomes ambiguous if the conflict is an interpretive-split instead. The detail now reads "another {kind} proposal ({id}) against parent {pid} is already pending; resolve or withdraw it before re-proposing." Both arms format identically.
- **No new `RejectionReason` values.** Decomposition_logic already added `'target-entity-not-found'`; rule 1 reuses it. Rules 2 and 3 reuse the `'illegal-state-transition'` umbrella (same arguments decomposition_logic made: the rule shape is "the parent is in a state that blocks this transition" — not a referential failure, not a facet inapplicability).
- **Where the handler lives.** `propose.ts` itself, factored locally as `validateInterpretiveSplitProposal(projection, action)`. Mirrors decomposition_logic's choice — see its refinement for the factoring rationale. If `propose.ts` grows past ~300 lines as more sub-kinds tighten, factor into `apps/server/src/methodology/handlers/proposeKinds/<sub-kind>.ts`.
- **Structural shape is upstream's concern.** The Zod schema in `interpretiveSplitProposalSchema` enforces:
  - `readings: z.array(proposalComponentSchema).min(2).max(10)` — the 2..10 bound (R27).
  - `proposalComponentSchema.wording: z.string().min(1)` — non-empty wording.
  - `proposalComponentSchema.classification: statementKindSchema` — valid statement kind.
  Per ADR 0021 the validator runs at the API ingress before the methodology engine sees the action. The methodology validator relies on this layering and does not re-check. The Vitest test file asserts this layering by calling `interpretiveSplitProposalSchema.safeParse` directly on the 1-, 11-, and empty-wording cases.
- **Read-side `findConflictingProposalAgainst` semantics.** Walks `projection.pendingProposals()` only — not `committedProposals` or `unresolvedMetaDisagreements`. A *committed* decompose or interpretive-split against the same parent will have already flipped `parent.visible = false`, so rule 2 catches that case structurally. A *meta-disagreement-marked* proposal isn't in flight any more. Same reasoning decomposition_logic made for `decomposeConflictsWith`; carries over directly to the generalized walker.
- **Test layout.** Vitest at `apps/server/src/methodology/handlers/proposeInterpretiveSplit.test.ts` (new file — same naming pattern as `proposeDecompose.test.ts`). Cucumber + pglite at `tests/behavior/methodology/propose-interpretive-split.feature` with step defs in `tests/behavior/steps/methodology-propose-interpretive-split.steps.ts`. Reuses `event-rows.ts` and the shared Then steps. The step file uses a distinct UUID prefix (`f0...` series) and distinct session id to avoid collisions with the propose-decompose step file's `e0...` prefix.

## Open questions

- **Commit-time multi-event emission for interpretive-split.** Mirrors decomposition_logic's identical open question — see [decomposition_logic.md](decomposition_logic.md) section "Open questions" for the three resolution paths (P1: `commit_logic` grows a per-sub-kind branch; P2: `interpretive_split_logic` registers a commit-side validator; P3: follow-up proposals per reading). The current task does not settle this for interpretive-split either — it's flagged for the next iteration to resolve, likely in a unified `decomposition_commit_logic` follow-up that handles both decompose and interpretive-split commit-time fan-out (the structural effect is identical — parent flips invisible, N reading-/component-nodes need creation; a single commit-side handler for both makes sense). For now the gap means a propose interpretive-split lands but cannot be committed; the integration tests in this task cover only the propose-side path and assert nothing about commit.
