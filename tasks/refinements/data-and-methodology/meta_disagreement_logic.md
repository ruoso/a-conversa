# Meta-disagreement logic — moderator marks a stuck proposal as meta-disagreement

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.meta_disagreement_logic`
**Effort estimate**: 1d
**Inherited dependencies**: depends on `methodology_engine.agreement_state_machine` (the framework — `MethodologyAction`, `validateAction`, `RejectionReason`, the primitives — `requireModerator`, `findProposal` — and the per-action-handler module shape factored out by `commit_logic`); siblings `commit_logic` and `withdrawal_logic` already landed (the `committedProposals` and per-facet `perParticipant` state the meta-disagreement validator reads against). Concretely settled: the methodology framework, the placeholder `placeholderMarkMetaDisagreementHandler` (`apps/server/src/methodology/handlers/markMetaDisagreement.ts`), the projection's `handleMetaDisagreementMarked` (`apps/server/src/projection/replay.ts`) which on the read side transitions the affected facet's status to `'meta-disagreement'`, and `deriveFacetStatus` (`apps/server/src/projection/facet-status.ts`) which carries the `'meta-disagreement'` status as a top-level rule.

## What this task is

Replace the placeholder `mark-meta-disagreement` handler that ships with `agreement_state_machine` with the real methodology-engine validator. A `mark-meta-disagreement` action is legal only when **(a)** the requester is the session's moderator, **(b)** the proposal exists in `pending` state (not already committed or meta-disagreed), and **(c)** the proposal has run through the methodology far enough to be stuck — concretely, the affected facet's `perParticipant` map contains at least one `dispute` vote against this proposal. On `Valid` the handler emits a single `EventToAppend` of kind `meta-disagreement-marked` whose payload is the matching `MetaDisagreementMarkedPayload` from `@a-conversa/shared-types` (`proposal_id`, `moderator`, `marked_at`). On any rule failure the handler returns a typed `Rejected` with a `RejectionReason` already minted by the prior task's union.

## Why it needs to be done

`docs/methodology.md` lines 203–208 settle meta-disagreement as the **last-resort fallback** when participants exhaust diagnostic tests and decomposition. The placeholder handler from `agreement_state_machine` returns `Valid` for *any* meta-disagreement-mark action that passes the universal checks (session match / sequence match / participant gate) — including marks from debaters, marks of nonexistent proposals, marks of proposals nobody has disputed, and double-marks. Without this task, the API layer would accept all those illegal marks; in particular it would let a moderator escape-hatch around any fresh proposal that hasn't even been engaged with — which is the opposite of what the methodology says meta-disagreement is for.

The boundary is precise: this task validates the *intent* to mark on the **write** side (does the request pass methodology rules?). The projection's `handleMetaDisagreementMarked` (`replay.ts`) applies the *structural effect* on the **read** side (move the proposal from `pendingProposals` to `unresolvedMetaDisagreements`, set the affected facet's status to `'meta-disagreement'`). The two layers run in order: `validateAction` returns `Valid`, the API layer appends the event, then `applyEventIncremental` calls `handleMetaDisagreementMarked`. They stay in sync by construction — once the validator passes the dispute-exists check, the read side has the matching state to transition.

## Constraints / requirements

- The handler **does not write events**; it returns a `ValidationResult`. On `Valid` it emits exactly one `EventToAppend` of kind `meta-disagreement-marked`.
- The handler **does not mint timestamps or ids**; the API layer mints `eventId`, `createdAt`, and `markedAt` before calling the engine.
- The handler **does not call `validateEvent`**; the API layer runs the structural validator separately (ADR 0021). Methodology validation is on top of structural validation.
- No new event payloads, `ProjectionChange` discriminators, or shared-types schemas. No new `RejectionReason` values — the prior task's union covers every case.
- Don't modify `apps/server/src/projection/*`. The projection's `handleMetaDisagreementMarked` and `deriveFacetStatus` are stable; this task validates *before* they run.
- Don't pre-empt the sibling tasks: `decomposition_logic`, `axiom_mark_logic`, `interpretive_split_logic`, etc.

## Acceptance criteria

- `apps/server/src/methodology/handlers/markMetaDisagreement.ts` exports a default `Validator<MarkMetaDisagreementAction>` that:
  1. Calls `requireModerator(projection, action.requester)`. On failure returns the rejection (reason `'not-a-moderator'` for joined-but-wrong-role; `'not-a-participant'` for not-joined-at-all — the latter is technically unreachable because `validateAction` already ran the universal participant gate, but the helper's contract still distinguishes the two).
  2. Calls `findProposal(projection, action.proposalEventId)`. Branches:
     - `null` → reject `'proposal-not-found'`.
     - `state: 'committed'` → reject `'proposal-already-committed'`.
     - `state: 'meta-disagreement'` → reject `'proposal-already-meta-disagreement'`.
     - `state: 'pending'` → continue.
  3. **Methodology-exhaustion gate (Option A — structural).** For the four facet-targeting proposal sub-kinds (`classify-node`, `set-node-substance`, `set-edge-substance`, `edit-wording`), the affected facet's `perParticipant` map must contain at least one `dispute` vote whose `proposalEventId` matches the proposal id. On absence → reject `'methodology-not-exhausted'` with a detail explaining "the proposal has not been disputed; meta-disagreement is the last resort after the methodology has run." For the structural sub-kinds (`decompose`, `interpretive-split`, `axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate`), defer with `'illegal-state-transition'` and a sub-kind-naming detail — same boundary `commit_logic` draws.
  4. On `Valid` emit one `EventToAppendEnvelope<'meta-disagreement-marked'>` whose payload is `{ proposal_id: action.proposalEventId, moderator: action.requester, marked_at: action.markedAt }`. Envelope fields are mirror-copied from the action.
- A new primitive `proposalHasAnyDispute(projection, proposalEventId) → boolean` is added to `apps/server/src/methodology/primitives.ts`. It resolves the affected facet via the same `facetTargetForProposal` shape used by `commit.ts` and `vote.ts`, walks the `perParticipant` map, and returns `true` iff at least one record has vote `'dispute'` AND `proposalEventId` matches. ## Decisions

- **Where the handler lives.** `apps/server/src/methodology/handlers/markMetaDisagreement.ts`. Replaces the placeholder of the same name; mirrors the per-action-module shape factored out by `commit_logic`.
- **Rule set (numbered, in evaluation order).** Mirrors `commit.ts`:
  1. **Moderator gate.** `requireModerator(projection, action.requester)` must succeed. → `'not-a-moderator'` (or `'not-a-participant'` if not joined; unreachable in practice because the universal gate already filtered).
  2. **Proposal exists.** `findProposal(projection, action.proposalEventId)` must return non-null. → `'proposal-not-found'`.
  3. **Proposal is pending.** The found state must be `'pending'`, not `'committed'` or `'meta-disagreement'`. → `'proposal-already-committed'` / `'proposal-already-meta-disagreement'`.
  4. **Methodology-exhaustion gate (Option A — structural).** For the four facet-targeting sub-kinds, the affected facet's `perParticipant` map must contain at least one record with vote `'dispute'` and `proposalEventId === action.proposalEventId`. For the structural sub-kinds the rule is deferred (`'illegal-state-transition'`). → `'methodology-not-exhausted'` for the facet-targeting case; `'illegal-state-transition'` for the structural case.
- **Methodology-exhaustion-gate semantics: chose Option A (structural — require at least one dispute).** The task description offered three options:
  - **(A)** structural — require at least one recorded `dispute` vote on the affected facet.
  - **(B)** UX-trust — no structural check; trust the moderator's judgment.
  - **(C)** mixed — require at least one dispute (A) plus an informational warning if no decompose / interpretive-split / defeater attempt is on record.
  Chose **A**. Rationale: (1) `docs/methodology.md` line 208 says meta-disagreement is "a last resort; the methodology is designed so that decomposition resolves most cases before this fallback is needed" — that phrasing presupposes "an actual stuck point," and a proposal with no dispute is by definition not stuck. A fresh proposal with no engagement is in `proposed` (the methodology's neutral starting state), not `disputed`; the methodology hasn't even started. (2) The `RejectionReason` union already carries `'methodology-not-exhausted'` exactly for this case — the prior task minted the value with this case in mind. (3) Option A is structurally enforceable from projection state (one walk of `perParticipant`), whereas Option B punts the rule entirely and Option C adds a warning channel the engine doesn't currently have. (4) Option A is the only one that catches the obvious abuse case — a moderator escape-hatching past a fresh proposal nobody objected to — without depending on the moderator's discretion. The "fuzzy" judgment about whether the methodology was *truly* exhausted (did the participants run the operationalization test? did they try decomposition?) is **out of scope for the structural validator** — those are UX-level prompts the moderator console can surface, not write-side gates. The validator's job is to refuse the *clearly-too-early* case; everything past "at least one dispute exists" is the moderator's call.
- **Rule 4 implementation — new primitive `proposalHasAnyDispute`.** Adds a small helper to `primitives.ts` mirroring `findParticipantVoteOnProposal`'s shape: resolves the affected facet via the local `facetTargetForProposal`, walks `perParticipant`, returns true on the first match. The walk is O(participants) which is tiny in practice. The primitive returns `false` for the structural sub-kinds; the handler short-circuits to `'illegal-state-transition'` before consulting it for those sub-kinds.
- **Structural-sub-kind boundary.** Same shape as `commit_logic`: the seven structural sub-kinds (`decompose`, `interpretive-split`, `axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate`) don't have per-participant vote state on the projection — `handleVote` only writes to `perParticipant` for the four facet-targeting sub-kinds. The validator rejects mark attempts on structural sub-kinds with `'illegal-state-transition'` and a sub-kind-naming detail. The sibling tasks (`decomposition_logic`, etc.) will tighten when they land — they may decide their sub-kind cannot be meta-disagreed at all (some structural ops are atomic), or that the dispute signal lives on a different projection structure. For v1 the rejection is honest: the validator doesn't know how to judge exhaustion for these sub-kinds, so it declines.
- **Participant-leaves semantics.** Inherited from `commit_logic` by construction: `proposalHasAnyDispute` walks the facet's `perParticipant` map directly, which contains historical records including those from left participants. A `dispute` vote cast by a participant who later left **does** count toward the methodology-exhaustion gate. Rationale: the dispute did happen — the proposal was contested — and the methodology's "this is stuck" signal is the historical fact, not the current-participant fact. This differs from `commit_logic` rule 4 (which only consults current participants for unanimity) because the two rules ask different questions: commit asks "do all currently-present people agree?" while meta-disagreement asks "has this proposal been contested at any point?" The historical answer is the meaningful one. (If the contesting participant left and the remaining current set all agrees, that's a different state — the moderator can commit, not mark meta-disagreement. The meta-disagreement path is for "we're stuck *now*," which only applies when there's a *current* contest. But validating "current contest" structurally is fragile — a participant could leave precisely to escape a stuck point. Accepting historical disputes is the safer rule.)
- **Boundary with the projection's existing meta-disagreement handling.** `handleMetaDisagreementMarked` in `apps/server/src/projection/replay.ts` already transitions the affected facet's `status` to `'meta-disagreement'` and moves the proposal from `pendingProposals` to `unresolvedMetaDisagreements`. **Our validator runs BEFORE that** — the validator gates whether the event is *legal to append*; the projection handler applies the state transition *after* the event is appended. The two layers do not overlap: the validator does not touch projection state, and the projection handler does not re-validate methodology rules. The projection handler's contract is "this event has been validated by the methodology engine; apply its effect."
- **No new `RejectionReason` values.** The prior task's union already includes every reason this handler needs: `'not-a-moderator'`, `'proposal-not-found'`, `'proposal-already-committed'`, `'proposal-already-meta-disagreement'`, `'methodology-not-exhausted'`, `'illegal-state-transition'`.
- **`EventToAppend` shape mirrors the placeholder.** Same payload, same envelope fields. The only thing this handler adds is the four-rule write-side gate before emitting.
- **Engine-test impact.** The existing dispatch test in `apps/server/src/methodology/engine.test.ts` (`'mark-meta-disagreement action emits a meta-disagreement-marked event'`) was written against the placeholder and asserts a `Valid` outcome on a proposal with no disputes. After this task that combination is rejected with `'methodology-not-exhausted'`. The dispatch test is updated to assert the rejection outcome — same pattern `commit_logic` used (the prior commit-action dispatch test was updated to assert `'unanimous-agree-required'` rejection). The full rule set is covered in the new `markMetaDisagreement.test.ts`; the engine test asserts dispatch only.
- **`installHandlers` wiring.** `engine.ts` swaps the import from `placeholderMarkMetaDisagreementHandler` to the real `markMetaDisagreementHandler`. `handlers/index.ts` re-exports the real handler; the placeholder export is removed (no consumers).
- **Test layout.** Vitest at `apps/server/src/methodology/handlers/markMetaDisagreement.test.ts` so the file sits next to the handler. Cucumber + pglite at `tests/behavior/methodology/mark-meta-disagreement.feature` with step defs in `tests/behavior/steps/methodology-mark-meta-disagreement.steps.ts`. Reuses `event-rows.ts` and the shared `Then 'the validation result is Valid'` / `Then 'the validation result is Rejected with reason "..."'` steps from the prior step files.
