## What this task is

Tighten the `propose` handler's `set-edge-substance` arm with the real methodology-engine validator. Today the arm falls through to the universal-pass placeholder path (per the propose handler header comment at [`apps/server/src/methodology/handlers/propose.ts:1106-1112`](../../../apps/server/src/methodology/handlers/propose.ts) and the dispatcher comment at [L1172-1176](../../../apps/server/src/methodology/handlers/propose.ts)); the structural-event builder at [L1312-1363](../../../apps/server/src/methodology/handlers/propose.ts) is what threads the new endpoint fields into the emitted `edge-created` envelope, but it performs zero cross-field referential checking — it trusts that whatever the schema accepted is methodologically coherent.

This task replaces the fall-through with three new methodology-specific rules for the **connecting-edge** shape (the substance-only re-vote shape stays unchanged: no endpoint fields, no extra validation, no structural events fire). The validator runs in two phases driven by the same fresh-edge predicate the structural-event builder uses:

1. **Symmetry phase.** If at least one of the three endpoint fields (`source_node_id`, `target_node_id`, `role`) is present on the proposal, all three must be present. A partial payload (two-of-three) signals client malformation; the rejection is `'illegal-state-transition'` with a detail naming the missing field(s). This phase short-circuits before any projection lookup.

2. **Referential phase.** When all three endpoint fields are present, four sub-rules run in order:
   - **2a. Source-node-visible.** `projection.getNode(source_node_id) !== undefined && node.visible === true` (i.e. `nodeIsVisible(projection, source_node_id)`). → `'target-entity-not-found'` (the source either doesn't exist on the projection or has been superseded by a prior decompose / interpretive-split / restructure).
   - **2b. Target-node-visible.** Symmetric to 2a against `target_node_id`. → `'target-entity-not-found'`.
   - **2c. Edge-agreement (when `edge_id` already exists).** If `projection.getEdge(edge_id) !== undefined`, the carried endpoint triple MUST agree with the existing edge's endpoints + role. I.e. `edge.sourceNodeId === source_node_id && edge.targetNodeId === target_node_id && edge.role === role`. Disagreement is `'illegal-state-transition'` with a detail naming both endpoint triples (the carried one and the projection's). Rationale: the existing edge is the authoritative source of truth; a proposal that names different endpoints against the same `edge_id` is structurally incoherent (cf. ADR 0027 — entity identity is fixed at `edge-created` time, not at proposal time).
   - **2d. Role-source-target compatibility.** Strict-pair check: for the `bridges-from` and `bridges-to` roles, the source node MUST be a warrant (i.e. the source carries an incoming or outgoing `bridges-from` or `bridges-to` edge, OR has a classification of `definitional`) — see Decision D3 for the rationale on choosing a **permissive** v1 strictness here, where 2d is **not implemented** in v1 and is documented as a follow-up methodology decision. The v1 validator stops at 2c; the role-compatibility check is a documented Open Question with rationale below, not a v1 acceptance criterion.

On `Valid` the validator returns `null` (the dispatcher proceeds to the structural-event builder and then to the proposal envelope construction at [`propose.ts:1199-1209`](../../../apps/server/src/methodology/handlers/propose.ts), exactly as the existing sibling validators do). On any rule failure the validator returns a typed `RejectedValidationResult` with the `RejectionReason` named above. The validator never emits events directly.

Scope is **propose-side only**, lives in the methodology engine (not the WS handler), and reuses the existing `RejectionReason` union — no new codes are minted in v1 per D2. Surface area is one new local function in `propose.ts` plus one dispatcher case-arm change, sibling to the eight existing per-sub-kind validators.

## Why it needs to be done

Three reasons compose:

**Validator-vs-emitter separation (the source-of-debt rationale).** Per [`mod_set_edge_substance_endpoint_carriage.md`](../moderator-ui/mod_set_edge_substance_endpoint_carriage.md) D2: "the propose handler's set-edge-substance arm does NOT check that `source_node_id` and `target_node_id` reference existing visible nodes on the projection, nor that the proposed role is valid for the source/target kinds. The structural-event builder's job is wire-shape emission, not methodology validation — keeping the two concerns separate matches the pattern the file's other arms already follow (the methodology validators are at the top of the file; the structural-event builder is at the bottom; each arm is one of the two, never both)." Landing the validator here is the closing move of that two-step factoring — the parent commit established the separation; this commit honors it.

**Failure-mode coverage that schema-on-write doesn't catch.** ADR 0021's `validateEvent` pass on the emitted `edge-created` envelope catches malformed UUIDs and out-of-enum role values (per the `edgeRoleSchema` re-use in the proposal payload). It does NOT catch: (a) a client that sends a well-formed UUID that doesn't reference any node in the session; (b) a client that sends a partial payload (e.g. `source_node_id` + `target_node_id` but no `role`); (c) a client that re-proposes against an existing edge with a different `(source, target, role)` triple than the projection records. All three failure modes are reachable today via direct-WS clients and would silently corrupt the engine's downstream computations (the structural-event builder would emit an `edge-created` against a non-existent source node — a `ReplayError` waiting to fire on the next projection rebuild, or worse, a phantom edge that the canvas projector renders against `undefined` source coordinates).

**Inverse-pair invariant survival (the lockstep with the structural-event builder).** Per the parent refinement's D3, the structural-event builder for `set-edge-substance` and the `entitiesToRetractForWithdraw` arm in `withdraw.ts` are an inverse pair — both keyed on the same fresh-edge predicate. A validator that rejects malformed connecting proposals **before** the structural-event builder runs preserves that invariant trivially: a rejected proposal emits nothing, so there is nothing to withdraw. Without this validator, a malformed connecting proposal that slipped through the schema (partial-payload case, missing-source-node case) would emit `edge-created` + `entity-included` against a broken topology, leaving the inverse-pair to "withdraw" entities the projection was never coherent about in the first place.

Downstream consumers benefit transitively:

- The moderator's canvas projector inherits the same correctness via the projection layer — no per-surface fix needed.
- The participant / audience canvas projectors (pending under M5 / M6) inherit it the same way.
- The defeater-precommit flow at [`apps/server/src/methodology/handlers/proposeDefeaterPreCommit.test.ts`](../../../apps/server/src/methodology/handlers/proposeDefeaterPreCommit.test.ts) (substance-only re-vote against an extant edge) is unaffected: the validator's first phase short-circuits when zero endpoint fields are present, so that test stays green without modification.
- Any future re-classify-edge flow (no concrete proposal yet, but the schema's optional-endpoints shape makes it possible) gets the agreement-with-existing-edge check (rule 2c) for free.

## Constraints / requirements

- **Validator lives in the methodology engine, not the WS handler.** The validator is added to `propose.ts` as a local function `validateSetEdgeSubstanceProposal(projection, action): RejectedValidationResult | null`, mirroring the eight existing siblings. It is called from the per-sub-kind switch in `proposeHandler`. Rationale per the user-supplied scope clarification and the source-of-debt refinement's "the validator should live in the methodology engine (not the WS handler) so it composes with replay and with any non-WS callers of propose" — the WS handler is one of N transports; the engine validator is the single gate every transport composes with.
- **Two-phase structure (symmetry then referential).** Phase 1 (symmetry) runs entirely off the proposal payload — no projection lookup needed; short-circuit fast. Phase 2 (referential) runs only when all three endpoint fields are present and indexes the projection. The split keeps the symmetry rejection cheap and the failure mode legible (a client sending two-of-three gets a distinct error message from a client sending three-of-three with an unknown source node).
- **No new `RejectionReason` value in v1 per D2.** All rule failures reuse existing codes: `'target-entity-not-found'` for the source-doesn't-exist / target-doesn't-exist cases (mirrors `break_edge_logic` rule 1, `meta_move_logic` rule 1a, `decomposition_logic` rule 1) and `'illegal-state-transition'` for the symmetry-violation and agreement-violation cases (mirrors `break_edge_logic` rule 2/3, `amend_node_logic` rule 2). The `detail` string carries the kind-specific specificity.
- **Validator does NOT re-check structural shape.** Per ADR 0021, `parseWsEnvelope` runs `setEdgeSubstanceProposalSchema` at the API ingress. The validator can assume `edge_id` is a UUID, each present endpoint field is a UUID, and `role` (if present) is one of the seven enum values. The validator only checks **referential** and **symmetry** constraints.
- **Validator does NOT emit events.** It returns `RejectedValidationResult | null` (the established sibling shape). The dispatcher in `proposeHandler` checks the return and either short-circuits with the rejection or proceeds to the structural-event builder. The structural-event builder's behavior is unchanged.
- **Substance-only re-vote path stays valid.** A proposal with zero endpoint fields skips Phase 2 entirely; Phase 1's "if any present, all must be present" check passes trivially (the antecedent is false). The `proposeDefeaterPreCommit.test.ts` baseline test passes without modification.
- **Rule 2c (agreement-with-existing-edge) handles a corner case the structural-event builder's predicate would otherwise hide.** Today, a proposal that carries all three endpoint fields against an **extant** edge whose stored endpoints differ from the carried ones would be silently accepted (the structural-event builder's predicate fails on `getEdge !== undefined`, so no `edge-created` fires, and the proposal envelope is emitted as a substance-only re-vote — but with a payload that lies about the endpoints). Rule 2c rejects this case with `'illegal-state-transition'`. The check costs one `projection.getEdge()` call plus three field comparisons.
- **Role-source-target compatibility (rule 2d) is NOT implemented in v1 per D3.** Documented as an Open Question with rationale below; the v1 acceptance criteria stop at rule 2c.
- **Test coverage layered per ADR 0022.** Vitest at a new file `apps/server/src/methodology/handlers/proposeSetEdgeSubstanceValidation.test.ts` — one happy-path accept case plus one negative case per rule. The existing `proposeSetEdgeSubstanceEndpoints.test.ts` continues to cover the structural-event-builder behavior end-to-end (the validator's accept path is implicitly exercised there too).
- **This is NOT a UI-stream task.** Per the user-supplied scope clarification, no Playwright e2e is required. The UI-stream e2e policy does not apply; validator-level Vitest unit tests are the right coverage. The connecting-edge canvas spec activated by the parent commit (Playwright Scenario 2) stays green throughout (the happy-path seeds the validator's accept case) but is not extended.

## Decisions

- **D1 — Validator lives in the methodology engine (in `propose.ts`'s per-sub-kind switch), not the WS handler.** Per the user-supplied scope context and the source-of-debt refinement's "validator should live in the methodology engine (not the WS handler) so it composes with replay and with any non-WS callers of propose." The methodology engine is the single gate every transport (WS, future REST, replay) composes with; placing the validator at the engine layer means a non-WS caller — e.g. a replay-time re-application that runs the engine against a historical action, or a future REST `POST /sessions/:id/propose` endpoint — gets the same validation for free. Alternatives considered:
  - **Validator in the WS handler at `apps/server/src/ws/handlers/propose.ts`.** Rejected — couples the validation to one transport; a future REST propose endpoint or a replay-time re-run would duplicate the check. The methodology engine is the right seam.
  - **Validator in the structural-event builder itself.** Rejected — D2 of the parent refinement explicitly says the structural-event builder's job is wire-shape emission, not methodology validation; merging the two concerns in one function breaks the file's established two-section layering (validators at the top, builders at the bottom).
  - **Validator at the Zod schema layer via `.refine()` / `.superRefine()`.** Rejected — schema validation runs without a projection; rules 2a / 2b / 2c require the projection state. A schema refine could only encode the symmetry rule (Phase 1); placing half the validator at the schema layer and the other half at the engine layer splits the validator across two seams. The validator stays whole at the engine layer.

- **D2 — Reuse existing `RejectionReason` codes (`'target-entity-not-found'` + `'illegal-state-transition'`), no new code in v1.** Both codes line up semantically with the failure modes the validator catches. `'target-entity-not-found'` is already the union member used by `break_edge_logic` / `decomposition_logic` / `meta_move_logic` / `axiom_mark_logic` for "the named entity does not reference a record on the projection" — the source / target node existence checks (rules 2a / 2b) fit that semantic shape exactly. `'illegal-state-transition'` is used by the same siblings for "the projection state is in conflict with the requested action" — symmetry violations (a partial payload says nothing coherent) and agreement violations (the carried endpoints disagree with the projection's record of truth) both fit. The `detail` string is where the kind-specific specificity lives, exactly as the siblings already do (`break_edge_logic` D summary: "All three rules reuse the existing `'target-entity-not-found'` and `'illegal-state-transition'` codes ... the `detail` string carries the kind-specific specificity"). Alternatives considered:
  - **Mint `'source-node-not-found'` / `'target-node-not-found'` / `'role-incompatible'` per the parent refinement's D2 suggestion.** Rejected for v1 — adds three single-call-site values to the `RejectionReason` union; the existing codes carry the same semantic shape; the `detail` string discriminates on the wire. The parent refinement's suggestion was a sketch ("e.g. ..."), not a binding choice — the precedent set by `break_edge_logic` and others is to NOT mint new codes when existing ones fit. If a future consumer (e.g. a client-side error-rendering layer) needs to discriminate the three failure modes at the typed level rather than via `detail` parsing, the codes can be minted then with a clear consumer driving the choice. The principle is **YAGNI** + **drift-resistance**: every new union member is a new arm in every consumer's switch.
  - **Mint just `'endpoint-payload-asymmetric'` for the symmetry rule.** Rejected — same reasoning; `'illegal-state-transition'` covers the case and the `detail` is more informative than the typed code would be.

- **D3 — Role-source-target compatibility (rule 2d) is NOT implemented in v1; documented as a follow-up Open Question.** Per `docs/data-model.md` L147-153 the role-vs-kind compatibility patterns are documented as **coherency guidance — advisory, not blocking** ("The system never blocks; it nudges"). The methodology contract explicitly reserves role compatibility as a coherency-hint dimension, not a validator-rejection dimension. Promoting role compatibility to a hard rejection in v1 would contradict the design principle that the system surfaces unusual configurations without blocking them; an "unusual" rebuts-from-definitional-to-value (the example at L151) should produce a hint, not a `Rejected` from the propose validator. The proper home for role-compatibility logic is the `diagnostics.coherency_hint_detection` task (already on the WBS at [`tasks/refinements/data-and-methodology/coherency_hint_detection.md`](coherency_hint_detection.md)), which fires advisory diagnostics. The v1 validator stops at rule 2c; the role-compatibility check is flagged in Open Questions for the coherency-hint detection task's author. Alternatives considered:
  - **Implement a strict bridges-from / bridges-to compatibility check** (only allow `bridges-from` / `bridges-to` when the source is a warrant). Rejected — promotes a pattern documented as "coherency guidance" to a hard rejection, which contradicts the design principle. Bridges semantics are also evolving (per `docs/data-model.md` L124-135 the warrant-vs-non-warrant distinction is structural, not classification-based — a warrant is defined by carrying bridges-from / bridges-to edges, which is a chicken-and-egg problem for the first bridges-from / bridges-to edge proposed against a fresh warrant).
  - **Implement a permissive check** (e.g. reject `contradicts` between two `definitional` nodes). Rejected — same reasoning; the data-model docs do NOT pin any role pair as forbidden; they only enumerate "typical" and "unusual" patterns.
  - **Implement a configurable check** (a config flag that flips the strictness). Rejected — adds configuration surface for a behavior the design docs settle one way. The check stays absent until a methodology decision pins which pairs are forbidden vs. unusual.

- **D4 — Rule 2c (agreement-with-existing-edge) IS implemented in v1.** Unlike role compatibility (D3), agreement with an existing edge is a **structural** coherence check, not a methodology coherence check: a proposal that names different endpoints against the same `edge_id` is lying about the entity's identity, and entity identity is fixed at `edge-created` time per ADR 0027. There is no defensible reading under which a client should be allowed to propose `set-edge-substance edge_id=X source=A target=B role=supports` when the projection records edge X as `(source=C, target=D, role=rebuts)`. The check is cheap (one `getEdge` lookup the structural-event builder is already making, plus three field comparisons) and the failure mode it catches is silent and corrupting (a substance-only re-vote with a payload that lies about the endpoints). Alternative considered: defer to a downstream layer (e.g. trust that no client would ever send such a payload). Rejected — the parent refinement's D2 explicitly names this case ("(d) if the edge_id is already known, the carried endpoint fields must agree with the existing edge's endpoints") as a v1 requirement.

- **D5 — Two-phase structure (symmetry, then referential).** The symmetry phase runs entirely off the proposal payload — no projection lookup needed. The referential phase indexes the projection. Splitting the two means:
  - A partial-payload client gets a fast rejection with a specific error message (no projection lookup, no risk of a spurious "target not found" message when the real bug is a missing field).
  - The referential phase only runs when the payload has the all-or-nothing shape it needs.
  - Future maintenance: adding a new symmetry rule (e.g. if a future field joins the endpoint triple) lands in Phase 1; adding a new referential rule lands in Phase 2; the two are separable.
  Alternative considered: a single flat list of rules. Rejected — interleaves projection lookups with payload-only checks; the rule numbering becomes less legible (which rules need projection state? which don't?). The two-phase structure makes the seam visible.

- **D6 — Test layout: new sibling file `proposeSetEdgeSubstanceValidation.test.ts`, NOT extending the existing `proposeSetEdgeSubstanceEndpoints.test.ts`.** Per the sibling-naming convention (`proposeBreakEdge.test.ts`, `proposeAmendNode.test.ts`, `proposeMetaMove.test.ts`, etc. — one file per validator). The existing `proposeSetEdgeSubstanceEndpoints.test.ts` pins the **structural-event builder's** behavior end-to-end (its three cases seed valid projections so the builder is exercised); the new file pins the **validator's** behavior in isolation (its cases seed invalid projections to exercise each rejection rule). Keeping the two concerns in separate files matches the file's two-section layering decision the parent commit established. Alternative considered: extend the existing file with the negative cases. Rejected — would mix builder-cover and validator-cover in one file, making the test boundary less crisp; the sibling-naming convention exists precisely to keep per-validator coverage discoverable. (`proposeSetEdgeSubstanceEndpoints.test.ts` is itself misnamed under the sibling convention — should be `proposeSetEdgeSubstance.test.ts` — but renaming it is out of scope for this task; the new file's name picks up the "Validation" suffix to distinguish it from the endpoints / builder file.)

- **D7 — Validator runs before the structural-event builder in the dispatcher.** The proposal envelope construction at [`propose.ts:1199-1209`](../../../apps/server/src/methodology/handlers/propose.ts) calls `buildStructuralEventsForPropose` after the per-sub-kind validator returns. Adding the new `set-edge-substance` validator at the standard position in the switch (which short-circuits with `return` on rejection) means a rejected proposal never reaches the builder. This is the correct order: a malformed proposal should emit nothing (neither entity-layer nor facet-layer events). The order matches the existing siblings (`validateBreakEdgeProposal` runs before any commit-side application; `validateAmendNodeProposal` runs before the proposal envelope is built). Alternative considered: run the validator inside the builder (the builder rejects by returning an empty array). Rejected — the builder's return type is `EventToAppend[]`, not `ValidationResult`; the rejection would have to flow through a side channel; the layering would be confused.

## Open questions

- **Role-source-target compatibility (rule 2d) — strict vs. permissive vs. coherency-hint.** Per D3 the v1 validator does NOT enforce role compatibility; the design docs treat role-vs-kind patterns as advisory coherency hints, not blocking validator rules. The full set of "typical / unusual / forbidden" role-pair patterns is documented at `docs/data-model.md` L147-153 as advisory only. The proper home for role-pair logic is the `diagnostics.coherency_hint_detection` task ([refinement](coherency_hint_detection.md)); when that task lands, its author should decide whether any role-pair patterns rise to **forbidden** (validator-blocking) status, or whether all role-pair guidance stays at the **hint** (non-blocking, surfaced for moderator attention) layer. This task's v1 validator is permissive on roles; the strict reading is flagged for the coherency-hint task to revisit.

- **Whether to mint `'source-node-not-found'` / `'target-node-not-found'` / `'role-incompatible'` typed codes.** Per D2 the v1 validator reuses `'target-entity-not-found'` + `'illegal-state-transition'` without minting new codes. If a future client-side error-rendering layer or a future test-discrimination need surfaces a use case for typed discrimination at the union level (rather than via `detail` parsing), the codes can be minted then. The parent refinement's D2 suggestion of these three codes was a sketch ("e.g. ..."); the v1 choice follows the established sibling precedent of reusing existing codes.
