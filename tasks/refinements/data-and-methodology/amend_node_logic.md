# Amend-node logic — propose-side validator for the `amend-node` proposal sub-kind

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.amend_node_logic`
**Effort estimate**: 0.5d
**Inherited dependencies**: depends on `methodology_engine.agreement_state_machine` (the framework — `MethodologyAction`, `validateAction`, `RejectionReason`, `requireParticipant`, etc.). Indirectly: `methodology_engine.decomposition_logic`, `interpretive_split_logic`, `axiom_mark_logic`, `meta_move_logic`, `reword_vs_restructure`, and `break_edge_logic` (the prior six propose-arm tightenings — their factoring (`validateXProposal(projection, action)` locals dispatched from `propose.ts`'s switch), their `nodeIsVisible` reuse, their conflict-walker pattern, and their structural-shape-is-upstream's-concern layering decision are the templates this task extends). Concretely settled: the Zod `amendNodeProposalSchema` in `packages/shared-types/src/events/proposals.ts` (enforces structural shape — `{ kind: 'amend-node', node_id: UUID, new_content: string min 1 }`); the projection's read-side `replay.ts/applyCommittedProposal` `amend-node` arm which mirrors reword semantics on commit (in-place wording update, lines 722–748); the `nodeIsVisible` and `findConflictingProposalAgainst` primitives from prior arms.

## What this task is

Tighten the `propose` handler's `amend-node` arm with the real methodology-engine validator. The propose handler currently dispatches `decompose`, `interpretive-split`, `axiom-mark`, `meta-move`, `edit-wording`, and `break-edge` to their real validators and lets the remaining five sub-kinds — including `amend-node` — fall through to the universal-pass placeholder path. This task replaces the fall-through for `amend-node` with four methodology-specific rules, in evaluation order:

1. The `node_id` references a node that exists in this session's projection.
2. The node is currently visible in the projection (`visible === true`). A not-visible node — already superseded by a prior decompose / interpretive-split / restructure per the visible-graph derivation in `docs/data-model.md` lines 273–285 — can't be amended: the wording change would attach to a node nobody can see.
3. No other proposal among the four node-touching structural sub-kinds (`decompose`, `interpretive-split`, `edit-wording`, `amend-node`) is currently pending against the same `node_id`. This task **extends** `CONFLICTING_PARENT_KINDS` to include `'amend-node'` and the `ConflictingParentKind` union accordingly. All four arms share the conflict-walker.
4. The node is currently a party (source or target) to a visible `contradicts` edge whose `substanceFacet` has been agreed (status `'agreed'` or `'committed'`, value `'agreed'`). This is the **strict** reading of docs/methodology.md line 219: amend-node is the contradiction-resolution path; without an agreed contradiction there is nothing to resolve.
5. The structural payload shape — `kind: 'amend-node'`, `node_id: UUID`, `new_content` non-empty — is already enforced upstream by the Zod schema (`amendNodeProposalSchema`, ADR 0021). The validator relies on that layering and does not re-check.

On `Valid` the handler emits exactly one `EventToAppend` of kind `proposal` whose payload is `{ proposal: action.proposal }`. On any rule failure the handler returns a typed `Rejected` with a `RejectionReason` that names the specific failure.

**Amend-node vs reword: the methodology distinction.** Both `amend-node` and `edit-wording(reword)` update a node's wording in place. The structural effect on commit is identical — `replay.ts/applyCommittedProposal`'s `amend-node` arm (lines 722–748) is a near-mirror of its `edit-wording(reword)` arm (lines 569–602). The difference is **intent**, not structure:

- **Reword** is the routine clarification path — the captured wording was imprecise; we agree on what was meant and tighten the phrasing.
- **Amend-node** is the contradiction-resolution path — the captured wording, taken at face value, implies a contradiction with another node; we agree to amend ONE side of an agreed `contradicts` edge to remove the conflict.

`docs/methodology.md` settles the distinction in two passages:

- Line 219 (in "Resolution of structural diagnostics → Blocking diagnostics → Contradiction"): "Decompose one or both nodes (most common — the apparent contradiction reveals that the two statements are about different things, or each is compound and the conflict is between specific components). **Amend one node so the conflict no longer holds.** The relevant participants each axiom-mark the position they hold; the `contradicts` edge stays."
- Line 74 (in "Kinds of dispute → Substance dispute on an edge"): "(`the contradiction doesn't actually hold` / `the data doesn't actually support the claim`) → propose a resolution: decompose, **amend**, mark as bedrock disagreement, or let it stay `disputed`."

And `docs/data-model.md` line 255 reinforces it: "`amend-node` — proposes changing a node's content beyond a wording edit (used in contradiction resolution). Payload: node, new content."

The schema-side commentary in `proposals.ts` line 240 is candid: "Distinct from `edit-wording.reword`: amend is the methodology-driven repair op; reword is the participant-driven wording tweak. (Whether these collapse into one is a methodology-engine concern; the schemas remain distinct here.)" — that decision has now been made: the methodology engine **does** keep them distinct, by enforcing rule 4 at the validator layer.

Scope is **propose-side only**. The commit-time structural application — the in-place wording update — already lives in `replay.ts/applyCommittedProposal`'s `amend-node` arm and has been complete since the projection landed. The methodology engine's commit-side gate is enforced by `commit_logic`'s existing rules.

## Why it needs to be done

Without this task the placeholder propose handler accepts amend-node proposals against:

- non-existent nodes (the projection's commit-side would later throw `ReplayError("commit/amend-node: node ${node_id} not present")`);
- already-superseded nodes (the commit would mutate `wording`/`wordingFacet` on an invisible node — meaningless work);
- nodes with another structural proposal (decompose / interpretive-split / edit-wording / amend-node) already pending — racing against the same wording / visibility dimension on commit;
- nodes that have no contradiction to resolve — collapsing the methodology distinction between reword and amend-node, and letting participants reach for the "contradiction-resolution" frame for routine wording cleanups.

The first three failures mirror the symmetric checks the sibling propose-arms enforce. The fourth is what makes `amend-node` semantically distinct from `edit-wording(reword)` at the validator layer — without it, the only thing distinguishing the two sub-kinds in practice would be the literal `kind` string on the wire, and participants would naturally use whichever name felt more weighty. Rule 4 pins the methodology distinction at the API boundary.

## Constraints / requirements

- The handler **does not write events**; it returns a `ValidationResult`. On `Valid` it emits exactly one `EventToAppend` of kind `proposal`.
- The handler **does not mint timestamps or ids**; the API layer mints `eventId` and `createdAt` before calling the engine.
- The handler **does not call `validateEvent`**; the API layer runs the structural validator (Zod) separately (ADR 0021). The methodology validator can rely on `node_id` being a valid UUID and `new_content` being non-empty.
- **No new `RejectionReason` value.** Rules 1, 2, and 3 reuse `'target-entity-not-found'` and `'illegal-state-transition'` exactly as the sibling propose-arms do. Rule 4 reuses `'methodology-not-exhausted'` (originally minted by `meta_disagreement_logic`) — the semantic shape lines up: the methodology hasn't reached the state where the contradiction-resolution path is appropriate yet. Alternatives considered: `'illegal-state-transition'` (the projection state isn't *illegal* — it's that this specific operation is the wrong tool for the current state) and minting a fresh `'no-contradiction-to-resolve'` reason (clearer but adds a single-call-site value to the union — not worth it for one rule).
- Add a primitive `nodeIsPartyToAgreedContradicts(projection: Projection, nodeId: string): boolean` in `primitives.ts`. Walks `projection.edges()`. For each visible edge whose `role === 'contradicts'` and whose source or target equals `nodeId`, inspects `substanceFacet.status` (must be `'agreed'` or `'committed'`) AND `substanceFacet.value === 'agreed'`. Returns `true` on the first match; `false` otherwise.
- Extend the node-scoped conflict walker: add `'amend-node'` to `ConflictingParentKind` and to `CONFLICTING_PARENT_KINDS`; teach `findConflictingProposalAgainst` to recognize the `amend-node` payload (target field is `node_id`, same as `edit-wording`).
- Symmetric updates to existing tests: `proposeDecompose.test.ts`, `proposeInterpretiveSplit.test.ts`, and `proposeEditWording.test.ts` each gain one new test case asserting the cross-kind conflict against a pending amend-node.
- Don't pre-empt the other proposal sub-kinds. `amend-node` is the seventh arm tightened; the remaining four (`classify-node`, `set-node-substance`, `set-edge-substance`, `annotate`) stay on the placeholder path until their own sibling tasks land.
- Verifications per ADR 0022: Vitest at `apps/server/src/methodology/handlers/proposeAmendNode.test.ts`; Cucumber + pglite scenarios at `tests/behavior/methodology/propose-amend-node.feature` with step defs in `tests/behavior/steps/methodology-propose-amend-node.steps.ts`.

## Decisions

- **Scope: propose-side only.** This task validates `propose amend-node`. The commit-time structural application — the in-place wording update — already lives in `replay.ts/applyCommittedProposal`'s `amend-node` arm and has been complete since the projection landed.
- **Rule set (numbered, in evaluation order).**
  1. **Node-exists.** → `'target-entity-not-found'`.
  2. **Node-visible.** → `'illegal-state-transition'`.
  3. **No conflicting decompose / interpretive-split / edit-wording / amend-node pending.** → `'illegal-state-transition'`.
  4. **Node is party to agreed contradicts edge.** → `'methodology-not-exhausted'`.
  5. (Structural shape — enforced upstream by Zod.)
- **Amend-node vs reword: distinct sub-kinds, distinct intents, identical structural effect on commit.** The methodology engine keeps them distinct by enforcing rule 4 at the validator layer. Reword is the routine clarification path (`docs/methodology.md` line 157+ — "the edit produces a clearer phrasing of the same statement"); amend-node is the contradiction-resolution path (`docs/methodology.md` line 219 — "amend one node so the conflict no longer holds"). Without rule 4, the only thing distinguishing the two on the wire would be the `kind` string and participants would naturally reach for whichever name felt more weighty. Rule 4 pins the methodology distinction at the API boundary. Per `proposals.ts` line 240's note, "Whether these collapse into one is a methodology-engine concern; the schemas remain distinct here." — that decision has now been made.
- **Strict reading of rule 4 chosen over permissive.** Two readings of the methodology docs were considered:
  - **Strict** (chosen): the propose-side validator rejects amend-node when the target node is not currently a party to a visible `contradicts` edge whose substance facet is `agreed` or `committed`. Forces participants to use `edit-wording(reword)` for routine wording cleanups, reserving `amend-node` for actual contradiction resolution. Cited support: `docs/methodology.md` line 219 names amend-node specifically as one of the three contradiction-resolution paths (alongside decompose-one-or-both and accept-as-bedrock); `docs/data-model.md` line 255 says "used in contradiction resolution." Both passages frame amend-node as a methodology-driven repair operation, not a generic wording op.
  - **Permissive**: amend-node is just like reword but with different intent labeling. The propose-side validator wouldn't enforce the contradiction-must-exist rule — participants self-discipline on which sub-kind to reach for. Defensible if the docs were silent on enforcement, but they aren't: the consistent framing across both `methodology.md` and `data-model.md` is that amend-node is a contradiction-resolution operation, not a generic one.
  The strict reading is what's implemented. The trade-off: a participant who wants to clean up wording will sometimes get a `'methodology-not-exhausted'` rejection and have to re-issue as an `edit-wording(reword)`. The API layer (eventually — backend.api_skeleton or a frontend hint) might surface a "did you mean reword?" suggestion when this happens; that's a UX concern downstream of the validator, not a validator concern. Flagged in Open Questions.
- **`'methodology-not-exhausted'` for rule 4 (not a new RejectionReason value).** The reason was minted by `meta_disagreement_logic` for "the methodology hasn't reached the state where this resolution path is appropriate yet." Amend-node-without-an-agreed-contradiction matches that shape: the participant is reaching for the contradiction-resolution path before there is a contradiction to resolve. Alternatives:
  - `'illegal-state-transition'` — the existing structural reason. *Rejected.* The projection state isn't *illegal*; it's that this specific methodology operation is the wrong tool for the current state. Conflating the two would lose the distinction between "you've broken a rule" and "you've reached for the wrong tool." `'methodology-not-exhausted'` is the right semantic shape.
  - Mint a fresh `'no-contradiction-to-resolve'` reason. *Rejected.* Adds a single-call-site value to the `RejectionReason` union for one rule. Reusing the existing reason keeps the union stable; the `detail` string carries the kind-specific specificity ("amend-node is the contradiction-resolution path; use edit-wording(reword) for routine wording clarifications").
- **Conflict-walker extension: add `'amend-node'` to the existing node-scoped walker, not a separate primitive.** The four node-touching structural sub-kinds (`decompose`, `interpretive-split`, `edit-wording`, `amend-node`) all compete on the same node — at most one may be pending at a time. Treating them uniformly under `findConflictingProposalAgainst` keeps the design symmetric. The walker normalizes the target-id field name across the three target-naming patterns (`parent_node_id` for decompose / interpretive-split; `node_id` for edit-wording / amend-node). The walker now matches the same shape as before with one additional `payload.kind === 'amend-node'` branch reading `payload.node_id`.
- **Substance-facet check: status `'agreed'` OR `'committed'`, value `'agreed'`.** Per the `FacetStatus` widening in `projection/types.ts` lines 37–43, the agreement-layer values are `'proposed' | 'agreed' | 'disputed' | 'meta-disagreement'`; `deriveFacetStatus` may produce `'committed'` and `'withdrawn'` on top. A contradicts edge's substance facet that has been committed lives at status `'agreed'` (the dispatcher writes that on commit) and may be widened to `'committed'` by the derivation. Both should satisfy rule 4 — they both mean "the substance has been agreed and committed." `'disputed'` and `'withdrawn'` should fail (the contradiction isn't agreed); `'proposed'` should fail (no commit yet); `'meta-disagreement'` should fail (the substance is irreducibly contested — there's no agreed contradiction to resolve, and amend-node would be premature).
- **Source OR target of the contradicts edge.** The contradicts edge is directional in the data model (per `docs/data-model.md` line 120: "Directed. If a contradiction is genuinely symmetric (each rules out the other in the same way), it is represented as **two** `contradicts` edges in opposite directions") but the methodology distinction is symmetric — if A contradicts B and the participants agree to amend B, that's just as legitimate as amending A. Rule 4 checks both `sourceNodeId === nodeId` and `targetNodeId === nodeId`.
- **Where the handler lives.** `propose.ts` itself, factored locally as `validateAmendNodeProposal(projection, action)`. Mirrors the sibling validators' choice. If `propose.ts` grows past ~1000 lines as the remaining four sub-kinds tighten, factor into per-sub-kind files.
- **Structural shape is upstream's concern.** The Zod schema in `amendNodeProposalSchema` enforces all structural constraints. Per ADR 0021 the validator runs at the API ingress before the methodology engine sees the action. The methodology validator relies on this layering and does not re-check.
- **`nodeIsVisible` reuse.** Already landed by `decomposition_logic`. Same predicate, same semantics — the rule-2 invariant is symmetric to the decompose / interpretive-split / edit-wording cases.
- **Test layout.** Vitest at `apps/server/src/methodology/handlers/proposeAmendNode.test.ts` (new file — same naming pattern as the sibling `proposeXxx.test.ts` files). Cucumber + pglite at `tests/behavior/methodology/propose-amend-node.feature` with step defs in `tests/behavior/steps/methodology-propose-amend-node.steps.ts`. Reuses `event-rows.ts` and the shared Then / When steps. The step file uses a distinct UUID prefix (`c4...`) to avoid collisions with the sibling step files.
- **Symmetric test updates.** Each of the existing `proposeDecompose.test.ts`, `proposeInterpretiveSplit.test.ts`, and `proposeEditWording.test.ts` gains one new test case asserting the cross-kind conflict against a pending amend-node. Same pattern as when `edit-wording` was added by `reword_vs_restructure` — the conflict-walker symmetry is what matters; the existing arms must reject when an amend-node is pending against the same node.

## Open questions

- **Reword-fallback hint in the API / UI layer.** The strict rule 4 will reject amend-node proposals against nodes with no agreed contradiction. The participant's natural next step is to re-issue as `edit-wording(reword)`. The validator's `detail` string already names this ("use edit-wording(reword) for routine wording clarifications"), but the API layer / moderator UI may want to surface a "did you mean reword?" suggestion when the rejection lands. That UX concern is downstream of the validator — `backend.api_skeleton` or one of the UI tasks (`moderator-ui`) is the right place to settle it. Flagged here for that task's author.
- **Commit-time structural-sub-kind support.** The decompose / interpretive-split / axiom-mark / meta-move / edit-wording / break-edge arms have flagged commit-side gaps where `commit_logic`'s rule 4 still rejects commits of structural sub-kinds with `'illegal-state-transition'` despite the projection arm being complete. The same observation applies to amend-node: `replay.ts`'s `amend-node` arm updates wording correctly, but a `commit` action against an amend-node proposal is currently rejected by `commit_logic` for being structural. Resolving the gap is the shared cross-sub-kind follow-up; this task does not settle it.
- **Substance-facet derivation cross-check.** Rule 4 reads `substanceFacet.status` directly. The widening to `'committed'` lives in `deriveFacetStatus` (a read-side derivation). When `replay.ts` commits a `set-edge-substance` proposal it writes `status = 'agreed'` (the agreement-layer value). For most projections the rule-4 check sees `'agreed'`; the `'committed'` widening shows up only when downstream code calls `deriveFacetStatus`. Including `'committed'` in the rule-4 acceptance set future-proofs against any future commit-path rewrite that writes the wider value directly. Documented as a defensive choice; no test exercises the `'committed'` arm today (the writer-side never produces it) but the predicate handles it correctly.

(All other questions settled.)
