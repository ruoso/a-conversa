## What this task is

Land the canonical end-to-end integration test that proves the projection materialises the example-walkthrough's final visible-graph state from the event log alone — no per-test event construction, no methodology-engine cheating, no out-of-band state.

Concretely the deliverable is two artefacts:

1. **The `walkthrough` test fixture** at [`packages/test-fixtures/src/fixtures/walkthrough/`](../../../packages/test-fixtures/src/fixtures/) — the deferred fixture from [`seed_data_for_tests.md`](./seed_data_for_tests.md)'s Status block. Files mirror the `empty` fixture's shape (`meta.json`, `users.json`, `session.json`, `participants.json`, `events.json`); contents encode the 22-turn debate from [`docs/example-walkthrough.md`](../../../docs/example-walkthrough.md) as a deterministic event log. UUIDs are stable (per the `seed_data_for_tests` R21 contract); the canonical identifiers (`Anna`, `Ben`, `Maria` participants; N1–N19 nodes; E1–E15 edges; A1–A3 annotations; "Segment 1 close" snapshot label) are mapped to stable test-fixture UUIDs via a header comment in `meta.json`.

2. **Cucumber scenarios** at [`tests/behavior/projection/walkthrough-replay.feature`](../../../tests/behavior/projection/) — a dedicated feature file (D3) sibling to `from-log.feature` (NOT appended to it; the walkthrough's assertion surface is too large to share a file without burying the simpler `empty`-fixture scenarios). The feature loads the `walkthrough` fixture via the existing `loadFixture` step, calls `projectFromLog` against the loaded event log (per the existing `from-log.feature` Background pattern), then walks a declarative assertion table that pins the walkthrough's coda checklist: committed entities, disputed-or-live entities, per-participant axiom-mark matrix, and the snapshot's existence.

Per D5 the test exercises ONE comprehensive scenario plus a small number of focused supplemental scenarios:

- **Scenario 1 — Full walkthrough projects to canonical final state.** The load-bearing scenario. Loads the fixture, projects, walks the coda checklist as a table of `Given the projection / Then node <id> classification facet is <status>` style assertions. This is the contract that the walkthrough's stated outcome IS the projection's computed outcome.
- **Scenario 2 — Per-participant axiom-marks on N12.** Carved out because the per-participant axiom-mark surface is unique to this walkthrough (turn 16 — both Anna and Ben mark N12) and warrants its own focused assertion (`getNode(N12).axiomMarks` contains both participant ids; the marks are independent agreements per `axiom_mark_logic`).
- **Scenario 3 — Defeater pre-commit + deferred edge activation.** Carved out because turn 11 (E5 = N8 rebuts N6, edge substance pre-committed agreed while N8's substance stays proposed) is the canonical example of "edge substance and node substance commit independently"; the walkthrough is the integration cover that `active_firing_computation` + `set_edge_substance_endpoint_validation` interact correctly at scale.
- **Scenario 4 — Disputed entities stay disputed.** The walkthrough's coda lists A2 (Anna's reframe), N17 (metaphysical reading), and the contradicts-edge E15 (N19 contradicts A2) as disputed-or-live. Asserts each via `deriveFacetStatus` returning the non-`'committed'` status the walkthrough's narrative demands.
- **Scenario 5 — Snapshot-created at "Segment 1 close" lands.** Pins the snapshot event is encoded and that `projection.snapshotCount` / equivalent reflects it. (The exact assertion API depends on what projection exposes for snapshots; see open question Q1 in the original list — D7 settles it inline.)

The fixture is the harder of the two artefacts; the Cucumber file is mostly mechanical once the fixture is right.

## Why it needs to be done

Three reasons compose:

**The walkthrough IS the load-bearing demonstration of the platform's procedure.** [DESIGN.md L39](../../../DESIGN.md) names [`docs/example-walkthrough.md`](../../../docs/example-walkthrough.md) as "the simulated debate ... [that] exercises the platform's procedure end-to-end." Today that demonstration is prose; an authoritative test pins the projection's output against the documented outcome. Without it, drift between the walkthrough's narrative and the engine's actual behaviour goes silently undetected — a regression in any of the ~10 methodology operations the walkthrough composes would not be caught at the cross-cutting level until a participant noticed in production. With this task, every PR that touches the projector / methodology engine runs the walkthrough scenario; a divergence between the doc and the code fails CI.

**It is the canonical "substantial debate" fixture for downstream tests.** [`seed_data_for_tests.md`](./seed_data_for_tests.md) line 51 pins "the walkthrough fixture is canonical — every other test that needs 'a substantial debate' uses this one." Without it, every downstream test that needs a non-trivial projection invents its own fixture (multi-warrant detection, cross-session reference, replay-from-position, audience-broadcast at scale — all listed in the same Status block as deferred-pending-walkthrough). Landing the fixture unblocks the rest of the seed-data deferrals AND gives every Playwright / Cucumber test a realistic starting state to build on.

**It closes the integration gap between unit-tested operations and a real end-to-end procedure.** The per-operation Vitest cover (`decomposition_logic.test.ts`, `interpretive_split_logic.test.ts`, etc.) and per-mechanic Cucumber feature files (`commit.feature`, `withdraw-agreement.feature`, etc.) each pin their isolated operation. Composing them into a 22-turn debate — with proposals depending on earlier commits, with per-participant axiom-marks accumulating, with defeater edges firing only once both endpoints' substances commit — is the integration concern none of the per-operation tests covers. The walkthrough replay is that integration cover at the projection layer.

Downstream consumers benefit transitively:

- [`data_methodology_tests.dm_behavior_tests.scenarios_*`](../../10-data-and-methodology.tji) (lines 456-493 — pending) — those nine focused Cucumber scenarios cover individual mechanics; the walkthrough is the at-scale companion. Once both exist, the layered cover is: per-event Vitest (`*_unit`) → per-mechanic Cucumber (`scenarios_*`) → end-to-end Cucumber (this task).
- [`data_methodology_tests.dm_e2e_tests.projection_invariants_e2e`](../../10-data-and-methodology.tji) (line 501 — pending) — property-based tests benefit from a non-trivial seed projection.
- [`replay_test.*`](../../60-replay-and-test-mode.tji) (pending) — replay-at-position queries need a long event log with a snapshot in the middle; the walkthrough is exactly that.
- [`audience.*`](../../50-audience-and-broadcast.tji) (pending) — audience surfaces need a "looks like a real debate" projection to render; the walkthrough is the canonical such projection.

## Constraints / requirements

- **The fixture is authored to match the walkthrough's coda checklist field-for-field.** Every committed / disputed entity the coda names must have an unambiguous fixture-id mapping. The fixture's `meta.json` carries a header comment ("// walkthrough identifiers — N1=…, N2=…, …") so future readers can correlate the walkthrough's narrative ids to the fixture's UUIDs.
- **Fixture is deterministic.** All UUIDs are stable strings of the form `00000000-0000-4000-8000-<hex>` (matching the `evId(n)` convention from `replay.test.ts` L57-72 and the `seed_data_for_tests` R21 contract). All timestamps are predictable ISO-8601 strings, monotonically increasing along the event sequence. A second run of the fixture loader produces an identical projection (per `seed_data_for_tests` R21 + R22).
- **Fixture events are valid per ADR 0021's `validateEvent`.** Each event record satisfies the per-kind Zod schema; the fixture loader's JSONB → typed-Event pass succeeds on every record. (The loader's existing `// TODO(R23):` validation gap — events go in via raw INSERT today — is acknowledged in D6 below; the fixture's events are AUTHORED to satisfy the schemas even though enforcement is deferred to R23.)
- **The full walkthrough is encoded — not a slim summary.** Authoring shortcut "encode only the operations the coda asserts on" is rejected: the coda's "live segment 1 close" implicitly requires every prior turn's events to be in the log for `projectFromLog` to reach the asserted state. The fixture encodes all 22 turns end-to-end.
- **The snapshot-created event lands at the correct log position.** Per L211 ("event log position recorded") the snapshot's payload includes the sequence number of the last event before it; this must be the sequence of turn 22's final commit (or whichever event the walkthrough's segment-close marks).
- **No new event kinds, no schema changes.** The walkthrough is authored against the EXISTING event schema; if a turn's behaviour cannot be expressed with the existing event kinds, that gap is registered as a future-task per the tech-debt registration section below — NOT papered over with a new event kind in this task.
- **Per-participant axiom-marks on N12 (turn 16) are encoded as two distinct `axiom-mark` proposals + commits.** Per `axiom_mark_logic` the per-participant axiom-mark is one event per participant marking — the walkthrough's two marks are two separate proposal-then-commit pairs, not a single multi-participant event.
- **Defeater pre-commit (turn 11 + 13 + 17) is encoded as edge substance pre-committed agreed while target node substance stays proposed.** Per `set_edge_substance_endpoint_validation` + `defeater_capture_logic` the edge substance can be agreed before the target node's substance — the walkthrough exploits this for E5 (N8 rebuts N6, edge agreed but N8 substance pending), E10 (N13 rebuts N11, same shape), E11a (N16 rebuts N11), E11b (N17 rebuts N11). The fixture must encode each defeater's edge-substance commit BEFORE the target node's substance commit (or hold it pending per the walkthrough's narrative).
- **Cucumber scenarios run on pglite per ADR 0007.** No real Postgres; per-scenario migration + truncate via the existing `tests/behavior/pglite/` harness.
- **Test discipline per ADR 0022.** Vitest delta and Cucumber delta are observable on `pnpm run test:smoke`. No verification step lives outside committed test code; the fixture-author's check that the projection reaches the coda state IS this committed test.
- **No new ADR.** Every architectural input is settled.
- **No new Playwright cover.** Per the UI-stream e2e policy at [tasks/refinements/README.md](../README.md) UI-stream tasks get Playwright cover; this task is under `data_and_methodology.*` (backend / projection) NOT under `moderator_ui.*` / `participant_ui.*` / `audience.*` / `replay_test.*`. The Cucumber scenario IS the right e2e level.

## Decisions

- **D1 — Cucumber + pglite (NOT Vitest, NOT Playwright).** This task is bucketed under `dm_e2e_tests` ("End-to-end / integration tests (no UI)"). Per ADR 0007, integration / DB-touching tests run on Cucumber + pglite. Rationale:
  - **The fixture's value is realised through the loader → DB → `projectFromLog` integration path.** A pure-Vitest test could load the fixture's `events.json` directly and call `projectFromLog` against the in-memory array — but that path bypasses the JSONB → typed-Event boundary (where `validateEvent` runs per ADR 0021) and the `loadFixture` seam (which is itself the fixture-correctness contract). Routing through pglite makes the test exercise the same code path that production reads share at session-rehydration time.
  - **The existing `from-log.feature` is the precedent.** Its header note at L1-13 articulates the exact framing: "events round-tripped through pglite's `session_events` table — JSONB-encoded payloads, TIMESTAMPTZ-formatted timestamps, BIGINT sequences — read back out, mapped to typed `Event` envelopes via `validateEvent`." The walkthrough's replay scenario is the at-scale extension of from-log.feature's empty-fixture scenario.
  - **Playwright is the wrong layer.** This task is not UI-stream (per the refinements README's bucket convention — `data_and_methodology.*` ≠ UI-stream); no rendering surface is involved; no user flow is exercised. The projection is observable purely at the server layer.
  - **The existing 12-event Vitest walkthrough at `replay.test.ts:416-507` stays.** It is a smaller, faster, in-memory cover; it is NOT replaced or made redundant by this task. The two walkthroughs cover different layers: Vitest pins the projector arithmetic, Cucumber pins the DB-round-trip + at-scale composition. Per the [refinements README test discipline](../README.md) (ADR 0006 / 0007 / 0022 cite) both layers are healthy at the same time.
  - **Alternative considered: pure Vitest at `apps/server/src/projection/replay.test.ts`.** Rejected — bypasses the JSONB integration surface, leaves `validateEvent` uncovered on the walkthrough's payloads, does not give the fixture its on-disk → DB cycle. The benefit (faster test) does not outweigh the cost (lost integration cover).
  - **Alternative considered: Playwright spec under `tests/e2e/`.** Rejected — Playwright is the UI/browser layer; no UI is involved here. Mounting a browser to exercise a backend-only contract would be a category error.

- **D2 — Fixture authored as JSON-by-hand against canonical schemas, not generated via a methodology-engine harness.** Rationale:
  - **Determinism + diffability.** Per `seed_data_for_tests` R21 fixtures live in version control as plain JSON; a generated fixture rebuilds the file on every change and breaks diffs.
  - **Fixture-authoring catches schema divergence early.** If a methodology operation's payload schema drifts, the fixture's event records start failing `validateEvent` — the test fails loudly, not silently with a regenerated fixture. The pain of hand-authoring IS the early-warning system.
  - **Generation requires the methodology engine to be runnable in fixture-author mode.** That capability does not exist today and adding it for this task is scope creep.
  - **Alternative considered: write a "walkthrough.ts" script that drives the methodology engine to produce the events, then snapshot the output as JSON.** Rejected — adds a generator path to maintain; brittle to engine-API changes; the schema-divergence early-warning is lost.

- **D3 — Dedicated `walkthrough-replay.feature` file, NOT appended to `from-log.feature`.** Rationale:
  - **The walkthrough's assertion surface is large** (5 scenarios, ~50 declarative assertions, ~200 events in the fixture). Appending to `from-log.feature` would bury its existing `empty`-fixture and DB-round-trip scenarios under the walkthrough's bulk; future readers searching for the empty-fixture cover would have to scroll past the walkthrough.
  - **The walkthrough is the load-bearing fixture per `seed_data_for_tests` R51.** Giving it a dedicated feature file signals its canonical status in the test surface.
  - **`from-log.feature`'s header note specifically frames it as "events round-tripped through pglite ... read back out, mapped to typed `Event` envelopes ... and replayed."** That is the layer's contract; both `from-log.feature` and `walkthrough-replay.feature` honor it. The split is on fixture identity (which fixture is loaded), not on layer.
  - **Alternative considered: append the walkthrough as scenarios in `from-log.feature`.** Rejected — buries the existing scenarios; the file would become 500+ lines; reading it would no longer convey the layer's contract concisely.
  - **Alternative considered: nest under `tests/behavior/walkthrough/walkthrough-replay.feature` as its own directory.** Rejected — the layer is "projection" not "walkthrough"; the directory split should remain by-layer for the existing Cucumber discipline.

- **D4 — Coda assertions read through `deriveFacetStatus` + `Projection` accessors, NOT through `session_events` table reads.** Per ADR 0027 (entity and facet layers strictly separate) the assertion target is facet state. The walkthrough's coda is a graph-state checklist, not an event-log checklist. Mechanically: each Gherkin step like `Then the node "N2" classification facet is "committed"` resolves to `expect(deriveFacetStatus(projection, 'node', n2Id, 'classification')).toBe('committed')`.
  - **Alternative considered: assert by counting events of each kind in the log.** Rejected — that is the structural-event-timing seam ADR 0027 names as a violation. The facet status is the correct seam.
  - **Open question handled inline — how to distinguish "disputed" from "proposed" in the coda.** The walkthrough names some entities as "live / disputed" (A2, N17) and some as "agreed / committed but referenced in disputed edges" (N19). The fixture's narrative position determines which status `deriveFacetStatus` returns; the Gherkin step accepts a parameter (`Then the node "N17" classification facet is "<status>"`) and the implementer writes the asserted status based on the fixture's actual computed state. If the asserted status diverges from the walkthrough's narrative intent, that surfaces as a fixture-authoring bug, not a projector bug.

- **D5 — Five scenarios instead of one monolithic scenario.** Rationale:
  - **Diagnostic readability.** A single-scenario failure ("Scenario X failed at line 47 of 312") is much harder to triage than five focused-scenario failures.
  - **Each scenario carries a distinct contract.** Scenario 1 is the full-state checklist; Scenario 2 is per-participant axiom-marks; Scenario 3 is defeater pre-commit + deferred edge activation; Scenario 4 is disputed-entity persistence; Scenario 5 is snapshot presence. Each surfaces a distinct regression class.
  - **Background-and-fixture-load runs ONCE per scenario in Cucumber.** Five scenarios means five fixture loads; this is acceptable given pglite's per-scenario isolation (each scenario starts from a clean DB anyway). The wall-clock cost is bounded by the fixture's size (~200 events × 5 = ~1000 inserts ≈ sub-second on pglite).
  - **Alternative considered: ONE scenario with all assertions.** Rejected — diagnostic readability is lost; a single line failure on the 47th assertion gives no information about whether the surrounding 46 assertions still hold.
  - **Alternative considered: ONE scenario per coda assertion (~50 scenarios).** Rejected — fixture-load amortization vanishes (50 fixture loads), and the diagnostic value is poor (each scenario is one line of assertion).

- **D6 — This task does NOT block on R23 (replay-through-append-API loader rewrite).** Per `seed_data_for_tests`' Status block R23 is deferred until `event_validation` + `backend.api_skeleton` land. The walkthrough fixture is authored to satisfy the schemas TODAY; when R23 lands, the same fixture will round-trip through the append API without changes (the Zod schemas are the same on both paths). Rationale:
  - **The walkthrough's value (load-bearing end-to-end test) does not depend on R23.** The current truncate-then-INSERT loader produces the same projection as the future append-through-API loader, assuming the fixture's events are valid (which D2 enforces).
  - **Authoring the fixture forces the schemas to be exercised at scale.** That is itself a value-add — if a schema gap exists, the fixture-authoring discovers it.
  - **Registering `r23_loader_replay_through_append_for_walkthrough` as future-task** (per the tech-debt registration above) names the follow-up explicitly; the closer files it in the WBS.

- **D7 — Snapshot assertion via the projection's snapshot-tracking surface; if that surface does not exist, fall back to event-log read-back.** Rationale:
  - **The projection's snapshot-tracking surface is the cleanest assertion target.** If `Projection` exposes `getSnapshots()` / `lastSnapshotPosition` / similar, the assertion is a one-liner.
  - **If no such surface exists today (the snapshot-created event handler may simply mutate projection state without exposing it), the assertion falls back to "the `walkthrough` fixture's `events.json` contains exactly one `snapshot-created` event with label 'Segment 1 close' at sequence N."** This is a weaker assertion (it pins the fixture, not the projection's read of it), but it is the minimum viable cover.
  - **Implementer discovers which path applies during implementation;** if the fallback is needed, register `projection_snapshot_query_surface` as a future-task naming the gap.
  - **Alternative considered: write a `projection.getSnapshots()` API in this task.** Rejected — scope creep. The snapshot-query surface is its own concern; if missing, the gap is registered as a future-task; this task does not block on adding it.

- **D8 — Turn 17's reframe is encoded as an `annotate` proposal with `kind: 'reframe'`, NOT as a `meta-move` proposal.** Per the walkthrough's narration at L168-175 the reframe is captured as an annotation (A2) AND the meta-move is the meta-level act of repositioning N6's normative claim. The doc's own framing is "Anna's reframe is captured as an annotation; Maria leaves it alone." Rationale:
  - **The annotation-as-reframe surface is what the walkthrough's narrative names.** Encoding it as a `meta-move` proposal would over-claim — meta-moves are first-class methodology operations per [`docs/methodology.md`](../../../docs/methodology.md); annotations of kind `reframe` are participant commentary. The walkthrough is the latter.
  - **The `meta-move` proposal kind IS still exercised** — but only via the scenarios_meta_move_contested sibling task's coverage (a focused unit per the dm_behavior_tests bucket); the walkthrough's interaction at turn 17 stays at the annotation level.
  - **Conditional follow-up registered.** Per the tech-debt registration above, if a subsequent design conversation decides turn 17 SHOULD be a `meta-move` proposal, the `walkthrough_fixture_meta_move_carveout_clarification` future-task captures the revision.
  - **Alternative considered: encode turn 17 as a `meta-move` proposal AND an `annotate` of kind `reframe`.** Rejected — over-encodes; the doc names ONE event (the annotation) for the reframe; adding the meta-move proposal would diverge from the canonical narrative without acceptance-criteria justification.

- **D9 — `pendingProposalCount()` is the open-disputes count at segment-1 close, NOT zero.** The walkthrough's segment 1 closes with A2 disputed (Ben never agreed) and N17 + E11b in interpretive-split-pending state. These leave at least 2-3 pending proposals on the projection. The exact count is determined by the fixture's encoded narrative; the assertion is `pendingProposalCount() === <N>` where N is the number of open disputes. Per D4 the implementer computes N from the fixture; if it diverges from the walkthrough's narrative intent, the divergence is a fixture-authoring bug to fix.
