## What this task is

Settle the methodology-engine treatment of **defeater capture** — the F6 moderator-UI operation per `docs/moderator-ui.md` lines 101–112 and the operationalization-test side-effect per `docs/methodology.md` line 114. The task **does not** add a new proposal sub-kind, a new `MethodologyAction` variant, or a new `RejectionReason`. The methodology engine's surface is unchanged. The task settles **how defeater capture maps onto the existing event vocabulary** and **documents the layering** so future contributors (and the moderator-UI implementation) build the same way.

The defeater-capture flow per `docs/moderator-ui.md` F6 decomposes into **three existing event-stream operations** that the methodology engine already validates (or accepts on the placeholder path):

1. **`node-created`** — the new defeater node X. A direct entity-creation event (kind `node-created` per `packages/shared-types/src/events.ts` lines 241–252). Materializes a node with `substance` in `proposed` (X has not been established).
2. **`edge-created`** — the `rebuts` edge from X to the defeated target. A direct entity-creation event (kind `edge-created` per the same module lines 254–263, `role: 'rebuts'` per the `edgeRoleSchema` in `events/enums.ts`).
3. **`propose set-edge-substance`** — a proposal against the rebut edge with `value: 'agreed'`. The proposing participant is the one declaring the retraction condition (the "pre-commitment" — `docs/methodology.md` line 114: "The participant proposes the rebuts edge's `substance` as agreed"). This goes through the **normal vote-and-commit lifecycle** (per `docs/moderator-ui.md` F6 step 6: "Vote and commit through the normal lifecycle"). The "pre-commitment" language is descriptive of *when* the substance is proposed-agreed (at capture time, before the source-node substance is established), **not** prescriptive of *how* (no special skip-vote / skip-commit semantics).

The defeater node X's own substance facet is NOT proposed at capture time — it stays in `proposed` per `docs/methodology.md` line 114 and `docs/moderator-ui.md` F6 step 5. The defeater "sits in the graph and does not currently fire"; it activates only if X's substance later becomes `agreed` through its own substantiation path. Per `docs/data-model.md` line 102 and `apps/server/src/projection/active-firing.ts` lines 30–40, the conditional reading of edge substance + `is-source-agreed?` does the right thing at projection-read time without any defeater-specific code.

The end state: a defeater is a **structural pattern** — node + rebuts-edge with agreed edge substance + proposed source substance — recognizable on the projection but not a first-class event-shape distinction. **No methodology-engine handler is added** because no step in the flow requires a defeater-specific rule that isn't already enforced (or pending in a sibling task) by the existing sub-kind validators.

## Why it needs to be done

The `defeater_capture_logic` slot exists in the WBS because earlier work flagged "defeater capture" as a methodology operation that needed an explicit position in the engine. Reading the docs together (`docs/methodology.md` line 114, `docs/data-model.md` lines 100–102, `docs/moderator-ui.md` F6, `docs/example-walkthrough.md` turns 11 / 14–15 / 23–24) settles that the operation is **expressible in existing event vocabulary** with no methodology-engine extension needed. The task's deliverable is therefore primarily this refinement — a documented layering choice that:

1. Pins which option the project picks (Option B per the task prompt's options A/B/C menu).
2. Cites the doc passages that justify the pick (so the next person reading the docs reaches the same conclusion).
3. Names the open question this task does NOT settle (commit-time multi-event emission — shared with five sibling refinements).
4. Adds the minimum verification surface: a Vitest that asserts the placeholder path on `set-edge-substance` doesn't crash on a propose-agreed-rebut, and a Cucumber scenario that walks the full three-event defeater pattern against pglite and asserts the projection's edge-firing predicate fires when (and only when) the source node's substance later becomes agreed.

Without this refinement the WBS slot stays ambiguous, the moderator-UI implementer can't know whether to call a single `captureDefeater` engine endpoint (Option A) or stitch three propose calls together (Option B), and the "pre-committed substance" phrase floats as load-bearing but unowned.

## Decisions

- **Option B: defeater capture is a UI-level macro built on existing primitives.** Three event-stream operations — `node-created` + `edge-created` (`role: 'rebuts'`) + `propose set-edge-substance` (`value: 'agreed'`) — fully express the F6 flow. No new methodology-engine handler, action variant, proposal sub-kind, event kind, or rejection reason. **Rationale**:
  - `docs/methodology.md` line 114 explicitly says "The participant **proposes** the rebuts edge's `substance` as agreed." Propose-set-edge-substance with `value: 'agreed'` is exactly that operation. The "pre-commitment" language is descriptive of *when* (at capture time, before X is established), not prescriptive of *how* (no skip-vote / skip-commit semantics).
  - `docs/moderator-ui.md` F6 step 6 closes the loop: "Vote and commit through the normal lifecycle." If the defeater's substance proposal went through a special skip-vote path, this step would say so. It doesn't.
  - `docs/data-model.md` line 102's "the pre-commitment is structural" supports the same read: structural means "the *shape* of the graph (agreed-substance edge over a proposed-substance source) encodes the pre-commitment," not "the methodology engine has a special path that writes the commit without a vote round."
  - `docs/example-walkthrough.md` turn 11 + turn 14–15: each defeater materializes as the three-event pattern. The walkthrough is the canonical worked example. (The walkthrough uses the platform-action shorthand "pre-committed by Ben as `agreed`" but the next sentence each time leaves the source node's substance as `proposed`, consistent with the standard propose-vote-commit lifecycle for the edge-substance proposal — the same as any other edge-substance proposal.)
  - Adding a `captureDefeater` action (Option A) would prematurely settle the **commit-time multi-event emission** architectural question for one narrow case while leaving the same question open for five siblings (decompose / interpretive-split / restructure / axiom-mark / meta-move all have it). The right time to settle the question is when it's settled uniformly across all six (the likely follow-up: a `commit_logic` extension or a unified `structural_commit` task — see Open Questions).
- **The "Capture defeater" mode banner in the moderator UI is a UI affordance, not a methodology variant.** Per `docs/moderator-ui.md` line 177 the mode is one of six listed (Capture statement / Capture decomposition components / Capture defeater / Capture meta-move / Elicit warrant / Run operationalization test / Snapshot label). Modes communicate "what kind of capture is in flight" to moderator and audience; they shape the UI's input rendering (the wording field captures Y, the target picker is locked to X) and they bound which keyboard shortcuts are active. None of that is methodology-engine concern. The moderator-UI implementation owns the mode → event-sequence translation.
- **The moderator-UI builds the three-event sequence client-side.** When the moderator triggers F6: (1) the UI asks the API to write a `node-created` for Y; (2) the UI asks the API to write an `edge-created` for the rebut Y → X; (3) the UI asks the API to call the methodology engine with a `propose set-edge-substance` action carrying `value: 'agreed'` against the new edge. Each call has its own validation surface (Zod on the entity-creation events per ADR 0021; methodology engine on the propose action). The moderator UI is responsible for sequencing the three calls and rolling back / surfacing errors if any step fails. (The transactional shape — "all three or none" — is a moderator-UI concern; not in scope here.)
- **No defeater-specific validator.** The propose-handler's `set-edge-substance` arm is unchanged (placeholder fall-through today; will tighten when its own sibling task lands). A defeater's pre-committed substance proposal is structurally identical to any other propose-set-edge-substance — same rules will apply.
- **The "pre-committed" word in the docs is descriptive, not load-bearing for the methodology engine.** It tells the reader *when* the substance proposal happens (at capture time, before X is established) and *who* proposes it (the participant declaring the retraction condition). It does not name a special engine path. The propose-vote-commit cycle runs as it does for any other edge-substance proposal — the only difference is that at the time of capture, the source node's substance facet is still `proposed`, so even after the edge-substance is committed-agreed, the edge does not fire until the source is also committed-agreed (which the projection's existing edge-firing predicate handles without modification).
- **Verification surface (per ADR 0022).** A Vitest unit case pins the placeholder path's behavior on a defeater-shaped set-edge-substance proposal. A Cucumber+pglite scenario walks the full three-event pattern end-to-end (entity-creation events + the propose-vote-commit cycle for the edge substance) and asserts the edge-firing predicate computes correctly across the source-substance transition. Together these two layers verify the load-bearing claim from `docs/data-model.md` line 102 ("the rebut sits in the graph but does not currently fire... if the source ever becomes substantively established, the rebut activates").
- **File naming.** Vitest at `apps/server/src/methodology/handlers/proposeDefeaterPreCommit.test.ts` (the name reads as the operation a future reader searches for, even though it's structurally a `propose set-edge-substance` test). Cucumber at `tests/behavior/methodology/defeater-capture.feature`. Step defs at `tests/behavior/steps/methodology-defeater-capture.steps.ts`.

## Open questions

- **Commit-time multi-event emission for structural / composite operations (shared with decompose / interpretive-split / restructure / axiom-mark / meta-move).** The defeater-capture flow needs three events written across two paths: two direct entity-creation events (`node-created`, `edge-created`) and one proposal-and-commit cycle (`set-edge-substance` with `value: 'agreed'`). Today the moderator UI is responsible for sequencing the three calls; the API-level transactional guarantee ("all three or none") is unsettled. The five sibling refinements all flag the same shape — when a commit lands for a structural sub-kind, who emits the paired `node-created` / `edge-created` / `annotation-created` events for the materialized entities? A future unified task (likely a `commit_logic` extension or a new `structural_commit` task) is expected to settle the question uniformly. Defeater capture inherits the resolution; it does not need its own answer.
- **Should the moderator-UI's "Capture defeater" mode trigger a server-side macro endpoint?** Adjacent to the prior question. If/when a `captureDefeater` macro endpoint lands (Option A on a future redesign), this refinement's Decisions section is the historical record of why the project shipped without one first. The macro would not change methodology semantics — it would batch the three calls behind one API request for atomicity. Out of scope here.
