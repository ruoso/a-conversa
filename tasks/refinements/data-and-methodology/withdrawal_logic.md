# Withdrawal logic — withdrawal of a prior agree on a committed proposal, plus the rest of the per-vote validation

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.withdrawal_logic`
**Effort estimate**: 1d
**Inherited dependencies**: depends on `methodology_engine.commit_logic` (the prior sibling — `commit_logic` lands the `committedProposals` bucket the withdrawal arm reads against; it also factored the per-action handler module shape and the `installHandlers` wiring this task slots into). Concretely settled: the methodology framework (`MethodologyAction`, `validateAction`, `RejectionReason`, the primitives — `requireParticipant`, `findProposal`, `currentParticipants` — and the per-action-handler module layout); the placeholder `placeholderVoteHandler` (`apps/server/src/methodology/handlers/vote.ts`); the projection's `handleVote` (`apps/server/src/projection/replay.ts`) which writes the per-facet `perParticipant` map; `deriveFacetStatus` (`apps/server/src/projection/facet-status.ts`) which on the read side returns `'withdrawn'` when a withdraw vote arrives against a committed facet (rule 3).

## What this task is

Replace the placeholder `vote` handler that ships with `agreement_state_machine` with the real methodology-engine validator. The vote action carries one of three vote values (`agree` / `dispute` / `withdraw`); this task validates all three. The **load-bearing** case is `withdraw` — a participant withdraws their prior `agree` on a *committed* proposal, returning the affected facet to `'withdrawn'` (per `deriveFacetStatus` rule 3 on the read side). The other two arms (`agree`, `dispute`) are validated here because the WBS at `tasks/10-data-and-methodology.tji` doesn't list `vote_logic` as a separate sibling — the `vote` action handler is owned in full by this task. On `Valid` the handler emits one `EventToAppend` of kind `vote` matching `VotePayload` (`{ proposal_id, participant, vote, voted_at }`). On any rule failure the handler returns a typed `Rejected` with a `RejectionReason` already minted by the prior task's union.

## Why it needs to be done

`docs/methodology.md` line 25: "A participant may **withdraw agreement** they previously gave. An `agreed` facet transitions back to `disputed`; the original commit and the withdrawal are both recorded in the change history." The read-side derivation (`deriveFacetStatus` rule 3) was already wired by `per_facet_status_derivation` — when `wasCommitted && hasWithdraw`, the facet reads `'withdrawn'`. But the write-side gate was deferred: the placeholder vote handler accepts *any* vote — including a withdraw from a participant who never agreed, an agree on an already-committed proposal (which the system would re-record but never act on), or a vote from a participant who isn't currently joined (the universal participant gate filters this last one, but the placeholder still emits an event). Without this task, the API layer would accept all those illegal votes.

The boundary is precise: this task validates the *intent* to vote on the **write** side (does the request pass methodology rules?). The projection's `handleVote` (`replay.ts`) applies the *structural effect* on the **read** side (record the vote into the affected facet's `perParticipant` map). `deriveFacetStatus` then derives the overall facet status (`'agreed'` / `'disputed'` / `'committed'` / `'withdrawn'` / etc.) from that map plus the commit marker. The three layers run in order: `validateAction` returns `Valid`, the API layer appends the event, then `applyEventIncremental` calls `handleVote`. They stay in sync by construction.

## Acceptance criteria

- `apps/server/src/methodology/handlers/vote.ts` exports a default `Validator<VoteAction>` that:
  1. Calls `requireParticipant(projection, action.requester)`. On failure returns the rejection (reason `'not-a-participant'`). Technically unreachable because the universal participant gate already filtered; the helper's contract still distinguishes the two.
  2. Calls `findProposal(projection, action.proposalEventId)`. Branches:
     - `null` → reject `'proposal-not-found'`.
     - `state: 'meta-disagreement'` → reject `'proposal-already-meta-disagreement'` (no votes are accepted on meta-disagreed proposals regardless of arm).
     - `state: 'committed'` → only `vote === 'withdraw'` is legal; `agree` and `dispute` reject with `'proposal-already-committed'` (the commit has landed, normal voting on this proposal is done; the only remaining action is `withdraw`).
     - `state: 'pending'` → only `vote === 'agree'` and `vote === 'dispute'` are legal; `withdraw` rejects with `'no-prior-agree'` (nothing to withdraw from — withdrawal is "second thoughts after a commit").
  3. For the four facet-targeting sub-kinds, look up the participant's prior vote record on the affected facet via `findParticipantVoteOnProposal(projection, action.proposalEventId, action.requester)`:
     - For `vote === 'agree'`: if the participant has a prior record on this proposal with vote `'agree'` → reject `'already-voted'` (no-op double-agree). If the participant has a prior record with vote `'dispute'` → accept (vote switch; documented as the chosen mutability semantics).
     - For `vote === 'dispute'`: mirror — prior `'dispute'` → reject `'already-voted'`; prior `'agree'` → accept (vote switch).
     - For `vote === 'withdraw'`: the participant **must** have a prior record on this proposal with vote `'agree'` → otherwise reject `'no-prior-agree'`. (The state-check in step 2 already required `state === 'committed'`; this is the second half of the "withdraw requires prior agree on a committed proposal" rule.)
  4. For the structural sub-kinds (`decompose`, `interpretive-split`, `axiom-mark`, `meta-move`, `break-edge`, `amend-node`, `annotate`), per-participant vote state isn't tracked on a single facet's `perParticipant` map; the per-participant state lives elsewhere (or is not yet tracked at all). The handler **defers** to the sibling task that owns each sub-kind's voting semantics — for now, the vote on a structural sub-kind is accepted at the framework level (the event is emitted), but no per-record cross-check is performed. The vote event still lands in `session_events`; the projection's `handleVote` already no-ops the per-facet write for structural sub-kinds. Sibling tasks (`decomposition_logic`, `axiom_mark_logic`, etc.) will tighten this as they land. Documented under "Decisions" — this is the only boundary withdrawal_logic has to draw inside its rule set.
  5. On `Valid` emit one `EventToAppendEnvelope<'vote'>` whose payload is `{ proposal_id: action.proposalEventId, participant: action.requester, vote: action.vote, voted_at: action.votedAt }`. Envelope fields are mirror-copied from the action.
- A new primitive `findParticipantVoteOnProposal(projection, proposalEventId, participantId) → PerParticipantVote | null` is added to `apps/server/src/methodology/primitives.ts`. It resolves the affected facet (via the same `facetTargetForProposal` shape used by `commit.ts`), looks up `facet.perParticipant.get(participantId)`, and returns the vote value **only if the record's `proposalEventId` matches the queried proposal id**. Returns `null` for the structural sub-kinds (which don't have per-facet vote tracking) and for "no prior vote." The proposalEventId match guards against false-positives across proposals: the per-participant map is keyed by participant only and overwritten on re-vote, so the record's own `proposalEventId` field is the ground truth for which proposal the recorded vote refers to.
- `apps/server/src/methodology/handlers/vote.test.ts` covers:
  - **agree path**:
    - Reject when proposal doesn't exist (`'proposal-not-found'`).

## Decisions

- **Scope: this task owns the entire `vote` action handler, not just the `withdraw` arm.** The WBS at `tasks/10-data-and-methodology.tji` doesn't list a separate `vote_logic` sibling — the agree/dispute arms have to live somewhere. Documenting them in the placeholder file isn't honest: the placeholder accepts *any* vote (it doesn't check `proposal-not-found`, `proposal-already-committed`, or `already-voted`), and the task description note ("the sibling `vote_logic` task will replace it with the real validator that enforces ...") explicitly anticipates an unsibling. The cleaner alternative — load the whole vote handler into this task — keeps the rule set co-located and avoids needing a second refinement document. The trade-off is this task does slightly more than its name ("withdrawal_logic"); the alternative was to leave agree/dispute permanently under the placeholder, which fails ADR 0022's "every behavior is regression-tested" bar.
- **Rule set (numbered, in evaluation order).**
  1. **Participant gate.** `requireParticipant(projection, action.requester)` must succeed. → `'not-a-participant'` (unreachable in practice because `validateAction` already filtered).
  2. **Proposal exists.** `findProposal(projection, action.proposalEventId)` must return non-null. → `'proposal-not-found'`.
  3. **Proposal-state vs. vote-arm matrix.**
     - `meta-disagreement` (any arm) → `'proposal-already-meta-disagreement'`.
     - `committed` + `agree` or `dispute` → `'proposal-already-committed'`. Only `withdraw` is legal on a committed proposal.
     - `pending` + `withdraw` → `'no-prior-agree'` (the pending proposal has no commit to withdraw from).
     - All other combinations → continue to rule 4.
  4. **Per-participant prior-vote check.** Look up the requester's prior vote on the affected facet via `findParticipantVoteOnProposal`:
     - `agree` arm: prior `agree` on this proposal → `'already-voted'`; otherwise accept (no prior vote OR prior `dispute` — the dispute→agree switch is legal).
     - `dispute` arm: prior `dispute` on this proposal → `'already-voted'`; otherwise accept (the agree→dispute switch is legal).
     - `withdraw` arm: prior `agree` on this proposal **required** — anything else (no prior vote, prior `dispute`, prior `withdraw`) → `'no-prior-agree'`.
  5. **Structural-sub-kind boundary.** For the seven structural sub-kinds, `findParticipantVoteOnProposal` returns `null` (no per-facet vote tracking exists). Rule 4 is short-circuited as "no prior vote": for `agree` / `dispute` arms this means accept (the vote is emitted; the per-sub-kind sibling will tighten later); for `withdraw` this means reject (`'no-prior-agree'` — withdrawal of a structural commit is deferred to the sibling task that owns the sub-kind). Documented; the alternative ("reject with `'illegal-state-transition'` and a sub-kind-naming detail") was considered but is more aggressive than necessary — the placeholder did emit the event, and this task tightens only the cases it can validate. Structural sub-kind sibling tasks (`decomposition_logic`, `axiom_mark_logic`, etc.) will revisit if they want their own per-vote restrictions.
- **Self-vote semantics.** `docs/methodology.md` line 13 ("agreement is per-facet and per-participant. Each facet of an entity tracks each participant's stance individually") and line 9 ("all participants — both debaters and the moderator — must agree on every change to the graph before it lands") together imply: yes, all participants vote on all proposals — including the moderator on a proposal the moderator made, and either debater on a proposal that debater made. The `'self-vote-not-allowed'` `RejectionReason` exists in the union (minted by `agreement_state_machine`) but is **not used here** — the handler does not check whether the requester proposed the proposal. The reason value remains in the union for a sibling task that may need it (e.g. annotation logic), but for the vote handler, self-vote is legal. Rationale: `commit_logic` requires unanimous agree across all current participants for a commit to land; if the proposer's own agree didn't count, no proposal could ever commit (someone proposing always abstains).
- **Dispute ↔ agree mutability.** `docs/methodology.md` doesn't explicitly address whether a participant can switch their vote between agree and dispute before commit. Two cleaner options: **(a)** mutable — vote switching is legal until commit (the participant changes their mind based on the discussion); **(b)** immutable — once a vote is cast, it's the final word and a participant must wait for a new proposal cycle. **Decision: vote switching is legal until commit (option a).** Rationale: the methodology rewards genuine deliberation ("real reasoning has second thoughts" per line 25 — withdrawal's stated motivation), and a participant who hears further argument may legitimately want to update their stance. Forbidding the switch would force a no-op + a fresh-proposal dance, which adds ceremony without protecting anything. The `already-voted` rejection fires only when re-casting the *same* vote value (an actual no-op). Implementation-wise this is a small simplification: the per-participant map's "overwrite on re-vote" behaviour from `handleVote` already supports the switch; the handler just has to allow it.
- **`findParticipantVoteOnProposal` implementation.** New primitive in `primitives.ts`. Mirrors `commit.ts`'s (entity, facet) target resolution (we duplicate the `facetTargetForProposal` shape locally in `vote.ts` rather than extracting it to a shared module — keeping the duplication symmetric with `commit.ts`'s own local copy, per the prior refinement's "if a future refactor extracts them into a shared module, both call sites can switch"). The lookup returns the vote value **only if the record's `proposalEventId` field matches the queried proposal id**. This guard matters: the per-facet `perParticipant` map is keyed by participant — only one record per participant per facet — and `handleVote` overwrites on re-vote. If the same facet hosted votes for two different proposals over time (e.g. a classification facet had proposal A → withdraw → proposal B), the map's record reflects only the latest proposal. The `proposalEventId` field on the record is the ground-truth source for "which proposal does this recorded vote refer to."
- **No new `RejectionReason` values.** The prior task's union already includes every reason this handler needs: `'not-a-participant'`, `'proposal-not-found'`, `'proposal-already-committed'`, `'proposal-already-meta-disagreement'`, `'already-voted'`, `'no-prior-agree'`. The `'proposal-not-pending'` value is in the union but unused by this handler — the more specific values cover the two reachable cases (`'proposal-already-committed'` and `'proposal-already-meta-disagreement'`).
- **`EventToAppend` shape mirrors the placeholder.** Same payload, same envelope fields. The only thing this handler adds is the four-rule write-side gate before emitting.
- **Cross-check with `deriveFacetStatus`.** The withdraw-accept path includes an in-test assertion that the read-side derivation returns `'withdrawn'` after the vote event is applied to the projection. The two layers (write-side validator and read-side derivation) must agree on the same projection state; this is the same "stay in sync by construction" check `commit_logic` already enforces for its accept path.
- **Test layout.** Vitest at `apps/server/src/methodology/handlers/vote.test.ts` so the file sits next to the handler. The framework-level engine test (`apps/server/src/methodology/engine.test.ts`) keeps covering dispatch; the existing test "vote action emits a vote event payload" continues to pass because a fresh agree on a pending proposal is a legal vote.
- **`installHandlers` wiring.** `engine.ts` swaps the import from `placeholderVoteHandler` to the real `voteHandler`. `handlers/index.ts` re-exports `voteHandler`; the `placeholderVoteHandler` export is removed (no consumers).

## Open questions
