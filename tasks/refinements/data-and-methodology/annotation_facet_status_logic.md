# Annotation facet-status logic — route meta-move votes into per-annotation facet status

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.annotation_facet_status_logic`
**Effort estimate**: 1d (as registered; see Open questions — the deliverable splits into a reachable core and an unreachable headline, which the closer should weigh)

**Inherited dependencies**:

- `methodology_engine.meta_move_commit_logic` (**settled**, landed 2026-06-02) — the predecessor that makes a committed meta-move emit `annotation-created`. It established the emission contract this task leans on for vote→annotation correlation: `buildStructuralEventsForCommit`'s meta-move arm returns `[annotation-created, commit]` in that order, with the `annotation-created` envelope taking `sequence = action.sequence` and the `commit` envelope the next slot (commit.ts:404–434). So the annotation-created event for a committing meta-move is the event immediately preceding that meta-move's `commit` in the log — a deterministic structural link with no provenance field required. See [`meta_move_commit_logic.md`](meta_move_commit_logic.md).
- Transitively settled: `meta_move_logic` (the propose-side validator; meta-move targets a node/edge per ADR 0036), `commit_logic` (the structural-unanimity commit gate), and the render-seam work in `mod_meta_move_disputed_visibility` (AC §4/§11) which shipped `AnnotationRenderData`, `AnnotationFacetStatusIndex`, `enrichAnnotationsWithFacetStatuses`, and the `<AnnotationBadge>` `data-facet-status` test seam — all wired with an **empty** index awaiting this task's populator.

## What this task is

The moderator render seam `enrichAnnotationsWithFacetStatuses` (apps/moderator/src/graph/selectors.ts:260–275) joins each projected annotation against an `AnnotationFacetStatusIndex` — `ReadonlyMap<annotationId, Readonly<Partial<Record<FacetName, FacetStatus>>>>` (selectors.ts:230–233) — but `selectAnnotations` always hands it the `EMPTY_ANNOTATION_FACET_STATUS_INDEX` singleton (selectors.ts:294–304), so no annotation ever receives `facetStatuses`. This task **builds the populator**: extend the shell's `computeFacetStatuses` walk so it derives a per-annotation facet-status bucket from the per-participant votes cast on the originating meta-move proposal, and wire `selectAnnotations` to pass `computeFacetStatuses(session.events).annotations` instead of the empty singleton.

Concretely:

1. **Engine (the real work).** Extend `FacetStatusIndex` with an `annotations` bucket and extend `computeFacetStatuses` (packages/shell/src/facet-status/facet-status.ts:269+) so that, for each committed meta-move, the per-participant `agree`/`dispute` votes accumulated against that meta-move proposal are run through the existing per-facet state machine and recorded under the resulting annotation's id, keyed on the annotation's **substance** facet. The annotation id is recovered from the log via the commit-batch adjacency the predecessor guarantees (the `annotation-created` event preceding the meta-move's `commit`).
2. **Wiring (cleanup).** Point `selectAnnotations` at the populated `.annotations` bucket; drop the `EMPTY_ANNOTATION_FACET_STATUS_INDEX` hand-off.

## Why it needs to be done

This is the third and final link in the chain that `mod_meta_move_disputed_visibility` registered as backend debt (AC §11): `mod_meta_move_disputed_visibility` built the render seam (empty index); `meta_move_commit_logic` made committed meta-moves materialize as annotations; **this task** routes the votes into per-annotation facet status and lights the seam up. Without it the `AnnotationFacetStatusIndex` is permanently empty and the moderator's annotation badges can never reflect deliberation state — the disputed-visibility render work shipped but stays inert.

The per-annotation facet status is observable at the projection seam (a pure function of the event log), so per the backend-task coverage policy and ADR 0022 it warrants a Cucumber+pglite pin in addition to Vitest, alongside the moderator-side selector wiring test.

**A reachability finding shapes this task's scope** (see Decisions §1 and Open questions). The headline the source-of-debt note attached — a Playwright `[data-facet-status="disputed"]` round-trip — is **not reachable through today's event vocabulary**: meta-move commit is gated on *unanimous agree* (commit.ts:289–344, `checkUnanimousAgreeStructural`), so at the instant an annotation exists every routed vote is `agree`; and there is no post-commit vote surface that targets an annotation's facets (`VoteEntityKind = 'node' | 'edge'`, useVoteAction.ts:74–75; the vote handler returns no facet target for structural sub-kinds, replay.ts:711–722). A *disputed* annotation therefore cannot be produced end-to-end. This task delivers the reachable core (the projector + wiring + the agreed/committed status it can actually compute) and defers the disputed-state behavior — and its Playwright e2e — to a named follow-up that builds the missing annotation-facet vote seam.

## Constraints / requirements

- **Reuse the existing walk and the existing state machine.** The per-annotation status is derived by the same `deriveStatus` rules already applied to nodes/edges, fed by the same current-participants filter (non-moderator, currently-joined). No second pass over the log, no duplicated participant bookkeeping.
- **Correlate annotation ↔ meta-move via the guaranteed emission order, not a new wire field.** The predecessor emits `[annotation-created, commit]` with the `annotation-created` immediately preceding the meta-move's `commit` (same actor/session, `commit.sequence - 1`). The walk pairs them structurally. **No new payload field, no schema change, no ADR-level wire change.** (See Decision §2 for why provenance-by-field was rejected.)
- **Route onto the `substance` facet only.** Meta-move votes are whole-proposal `agree`/`dispute`, not facet-keyed; the annotation's `wording` is inline-agreed at creation (`content`), and `substanceFacet` is the deliberation facet typed `'agreed' | 'disputed'`. Record the routed status under `substance` (Decision §3).
- **`FacetStatusIndex` gains `annotations`; `.nodes`/`.edges` are untouched.** Existing consumers read `.nodes`/`.edges` and are unaffected by the added field. The new bucket's value shape equals `AnnotationFacetStatusIndex`'s value shape so `selectAnnotations` consumes it directly.
- **Empty/short-circuit identity preserved.** When there are no annotation statuses, `selectAnnotations` must still hand `enrichAnnotationsWithFacetStatuses` a stable empty reference (reuse `EMPTY_ANNOTATION_FACET_STATUS_INDEX`, or return the engine bucket only when non-empty) so the memoization contract from `mod_meta_move_disputed_visibility` Decision §7 holds.
- **No change to `enrichAnnotationsWithFacetStatuses`, `AnnotationBadge`, the wire schemas, the server projection, or the read-side replay handlers.** This task is the engine projector + the one selector call site.
- **Do not invent a disputed path.** Producing `disputed` requires the annotation-facet vote seam that does not exist; building that is the deferred follow-up, not this task. This task must not, e.g., fabricate a synthetic dispute event or special-case the badge.
- Verifications per ADR 0022: Vitest for the projector and the `selectAnnotations` wiring; Cucumber+pglite at the projection seam. No throwaway scripts.

## Decisions

- **§1 — The disputed headline is unreachable; deliver the reachable core and defer the rest honestly.** The source-of-debt note framed the task around a `[data-facet-status="disputed"]` Playwright round-trip. Investigation established two independent blockers: (a) meta-move commit is gated on unanimous `agree` (`checkUnanimousAgreeStructural`, commit.ts:289–344) — a single `dispute` blocks the commit, so no annotation exists while any vote is a dispute; (b) there is no event that votes on an annotation's facets after creation (`VoteEntityKind = 'node' | 'edge'`, useVoteAction.ts:74–75; structural votes resolve to no facet target, replay.ts:711–722). So `disputed` is not producible end-to-end today. **Chosen:** build the projector + wiring that the index needs (correct and forward-compatible: the same code yields `disputed` the moment a dispute vote can reach an annotation), cover the reachable agreed/committed state with Vitest + Cucumber, and defer the disputed behavior + its e2e to a named seam task. **Rejected:** (i) writing the disputed e2e against fabricated projection state — it would bypass the event stream and only re-assert what `AnnotationBadge.test.tsx` already pins, violating the spirit of ADR 0022; (ii) silently shipping an always-empty-in-practice index and calling the task done — that buries the gap the next task would rediscover.
- **§2 — Correlate annotation ↔ meta-move by commit-batch adjacency, not a provenance field.** The annotation gets a fresh `randomUUID()` at commit with no back-pointer to the proposal. **Chosen:** recover the link from the predecessor's guaranteed emission order — the `annotation-created` event is the one immediately preceding the meta-move's `commit` (`commit.sequence - 1`, same actor/session), per `meta_move_commit_logic` AC. The projector already walks the log in order, so this is a local pairing with zero wire change. **Rejected:** adding `source_proposal_id` to the `annotation-created` payload — it is the "cleaner" provenance model, but it is a wire-schema change crossing the protocol boundary (new ADR, reprojection concerns, re-opening the just-shipped `meta_move_commit_logic` emitter) for a correlation the existing event order already determines unambiguously. Per the brief's "reuse existing seams / simpler abstraction" bias, adjacency wins; if a future task needs annotation provenance for an independent reason, that task can introduce the field with its own ADR. **Rejected also:** matching annotation to proposal by `(target, kind, content)` — it collides when a target receives two meta-moves of the same kind/content (the exact ambiguity that retired the old deterministic-string id), whereas commit-batch adjacency is collision-free.
- **§3 — Route meta-move votes onto the annotation's `substance` facet.** Meta-move votes are whole-proposal `agree`/`dispute` with no facet field (metaMoveProposalSchema, proposals.ts:412–422). The annotation has `wording` (inline-agreed at creation: it *is* the `content`) and `substance` (typed `FacetState<'agreed' | 'disputed'>`, projection types.ts:250–261 — the deliberation facet). **Chosen:** map the whole-proposal accept/dispute vote to the `substance` facet, since that is the facet the data model reserves for "do we agree with this annotation's substance." **Rejected:** routing onto `wording` (the wording is fixed at creation; voting it makes no sense) or onto both facets (double-counts one vote into two facets, inflating the rollup).
- **§4 — Deferred follow-up is a concrete seam, not an "audit."** The disputed behavior is blocked on a *buildable* seam — a post-commit annotation-facet vote surface — not on a judgment call. The standing `substanceFacet: FacetState<'agreed' | 'disputed'>` typing and the useVoteAction.ts "follow-up" comment show the data model already anticipates it. **Chosen:** name `data_and_methodology.methodology_engine.annotation_facet_vote_seam` (widen `VoteEntityKind` to include `'annotation'`; add the vote-handler arm that resolves an annotation-targeted facet vote; extend the `computeFacetStatuses` annotation routing to fold in those direct votes so a committed annotation can reach `disputed`) and its UI rider `moderator_ui.mod_graph_rendering.mod_annotation_dispute_e2e` (the participant affordance to dispute an annotation + the moderator Playwright `[data-facet-status="disputed"]` round-trip). Both are agent-implementable. **Rejected:** an "audit"/"revisit" task — explicitly disallowed by the brief and would self-perpetuate.
- **§5 — Extend `FacetStatusIndex` rather than add a parallel function.** **Chosen:** add the `annotations` bucket to the existing single-pass `computeFacetStatuses`, which already tracks participants, proposals, votes, and commits — the annotation routing is an incremental addition on the same walk, and the bucket's value shape already equals `AnnotationFacetStatusIndex`. **Rejected:** a separate `computeAnnotationFacetStatuses(events)` — it would re-walk the log and duplicate the participant/vote bookkeeping for no benefit, and `selectAnnotations` would have to call two functions over the same events.
- **§6 — No server-side mirror change.** The moderator's `AnnotationFacetStatusIndex` is computed client-side from the WS event log via the shell's `computeFacetStatuses`; the server's `apps/server/src/projection/facet-status.ts` mirror serves a different consumer and is out of scope here. The `ProjectedAnnotation` facets exist already; this task neither reads nor writes them.
