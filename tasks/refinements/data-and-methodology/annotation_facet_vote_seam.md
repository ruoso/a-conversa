# Annotation-facet vote seam — make a committed annotation disputable post-commit

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.annotation_facet_vote_seam`
**Effort estimate**: 1d (as registered).

**Inherited dependencies**:

- `methodology_engine.annotation_facet_status_logic` (**settled**, landed 2026-06-02) — the predecessor that built the client-side populator routing a committed meta-move's per-participant votes onto the resulting annotation's `substance` facet (`computeFacetStatuses(events).annotations`), and rewired `selectAnnotations` to feed `enrichAnnotationsWithFacetStatuses` the populated bucket. It reached the *agreed/committed* annotation state but **could not reach `disputed`** and named this task as the seam that closes the gap. See [`annotation_facet_status_logic.md`](annotation_facet_status_logic.md) Decision §4 (which scoped this task verbatim: "widen `VoteEntityKind` to include `'annotation'`; add the vote-handler arm that resolves an annotation-targeted facet vote; extend the `computeFacetStatuses` annotation routing to fold in those direct votes so a committed annotation can reach `disputed`").
- Transitively settled: `meta_move_commit_logic` (committed meta-move emits `annotation-created`), `mod_meta_move_disputed_visibility` (shipped `<AnnotationBadge>`'s rose `data-facet-status="disputed"` marker + `enrichAnnotationsWithFacetStatuses` join, both inert until a dispute can be produced), and the per-facet vote machinery of ADR 0030 (the `target`-discriminated facet-keyed vote arm this task extends to annotations).

**Settled by this task's ADR**: the standing product question *"should annotations be disputable post-commit?"* (parking lot, 2026-06-02) is resolved **yes** by [ADR 0038](../../../docs/adr/0038-annotations-disputable-post-commit-via-substance-facet-votes.md), grounded in `docs/data-model.md:141` ("an annotation can itself be disputed") and `docs/methodology.md:25` (second-thoughts principle). The seam is therefore built to the methodology's intent, not merely to the `substanceFacet` type's incidental shape.

## What this task is

Today a committed annotation's `substance` facet can never become `disputed`, because nothing in the system votes against an annotation after it exists: the participant hook narrows `VoteEntityKind` to `'node' | 'edge'` (useVoteAction.ts:75), the facet-keyed vote wire message + persisted event narrow `entity_kind` to `['node', 'edge']` (ws-envelope.ts:480/614, events.ts:441), and the engine vote handler's `facetStateForTarget` resolves node/edge only (vote.ts:80–103). This task **builds the missing vote seam end-to-end** so a participant can cast a facet-keyed dispute vote on a committed annotation's `substance` facet, and that dispute surfaces as the rose `disputed` badge.

Concretely, in lockstep:

1. **Wire + event schema (the contract).** Widen `entity_kind` from `['node', 'edge']` to `['node', 'edge', 'annotation']` on the two facet-vote *wire message* schemas (ws-envelope.ts:480, 614) and the persisted facet-vote *event* schema (`facetVotePayloadSchema`, events.ts:441). Leave `facetCommitPayloadSchema`, `facetMetaDisagreementPayloadSchema`, and `withdrawAgreementPayloadSchema` at `['node', 'edge']` — annotations are *disputable*, not committable / meta-disagreeable / withdrawable through those arms (ADR 0038 §1).
2. **Participant hook (the dispatch capability).** Widen `VoteEntityKind` to `'node' | 'edge' | 'annotation'` (useVoteAction.ts:75) and update its doc comment (the existing comment already flags the widening as "a follow-up"). The facet arm of `castVote` already passes `entity_kind` through generically — no other hook change. **No UI affordance is added here**; the participant dispute-annotation gesture is the downstream task (see Acceptance criteria / Tech-debt).
3. **Engine vote handler (the accept + emit).** Add an `entity_kind === 'annotation'` arm to the vote handler's `facetStateForTarget` (vote.ts:80–103) returning the annotation's `substanceFacet` via `projection.getAnnotation(entityId)` — mirroring the annotation arm already in `primitives.ts:209` and `commit.ts:147`. Relax the committed-facet gate **for annotations only**: a vote (notably `dispute`) is legal on a committed/agreed annotation substance facet, unlike node/edge facets where `committed` rejects with `'proposal-already-committed'` (vote.ts:181–186). The per-participant already-voted guard (vote.ts:209–224) still applies. Restrict the annotation arm to `facet === 'substance'`.
4. **Read side + engine projector (record the vote).** Add the `entity_kind === 'annotation'` arm to the replay facet-vote handler (replay.ts:651–682) so an annotation facet vote records onto the projected annotation's substance facet rather than throwing, and to the shell's `computeFacetStatuses` facet-vote arm (facet-status.ts:438–448) so the vote folds into the per-annotation `substance` accumulator (`annotationStates`). The derivation then yields `disputed` with no rule change (Rule 5 already fires before Rule 6).

## Why it needs to be done

This is the final link that makes the disputed-annotation render seam reachable. `mod_meta_move_disputed_visibility` built the badge; `meta_move_commit_logic` materialized annotations; `annotation_facet_status_logic` lit the agreed/committed status — but all three are inert with respect to `disputed` because no event can dispute a committed annotation. This task supplies that event path. Without it, `ProjectedAnnotation.substanceFacet`'s `'disputed'` variant is permanently dead, the rose badge can never render against real data, and the downstream `mod_annotation_dispute_e2e` round-trip cannot be written.

Because the change crosses the protocol boundary (the wire vote message and the persisted vote event widen) **and** the replay boundary (the projector records a new entity kind on the facet-vote arm), it warrants a Cucumber + pglite pin per the backend-task coverage policy and ADR 0022, in addition to Vitest at the in-memory helpers.

## Constraints / requirements

- **Widen only the vote payloads.** `entity_kind` gains `'annotation'` on the facet-vote wire messages (ws-envelope.ts:480, 614) and the persisted facet-vote event (events.ts:441) — and nowhere else. Commit / meta-disagreement / withdraw-agreement facet payloads stay `['node','edge']` (ADR 0038 §1). Keep the wire-in and event schemas in sync (both widen together) so a valid wire vote round-trips to a valid event.
- **`substance` only.** The annotation arm of `facetStateForTarget` resolves `substanceFacet`; a vote naming any other facet on an annotation target is rejected (`target-entity-not-found` / a dedicated rejection). `wording` is inline-agreed at creation; `classification`/`shape` don't apply to annotations.
- **Reuse the existing annotation arm pattern.** The vote handler's `facetStateForTarget` annotation branch mirrors `primitives.ts:209` / `commit.ts:147` (`projection.getAnnotation`). Do not invent a parallel accessor.
- **Post-commit dispute is legal for annotations, by design (ADR 0038 §3).** The committed-facet rejection that applies to node/edge (vote.ts:181–186) must not apply to an annotation substance facet; a `dispute` on a committed annotation is the whole point. The already-voted guard (no double-dispute/double-agree) still applies. Add a comment at vote.ts citing ADR 0038 so the asymmetry isn't "fixed" by a later refactor.
- **No derivation-rule change.** `disputed` is produced by the existing Rule 5; do not add an annotation-specific rule. Do not touch `enrichAnnotationsWithFacetStatuses`, `AnnotationBadge`, `selectAnnotations`, or the predecessor's `FacetStatusIndex` shape — those are already correct and merely receive a status they couldn't see before.
- **Replay must not throw on an annotation facet vote.** The read-side projector records the annotation facet vote (onto the projected annotation's substance facet) so server projection and client badge stay consistent; a thrown `ReplayError` would break projection for the whole session.
- **Empty/short-circuit identity preserved.** No change to the predecessor's stable-empty-index contract (`EMPTY_ANNOTATION_FACET_STATUS_INDEX`); a session with no annotation disputes behaves exactly as before.
- Verifications per ADR 0022: Vitest for the shell projector + schema widening + hook type; Cucumber + pglite at the protocol/replay seam. No throwaway scripts.

## Acceptance criteria

- **Schema.** `facetVotePayloadSchema.entity_kind` (events.ts:441) and the two facet-vote wire `entity_kind` enums (ws-envelope.ts:480, 614) accept `'annotation'`; a vote payload `{ target:'facet', entity_kind:'annotation', entity_id, facet:'substance', choice:'dispute' }` parses. `facetCommitPayloadSchema` / `facetMetaDisagreementPayloadSchema` / `withdrawAgreementPayloadSchema` still reject `'annotation'`.
- **Hook.** `VoteEntityKind` includes `'annotation'`; `useVoteAction({ entity_kind:'annotation', entity_id, facet:'substance' }).castVote('dispute')` dispatches the facet-arm wire payload with `entity_kind:'annotation'`.
- **Engine.** The vote handler accepts a `dispute` vote on a committed annotation's `substance` facet (emits the `vote` event) and rejects: a vote on a non-existent annotation (`target-entity-not-found`); a second identical vote by the same participant (`already-voted`); a vote naming a non-`substance` facet on an annotation target.

The node/edge buckets are unaffected. Replay of the same log records the vote on the projected annotation's substance facet without throwing.

## Decisions

- **§1 — Annotations are disputable post-commit; build the vote seam. ([ADR 0038](../../../docs/adr/0038-annotations-disputable-post-commit-via-substance-facet-votes.md))** The parked product question is resolved **yes** by the canonical docs: `data-model.md:141` ("an annotation can itself be disputed, decomposed, or retracted") and `methodology.md:25` (the second-thoughts principle behind withdraw-of-agreement). The `substanceFacet: FacetState<'agreed' | 'disputed'>` typing is therefore intentional, not incidental. **Rejected:** the append-only-commentary reading (which would descope this task and narrow `substanceFacet` to `'agreed'`) — it contradicts the explicit data-model statement. An ADR was written because this is a genuinely new architectural seam (a new entity kind on the vote payload + a deliberate divergence in the committed-facet gate) and because the parking-lot item explicitly noted no ADR recorded the methodology's intent.
- **§2 — Reuse the facet-keyed vote arm; do not invent a `dispute-annotation` proposal kind.** ADR 0030's `target:'facet'` vote arm already models direct per-facet dissent against an entity without a proposal lookup (it serves inline-seeded facets like an edge's `shape`). Disputing an annotation is exactly that shape with a new `entity_kind`. **Rejected:** a new proposal sub-kind + agreement lifecycle — over-built, and it would collide with the facet-vote arm. **Rejected:** reusing `withdraw-agreement` — it yields `withdrawn`, not the `disputed` the badge renders, and imposes a heavier ceremony than the methodology's "first-class disputable" framing.
- **§3 — Post-commit legality diverges for annotations, deliberately.** Node/edge committed facets reject votes (re-open via `withdraw-agreement` → `withdrawn`); an annotation substance facet accepts a `dispute` even when committed (→ `disputed`). This asymmetry is the methodological distinction between a structural commitment (re-opened by an explicit withdraw) and commentary (contested by a vote). It is pinned by tests and a code comment citing ADR 0038. **Rejected:** forcing annotations through the node/edge withdraw path for uniformity — it cannot produce `disputed`.
- **§4 — `substance` is the only disputable annotation facet.** Meta-move votes are whole-proposal and route onto `substance` (predecessor Decision §3); `wording` is fixed at creation; `classification`/`shape` are node/edge-only. The annotation arm resolves and accepts votes against `substance` exclusively.
- **§5 — Record the annotation facet vote on the read side, don't no-op it.** Replay writes the vote onto the projected annotation's substance facet (mirroring the node/edge facet arm) rather than skipping it, so the server projection and the client-computed badge stay consistent and a snapshot-hydrating late-joiner sees the same disputed state. **Rejected:** a replay no-op for annotation facet votes — it would diverge the server projection from the client badge and risk a `ReplayError` if not explicitly handled.
- **§6 — Orthogonal to ADR 0036.** 0036 forbids *meta-moves from targeting annotations* (a meta-move creates an annotation → infinite regress). A dispute vote creates no annotation; it only flips an existing annotation's substance status. No regress, 0036 untouched. (Documented in ADR 0038 Context so the "projection-layer residue" framing of 0036 is not misread as foreclosing annotation disputes.)
- **§7 — No server-side facet-status mirror rework.** Consistent with the predecessor's Decision §6: the moderator badge is computed client-side by the shell's `computeFacetStatuses` over the WS event log. This task adds the annotation arm there and on the read-side replay (§5); it does not re-architect `apps/server/src/projection/facet-status.ts`.
