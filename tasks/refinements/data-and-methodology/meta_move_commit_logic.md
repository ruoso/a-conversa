# Meta-move commit logic — committed meta-move emits `annotation-created`

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.meta_move_commit_logic`
**Effort estimate**: 1.5d
**Inherited dependencies**:
- `methodology_engine.meta_move_logic` (**settled**, landed 2026-05-10) — the propose-side meta-move validator. It established that a meta-move targets a visible node or edge (`metaMoveProposalSchema`: `meta_kind ∈ {reframe, scope-change, stance}`, `target_kind ∈ {node, edge}`, `target_id`), and it left a deliberate TODO in `replay.ts`'s `applyCommittedProposal` meta-move arm: synthesize-an-annotation-on-commit was deferred, the arm is a structural no-op, and the comment explicitly hands annotation-creation semantics to "the methodology engine … alongside `commit_logic`'s per-sub-kind support."
- `methodology_engine.commit_logic` (**settled**, landed 2026-05-10) — the write-side commit validator. As shipped it deferred all structural sub-kinds (meta-move included) with `'illegal-state-transition'`. That boundary has since been **tightened** (see Decisions): the current commit handler validates structural sub-kinds through `checkUnanimousAgreeStructural` and emits a structural fan-out via `buildStructuralEventsForCommit` — the exact seam this task extends. `annotate` already rides that seam to emit `annotation-created` on commit; meta-move does not yet.

## What this task is

Make a committed meta-move materialize as an annotation. When a `meta-move` proposal commits, the commit handler must emit an `annotation-created` event — exactly the way a committed `annotate` proposal already does — so the projection gains a visible annotation of kind `reframe` / `scope-change` / `stance` tied to the meta-move's target node or edge. The annotation's `kind` is the proposal's `meta_kind` (the annotation-kind enum already contains those three values), its `content` is the proposal's `content`, and its target rides in `target_node_id` / `target_edge_id` per the meta-move's `target_kind` / `target_id`.

Concretely the change has two halves:
1. **Write side (the real work).** Add a `meta-move` arm to `buildStructuralEventsForCommit` in `apps/server/src/methodology/handlers/commit.ts` that builds the `annotation-created` envelope, mirroring the existing `annotate` arm.
2. **Read side (cleanup).** The `applyCommittedProposal` meta-move arm in `apps/server/src/projection/replay.ts` currently synthesizes a non-UUID annotation id (`meta-move:${target_id}:${meta_kind}:${content.length}`) and `void`s it — a dead M1 sketch. Once the annotation rides its own `annotation-created` event (applied by the existing `handleAnnotationCreated` read-side handler), the commit arm stays a deliberate no-op; the dead synthesis and its misleading TODO are removed and replaced with a one-line pointer to the write-side emission.

The commit-side **acceptance** of meta-move is already in place (see Decisions §"The debt note's two halves") — so the half the source-of-debt note framed as "flip commit_logic's rule 4 rejection to accept" is already true and this task only pins it with a test.

## Why it needs to be done

This is the load-bearing middle link in a three-task chain that the moderator's disputed-visibility work (`moderator_ui.mod_graph_rendering.mod_meta_move_disputed_visibility`, **Done** 2026-06-01) registered as backend debt:

- `mod_meta_move_disputed_visibility` (AC §10/§11) built the **render** seam — `AnnotationRenderData`, `AnnotationFacetStatusIndex`, and the `enrichAnnotationsWithFacetStatuses` selector in `apps/moderator/src/graph/selectors.ts`. That seam is wired with an **empty** index today: it receives no annotations to enrich because committed meta-moves produce no annotation.
- **This task** (`meta_move_commit_logic`) makes committed meta-moves actually emit `annotation-created`, so annotations begin to exist in the projection and reach the render seam.
- `annotation_facet_status_logic` (already registered, 1d, `depends !meta_move_commit_logic`) then routes per-participant votes on the meta-move into per-annotation facet status, populates the `AnnotationFacetStatusIndex`, and lands the Playwright e2e for the full F8 → commit → dispute → disputed-badge round-trip.

Without this task the annotation never comes into being, so neither the facet-status routing nor the disputed-badge rendering downstream has anything to act on. The change is observable at the protocol/replay seam (a committed meta-move now appends an `annotation-created` event to the log, and replay surfaces an annotation on the target), so it warrants a Cucumber+pglite pin in addition to Vitest (per the backend-task coverage policy and ADR 0022).

## Constraints / requirements

- **Emit on the write side, not the read side.** The new `annotation-created` envelope is built in `buildStructuralEventsForCommit` (`commit.ts`), composed into the commit handler's returned `events` array. `replay.ts` is the read-side projection that *consumes* events; it must not append/synthesize new events (doing so would break replay determinism and double-count on re-projection).
- **`annotation_id` is a fresh `randomUUID()`** minted once at commit time (exactly like the `annotate` arm). It is persisted in the event log, so it is stable across every subsequent replay. The previous deterministic-string synthesis is removed — it would fail `annotationCreatedPayloadSchema`'s UUID constraint anyway.
- **`kind` is the proposal's `meta_kind` verbatim** — `annotationKindSchema` already admits `reframe` / `scope-change` / `stance`. No mapping table, no new enum value, no new schema.
- **Target XOR.** `target_kind === 'edge'` → `{ target_node_id: null, target_edge_id: target_id }`; otherwise → `{ target_node_id: target_id, target_edge_id: null }`. Per ADR 0036 the target is always a node or edge, never an annotation.
- **No new event kinds, payload schemas, `RejectionReason` values, or `ProjectionChange` discriminators.** Reuse `annotation-created` and the existing `handleAnnotationCreated` read path end to end.
- **The read-side meta-move arm stays a no-op.** Annotation creation rides the separate `annotation-created` event applied by `handleAnnotationCreated`; `applyCommittedProposal`'s meta-move arm must not also create an annotation (no double-create; `handleAnnotationCreated` would otherwise throw on a duplicate id). The arm's dead synthesized id and stale TODO are removed and replaced with a one-line comment pointing at the write-side emission.
- **Event ordering matches `annotate`.** Structural events take the leading sequence slot(s); the `commit` envelope is last (`[...structuralEvents, commitEvent]`). For meta-move there is exactly one structural event, so the order is `[annotation-created, commit]`.
- **Don't touch the downstream surfaces.** `computeFacetStatuses`, `AnnotationFacetStatusIndex` population, `selectAnnotations` wiring, and the Playwright e2e are `annotation_facet_status_logic`'s scope. This task only makes the annotation exist.
- Verifications per ADR 0022: Vitest at `apps/server/src/methodology/handlers/commit.test.ts`; Cucumber+pglite at `tests/behavior/methodology/commit-meta-move.feature` with step defs under `tests/behavior/steps/`. No throwaway scripts.

## Decisions

- **Emission seam: write-side `buildStructuralEventsForCommit`, not read-side `replay.ts`.** The source-of-debt note (and `meta_move_logic`'s in-code TODO) pointed at `replay.ts`'s `applyCommittedProposal` meta-move arm because that was the architecture when those documents were written — at the time there was no write-side structural fan-out. Since then `commit_logic` (and the subsequent `annotate`-commit work) established `buildStructuralEventsForCommit` as the canonical place a commit mints follow-on events; `annotate` already emits `annotation-created` there. **Chosen:** add a parallel `meta-move` arm to that same function. *Rejected:* emitting from `replay.ts` — the read-side projection consumes events and must stay a pure fold over the log; appending new events during projection would break replay determinism (the event wouldn't be persisted, and re-projection would re-mint a different id) and double-count against the persisted commit. Reusing the existing `annotate` seam is the simpler abstraction with an established precedent (the brief's "reuse existing seams" bias).
- **The debt note's two halves — one is already done.** The note framed scope as *(a)* emit `annotation-created`, and *(b)* "flip `commit_logic`'s rule 4 rejection of meta-move commits to accept." Part (b) is **already true**: the shipped `commit_logic` deferred structural sub-kinds with `'illegal-state-transition'`, but the commit handler has since grown `checkUnanimousAgreeStructural`, which validates structural sub-kinds (meta-move included, with no special case) and reaches the accept path. So this task does **not** implement a rule-4 flip; it confirms the accept path with a Vitest case and the Cucumber scenario. The only net-new behavior is the emission (part a) plus the read-side dead-code cleanup. This is documented rather than silently dropped so the closer/implementer don't go hunting for a rejection that no longer exists.
- **`kind = meta_kind` directly; no mapping, no new ADR.** `annotationKindSchema` already enumerates `reframe` / `scope-change` / `stance` — the data model was built anticipating that committed meta-moves surface as annotations of those kinds (the render seam in `mod_meta_move_disputed_visibility` and `meta_move_logic`'s "synthesize a visible artifact" TODO corroborate this). The architectural decision (annotation kinds include the meta-move kinds; meta-move targets node/edge only) is already captured by the enum and by ADR 0036. Materializing the annotation is an *extension of an existing, ADR-implied pattern* through an existing seam — not a new architectural seam — so **no new ADR is warranted**. *Rejected:* a mapping table from `meta_kind` to a distinct annotation `kind` (unnecessary — the vocabularies coincide by design) and a new ADR (no new dependency, seam, or security trade-off).
- **`annotation_id` is a write-time `randomUUID()`.** Minted once in the handler and persisted, so it is stable across replays — which is what `meta_move_logic`'s replay-side comment was straining to achieve with a deterministic string. *Rejected:* the deterministic `meta-move:${target_id}:${meta_kind}:${content.length}` id — it isn't a UUID (fails `annotationCreatedPayloadSchema`), and it collides if a target receives two meta-moves of the same `meta_kind` and content length. Minting at the write side sidesteps both: the id lives in the log, not in a projection-time computation.
- **Read-side arm stays a no-op; remove the dead synthesis.** With the annotation created by its own `annotation-created` event (applied by `handleAnnotationCreated`), `applyCommittedProposal`'s meta-move arm has nothing structural to do. Leaving the `void annotationId` sketch in place is misleading (it reads as "we compute an id here") and risks a future double-create if someone wires it up. *Chosen:* delete the synthesis, keep an explicit no-op with a comment pointing at the write-side emission. This is in-scope cleanup, not a behavior change.
- **Coverage: Cucumber at the protocol seam + Vitest for the handler.** The change is observable at the replay/protocol boundary (a committed meta-move now appends `annotation-created` to the log and surfaces an annotation on replay), so per the backend-task coverage policy and ADR 0022 it gets a Cucumber+pglite scenario, not Vitest alone. Vitest pins the handler's event-shaping (the `[annotation-created, commit]` ordering and payload fields) the way the `annotate` test already does.
