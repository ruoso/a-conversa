# Interpretive-split edge inheritance — reading nodes inherit the parent's committed edges

**TaskJuggler entry**: [tasks/10-data-and-methodology.tji](../../10-data-and-methodology.tji) — task `data_and_methodology.methodology_engine.interpretive_split_edge_inheritance`

## What this task is

Make interpretive-split commits actually produce the inherited edges the design docs specify. Today, when an `interpretive-split` proposal commits, the reading nodes (already minted at propose time) replace the parent in the visible graph — but the parent's committed edges vanish with it (invisible by missing endpoint, per the visible-graph derivation in `docs/data-model.md`), and nothing takes their place. Per `docs/example-walkthrough.md` turn 17 (L165–169), each reading node must instead take an inherited copy of each of the parent's committed outgoing edges, with the edge substance pre-commitment carrying over: when N14 (which rebuts N11, substance pre-committed agreed) splits into readings N16/N17, the commit yields E11a (N16 rebuts N11) and E11b (N17 rebuts N11), each substance-committed by carry, while N16's and N17's own substance stays proposed.

The design is settled by [ADR 0046](../../../docs/adr/0046-interpretive-split-edge-inheritance-commit-time-carry.md), written with this refinement:

1. **Commit-time mirror.** The commit handler's structural fan-out (`buildStructuralEventsForCommit`, `apps/server/src/methodology/handlers/commit.ts` L366–435 — the seam the `annotate` and `meta-move` branches already use) grows an `interpretive-split` branch. At commit, for each reading node × each qualifying parent edge it emits `edge-created` (fresh id; the parent edge's `role`; the reading node as source; the parent edge's target endpoint carried verbatim, node or annotation) + `entity-included`.
2. **Qualifying parent edge** = parent is the **source** (`projection.getEdgesBySource(parent_node_id)`, `apps/server/src/projection/projection.ts` L365–371), edge currently included and visible, and substance facet has a **landed commit** (the terminal state the `set-edge-substance` commit arm produces — `replay.ts` L1177–1192: value set, status `'agreed'`, `committedProposalEventId` recorded). Target-side edges and proposed-substance edges never inherit.
3. **Substance carry as an explicit facet-layer event.** Per inherited edge, the fan-out also emits a facet-keyed `commit` event with the new optional payload field `carried_from_edge_id` on `facetCommitPayloadSchema` (`packages/shared-types/src/events.ts` L517–525). Appliers resolve the referenced parent edge at apply time and land the inherited edge's substance facet in the same terminal state, copying its committed value. Per-participant vote state is **not** copied — provenance lives in the event field; the inherited facet behaves like any committed facet afterwards (withdrawal supersedes normally).
4. **Cluster ordering.** Per inherited edge: `edge-created`, `entity-included`, `commit{facet, carried_from_edge_id}`; the whole fan-out precedes the split's own proposal-keyed `commit` (the existing sequence offsetting at `commit.ts` L547–561 already does this), so appliers see the parent edge still in place when resolving the carry.

Consumers to extend: the server projection's facet-keyed commit application path in `replay.ts` (the path that lands `set-edge-substance` commits) learns `carried_from_edge_id` and emits the corresponding `facet-updated` change-feed entry; the shared client facet-status walker (`packages/shell/src/facet-status/facet-status.ts`, commit arm L498–551, facet-keyed at L505–514) learns the same carry. The canvas projectors need **no** split-specific edge logic — inherited edges arrive as ordinary `edge-created` events through the generic arm (`packages/graph-view/src/projectGraph.ts` L546–595), which both the moderator and participant/audience surfaces consume; sweep for any remaining facet-status mirrors (`grep -rn "deriveFacetStatus\|computeFacetStatuses" apps/ packages/`) and extend any that still derive edge facet state independently.

Finally, the task closes the loose end in the predecessor: add a note-level pointer in `tasks/refinements/data-and-methodology/interpretive_split_logic.md` (under its Open questions / Status area, without rewriting historical sections) recording that commit-time edge inheritance is settled by this task and ADR 0046.

## Why it needs to be done

`docs/example-walkthrough.md` turn 17 is explicit and the methodology owner confirmed at the 2026-06-10 parking-lot triage that inheritance is intended behavior. The implementation has none of it: `propose.ts` mints the reading nodes but emits zero `edge-created` events for inherited edges, and the projection's commit arm only flips parent visibility. The observable consequence is methodological, not cosmetic — a committed rebut relation (a participant's pre-commitment, `docs/methodology.md` L119–121) silently evaporates when the moderator splits the statement it hangs off, destroying agreement state that participants never withdrew. The conditional reading of edge substance (`docs/data-model.md` L100–104) is what makes the carry sound: each inherited edge sits pre-committed but inert until its reading node's own substance is established — exactly the defeater semantics.

The e2e coverage that should have caught this is itself debt: `tests/e2e/full-session-walkthrough.spec.ts` AC-5 (L804–913) asserts "at least one `rebuts` edge visible" after the split (L910–912), which passes incidentally via an unrelated edge from AC-3b. This task tightens that assertion to pin the true inherited edges.

The milestone `m_audits` ("M8-audits", `tasks/99-milestones.tji` L99–103) depends on this task; it was registered from the 2026-06-10 parking-lot triage alongside its sibling conformance fix `mod_decompose_split_parent_visibility`.

## Constraints / requirements

- **Layer separation (ADR 0027) is strict.** The entity events (`edge-created` / `entity-included`) carry no facet effect; the substance carry is its own facet-keyed `commit` event. No applier may set facet state from an entity event.
- **The inheritance predicate is computed at commit time** against the live projection — not snapshotted at propose time. Source-side only; committed-substance only; included-and-visible only. Each qualifying edge mirrors onto *each* reading node (N readings × M edges).
- **Vote state does not carry.** The inherited edge's substance facet gets value + terminal status + this commit event's id/timestamp; `perParticipant` stays empty; original attribution is reachable via `carried_from_edge_id`.
- **Interpretive-split only.** Decompose components do not inherit (ADR 0046 Decision §4 — bundle-level relations don't distribute over distinct claims). Don't generalize the branch.
- **Schema change is additive** — `carried_from_edge_id` optional on `facetCommitPayloadSchema`; all existing `commit` payloads stay valid; no SQL migration (the `CHECK` constrains kinds only).
- **Replay determinism.** A carried commit whose `carried_from_edge_id` doesn't resolve in the projection is a `ReplayError` (same posture as the existing commit arms' missing-entity checks).
- **Edge ids are minted by the commit handler** (fresh UUIDs, as the `annotate` branch does for annotations); `created_by` is the committing actor, `created_at` the commit timestamp.
- **Zero-inheritance commits stay unchanged.** A split whose parent has no qualifying edges emits exactly today's event shape (no empty-cluster artifacts).
- Build + full test suite green before commit (global gate); doc-only pointer edit to the predecessor refinement rides the same commit.

## Decisions

- **Commit-time mirror, not propose-time minting.** *Alternatives*: (a) emit inherited `edge-created`s in `buildStructuralEventsForPropose` alongside the reading nodes — *rejected*: the inherited set would be a propose-time snapshot, silently stale if another edge on the parent commits mid-vote; a rejected/withdrawn split would leave mirrored-edge debris; and the walkthrough times inheritance at commit ("when split commits … each with rebut edges"). (b) Commit-time (chosen) — legitimate under ADR 0027 §2's commit-time-creation carve-out (the mirrors are a consequence of supersession computed from graph state at commit, not proposal content), and the existing `buildStructuralEventsForCommit` seam already does per-sub-kind commit-time fan-out for `annotate`/`meta-move`. Participants voting on the split accept the deterministic inheritance *rule*, not a frozen edge list. Recorded in ADR 0046.
- **Carry via facet-keyed `commit` + `carried_from_edge_id`, not entity-event side effects or projector-side derivation.** *Alternatives*: an `inherited_from_edge_id` on `edge-created` whose applier also sets substance — *rejected*, entity event with facet effect is the exact conflation ADR 0027 bans; a bare facet commit with each consumer re-deriving the inheritance — *rejected*, duplicates methodology logic across server projection and every client surface (the same reason ADR 0027 rejected ghost synthesis). The explicit carried field keeps the event log self-describing and both appliers one-lookup simple. Recorded in ADR 0046.
- **Source-side committed edges only.** Inbound edges with agreed substance + agreed source would *fire immediately* against readings nobody evaluated — manufacturing agreement, the precise confusion the split exists to dissolve. Outbound carries are safe under the conditional reading (firing stays gated on the reading's own proposed substance). Proposed-substance edges are pending negotiations about a statement that's being superseded; participants re-propose against the readings explicitly, matching the restructure posture (`docs/data-model.md` L304–305). Recorded in ADR 0046.
- **No decompose inheritance.** `docs/methodology.md` L187's "same graph treatment" covers parent replacement, not edge carry; readings are alternative construals of one claim (relations transfer), components are distinct bundled claims (relations against the bundle don't distribute over conjuncts). Nothing in the docs specifies decompose-side inheritance and the triage registered none. Deliberately out of scope — not registered as future work. Recorded in ADR 0046.
- **Vote state not copied.** Copying `perParticipant` wholesale would entangle withdrawal semantics with phantom votes the participants never cast against *this* edge. The carry transfers the committed outcome; attribution ("pre-committed agreed by Ben") is reachable through the provenance field. Walkthrough fidelity is preserved at the level it specifies — the pre-commitment carries.
- **AC-5 tightened in place rather than a new spec.** The assertion exists and is wrong (incidentally green); tightening it pays down the debt the WBS note names. The deliberately-deferred parent-removal canvas assertion stays with `mod_decompose_split_parent_visibility` to keep the two triage-registered tasks' scopes disjoint.
