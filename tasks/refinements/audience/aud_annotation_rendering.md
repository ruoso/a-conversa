## What this task is

The 1d leaf that lands **per-annotation badges** on the audience broadcast canvas — a row of small amber pills painted just below every node that carries at least one committed annotation, one pill per `annotation-created` event targeting the node, each carrying the localized annotation-kind label (`Note` / `Reframe` / `Scope change` / `Stance` for en-US, translated equivalents for pt-BR / es-419) plus the annotation's full `content` body surfaced via the `title` attribute for hover-tooltip readability. Each badge is a small rounded-pill with the kind label rendered in `uppercase tracking-wide text-[10px]` amber-100/amber-900 Tailwind classes — same shape and palette as the moderator's `<AnnotationBadge>`. Multiple annotations on the same node stack as a horizontal flex-row below the axiom-mark row — same "row of small chips per node" shape as the per-facet pill row and the axiom-mark badge row, completing the audience canvas's three-decoration-layer vocabulary (agreement layer above the node; per-participant disposition layer below the node; meta-commentary layer below the disposition layer).

Because the audience renders to a Cytoscape `<canvas>` and there is no React tree per node, the badge row CANNOT live inside the node card (Cytoscape's nodes are pixels, not React subtrees). The badge row lands as a **DOM-overlay sibling** of the Cytoscape mount — the third overlay added to the existing `audience-graph-root-wrapper` positioning ancestor (the per-facet pill overlay is the first, the axiom-mark overlay is the second, the annotation overlay is the third). The overlay subscribes to the Cytoscape `Core`'s pan / zoom / render / position / add-remove-data events and re-paints (rAF-batched) so the badge rows track each node's screen position as the viewport moves — same singleton-rAF pattern, same subscription set, same `useState<Core | null>` slot.

This leaf is the **seventh** sibling under `aud_graph_rendering` (after the four state-styling refinements, the per-facet pill overlay, the axiom-mark badge overlay, and the stylesheet-extraction housekeeping) and the **third** DOM-overlay surface on the audience workspace. The pattern lift from `<AudienceAxiomMarkOverlay>` is direct — same wrapper structure (`pointer-events-none absolute inset-0`), same subscription set (`render pan zoom resize` + `position node` + `add remove data`), same singleton-rAF handle, same `<useState<Core | null>>` arrival pattern, same `containerRef` prop shape. The differences are local: the per-element field read is `data.annotations` (not `data.axiomMarks`); the per-element child is a flex-row of `<AudienceAnnotationBadge>` chips iterated in commit-arrival order from the `Annotation[]` array (same iteration shape as axiom marks — array-order is methodology-natural here because annotations don't have a canonical "reading order" like the facet `wording → classification → substance`); the anchor is below the axiom-mark row (`renderedBoundingBox().y2 + ANNOTATION_ROW_OFFSET_Y` with `ANNOTATION_ROW_OFFSET_Y = 30`) rather than just below the node bounding box.

## Why it needs to be done

The `m_audience_mvp` milestone in [`tasks/99-milestones.tji`](../../99-milestones.tji) names the entire `aud_graph_rendering` group transitively. The four state-styling siblings, the per-facet pill overlay, and the axiom-mark badge overlay land the **agreement layer** (proposed / agreed / disputed / per-facet pills) and the **per-participant disposition layer** (axiom marks) on the broadcast canvas. But the **meta-commentary layer** — the methodology's mechanism for capturing reframes, scope-changes, stance-pins, and free-form notes on existing entities — stays invisible until this leaf ships. From [`docs/methodology.md` § "Annotations"](../../../docs/methodology.md): annotations are first-class methodology entities that record participant context (scope boundaries, methodological stances, concerns) without being part of the argument itself; from [`docs/data-model.md` § "Annotations"](../../../docs/data-model.md), the schema treats annotations as a third entity kind alongside nodes and edges (their own `session_annotations` M-N join table at [`packages/shared-types/src/events/enums.ts:45`](../../../packages/shared-types/src/events/enums.ts#L45)).

The methodology assumes the broadcast audience sees the same annotation signal the moderator does. Without this leaf, an `annotation-created` event lands silently in the audience event log and the broadcast viewer has no visual confirmation that meta-commentary on an entity has been recorded — defeating the methodology's "meta-commentary layer is observable on both sides" assumption on the very surface designed to expose the conversation's narrative to a viewing audience.

The audience reads the **per-annotation list** (NOT just the per-target boolean the participant chose) for the same reason `aud_axiom_mark_decoration` rendered per-participant chromatic badges instead of a boolean overlay: the methodology's meta-commentary content IS the methodology — a reframe and a stance-pin are different narrative moves, and the broadcast viewer needs to read the kind label at minimum (and the content body via hover tooltip) to follow the conversation. Collapsing all annotations on a node to "this node has annotations" would silently erase the methodology's meta-commentary content for broadcast viewers.

Concretely:

- **`aud_session_url`** ([`tasks/50-audience-and-broadcast.tji:399`](../../50-audience-and-broadcast.tji#L399)) wires the audience surface into a `/sessions/:id` route. Once that route lands, broadcast viewers see the full canvas vocabulary — state styling + per-facet pills + axiom-mark badges + annotation badges — every published audience leaf renders together. The methodology's narrative becomes complete on the broadcast surface only after this leaf ships.
- **`aud_visual_regression`** ([`tasks/50-audience-and-broadcast.tji:469`](../../50-audience-and-broadcast.tji#L469)) is the pixel-comparison task that already inherits typography + layout + per-state-styling + per-facet pill-row + axiom-mark-badge pixel-stability deferrals; the annotation badge palette + position pixel-stability lands here too per Decision §6.
- **`shell_package.extract_cytoscape_projectors`** ([`tasks/27-shell-package.tji:101-108`](../../27-shell-package.tji#L101)) is the already-registered 1d task that lifts `projectAnnotations` + `groupAnnotationsBy{Node,Edge}` (alongside other multi-caller projectors) into `@a-conversa/shell`. The audience leaf is the third caller — the documented trigger fires here. No new follow-up task is registered; the closer references `extract_cytoscape_projectors`'s existing note when adding the `complete 100` marker to this leaf.

Architecturally, this leaf parallels the moderator's `mod_annotation_rendering` (shipped 2026-05-11) but renders via a Cytoscape-canvas DOM overlay rather than ReactFlow custom-node decoration. The moderator's `<StatementNode>` is a ReactFlow React subtree; the audience's nodes are Cytoscape canvas pixels. The DOM-overlay pattern proven by `<AudiencePerFacetPillOverlay>` and `<AudienceAxiomMarkOverlay>` is the canonical answer to "render arbitrary React per Cytoscape element"; this leaf adopts it directly. Same amber pill, same per-kind label vocabulary, same `data-annotation-kind` seam, same `title` hover content — only the rendering technology differs.

## Decisions

### §1 — Per-annotation DOM-overlay badges with localized kind labels (NOT boolean+count overlay; NOT Cytoscape-native; NOT deferred to detail panel)

The participant's `part_annotation_render` (Decision §1 there) chose a **boolean `hasAnnotation` + numeric `annotationCount`** overlay painted via Cytoscape stylesheet selectors (`node[?hasAnnotation]` + `edge[?hasAnnotation]` with amber `overlay-*` / line-gradient styling), explicitly deferring per-annotation chromatic identity / content / kind labels to a future detail-panel React surface (`part_entity_detail_panel`). Three options for the audience:

- **(A — chosen)** Render **per-annotation badges** as a DOM-overlay sibling — one amber-pill badge per `annotation-created` event targeting the node, each carrying the localized annotation-kind label (`Note` / `Reframe` / `Scope change` / `Stance`) plus the full `content` body via the `title` attribute. Stacked as a horizontal flex-row below the axiom-mark row. The badge is a verbatim port of the moderator's `<AnnotationBadge>` (same Tailwind class set, same `useTranslation` key resolution, same `data-annotation-kind` seam, same `title={content}`). The overlay is a verbatim transposition of the axiom-mark overlay pattern (positioning ancestor + `pointer-events-none absolute inset-0` sibling + rAF-batched subscription).
- **(B)** Render a **boolean+count overlay** like the participant — one Cytoscape stylesheet selector `node[?hasAnnotation]` with an amber tint on annotated nodes; no per-annotation kind label, no content surfacing. Cost: the methodology's meta-commentary content ("Ben notes the accredited/unaccredited boundary does argumentative work" — actual prose authored by a participant during the conversation) becomes invisible on the broadcast surface; the broadcast viewer sees "this node has annotations" with no indication of *what* the annotations say. The participant tablet can defer because it has a tap-to-detail panel future-home; the broadcast surface has no detail panel, no tap interaction, and no fallback path for surfacing annotation content. Rejected — same rejection rationale as `aud_per_facet_visualization` (Decision §1) and `aud_axiom_mark_decoration` (Decision §1): when the participant's deferral path doesn't exist on the audience (no detail panel, no tap), the audience inverts the participant's collapse and surfaces the load-bearing detail on the canvas itself.
- **(C)** Defer per-annotation rendering entirely — ship rollup-only on the audience and register a future task. Cost: the methodology's meta-commentary layer stays invisible on the broadcast canvas. Same rejection rationale as the per-facet pill overlay's (C) option and the axiom-mark overlay's (C) option: deferring a methodology-load-bearing surface means the broadcast viewer cannot read a layer of the conversation. Rejected.

The choice (A) is the consistent translation of the per-facet pill overlay's and axiom-mark badge overlay's posture: the audience surface inverts the participant's detail-panel-deferral whenever the load-bearing semantics are broadcast-visible. The methodology's meta-commentary layer IS the annotation content — collapsing it to a boolean+count for the broadcast surface would erase the narrative content.

Alternative — render the badge row INSIDE the per-facet pill row (combine both overlays into one). Rejected — the two overlays surface different layers (per-facet status is the **agreement** layer; annotations are the **meta-commentary** layer; per ADR 0027 the layers should not visually compete). The spatial separation (pills above the node, axiom marks below the node, annotations below the axiom-mark row) makes each layer boundary visible to broadcast viewers (Decision §4 below).

Alternative — render the kind label only (no `title={content}` carrier). Rejected — the methodology's load-bearing content IS the annotation's content body; surfacing only the kind label ("Note", "Reframe") without any way to read what the note or reframe says would amount to the same erasure (B) above does, just with a slightly richer at-a-glance signal. The `title` attribute is the cheap-baseline hover carrier the moderator already chose; the audience adopts it verbatim.

### §2 — Scope to node-targeted annotations only; register `aud_annotation_rendering_edges` (~1d) as named-future-task for edge-targeted

The wire schema is XOR by construction — `(target_node_id === null) !== (target_edge_id === null)` per [`packages/shared-types/src/events.ts:333`](../../../packages/shared-types/src/events.ts#L333) — and the methodology assigns equal weight to node-targeted and edge-targeted annotations. But landing a DOM-overlay anchored to a Cytoscape edge is non-trivial:

- Edges have **midpoints**, not bounding boxes (`renderedBoundingBox()` works for nodes; edges expose `midpoint()` and `sourceEndpoint()` / `targetEndpoint()` instead).
- The badge geometry would need to account for source/target node positions, edge curvature, the existing role label that already occupies the edge midpoint zone, and label collisions when edges cross.
- The audience's breadthfirst layout (per `aud_layout_engine`) produces fan-out edges from a single root; multiple edges sharing a node can produce overlapping midpoint zones.

Three options:

- **(A — chosen)** Scope this leaf to node-targeted annotations and register **`aud_annotation_rendering_edges`** (~1d, named-future-task — closer adds to WBS at task close) as the follow-up. Mirrors the `aud_per_facet_visualization_edges` deferral pattern (registered by `aud_per_facet_visualization` for the same reason).
- **(B)** Scope this leaf to both targets — node + edge — in 1d. Rejected: the edge-midpoint geometry is its own ~0.5d concern (Cytoscape midpoint computation, collision with the role-label overlay, source/target endpoint handling for curved edges); bundling it into 1d inflates scope past the budget. The moderator's `mod_annotation_rendering` had 1d for both targets, but it landed in a ReactFlow surface with native React subtrees per edge label (`<EdgeLabelRenderer>` at the edge midpoint already handles the geometry); the audience's Cytoscape edges have no native React mount per edge, so re-creating the geometry primitive is the bulk of the work.
- **(C)** Render edge-targeted annotations via a Cytoscape stylesheet selector (the participant's path — `edge[?hasAnnotation]` with a gradient stroke). Rejected — the audience inverted the participant's boolean overlay for nodes (Decision §1); inverting it for nodes and NOT for edges would produce an asymmetric surface where node annotations carry per-kind labels and edge annotations carry only an amber stroke. The asymmetry would be more confusing than uniform deferral; deferring both edge concerns to the future task keeps the audience canvas's annotation vocabulary uniform once `aud_annotation_rendering_edges` ships.

The named-future-task `aud_annotation_rendering_edges` description (for the closer to register):

> Extend `<AudienceAnnotationOverlay>` to also render annotations whose `target_edge_id` is set. Imports `groupAnnotationsByEdge` from `@a-conversa/shell` (post-`extract_cytoscape_projectors`) or from `apps/audience/src/graph/annotations.ts` (pre-extract, if that follow-up hasn't landed; this task adds the helper to the audience-local file too). Edges anchor badges at the rendered edge midpoint (`cy.edges().forEach(edge => edge.midpoint())`); `AudienceEdgeData` grows `annotations: readonly Annotation[]`; tests pin the symmetric edge-rendering case. Effort 1d. Inherits Decisions §1, §3, §4-modified-for-edges, §5, §6 from `aud_annotation_rendering`.

Cross-surface consistency: `aud_per_facet_visualization` registered the same shape of follow-up for the same shape of reason. The "scope to nodes; register edge follow-up" posture is the audience's canonical answer when the wire schema admits both targets but the Cytoscape edge geometry adds enough complexity to warrant its own ~1d task.

### §4 — Badge row anchor: BELOW the axiom-mark row at `y2 + 30` (not above the pill row, not inside the node, not horizontal)

The audience canvas's vertical decoration stack is now well-established:

- ABOVE the node bounding box (at `y1 - PILL_ROW_OFFSET_Y = y1 - 6`): per-facet pill row (agreement layer, per `aud_per_facet_visualization` Decision §2).
- INSIDE the node bounding box: the entity content (wording — Cytoscape canvas pixels, the methodology entity layer per ADR 0027).
- BELOW the node bounding box at `y2 + AXIOM_BADGE_ROW_OFFSET_Y = y2 + 6`: axiom-mark badge row (per-participant disposition layer, per `aud_axiom_mark_decoration` Decision §4).

Four approaches to the annotation badge row's anchor position:

- **(A — chosen)** Anchor at `y2 + ANNOTATION_ROW_OFFSET_Y` with `ANNOTATION_ROW_OFFSET_Y = 30` — below the axiom-mark row. The axiom-mark row sits at `y2 + 6` and is ~20 px tall (the `h-5 w-5` ring-1 chip per `<AudienceAxiomMarkBadge>`); 6 + 20 = 26 px would touch the axiom-mark row's bottom edge, so 30 leaves a 4 px breathing gap. When the axiom-mark row is empty (no committed marks on the node), the annotation row sits at `y2 + 30` standalone — the small gap between the node's bottom edge and the annotation row signals "decoration here" without ambiguity about which layer it belongs to. The fixed offset is decoupled from the per-element axiom-mark row's actual rendered height (badges are uniformly sized; multi-row stacks don't appear because axiom-marks are a single flex-row per node).
- **(B)** Anchor ABOVE the per-facet pill row (at `y1 - PILL_ROW_OFFSET_Y - PILL_ROW_HEIGHT - ANNOTATION_ROW_OFFSET_Y_ABOVE`). Rejected: stacks three rows above the node, which competes with the breadthfirst layout's vertical flow (nodes one layer up sit at the same x-band as the current node; their annotation rows would collide). The "decorations below the node" posture set by `aud_axiom_mark_decoration` is the more compositionally robust choice — vertical stack growing downward from the node.
- **(C)** Anchor INSIDE the node (overlay the badge row on top of the node's wording). Rejected — overlays the entity-layer content (wording) with the meta-commentary-layer decoration; per ADR 0027 the layers should not visually compete. The "decorations adjacent to the node" posture is what the predecessor overlays established.
- **(D)** Anchor horizontally to the right of the node (at `x2 + ANNOTATION_ROW_OFFSET_X`). Rejected — same reason as the per-facet pill row's left/right rejection (per `aud_per_facet_visualization` Decision §2) and the axiom-mark row's left/right rejection (per `aud_axiom_mark_decoration` Decision §4): the 200×80 node box is wider than tall; horizontal anchors compete with edge labels and don't compose with the breadthfirst layout's left-to-right flow.

Cross-surface consistency: the moderator's `mod_annotation_rendering` places annotation badges in a row INSIDE the node card (`mt-1 flex flex-wrap gap-1` below the kind-label decoration row at `apps/moderator/src/graph/StatementNode.tsx`). The audience cannot mirror "inside the node" because the node body is Cytoscape canvas pixels; the next-best translation is "spatially below the node, after the per-participant disposition layer" — the meta-commentary layer is methodologically the layer furthest from the entity content, so it sits furthest in the spatial stack too.

Alternative — anchor at `y2 + 6` and place annotations BEFORE axiom marks (above them spatially). Rejected — the axiom-mark layer is per-participant disposition (a per-participant statement about the node), which is methodologically closer to the entity content than the meta-commentary layer (which is commentary ABOUT the disposition). The "closer-to-entity layer sits closer to the entity in screen space" principle puts axiom marks at `y2 + 6` and annotations at `y2 + 30`.

### §5 — Subscribe to `render pan zoom resize` + `position node` + `add remove data`; rAF-batched; reuse the existing `cyState` slot

Direct mirror of `aud_axiom_mark_decoration`'s Decision §5 — same subscription set, same singleton-rAF handle, same `useState<Core | null>` slot in `<AudienceGraphView>` (the slot exists at [`apps/audience/src/graph/GraphView.tsx:218`](../../../apps/audience/src/graph/GraphView.tsx#L218); no new slot needed; all three overlays receive the same `cy` prop).

Adding a third subscriber to the same `Core` instance is mechanically safe — Cytoscape's `cy.on(...)` is a multi-subscriber dispatcher; the three overlays' listeners coexist without interaction. Each overlay owns its own `cy.off(...)` teardown set keyed on the handler reference passed to `cy.on(...)`, so cleanups are isolated.

Alternative — share a single rAF batch across all three overlays (centralize the scheduler in `<AudienceGraphView>` and pass a "scheduler" prop down). Rejected — the orchestrator brief's "two callers is YAGNI" applies but the inverse also applies: three callers is the threshold to *validate* the pattern, not the threshold to abstract the scheduler. The per-overlay rAF singleton is a small primitive (~5 lines per overlay); centralizing would couple the three overlays' lifecycles and require routing scheduler ownership through props or a new context. The duplication cost is bounded and the abstraction would couple consumers that should stay independent (a future fourth overlay should be able to land without touching the existing three).

If a future audience leaf adds a fourth or fifth overlay sibling, the question of centralizing the scheduler may warrant its own refinement; today three instances of the same five-line primitive is below the threshold for abstraction.
