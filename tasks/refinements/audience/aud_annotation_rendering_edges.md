## What this task is

The 1d follow-up that completes the meta-commentary layer on the audience broadcast canvas by extending `<AudienceAnnotationOverlay>` to also paint badge rows at the rendered midpoint of every Cytoscape edge carrying at least one committed annotation. The parent leaf shipped node-targeted annotations only (per its Decision §2's explicit scope cap), leaving every `annotation-created` event whose `target_edge_id` is set silently invisible on the broadcast canvas — even though such events ARE absorbed by `projectAnnotations(events)`, they're skipped by `groupAnnotationsByNode`'s XOR-aware filter, and no edge sees them. This leaf closes that gap.

## Why it needs to be done

The methodology assigns equal weight to node-targeted and edge-targeted annotations: from [`docs/methodology.md` § "Annotations"](../../../docs/methodology.md), an annotation may target an existing node OR an existing edge to record meta-commentary on either kind of entity. From [`docs/data-model.md` § "Annotations"](../../../docs/data-model.md), the schema treats annotations as a third entity kind alongside nodes and edges with the polymorphic-FK XOR encoding ([`packages/shared-types/src/events.ts:333`](../../../packages/shared-types/src/events.ts#L333)). The wire schema, the moderator surface, and the participant surface all support both targets; the audience's parent leaf was deliberately scoped to node-only to keep the 1d budget honest, with the understanding that this leaf would close the gap.

Without this leaf, `annotation-created` events with `target_edge_id` set land silently in the audience event log — `projectAnnotations` absorbs them (they appear in the flat array), but `groupAnnotationsByNode` skips them (XOR), and no consumer reads them. A broadcast viewer watching a session where the moderator annotates a `rebuts` edge ("Ben notes this rebuttal applies only to the accredited subset of cases") would see the annotation appear in the moderator's view, see the annotation appear in the participants' tablets, and see NOTHING on the broadcast canvas — silently dropped. That asymmetry defeats the methodology's "meta-commentary layer is observable on both sides for both entity kinds" assumption on the very surface designed to expose the conversation's narrative to a viewing audience.

Concretely:

- **`aud_session_url`** ([`tasks/50-audience-and-broadcast.tji:399`](../../50-audience-and-broadcast.tji#L399)) wires the audience surface into a `/sessions/:id` route. Once that route lands, broadcast viewers see the full canvas vocabulary — state styling + per-facet pills + axiom-mark badges + annotation badges (now on both node and edge entities). The methodology's narrative becomes complete on the broadcast surface only after this leaf ships AND `aud_session_url` lands.
- **`aud_visual_regression`** ([`tasks/50-audience-and-broadcast.tji:469`](../../50-audience-and-broadcast.tji#L469)) is the pixel-comparison task that already inherits seven sibling deferrals' work; this leaf extends the inherited deferral note to cover the edge case alongside the node case at the same broadcast-resolution matrix.
- **`shell_package.extract_cytoscape_projectors`** ([`tasks/27-shell-package.tji:101-108`](../../27-shell-package.tji#L101)) — the third-caller trigger already fired with the parent leaf; this leaf is mechanically the same shape of caller (consumes `groupAnnotationsByEdge` from the audience-local file today; closer documents whether the import path moves to `@a-conversa/shell` at this leaf's task-close depending on whether the extract has landed first).

Architecturally, this leaf parallels the moderator's `<StatementEdge>` annotation rendering at [`apps/moderator/src/graph/StatementEdge.tsx:308-317`](../../../apps/moderator/src/graph/StatementEdge.tsx#L308) (which renders the badge row directly under the role-label pill via ReactFlow's `<EdgeLabelRenderer>`). The audience's Cytoscape canvas has no native React mount per edge — the DOM-overlay-sibling pattern proven by `<AudienceAnnotationOverlay>` already extends to nodes; this leaf extends the same overlay's commit pass to also iterate `cy.edges()`. Same amber pill, same per-kind label vocabulary, same `data-annotation-kind` seam, same `title` hover content — only the placement source differs (`renderedBoundingBox()` center on the edge collection vs. `y2 + ANNOTATION_ROW_OFFSET_Y` on nodes).

## Decisions

### §1 — Single overlay component iterates both `cy.nodes()` AND `cy.edges()` in the same commit pass (NOT two separate overlay components, NOT a separate edge-overlay sibling)

Three options for where the edge branch lives:

- **(A — chosen)** Extend the existing `<AudienceAnnotationOverlay>`'s `useEffect` commit closure to also iterate `cy.edges()`. The render path doesn't care about the source kind (it iterates `placements` and emits one positioned row per placement); the only branch-specific concern is the placement-coords computation (node uses `y2 + ANNOTATION_ROW_OFFSET_Y`; edge uses `midpoint + EDGE_ANNOTATION_OFFSET_Y`). The placement record can keep its existing flat shape (no `kind` discriminator); the consumer's iteration is uniform.
- **(B)** Land a separate `<AudienceAnnotationEdgeOverlay>` component as a fifth child of `audience-graph-root-wrapper`. Each overlay subscribes to its own listeners. Cost: doubles the subscription set on the same `Core` instance (Cytoscape coalesces, but it's still extra closure setup at mount), routes the same `cy` prop through two parallel components, requires a parallel test file. The architectural seam is the same regardless — both branches read the same `data.annotations` field — so the second component would be parallel scaffolding without parallel substance. The parent leaf's Decision §5 already approved adding a third subscriber to the same `Core` instance; this leaf would add a fourth subscriber, when one subscriber covers both branches.
- **(C)** Land a separate `<AudienceAnnotationEdgeOverlay>` component but composed inside `<AudienceAnnotationOverlay>` (an internal sub-component). Cost: the same cost as (B) without the parallel `<GraphView>` mounting — but the cost remains "two listener closures where one suffices" and "two test files where one (with new cases) suffices."

Chosen: (A). The cost of (B) and (C) is real (extra subscription, parallel scaffolding) and the savings are negligible (the edge-iteration branch is ~10 lines added to the existing commit closure). The parent leaf's overlay already has a placement-coords function on one element kind; this leaf parameterizes the placement-coords by element kind and keeps the rest uniform.

Alternative — emit a `kind: 'node' | 'edge'` discriminator on the placement record and branch the render path on the discriminator. Rejected — the render path is identical for both kinds (same wrapper, same badge children, same positioning style attributes); the discriminator would be pure inert metadata. The placement record stays flat.

Alternative — emit the placement record's `id` field as a namespaced string (`node-<id>` vs `edge-<id>`) to disambiguate in tests. Rejected — node and edge ids are globally unique in the data model (the entity-id space is shared across kinds; the wire schema's XOR enforces unique entity ids across both kinds for any single session). The id field stays as the raw entity id; tests asserting `data-element-id="<id>"` match the same id as the entity. (If a future schema change introduces id collisions across kinds, this assumption would need to be revisited.)

### §3 — Edge badge anchor at `edge.renderedBoundingBox()` center plus `EDGE_ANNOTATION_OFFSET_Y = 18` vertical offset (NOT `edge.midpoint()` with manual pan/zoom transform, NOT directly at the midpoint with no offset, NOT above the role label)

The audience renders the edge role label at the edge midpoint via the Cytoscape stylesheet (`label: 'data(roleLabel)'` at [`apps/audience/src/graph/stylesheet.ts:197`](../../../apps/audience/src/graph/stylesheet.ts#L197), with `font-size: 12px` per the typography token set). The Cytoscape-rendered text label occupies a bounding box of roughly 14-16 px vertical at the midpoint; the badge row needs to land somewhere clear of the label.

Four approaches:

- **(A — chosen)** Use `edge.renderedBoundingBox()` and compute the center as `((bb.x1 + bb.x2) / 2, (bb.y1 + bb.y2) / 2)`, then offset by `EDGE_ANNOTATION_OFFSET_Y = 18` vertically downward. The `renderedBoundingBox()` API is the symmetric primitive to the one the parent leaf already uses for nodes; it returns screen-space coords accounting for pan + zoom, no manual transform required. For straight edges, the bounding-box center equals the midpoint exactly. For curved or angled edges, the bounding-box center is approximate (a slight skew compared to the geometric midpoint), but the role label is rendered at the same midpoint and shifts under the same skew, so badge + label maintain their relative offset.

  The 18 px offset value: the role label at 12 px font + Cytoscape's text-padding renders ~14 px vertical; plus a 4 px visual breathing gap below the label bottom edge, the badge row's top edge sits 18 px below the bounding-box center, which is below the label's bottom edge. The badge row is itself ~16 px tall (matching the moderator's amber-100/amber-900 pill height with text-[10px]); the row extends from y+18 to y+34 below the midpoint.

  Symmetric to the node anchor: nodes use `y2 + 30` (below the bounding box, 30 px gap = clear-the-axiom-mark-row); edges use `midpoint + 18` (below the midpoint, 18 px gap = clear-the-role-label).

- **(B)** Use `edge.midpoint()` (model-space coords) and apply a manual `cy.pan() + cy.zoom()` transform to convert to screen coords. Cost: extra arithmetic in the commit closure that duplicates what `renderedBoundingBox()` already does internally; risk of drift if the Cytoscape pan/zoom conventions change. Rejected — `renderedBoundingBox()` is the documented Cytoscape API for screen-space element placement; using the manual transform reinvents a primitive.
- **(C)** Anchor directly at the rendered midpoint with no offset (y +0). Cost: badge row overlaps the role label, both rendering at the same screen-space pixels; the badge would obscure the role label or vice versa depending on Cytoscape canvas-vs-DOM z-order. Rejected.
- **(D)** Anchor ABOVE the role label (y -18 from the midpoint). Cost: edges in the breadthfirst-bottom layout have the source node below and the target node above; a badge row above the midpoint encroaches on the target node's vertical zone, particularly for short edges. The "below" anchor matches the predecessor overlays' "decorations sit below the related entity-content visual" posture (axiom marks below the node, annotations below the axiom-mark row). Rejected.

The choice (A) keeps the placement API symmetric with the node branch (both use `renderedBoundingBox()`-derived coords with a fixed offset) and respects the existing visual stack (Cytoscape-rendered role label at the midpoint; DOM-overlay annotation badges below).

Alternative — make `EDGE_ANNOTATION_OFFSET_Y` configurable via a prop. Rejected — the constant is a v1 baseline; if the role-label font size changes (per the typography work), the constant updates inline. No call site needs to override it today; the abstraction cost (prop threading + test setup variations) is unwarranted for one call site.

Symmetric to the parent leaf's `AudienceNodeData.annotations` stamping. The overlay reads the full per-edge list; no boolean+count collapse.

### §5 — Reuse the same `cy.on(...)` subscription set; no new event subscription (NOT add `position` listener on edges, NOT add a new event vocabulary)

The parent leaf's overlay subscribes to:

- `cy.on('render pan zoom resize', scheduleUpdate)` — viewport-coupled changes (pan, zoom, viewport resize all re-render the canvas; the `render` event fires on every Cytoscape render pass).
- `cy.on('position', 'node', scheduleUpdate)` — node-position changes (e.g., layout re-runs, programmatic position updates).
- `cy.on('add remove data', scheduleUpdate)` — topology + data changes.

Every change that moves an edge midpoint relative to the screen is one of these:

- **Pan / zoom**: caught by `render pan zoom resize`.
- **Node-position change**: caught by `position node`; edges follow their endpoints, so an endpoint move re-positions the edge's midpoint without firing a separate edge-position event (Cytoscape doesn't fire `position` events on edges; edges are derived geometry).
- **Edge added / removed**: caught by `add`.
- **Edge data changed (`annotations` re-stamped)**: caught by `data`.

So the existing subscription set already catches every viewport-coupled and topology-coupled change that affects edge midpoint coordinates. No new subscription is needed.

Alternative — add `cy.on('position', 'edge', scheduleUpdate)`. Rejected — Cytoscape doesn't fire position events on edges; the listener would never fire (verified by reading the Cytoscape.js docs and the existing per-facet pill overlay's posture, which subscribes to `position node` only).

Alternative — add a `cy.on('render')`-only listener and drop the others. Rejected — `render` fires on every canvas render pass, which is the most-granular signal; the others are subsets but cheaper to subscribe to and catch the precise change semantics (the parent leaf's set is the right baseline).
