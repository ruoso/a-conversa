## What this task is

The 1d leaf that lands **per-participant axiom-mark badges** on the audience broadcast canvas — a small row of colored "A"-glyph chips painted just below every node that carries at least one committed axiom-mark, one badge per (node, participant) pair, each colored via the same `axiomMarkColorFor` deterministic hash the moderator uses so that "Anna marked N9 here" and "Anna marked N12 there" read as the same participant identity across the canvas (and across the moderator+audience composite if the show producer composites both feeds). Each badge is a small rounded-square chip with a centered "A" glyph, a per-participant background/text/ring color triple, and an `aria-label` carrying the localized "Axiom mark from participant {participantId}" prose. Multiple participants marking the same node stack as a horizontal flex-row below the node card — same "row of small chips" shape as the per-facet pill row above.

Because the audience renders to a Cytoscape `<canvas>` and there is no React tree per node, the badge row CANNOT live inside the node card (Cytoscape's nodes are pixels, not React subtrees). The badge row lands as a **DOM-overlay sibling** of the Cytoscape mount — the second overlay added to the existing `audience-graph-root-wrapper` positioning ancestor (the per-facet pill overlay is the first; this is the second). The overlay subscribes to the Cytoscape `Core`'s pan / zoom / render / position / add-remove-data events and re-paints (rAF-batched) so the badge rows track each node's screen position as the viewport moves.

This leaf is the **sixth** sibling under `aud_graph_rendering` (after the four state-styling refinements, the per-facet pill overlay, and the stylesheet-extraction housekeeping) and the **second** DOM-overlay surface on the audience workspace. The pattern lift from `PerFacetPillOverlay` is direct — same wrapper structure (`pointer-events-none absolute inset-0`), same subscription set (`render pan zoom resize` + `position node` + `add remove data`), same singleton-rAF handle, same `<useState<Core | null>>` arrival pattern, same `containerRef` prop shape. The differences are local: the per-element field read is `data.axiomMarks` (not `data.facetStatuses`); the per-element child is a flex-row of `<AudienceAxiomMarkBadge>` chips iterated in commit-arrival order from the `AxiomMark[]` array (not three pills in canonical reading order); the anchor is below the node bounding box (`renderedBoundingBox().y2 + AXIOM_BADGE_ROW_OFFSET_Y`) rather than above.

## Why it needs to be done

The `m_audience_mvp` milestone in [`tasks/99-milestones.tji`](../../99-milestones.tji) names the entire `aud_graph_rendering` group transitively. The four state-styling siblings + the per-facet pill overlay land the **agreement-layer** vocabulary on the broadcast canvas (proposed / agreed / disputed / per-facet pills) — viewers see the conversation's per-facet state. But the methodology's **primary success state** — bedrock — stays invisible until this leaf ships. From [`docs/methodology.md:204`](../../../docs/methodology.md#L204): *"Axioms are not a defect. They are often the most valuable output of the exercise: the debate dead-ends at 'A holds X as bedrock, B holds Y as bedrock, and that is the real disagreement.' Surfacing axioms is a primary success state."* And from [`DESIGN.md:26`](../../../DESIGN.md#L26): *"Bedrock axioms — when 'nothing could change my mind', the system marks the node as an axiom. Surfacing axioms is a primary success state — the debate has identified the irreducible disagreement."*

The methodology assumes the broadcast audience sees the same axiom-mark signal the moderator does. Without this leaf, a committed axiom-mark lands silently in the audience event log and the broadcast viewer has no visual confirmation that bedrock has been located — defeating the methodology's "surface axioms as a primary success state" rule on the very surface designed to expose the conversation's narrative to a viewing audience.

A broadcast viewer needs the per-participant attribution specifically (NOT just the at-a-glance boolean the participant chose) because the methodology's axiom-mark semantics ARE per-participant — Ben's bedrock and Anna's bedrock are different recorded events with different implications, and the broadcast narrative "Anna and Ben both hit the same bedrock here" reads completely differently from "Anna marked N9, Ben marked N12" (the structural-shared-bedrock case is one of the methodology's interesting outcomes; the structural-different-bedrock case is the canonical irreducible-disagreement diagnostic). A single bedrock-flag overlay (the participant's choice) would conflate these two cases and silently erase the methodology's per-participant content for broadcast viewers. The chromatic-by-participant rendering is what makes the methodology readable on the broadcast canvas.

### §1 — Per-participant chromatic DOM-overlay badges (NOT boolean overlay; NOT Cytoscape-native; NOT deferred to detail panel)

The participant's `part_axiom_mark_decoration` (Decision §1 there) chose a **boolean** `isAxiom` overlay painted via a Cytoscape stylesheet selector (`node[?isAxiom]` with `border-style: 'double'` + `border-width: 3`), explicitly deferring per-participant chromatic identity to a future detail-panel React surface (`part_entity_detail_panel`). Three options for the audience:

- **(A — chosen)** Render **per-participant chromatic badges** as a DOM-overlay sibling — one badge per (node, participant) axiom-mark pair, colored via the deterministic `axiomMarkColorFor(participantId)` hash, stacked as a horizontal flex-row below the node card. The badge is a small rounded-square (`rounded-sm`) with a centered "A" Latin-anchor glyph, mirroring the moderator's `<AxiomMarkBadge>`. The overlay is a verbatim transposition of the per-facet pill overlay pattern (positioning ancestor + `pointer-events-none absolute inset-0` sibling + rAF-batched subscription).
- **(B)** Render a **boolean overlay** like the participant — one Cytoscape stylesheet selector `node[?hasAxiomMark]` with a visually distinct border-style on axiom-marked nodes; no per-participant detail. Cost: the methodology's per-participant semantics ("Anna's bedrock vs Ben's bedrock") become invisible on the broadcast surface — exactly the conflation the moderator's leaf was explicit about avoiding (Decision "Per-participant color via deterministic hash, not a per-session palette assignment" + "Visual distinctness from the annotation badge matters too"). The participant tablet can defer because it has a tap-to-detail panel; the broadcast surface has no detail panel, no tap interaction, and no fallback path for per-participant attribution. Rejected.
- **(C)** Defer per-participant axiom-mark rendering entirely — ship rollup-only on the audience and register a future task. Cost: the methodology's primary success state (bedrock surfacing) stays invisible on the broadcast canvas. The same rejection rationale as the per-facet pill overlay's (C) option: deferring a methodology-load-bearing surface means the broadcast viewer cannot read the conversation's primary success-state narrative. Rejected.

The choice (A) is the consistent translation of the per-facet pill overlay's posture: the audience surface inverts the participant's detail-panel-deferral whenever the load-bearing semantics are broadcast-visible. The methodology's primary success state IS the per-participant attribution — collapsing it to a boolean for the broadcast surface would erase the narrative content.

Alternative — render per-participant chromatic badges INSIDE the per-facet pill row (combine both overlays into one). Rejected — the two overlays surface different layers (per-facet status is the **agreement** layer; axiom-marks are the per-participant **disposition** layer). Per ADR 0027 ("entity and facet layers are strictly separate") the two should not visually compete; the spatial separation (per-facet pills above the node; axiom badges below) makes the layer boundary visible to broadcast viewers (Decision §4 below).

### §2 — Stamp `axiomMarks: readonly AxiomMark[]` on the projection output, not a boolean or a Cytoscape class

Two options for how the projector communicates the axiom-mark signal to the audience overlay:

- **(A — chosen)** Stamp the full `AxiomMark[]` list (one entry per committed axiom-mark targeting the node) on `data.axiomMarks`; default to the module-scope frozen `EMPTY_AXIOM_MARKS` for nodes with no marks. The overlay iterates the array and renders one badge per entry.
- **(B)** Stamp a boolean `data.hasAxiomMark: boolean` plus a separate lookup keyed on node id for the badge consumer. Rejected: two sources of truth (the data field + the lookup), with reconciliation risk on commit-time mutations; the overlay would have to consult both. The single-array-per-node shape collapses the two sources into one.
- **(C)** Stamp the boolean (`isAxiom`) and have the overlay re-derive the per-participant list from the events log on each render. Rejected: re-derivation per render defeats `useMemo`-style caching the per-facet-status pattern already established.

The chosen shape (A) mirrors the moderator's `StatementNodeData.axiomMarks: readonly AxiomMark[]` exactly (per `mod_axiom_mark_decoration` Constraints "**`StatementNodeData` extension**: add `axiomMarks: readonly AxiomMark[]`"). Cross-surface consistency: moderator and audience read the same per-node shape; the audience port is verbatim. The module-scope `EMPTY_AXIOM_MARKS` frozen empty array preserves stable React-memoization identity (same rationale as `EMPTY_FACET_STATUSES`).

Alternative — use a Cytoscape `addClass` per-participant (one class per participant id, dynamically registered to the stylesheet). Rejected for the same reason `part_axiom_mark_decoration` Decision §4 rejected it: the stylesheet is module-scope static (per `aud_stylesheet_module_extraction` Decision §1); injecting per-participant selectors would force a `cy.style()` swap on every axiom-mark commit; the data-field-with-selector pattern is what the predecessor stylesheet siblings already use. Rejected.

### §3 — Inline port of `axiomMarks.ts` + `<AxiomMarkBadge>` into the audience workspace; register `shell_axiom_marks_extraction` as named-future-task

The participant ported `axiomMarks.ts` verbatim from the moderator. Decision §2 there explicitly named the audience leaf as the third caller that triggers the lift-into-shell. Two options:

- **(A — chosen)** Verbatim inline port (one more copy of `axiomMarks.ts`; inline copy of `<AxiomMarkBadge>` minus the moderator-local `AxiomMark` import path); register the shell extraction as a named-future-task **`shell_axiom_marks_extraction`** (~0.5d). Rationale: the 1d budget for THIS leaf covers a complete per-participant-chromatic decoration feature on the audience; bundling the shell-extract refactor inside this leaf (which would also touch the moderator + participant code paths) inflates the scope past 1d and crosses surface boundaries that warrant a separate PR for review clarity. The named-future-task captures the debt explicitly per the orchestrator brief's tech-debt registration policy ("name the future task crisply — a stable id, an effort estimate, a one-line description — so the closer can register it mechanically").
- **(B)** Bundle the shell-extract into this leaf. Rejected — scope inflation past 1d; cross-surface refactor that spans three apps' import paths is a separate PR's worth of review. The third-caller trigger fires here, but the extract execution belongs in its own commit cluster.
- **(C)** Skip the shell-extract follow-up registration entirely; let some future commit author notice the duplication. Rejected — the orchestrator brief specifically warns: "When you defer something to a future task, name the future task crisply" and "Status-block notes are invisible to the orchestrator's pick-task pass. Only real WBS leaves get picked up. Every deferred follow-up must surface as a named-future-task in your refinement."

Chosen: (A). The closer registers `shell_axiom_marks_extraction` (~0.5d) at task close. Description: "Lift `AxiomMark` + `EMPTY_AXIOM_MARKS` + `projectAxiomMarks` + `groupAxiomMarksByNode` from `apps/{moderator,participant,audience}/src/graph/` into `packages/shell/src/axiom-marks/`; consolidate the three client copies as `@a-conversa/shell` imports; lift the moderator+audience `<AxiomMarkBadge>` component (and the participant tablet's future panel consumer) as a sibling cross-surface lift."

The header comment on the new audience `axiomMarks.ts` links to BOTH parallel ports (moderator + participant) so a future reader cross-referencing them sees the full duplication trail; this is the same posture the participant's leaf established.

### §4 — Badge row anchor: BELOW the node bounding box (not above, not inside)

Two approaches to the per-element anchor position:

- **(A — chosen)** Anchor the badge row's TOP edge to `renderedBoundingBox().y2 + AXIOM_BADGE_ROW_OFFSET_Y` (a few px below the node's bottom edge), with `translate(-50%, 0)` centering horizontally and leaving the row's top at the anchor point. `AXIOM_BADGE_ROW_OFFSET_Y = 6` matches the per-facet pill row's `PILL_ROW_OFFSET_Y` for visual symmetry: pills 6px above the node, badges 6px below.
- **(B)** Anchor ABOVE the node (collide with the per-facet pill row's anchor). Rejected — the per-facet pill row already occupies that space; two overlays stacked above the node would compete vertically and break the spatial layer separation (per ADR 0027) the per-facet pill row established (pills are the facet/agreement layer; badges are the disposition layer — separating them spatially makes the layer boundary readable).
- **(C)** Anchor INSIDE the node (overlay the badge row on top of the node's wording). Rejected — overlays the entity-layer content (wording) with the disposition-layer decoration; per ADR 0027 the two layers should not visually compete. The above-and-below-the-node spatial arrangement keeps the entity content (wording) centered and unobscured while the two decoration overlays read at glance distance.

The below-the-node anchor matches the participant's `OtherVotesOverlay` anchor (the audience's `aud_per_facet_visualization` Decision §2 noted this anchor was rejected for pills because vote-dots are per-proposal and pills are per-facet, but the per-participant axiom-mark badge IS conceptually closer to "this participant's vote on bedrock" semantically — a per-participant signal anchored below the node where the participant tablet shows per-participant vote dots today). Cross-surface consistency on per-participant signals anchored below the node is the principle.

Alternative — anchor on the left or right of the node. Rejected — same reason as the per-facet pill row's left/right rejection: the 200×80 node box is wider than tall; horizontal anchors compete with edge labels and don't compose with the breadthfirst layout's left-to-right flow.

### §5 — Subscribe to `render pan zoom resize` + `position node` + `add remove data`; rAF-batched; reuse the existing `cyState` slot

Direct mirror of the per-facet pill overlay's Decision §4 + §5 — same subscription set, same singleton-rAF handle, same `useState<Core | null>` slot in `<AudienceGraphView>` (the slot already exists at [`apps/audience/src/graph/GraphView.tsx:178`](../../../apps/audience/src/graph/GraphView.tsx#L178); no new slot needed; both overlays receive the same `cy` prop).

Adding a second subscriber to the same `Core` instance is mechanically safe — Cytoscape's `cy.on(...)` is a multi-subscriber dispatcher; the per-facet overlay's listeners and the axiom-mark overlay's listeners coexist without interaction. Each overlay owns its own `cy.off(...)` teardown set keyed on the handler reference passed to `cy.on(...)`, so cleanups are isolated.

Alternative — share a single rAF batch across both overlays (centralize the scheduler in `<AudienceGraphView>` and pass a "scheduler" prop down). Rejected — premature centralization; the rAF singleton in each overlay is a small primitive whose duplication cost is bounded (~5 lines per overlay) and whose centralization would couple the two overlays' lifecycles. The orchestrator brief's "two callers is YAGNI" applies here too: two overlay instances of the same pattern is the threshold to validate the pattern, not the threshold to abstract it.

### §6 — Playwright deferral lands on `aud_visual_regression`, NOT a new task

Per the orchestrator brief's deferred-e2e exception: this leaf adds a DOM-overlay to an audience-surface region that is still not reachable through any user-flow route ([`apps/audience/src/App.tsx`](../../../apps/audience/src/App.tsx) maps every path to the placeholder; the per-session route lands in `aud_url_routing.aud_session_url`). The axiom-mark badges are invisible until that route ships.

`aud_visual_regression`'s inherited deferred-e2e debt is already past the orchestrator brief's "2+ refinements, pay down" threshold (per the four state-styling siblings + the per-facet pill overlay's deferrals). The brief says: "If it's inheriting from 2+ refinements already, pay debt down instead — either scope a small Playwright spec inline, or split the deferral target into multiple smaller future tasks." The honest read of the situation: `aud_visual_regression` is the right destination because (a) the predecessor refinements established it as the canonical landing zone for the audience's pixel-stability work and (b) the inline-Playwright option is genuinely not available — the route doesn't exist and no inline spec can mount the `<AudienceGraphView>` without re-creating the future routing scaffolding `aud_session_url` will provide. The "split into multiple smaller tasks" option is a planning-debt move: the audience surface needs ONE visual-regression task that lights up after `aud_session_url` lands, not six.

So: the right destination is **`aud_visual_regression`** (2d, [`tasks/50-audience-and-broadcast.tji:318-347`](../../50-audience-and-broadcast.tji#L318)). The task already inherits five sibling deferrals; this leaf is the sixth. The closer extends `aud_visual_regression`'s existing note to also cover the axiom-mark badge palette + position + per-participant chromatic stability across Chrome / Firefox / WebKit at 720p / 1080p / 1440p.

The overlay's mount lifecycle, subscription set, per-element placement, per-participant chromatic determinism, and array-order iteration are fully pinned at the Vitest layer above — `aud_visual_regression` is for the *visible* rendering (pixel-level badge colors, "A" glyph font rendering, position-sync across the viewport), not the overlay's structure.

Alternative — defer to `aud_session_url`. Rejected per the "2+ refinements, pay down" rule and per the scope mismatch (route-mount assertions are not per-element overlay assertions).

Alternative — register a new `aud_pw_axiom_mark_smoke` task. Rejected — planning-debt for no architectural reason; `aud_visual_regression` already exists and inherits five sibling deferrals' work, which the axiom-mark overlay is a natural extension of (same broadcast-canvas, same DOM-overlay pattern, same not-yet-reachable surface).

Alternative — scope a thin Playwright spec inline that asserts the placeholder route's chrome includes a `[data-testid="audience-axiom-mark-overlay"]` element. Rejected — the placeholder chrome does NOT include the Cytoscape mount (the `<AudienceGraphView>` is not mounted from `App.tsx`); no inline spec is meaningful before `aud_session_url` lands.
